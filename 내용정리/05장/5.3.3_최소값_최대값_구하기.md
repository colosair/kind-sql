## 5.3.3 최소값/최대값 구하기

### 핵심 개념

`MIN`, `MAX`는 인덱스를 이용하면 전체 데이터를 읽지 않고도 빠르게 구할 수 있습니다. 인덱스는 정렬되어 있으므로 최소값은 왼쪽 끝, 최대값은 오른쪽 끝에 있습니다.

```text
MAX(sal)
   ↓
sal 인덱스 오른쪽 끝 탐색
   ↓
첫 번째 값 반환
```

```sql
create index emp_x1 on emp(sal);

select max(sal)
from emp;
```

실행계획의 핵심은 다음입니다.

```text
SORT AGGREGATE
  INDEX FULL SCAN (MIN/MAX)
```

`SORT AGGREGATE`가 보이지만 전체를 정렬한다는 뜻은 아닙니다. 인덱스 끝 값 하나를 읽어 집계 결과로 반환하는 구조입니다.

### First Row Stopkey 조건

조건절이 있을 때 전체 데이터를 읽지 않고 최소/최대값을 구하려면 조건절 컬럼과 `MIN/MAX` 대상 컬럼이 모두 인덱스에 포함되어야 합니다.

```sql
create index emp_x1 on emp(deptno, mgr, sal);

select max(sal)
from emp
where deptno = 30
and mgr = 7698;
```

```text
인덱스: deptno + mgr + sal
조건: deptno = 30, mgr = 7698
대상: max(sal)
   ↓
조건 구간의 오른쪽 끝으로 이동
   ↓
첫 행 읽고 중단
```

실행계획에는 `FIRST ROW`와 `INDEX RANGE SCAN (MIN/MAX)`가 나타납니다. 이를 책에서는 `First Row Stopkey` 알고리즘이라고 부릅니다.

### 컬럼 순서별 차이

| 인덱스 구성 | 동작 방식 |
|---|---|
| `deptno + mgr + sal` | `deptno`, `mgr`가 액세스 조건이고 해당 구간 끝 값 하나를 읽습니다. |
| `deptno + sal + mgr` | `deptno`는 액세스 조건, `mgr`은 필터 조건입니다. `sal` 큰 쪽부터 읽다가 `mgr` 조건을 만족하는 첫 행에서 멈춥니다. |
| `sal + deptno + mgr` | 조건절 컬럼이 선두가 아니므로 Index Full Scan으로 오른쪽 끝부터 찾습니다. |
| `deptno + sal` | `mgr`이 인덱스에 없으면 테이블 액세스 후 필터링해야 하므로 Stopkey가 작동하지 않습니다. |

핵심은 "테이블 액세스 없이 인덱스 안에서 조건 판단이 가능한가"입니다.

### Top N으로 최소/최대값 구하기

`MIN/MAX` 대신 Top N 쿼리로도 최대값을 구할 수 있습니다.

```sql
select *
from (
  select sal
  from emp
  where deptno = 30
  and mgr = 7698
  order by sal desc
)
where rownum <= 1;
```

이 방식은 `mgr` 컬럼이 인덱스에 없어도 `Top N Stopkey`가 작동할 수 있습니다. 인덱스에서 큰 `sal`부터 읽으며 테이블을 액세스하다가 `mgr = 7698` 조건을 만족하는 첫 행에서 멈추기 때문입니다.

```text
deptno = 30 구간의 sal 큰 쪽부터 역순 스캔
   ↓
테이블에서 mgr 조건 확인
   ↓
조건 만족 첫 행 발견
   ↓
rownum <= 1에 의해 중단
```

---

## ✅ 이 절(5.3.3)의 핵심 통찰

1. 인덱스를 이용하면 최소/최대값을 전체 스캔 없이 구할 수 있습니다.
2. `INDEX FULL SCAN (MIN/MAX)` 또는 `INDEX RANGE SCAN (MIN/MAX)`가 핵심 실행계획입니다.
3. 조건절 컬럼과 `MIN/MAX` 대상 컬럼이 모두 인덱스에 있어야 `First Row Stopkey`가 잘 작동합니다.
4. 인덱스에 없는 필터 조건이 있으면 `MIN/MAX` 방식은 불리해질 수 있습니다.
5. Top N 쿼리는 테이블 필터 조건이 있어도 첫 만족 행에서 멈출 수 있어 유리한 경우가 많습니다.

## 🔗 흐름 속 위치

5.3.2의 `Top N Stopkey`가 상위 N건 조회였다면, 5.3.3은 같은 원리를 `MIN/MAX` 조회에 적용합니다.

📌 한 줄 요약: 최소/최대값은 계산하는 것이 아니라 인덱스의 양 끝을 찾아가는 문제입니다.
