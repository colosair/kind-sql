## 5.3.1 Sort Order By 생략

### 핵심 개념

인덱스는 키 컬럼 순서로 정렬된 자료구조입니다. 따라서 `ORDER BY` 컬럼 순서와 인덱스 컬럼 순서가 맞으면 별도의 `SORT ORDER BY` 없이 결과를 반환할 수 있습니다.

```text
ORDER BY 필요
   ↓
정렬 기준과 같은 순서의 인덱스 존재
   ↓
인덱스를 순서대로 스캔
   ↓
SORT ORDER BY 생략
```

> "인덱스는 항상 키 컬럼 순으로 정렬된 상태를 유지한다."

### 인덱스가 부족한 경우

아래 쿼리에서 인덱스가 `종목코드`만으로 구성되어 있으면 `거래일시` 순서를 보장할 수 없습니다.

```sql
select 거래일시, 체결건수, 체결수량, 거래대금
from 종목거래
where 종목코드 = 'KR123456'
order by 거래일시;
```

```text
종목코드 인덱스 스캔
   ↓
종목코드 조건 만족 로우 읽기
   ↓
거래일시 순서 보장 불가
   ↓
SORT ORDER BY
```

### 소트 생략 가능한 인덱스

인덱스를 `종목코드 + 거래일시` 순으로 구성하면, `종목코드 = 상수` 조건 안에서 `거래일시` 순서가 이미 보장됩니다.

```text
인덱스: 종목코드 + 거래일시
조건: 종목코드 = 'KR123456'
정렬: order by 거래일시
   ↓
같은 종목코드 구간 내부가 거래일시 순서
   ↓
SORT ORDER BY 생략
```

실행계획에서는 `SORT ORDER BY`가 사라지고 `INDEX RANGE SCAN`이 정렬 역할까지 수행합니다.

### 부분범위 처리와의 연결

소트가 생략되면 결과집합을 모두 읽고 정렬할 필요가 없습니다. 인덱스 순서대로 읽으면서 곧바로 결과를 반환할 수 있으므로 부분범위 처리가 가능해집니다.

```text
소트 필요
   ↓
전체 후보 집합 읽기
   ↓
정렬 완료 후 반환

소트 생략
   ↓
인덱스 순서로 읽음
   ↓
앞쪽 결과부터 즉시 반환
```

---

## ✅ 이 절(5.3.1)의 핵심 통찰

1. 인덱스는 정렬된 자료구조이므로 `ORDER BY`를 대체할 수 있습니다.
2. `WHERE`의 등치 조건 뒤에 오는 인덱스 컬럼 순서가 `ORDER BY`와 맞으면 소트 생략이 가능합니다.
3. 소트 생략은 단순 CPU 절감이 아니라 부분범위 처리를 가능하게 합니다.
4. OLTP에서 빠른 초기 응답속도를 만들려면 인덱스 순서와 정렬 순서를 맞춰야 합니다.

## 🔗 흐름 속 위치

5.2가 소트 발생 SQL을 피하는 방법이었다면, 5.3은 소트가 필요한 SQL에서도 인덱스 정렬성을 이용해 실제 소트 연산을 생략하는 방법입니다.

📌 한 줄 요약: `ORDER BY`를 빠르게 처리하는 핵심은 정렬하는 것이 아니라 이미 정렬된 인덱스를 읽는 것입니다.
