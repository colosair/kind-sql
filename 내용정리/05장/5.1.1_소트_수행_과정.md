## 5.1.1 소트 수행 과정

### 핵심 개념

소트는 기본적으로 PGA에 할당된 `Sort Area`에서 수행됩니다. `Sort Area` 안에서 정렬을 끝내면 메모리 소트이고, 메모리가 부족해 Temp 테이블스페이스를 사용하면 디스크 소트입니다.

```text
정렬 대상 집합 읽기
   ↓
SGA 버퍼캐시를 통해 블록 읽기
   ↓
PGA Sort Area에서 정렬 시도
   ↓
Sort Area 안에서 완료 → 메모리 소트
Sort Area 부족 → Temp 사용 → 디스크 소트
```

> "소트는 기본적으로 PGA에 할당한 Sort Area에서 이루어진다."

| 구분 | 의미 | 성능 영향 |
|---|---|---|
| 메모리 소트 | 전체 정렬을 메모리 안에서 완료 | 가장 바람직합니다. Temp I/O가 없습니다. |
| 디스크 소트 | Sort Area 부족으로 Temp까지 사용 | 디스크 I/O가 발생해 성능이 급격히 나빠집니다. |

디스크 소트는 CS 관점에서 **External Merge Sort**와 같습니다.

```text
입력 데이터
   ↓
Sort Area 크기만큼 정렬
   ↓
정렬된 중간 집합 생성 = Sort Run
   ↓
Temp에 저장
   ↓
여러 Sort Run 병합
   ↓
최종 정렬 결과
```

`Sort Run`은 Sort Area가 찰 때마다 Temp 영역에 저장한 정렬된 중간 집합입니다. 각 Run은 이미 정렬되어 있으므로, 최종 단계에서는 다시 처음부터 정렬하는 것이 아니라 여러 정렬 리스트를 병합합니다.

### 튜닝 포인트

디스크 소트가 발생하면 SQL 성능은 나빠질 수밖에 없습니다. 이유는 단순히 정렬 계산이 많아서가 아니라, 메모리와 CPU뿐 아니라 Temp 디스크 I/O까지 사용하기 때문입니다.

```text
소트 대상 증가
   ↓
Sort Area 부족
   ↓
Temp Write
   ↓
Run Merge를 위한 Temp Read
   ↓
CPU + 메모리 + 디스크 I/O 동시 증가
```

특히 OLTP에서는 디스크 소트가 부분범위 처리를 막습니다. 인덱스를 이용하면 앞쪽 일부 결과를 바로 반환할 수 있지만, 큰 정렬이 필요하면 전체 후보 집합을 모아 정렬한 뒤에야 결과를 반환할 수 있습니다.

---

## ✅ 이 절(5.1.1)의 핵심 통찰

1. 소트는 먼저 PGA의 `Sort Area`에서 수행됩니다.
2. `Sort Area`에서 끝나면 메모리 소트, Temp를 쓰면 디스크 소트입니다.
3. 디스크 소트는 정렬된 중간 집합인 `Sort Run`을 만들고 다시 병합합니다.
4. 디스크 소트는 CPU, 메모리, 디스크 I/O를 모두 압박합니다.
5. OLTP에서는 소트가 부분범위 처리를 막아 응답속도를 크게 떨어뜨립니다.

## 🔗 흐름 속 위치

4장에서 소트 머지 조인과 해시 조인을 통해 PGA와 Temp 사용을 봤습니다. 5장에서는 이 메커니즘을 `ORDER BY`, `GROUP BY`, `DISTINCT`, 집합 연산 등 일반 소트 튜닝으로 확장합니다.

📌 한 줄 요약: 소트 튜닝의 첫 원칙은 소트를 없애고, 불가피하다면 Temp로 밀려나지 않게 작게 만드는 것입니다.
