## 5.1.2 소트 오퍼레이션

### 핵심 개념

5.1.2의 결론은 명확합니다. 실행계획에 `SORT`라는 단어가 보인다고 해서 항상 "데이터를 물리적으로 정렬한다"는 뜻은 아닙니다. 어떤 오퍼레이션은 실제 정렬을 수행하고, 어떤 오퍼레이션은 단지 `Sort Area`를 사용한다는 의미에 가깝습니다.

```text
SQL 수행 중 가공된 집합 필요
   ↓
정렬, 그룹핑, 중복 제거, 분석함수, 소트 머지 조인 등 발생
   ↓
실행계획에 SORT 계열 오퍼레이션 표시
   ↓
실제 정렬인지, 집계/중복제거/분석 처리를 위한 메모리 작업인지 구분 필요
```

5.1.1에서 본 핵심은 "소트는 PGA의 Sort Area에서 시작하고, 부족하면 Temp로 밀려난다"였습니다. 5.1.2는 한 단계 더 들어가서 **어떤 SQL 문법과 처리 방식이 소트 계열 오퍼레이션을 발생시키는가**를 분류합니다.

> "본격적으로 소트 튜닝을 설명하기에 앞서 소트를 발생시키는 오퍼레이션에 어떤 것들이 있는지부터 살펴보자."

| 오퍼레이션 | 주로 발생하는 경우 | 실제 정렬 여부 | 핵심 해석 |
|---|---|---|---|
| `SORT AGGREGATE` | 전체 집계 | 보통 정렬하지 않음 | 전체 로우를 하나의 집계 결과로 축약 |
| `SORT ORDER BY` | `ORDER BY` | 정렬함 | 결과 집합 순서 보장 |
| `SORT GROUP BY` | `GROUP BY` | 소팅 알고리즘 사용 | 그룹별 집계, 단 결과 순서 보장은 별도 문제 |
| `HASH GROUP BY` | `GROUP BY` + `ORDER BY` 없음 | 정렬하지 않음 | 해시 버킷으로 그룹별 집계 |
| `SORT UNIQUE` | `DISTINCT`, `UNION`, `MINUS`, `INTERSECT`, 서브쿼리 중복 제거 | 중복 제거 목적 정렬 | 유일 집합 생성 |
| `HASH UNIQUE` | `DISTINCT` + `ORDER BY` 없음 등 | 정렬하지 않음 | 해시 방식 중복 제거 |
| `SORT JOIN` | 소트 머지 조인 | 조인 키 기준 정렬 | 양쪽 집합을 정렬 후 병합 |
| `WINDOW SORT` | 분석함수 | 파티션/정렬 기준에 따라 정렬 | 윈도우 함수 계산 준비 |

---

### 1. Sort Aggregate

`Sort Aggregate`는 전체 로우를 대상으로 집계를 수행할 때 나타납니다. 이름에 `Sort`가 들어가지만, 실제로 데이터를 정렬하지는 않습니다. `Sort Area`에 집계용 변수를 두고 값을 누적한다고 이해하면 됩니다.

예제는 다음과 같은 형태입니다.

```sql
select sum(sal), max(sal), min(sal), avg(sal)
from emp;
```

실행계획은 다음과 같은 구조입니다.

```text
SELECT STATEMENT
  SORT AGGREGATE
    TABLE ACCESS FULL EMP
```

처리 과정은 변수 갱신에 가깝습니다.

```text
Sort Area에 변수 준비
SUM, MAX, MIN, COUNT
   ↓
첫 번째 SAL 값을 읽어 초기화
   ↓
다음 레코드부터 하나씩 읽으며 갱신
   ↓
SUM  : 누적
MAX  : 더 큰 값이면 교체
MIN  : 더 작은 값이면 교체
COUNT: NULL이 아닌 값이면 증가
   ↓
AVG = SUM / COUNT
```

CS 관점에서는 배열을 정렬하는 것이 아니라, 스트림을 한 번 순회하면서 accumulator를 갱신하는 방식입니다.

```text
for row in EMP:
    sum += row.sal
    max = max(max, row.sal)
    min = min(min, row.sal)
    count += 1
```

따라서 `SORT AGGREGATE`를 보고 곧바로 "정렬 비용이 크다"고 판단하면 안 됩니다. 핵심 비용은 정렬 자체가 아니라 **집계 대상 로우를 읽는 I/O**에 있습니다.

---

### 2. Sort Order By

`Sort Order By`는 이름 그대로 데이터를 정렬할 때 나타납니다.

```sql
select *
from emp
order by sal desc;
```

실행계획은 다음과 같은 형태입니다.

```text
SELECT STATEMENT
  SORT ORDER BY
    TABLE ACCESS FULL EMP
```

이 오퍼레이션은 실제 정렬을 수행합니다. 따라서 데이터량이 많고 인덱스로 정렬 순서를 대체하지 못하면 5.1.1에서 본 디스크 소트 위험이 생깁니다.

```text
EMP 읽기
   ↓
정렬 대상 로우 수집
   ↓
Sort Area에서 정렬
   ↓
메모리 부족 시 Temp 사용
   ↓
정렬 결과 반환
```

튜닝 관점에서는 `ORDER BY`를 무조건 없애는 것이 목표가 아닙니다. 필요한 정렬은 명시해야 합니다. 다만 인덱스 순서를 이용해 `SORT ORDER BY`를 생략할 수 있는지가 이후 5.3의 핵심 주제가 됩니다.

---

### 3. Sort Group By

`Sort Group By`는 소팅 알고리즘을 사용해 그룹별 집계를 수행할 때 나타납니다.

```sql
select deptno, sum(sal), max(sal), min(sal), avg(sal)
from emp
group by deptno
order by deptno;
```

실행계획은 다음과 같은 구조입니다.

```text
SELECT STATEMENT
  SORT GROUP BY
    TABLE ACCESS FULL EMP
```

중요한 점은 `GROUP BY`가 항상 전체 레코드를 정렬한 뒤 집계한다는 뜻은 아니라는 점입니다. 책은 부서별 급여 집계를 메모지 비유로 설명합니다.

```text
부서 10 메모지: SUM, MAX, MIN, COUNT
부서 20 메모지: SUM, MAX, MIN, COUNT
부서 30 메모지: SUM, MAX, MIN, COUNT
부서 40 메모지: SUM, MAX, MIN, COUNT
```

각 사원 레코드를 읽을 때마다 해당 부서의 메모지를 찾아 집계 값을 갱신합니다.

```text
사원 레코드 읽기
   ↓
부서번호 확인
   ↓
해당 그룹 찾기
   ↓
SUM, MAX, MIN, COUNT 갱신
   ↓
다음 레코드 반복
```

이때 그룹 개수가 적다면 처리할 레코드가 매우 많아도 필요한 메모리는 작을 수 있습니다.

```text
집계 대상 로우 수: 매우 큼
그룹 수: 작음
   ↓
Sort Area에는 그룹별 집계 상태만 유지
   ↓
Temp를 쓰지 않을 가능성 높음
```

핵심은 메모리 사용량이 "입력 로우 수"에만 비례하지 않는다는 것입니다. `GROUP BY`에서는 **그룹 개수와 그룹별 상태 크기**가 매우 중요합니다.

---

### 4. Hash Group By

오라클 10gR2부터는 `GROUP BY` 뒤에 `ORDER BY`가 없으면 많은 경우 `Hash Group By`로 처리합니다.

```sql
select deptno, sum(sal), max(sal), min(sal), avg(sal)
from emp
group by deptno;
```

실행계획은 다음과 같이 나타날 수 있습니다.

```text
SELECT STATEMENT
  HASH GROUP BY
    TABLE ACCESS FULL EMP
```

`Sort Group By`가 정렬 기반 자료구조에서 그룹을 찾는 방식이라면, `Hash Group By`는 해시 테이블에서 그룹을 찾는 방식입니다.

```text
레코드 읽기
   ↓
GROUP BY 컬럼 해시값 계산
   ↓
해시 버킷 탐색
   ↓
그룹별 집계 값 갱신
```

CS 대응은 `HashMap<group_key, aggregate_state>`입니다.

```text
deptno → {sum, max, min, count}
```

이는 4장의 해시 조인과도 같은 원리입니다. 차이는 다음과 같습니다.

| 구분 | 해시 조인 | Hash Group By |
|---|---|---|
| 해시 키 | 조인 컬럼 | 그룹핑 컬럼 |
| 해시 값이 가리키는 것 | 조인 대상 로우 또는 로우 체인 | 그룹별 집계 상태 |
| 목적 | 두 집합 연결 | 같은 그룹끼리 축약 |

---

### 5. 그룹핑 결과의 정렬 순서

이 절에서 가장 주의해야 할 부분입니다.

**`GROUP BY`는 결과 정렬을 보장하지 않습니다.** 정렬된 그룹핑 결과가 필요하면 반드시 `ORDER BY`를 명시해야 합니다.

```sql
select deptno, job, sum(sal), max(sal), min(sal)
from emp
group by deptno, job;
```

실행계획에 `SORT GROUP BY`가 표시되더라도 결과가 `deptno, job` 순으로 항상 정렬되어 나온다고 믿으면 안 됩니다.

```text
SORT GROUP BY 표시
   ↓
소팅 알고리즘으로 그룹핑했다는 뜻
   ↓
결과 출력 순서 보장과는 별개
```

정렬 순서가 필요하면 다음처럼 명시해야 합니다.

```sql
select deptno, job, sum(sal), max(sal), min(sal)
from emp
group by deptno, job
order by deptno, job;
```

여기서 오해하기 쉬운 점이 있습니다. `GROUP BY`만 있던 쿼리에 `ORDER BY`를 추가한다고 해서 반드시 "그룹핑 한 번, 정렬 한 번"으로 작업이 두 배가 되는 것은 아닙니다. 오라클은 내부 알고리즘을 활용해 처리할 수 있습니다. 다만 `Hash Group By`와 `Sort Group By`의 알고리즘 차이로 인한 성능 차이는 생길 수 있습니다.

```text
정렬 순서가 필요함
   ↓
ORDER BY 명시
   ↓
실행계획의 SORT GROUP BY 여부만으로 정렬 보장 판단 금지
```

### 6. Sort Unique

`Sort Unique`는 중복 제거가 필요할 때 나타납니다.

주요 발생 지점은 다음과 같습니다.

| SQL 요소 | 중복 제거 필요성 |
|---|---|
| `DISTINCT` | 같은 결과 로우 제거 |
| `UNION` | 두 집합을 합친 뒤 중복 제거 |
| `MINUS` | 차집합 처리를 위한 정렬/중복 제거 |
| `INTERSECT` | 교집합 처리를 위한 정렬/중복 제거 |
| 서브쿼리 Unnesting | 조인 전 중복 제거가 필요할 수 있음 |

예를 들어 `UNION`은 내부적으로 두 집합을 단순 결합한 뒤 중복을 제거합니다.

```sql
select job, mgr from emp where deptno = 10
union
select job, mgr from emp where deptno = 20;
```

실행계획은 다음처럼 나타날 수 있습니다.

```text
SELECT STATEMENT
  SORT UNIQUE
    UNION-ALL
      TABLE ACCESS FULL EMP
      TABLE ACCESS FULL EMP
```

여기서 중요한 점은 실행계획 내부에 `UNION-ALL`이 보인다는 것입니다. 오라클은 먼저 두 집합을 붙이고, 그 위에서 `SORT UNIQUE`로 중복을 제거합니다.

`MINUS`에서는 양쪽 집합 각각에 `SORT UNIQUE`가 나타날 수 있습니다.

```text
SELECT STATEMENT
  MINUS
    SORT UNIQUE
      TABLE ACCESS FULL EMP
    SORT UNIQUE
      TABLE ACCESS FULL EMP
```

`DISTINCT`도 마찬가지로 중복 제거가 필요합니다.

```sql
select distinct deptno
from emp
order by deptno;
```

```text
SELECT STATEMENT
  SORT UNIQUE
    TABLE ACCESS FULL EMP
```

중복 제거는 단순 필터가 아닙니다. 같은 값이 있었는지 판단하려면 후보 집합을 모아 비교해야 하므로, 데이터가 많을수록 소트 또는 해시 부담이 커집니다.

---

### 7. Hash Unique

오라클 10gR2부터는 `DISTINCT`에도 `Hash Unique` 방식을 사용할 수 있습니다. 특히 `ORDER BY`가 없을 때 그렇습니다.

```sql
select distinct deptno
from emp;
```

```text
SELECT STATEMENT
  HASH UNIQUE
    TABLE ACCESS FULL EMP
```

`Sort Unique`가 정렬 후 인접 값 비교로 중복을 제거하는 방식이라면, `Hash Unique`는 해시 자료구조를 사용해 이미 본 값인지 확인하는 방식입니다.

```text
레코드 읽기
   ↓
중복 제거 대상 컬럼 해시 계산
   ↓
해시 테이블에 없으면 추가
   ↓
이미 있으면 버림
```

CS 관점에서는 `HashSet`과 같습니다.

```text
seen = HashSet()
for row in input:
    key = row.deptno
    if key not in seen:
        seen.add(key)
```

단, 정렬 순서가 필요한 경우에는 해시 방식만으로는 부족합니다. 이때도 `ORDER BY`가 결과 순서를 결정하는 유일하게 신뢰할 수 있는 문법입니다.

---

### 8. Sort Join

`Sort Join`은 소트 머지 조인을 수행할 때 나타납니다.

```sql
select /*+ ordered use_merge(e) */ *
from dept d, emp e
where d.deptno = e.deptno;
```

실행계획은 다음과 같은 구조입니다.

```text
SELECT STATEMENT
  MERGE JOIN
    SORT JOIN
      TABLE ACCESS FULL DEPT
    SORT JOIN
      TABLE ACCESS FULL EMP
```

처리 과정은 4.2의 소트 머지 조인과 연결됩니다.

```text
DEPT를 조인 키 기준으로 정렬
EMP를 조인 키 기준으로 정렬
   ↓
양쪽 정렬 집합을 순차적으로 병합
   ↓
조인 결과 생성
```

즉, `SORT JOIN`은 소트 머지 조인을 위한 사전 정렬 단계입니다. 조인 키 기준으로 이미 정렬된 인덱스를 활용하거나, 정렬 부담보다 병합 이점이 클 때 유효한 방식입니다.

---

### 9. Window Sort

`Window Sort`는 윈도우 함수, 즉 분석함수를 수행할 때 나타납니다.

```sql
select empno, ename, job, mgr, sal,
       avg(sal) over (partition by deptno)
from emp;
```

실행계획은 다음과 같은 형태입니다.

```text
SELECT STATEMENT
  WINDOW SORT
    TABLE ACCESS FULL EMP
```

분석함수는 행을 단순히 하나씩 독립 처리하지 않습니다. `partition by`, `order by`, window frame 기준에 따라 같은 파티션의 다른 행들을 함께 봐야 합니다.

```text
EMP 읽기
   ↓
deptno 기준 파티션 구성
   ↓
필요하면 파티션 내부 정렬
   ↓
각 행에 분석함수 결과 부여
```

CS 관점으로는 "전체 배열을 단순 map 처리하는 것"이 아니라, `grouping + ordered scan + per-row derived value`에 가깝습니다.

## 실행계획 해독 포인트

이 절의 실행계획 해독 기준은 다음과 같습니다.

| 실행계획 표현 | 질문해야 할 것 |
|---|---|
| `SORT AGGREGATE` | 실제 정렬인가, 전체 집계용 변수 갱신인가 |
| `SORT ORDER BY` | 인덱스 순서로 생략할 수 있는가 |
| `SORT GROUP BY` | 결과 순서가 필요한데 `ORDER BY`를 빠뜨리지 않았는가 |
| `HASH GROUP BY` | 정렬 순서가 필요하지 않은 그룹핑인가 |
| `SORT UNIQUE` | 중복 제거가 정말 필요한가 |
| `HASH UNIQUE` | 정렬 순서가 필요하지 않은 중복 제거인가 |
| `SORT JOIN` | 소트 머지 조인을 선택한 이유가 타당한가 |
| `WINDOW SORT` | 분석함수의 파티션/정렬 기준 때문에 큰 정렬이 발생하지 않는가 |

---

## ✅ 이 절(5.1.2)의 핵심 통찰

1. 실행계획의 `SORT`는 모두 같은 의미가 아닙니다.
2. `SORT AGGREGATE`는 이름과 달리 실제 정렬이 아니라 전체 집계용 메모리 작업에 가깝습니다.
3. `ORDER BY`만 결과 순서를 보장합니다. `GROUP BY` 또는 `SORT GROUP BY` 실행계획을 보고 정렬 결과를 기대하면 안 됩니다.
4. `GROUP BY`는 소트 기반 또는 해시 기반으로 처리될 수 있습니다.
5. `DISTINCT`, `UNION`, `MINUS`, `INTERSECT`는 중복 제거 때문에 `SORT UNIQUE` 또는 `HASH UNIQUE`를 유발합니다.
6. `SORT JOIN`은 소트 머지 조인을 위한 조인 키 정렬입니다.
7. `WINDOW SORT`는 분석함수 처리를 위한 파티션/정렬 작업입니다.
8. 모든 판단은 결국 `reduce I/O`로 귀결됩니다. 불필요한 소트와 중복 제거를 줄이고, 필요한 경우에도 Temp로 밀려나지 않게 해야 합니다.

## 🔗 흐름 속 위치

5.1.1에서 소트가 PGA와 Temp를 어떻게 사용하는지 봤습니다. 5.1.2는 그 소트가 어떤 SQL 오퍼레이션에서 발생하는지를 분류합니다. 다음 5.2부터는 이 분류를 바탕으로 `UNION`, `DISTINCT`, 조인 방식 등을 조정해 소트 자체를 줄이는 방법으로 넘어갑니다.

📌 한 줄 요약: 실행계획의 `SORT`를 보면 먼저 "정렬인가, 집계인가, 중복 제거인가, 조인을 위한 준비인가"를 구분해야 합니다.
