## 5.2.3 조인 방식 변경

### 핵심 개념

인덱스로 정렬 순서를 만들 수 있어도 조인 방식이 그 순서를 깨뜨리면 `SORT ORDER BY`가 발생합니다. 따라서 소트 생략을 원한다면 인덱스뿐 아니라 조인 방식과 조인 순서까지 함께 봐야 합니다.

```text
정렬 순서에 맞는 인덱스 존재
   ↓
하지만 해시 조인 선택
   ↓
조인 결과 순서 보장 불가
   ↓
SORT ORDER BY 발생
```

### 해시 조인에서 소트 발생

계약 인덱스가 `지점ID + 계약일시` 순서라면 계약 테이블만 보면 `ORDER BY 계약일시 DESC`를 인덱스로 처리할 수 있습니다.

하지만 상품 테이블과 해시 조인을 하면 조인 결과의 순서가 깨지므로 다시 정렬해야 합니다.

```text
계약 인덱스에서 계약일시 순서로 읽기
   ↓
상품과 HASH JOIN
   ↓
해시 조인은 입력 순서를 보장하지 않음
   ↓
SORT ORDER BY 필요
```

### NL 조인으로 소트 생략

계약 테이블을 드라이빙 집합으로 삼고, 상품을 PK로 룩업하는 NL 조인을 사용하면 계약 인덱스의 정렬 순서를 유지할 수 있습니다.

```sql
select /*+ leading(c) use_nl(p) */
       c.계약번호, c.상품코드, p.상품명, p.상품구분코드,
       c.계약일시, c.계약금액
from 계약 c, 상품 p
where c.지점ID = :brch_id
and p.상품코드 = c.상품코드
order by c.계약일시 desc;
```

실행 흐름은 다음과 같습니다.

```text
계약_X01 인덱스 역순 스캔
   ↓
지점ID 조건 만족 데이터를 계약일시 DESC 순서로 읽음
   ↓
상품 PK로 NL 룩업
   ↓
계약 순서 유지
   ↓
SORT ORDER BY 생략
```

### 튜닝 포인트

조인 방식은 단순히 조인 속도만 결정하지 않습니다. 결과 집합의 순서 유지 가능성도 결정합니다.

| 조인 방식 | 정렬 순서 관점 |
|---|---|
| NL 조인 | 드라이빙 테이블의 인덱스 순서를 비교적 유지하기 쉽습니다. |
| 해시 조인 | 해시 테이블 기반이므로 입력 순서 보장이 어렵습니다. |
| 소트 머지 조인 | 조인 키가 정렬 기준이면 `SORT ORDER BY` 생략 가능성이 있습니다. |

---

## ✅ 이 절(5.2.3)의 핵심 통찰

1. 인덱스가 정렬 순서를 제공해도 조인 방식이 순서를 깨면 소트가 발생합니다.
2. 해시 조인은 결과 순서를 보장하지 않으므로 `SORT ORDER BY`가 필요할 수 있습니다.
3. NL 조인은 드라이빙 테이블의 인덱스 순서를 유지하는 데 유리합니다.
4. 정렬 기준이 조인 키라면 소트 머지 조인도 `SORT ORDER BY` 생략에 도움이 될 수 있습니다.
5. 소트 생략 튜닝은 인덱스 설계와 조인 방식 선택을 함께 봐야 합니다.

## 🔗 흐름 속 위치

5.2.1과 5.2.2가 집합 연산과 중복 제거 소트를 줄이는 방법이었다면, 5.2.3은 조인 방식 선택으로 `ORDER BY` 소트를 줄이는 방법입니다.

📌 한 줄 요약: 소트 생략은 인덱스만의 문제가 아니라 조인 순서와 조인 방식까지 포함한 실행 흐름의 문제입니다.
