## 5.4.3 Top N 쿼리가 아닐 때 발생하는 소트 부하

### 핵심 개념

페이징 쿼리에서 안쪽 `ROWNUM <= N` 조건을 제거하면 `Top N 소트`가 작동하지 않습니다. 겉으로는 같은 페이지 10건만 반환하더라도, 내부적으로는 전체 후보 집합을 정렬해야 할 수 있습니다.

```text
안쪽 ROWNUM <= N 있음
   ↓
SORT ORDER BY STOPKEY
   ↓
Top N 소트
   ↓
작은 메모리로 처리

안쪽 ROWNUM <= N 없음
   ↓
SORT ORDER BY
   ↓
전체 정렬
   ↓
Temp I/O 발생 가능
```

### 잘못된 페이징 패턴

다음 패턴은 간결해 보이지만 위험합니다.

```sql
select *
from (
  select rownum no, a.*
  from (
    select 거래일시, 체결건수, 체결수량, 거래대금
    from 종목거래
    where 종목코드 = 'KR123456'
    and 거래일시 >= '20180304'
    order by 거래일시
  ) a
)
where no between (:page - 1) * 10 + 1 and (:page * 10);
```

이 구조에서는 실행계획에서 `STOPKEY`가 사라질 수 있습니다.

```text
VIEW
  COUNT
    VIEW
      SORT ORDER BY
        TABLE ACCESS FULL
```

### 성능 차이

책의 테스트에서는 같은 690 블록을 읽고 정렬했지만 결과가 크게 달랐습니다.

| 패턴 | 실행계획 | Temp I/O |
|---|---|---|
| Top N 쿼리 | `SORT ORDER BY STOPKEY` | 없음 |
| Top N이 아닌 페이징 | `SORT ORDER BY` | Physical Read/Write 발생 |

즉, 읽은 데이터량이 같더라도 정렬 알고리즘이 다르면 디스크 소트 여부가 달라집니다.

```text
STOPKEY 있음
   ↓
N개 후보만 관리
   ↓
메모리 처리

STOPKEY 없음
   ↓
전체 후보 정렬
   ↓
Sort Area 부족
   ↓
Temp 사용
```

### 튜닝 포인트

페이징 쿼리에서는 반드시 안쪽에서 상한선을 걸어야 합니다.

```sql
select *
from (
  select rownum no, a.*
  from (
    /* 정렬된 SQL Body */
  ) a
  where rownum <= (:page * :page_size)
)
where no >= (:page - 1) * :page_size + 1;
```

---

## ✅ 이 절(5.4.3)의 핵심 통찰

1. 안쪽 `ROWNUM <= N`은 단순 조건이 아니라 Top N 소트를 작동시키는 조건입니다.
2. `STOPKEY`가 사라지면 전체 정렬이 발생할 수 있습니다.
3. 같은 10건을 반환해도 내부 처리량은 완전히 달라질 수 있습니다.
4. 잘못된 페이징 패턴은 Temp I/O를 유발합니다.
5. 실행계획에서 `SORT ORDER BY STOPKEY`와 `SORT ORDER BY`를 반드시 구분해야 합니다.

## 🔗 흐름 속 위치

5.4.2가 Top N 소트의 장점을 설명했다면, 5.4.3은 Top N 패턴을 깨뜨렸을 때 얼마나 큰 소트 부하가 생기는지 보여줍니다.

📌 한 줄 요약: 페이징 쿼리에서 `STOPKEY`를 잃는 순간 작은 페이지 조회가 전체 정렬 작업으로 변합니다.
