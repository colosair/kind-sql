## 5.4.2 Top N 쿼리의 소트 부하 경감 원리

### 핵심 개념

인덱스로 소트를 생략할 수 없어도 Top N 쿼리는 소트 부하를 크게 줄일 수 있습니다. 전체를 정렬하지 않고 N개짜리 후보 배열만 유지하기 때문입니다. 이를 `Top N 소트` 알고리즘이라고 합니다.

```text
전체 정렬
   ↓
모든 행을 Sort Area에 모아 정렬
   ↓
상위 N건 선택

Top N 소트
   ↓
N개 후보만 유지
   ↓
새 행이 후보보다 좋을 때만 후보 배열 갱신
   ↓
최종 N건 반환
```

### 알고리즘 직관

전교생 1,000명 중 키 큰 10명을 뽑는다고 하겠습니다. 전체 1,000명을 키순으로 정렬할 필요는 없습니다.

```text
처음 10명을 Top 10 후보로 세움
   ↓
나머지 학생을 한 명씩 확인
   ↓
현재 10등보다 크면 후보에 진입
   ↓
기존 10등은 탈락
   ↓
끝까지 반복
```

CS 관점에서는 크기 N짜리 bounded priority queue 또는 작은 정렬 배열을 유지하는 방식입니다.

```text
N = 10
candidate = first 10 rows sorted

for row in remaining_rows:
    if row.key < candidate.last.key:
        replace candidate.last with row
        reorder candidate
```

### 실행계획 신호

인덱스로 `ORDER BY`를 생략할 수 없으면 `SORT ORDER BY`가 나타납니다. 그러나 Top N 쿼리라면 다음처럼 `STOPKEY`가 붙습니다.

```text
COUNT STOPKEY
  VIEW
    SORT ORDER BY STOPKEY
      TABLE ACCESS FULL
```

`SORT ORDER BY STOPKEY`는 정렬은 하되 전체 정렬이 아니라 Top N 소트 알고리즘이 작동한다는 신호입니다.

### Sort Area 절감 효과

Top N 소트가 작동하면 대상 집합이 커도 N개 후보만 유지하면 됩니다.

```text
page = 1, page size = 10
   ↓
10개 원소 배열만 필요
   ↓
전체 정렬 불필요
   ↓
Sort Area 사용량 감소
   ↓
Temp 사용 가능성 감소
```

책의 예에서는 같은 690 블록을 읽었지만 `SORT ORDER BY STOPKEY`가 작동해 `physical reads`와 `sorts(disk)`가 0으로 나왔습니다.

---

## ✅ 이 절(5.4.2)의 핵심 통찰

1. Top N 쿼리는 인덱스로 소트 생략이 안 되어도 소트 부하를 줄일 수 있습니다.
2. `SORT ORDER BY STOPKEY`는 Top N 소트 알고리즘이 작동한다는 신호입니다.
3. 전체 정렬이 아니라 N개 후보만 유지하므로 Sort Area 사용량이 작습니다.
4. 이 원리는 페이징 처리에서 매우 중요합니다.
5. Top N 소트도 결국 Temp I/O를 줄이는 `reduce I/O` 전략입니다.

## 🔗 흐름 속 위치

5.3.2의 `Top N Stopkey`는 인덱스 정렬 순서로 N건에서 멈추는 방식이었습니다. 5.4.2는 인덱스로 소트 생략이 불가능할 때도 N개 후보만 유지해 소트 부하를 줄이는 방식을 설명합니다.

📌 한 줄 요약: Top N 쿼리는 전체를 정렬하지 않고 N개 후보만 관리하므로 소트 메모리를 크게 줄입니다.
