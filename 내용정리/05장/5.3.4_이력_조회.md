## 5.3.4 이력 조회

### 핵심 개념

이력 조회 튜닝의 핵심은 각 대상별 최신 이력을 찾을 때 `First Row Stopkey` 또는 `Top N Stopkey`가 작동하도록 인덱스와 SQL을 설계하는 것입니다.

```text
장비 1건
   ↓
상태변경이력에서 최신 변경일자/변경순번 조회
   ↓
인덱스 역순 또는 MIN/MAX 활용
   ↓
첫 행에서 중단
```

이력 테이블은 보통 현재 값까지 포함합니다. 그래야 특정 값이 언제 현재 상태로 바뀌었는지 이력 테이블에서 재현할 수 있습니다.

### 단순한 최신 이력 조회

상태변경이력 PK가 `장비번호 + 변경일자 + 변경순번`이면 장비별 최신 변경일자는 `MAX(변경일자)`로 빠르게 찾을 수 있습니다.

```sql
select 장비번호, 장비명, 상태코드,
       (select max(변경일자)
        from 상태변경이력 h
        where h.장비번호 = p.장비번호) 최종변경일자
from 장비 p
where 장비구분코드 = 'A001';
```

```text
장비 1건
   ↓
상태변경이력_PK에서 해당 장비번호 구간 탐색
   ↓
변경일자 최대값 쪽으로 이동
   ↓
첫 행 반환
```

### 인덱스 컬럼 가공의 위험

위 단순 조회는 `변경일자` 하나만 필요했기 때문에 `MAX(변경일자)`에 `INDEX RANGE SCAN (MIN/MAX)` + `FIRST ROW`가 깔끔하게 붙습니다. 그런데 **최종 변경순번까지** 한 번에 얻으려고 인덱스 컬럼을 결합/가공하면 `First Row Stopkey`가 작동하지 않습니다.

```sql
select 장비번호, 장비명, 상태코드
     , substr(최종이력, 1, 8)               최종변경일자
     , to_number(substr(최종이력, 9, 4))    최종변경순번
from (
  select 장비번호, 장비명, 상태코드
       , (select max(h.변경일자 || lpad(h.변경순번, 4))
          from   상태변경이력 h
          where  h.장비번호 = p.장비번호) 최종이력
  from   장비 p
  where  장비구분코드 = 'A001'
);
```

```text
max(변경일자 || lpad(변경순번, 4))
   ↓
인덱스 컬럼 가공 → MIN/MAX 최적화 불가
   ↓
INDEX RANGE SCAN (MIN/MAX 아님)
   ↓
장비별 이력 전체를 읽어 가공 후 비교
```

책의 실측이 차이를 분명히 보여줍니다. 가공하지 않은 단순 조회는 `INDEX RANGE SCAN (MIN/MAX)`로 장비당 한 건만 만지므로 **Buffers 22**에 그쳤지만, 컬럼을 가공한 위 쿼리는 같은 인덱스를 쓰면서도 장비별 이력을 모두 훑어 **A-Rows 1,825K, Buffers 6,380**까지 치솟았습니다.

장비별 이력이 적으면 문제가 작지만, 많으면 심각한 비효율이 됩니다.

### 여러 컬럼이 필요할 때: 스칼라 서브쿼리 분리

그렇다면 변경일자와 변경순번을 모두 얻으면서 `First Row Stopkey`를 유지하려면 어떻게 해야 할까요. 책의 해법은 **가공하지 않고 스칼라 서브쿼리를 컬럼별로 분리**하는 것입니다. 각 서브쿼리가 가공 없는 `MAX`이므로 모두 MIN/MAX 최적화를 받습니다.

```sql
select 장비번호, 장비명, 상태코드
     , (select max(h.변경일자)
        from   상태변경이력 h
        where  h.장비번호 = p.장비번호) 최종변경일자
     , (select max(h.변경순번)
        from   상태변경이력 h
        where  h.장비번호 = p.장비번호
        and    h.변경일자 = (select max(h2.변경일자)
                             from   상태변경이력 h2
                             where  h2.장비번호 = p.장비번호)) 최종변경순번
from   장비 p
where  장비구분코드 = 'A001';
```

```text
변경순번 서브쿼리
   ↓
안쪽: max(변경일자) → FIRST ROW (MIN/MAX)
   ↓
바깥: 그 변경일자 구간에서 max(변경순번) → FIRST ROW (MIN/MAX)
   ↓
가공이 없으므로 양쪽 모두 Stopkey 작동 (Buffers 22~47)
```

쿼리는 길고 복잡해지고 상태변경이력을 여러 번 조회하는 비효율은 있지만, `First Row Stopkey`가 잘 작동하므로 성능은 비교적 좋습니다. 출력 컬럼이 많아질수록 서브쿼리가 중첩되어 더 복잡해지는 것이 이 방식의 한계입니다.

### INDEX DESC + ROWNUM 방식

전통적으로 많이 쓰는 방식은 인덱스를 역순으로 읽고 `rownum <= 1`로 첫 행에서 멈추는 패턴입니다.

```sql
select /*+ index_desc(x 상태변경이력_pk) */
       변경일자 || lpad(변경순번, 4) || 상태코드
from 상태변경이력 x
where 장비번호 = p.장비번호
and rownum <= 1;
```

```text
장비번호 구간
   ↓
변경일자, 변경순번 역순 스캔
   ↓
첫 행 = 최신 이력
   ↓
COUNT STOPKEY
```

이 방식은 빠르지만 인덱스 구성이 바뀌면 결과가 달라질 수 있습니다. 성능은 좋지만 인덱스 구조에 강하게 의존합니다.

### 11g/12c 패턴과 Predicate Pushing

11g에서는 메인 쿼리 컬럼을 서브쿼리 내부 인라인 뷰에서 직접 참조하면 파싱 오류가 날 수 있었습니다. 이를 회피하기 위해 인라인 뷰 밖에서 장비번호 조건을 주면, 실제 실행 시 조건이 안쪽으로 밀려 들어가는 `Predicate Pushing`이 작동할 수 있습니다.

```text
인라인 뷰에서 이력 정렬
   ↓
바깥에서 장비번호 + rownum <= 1
   ↓
Predicate Pushing
   ↓
장비번호 조건이 안쪽으로 이동
   ↓
Top N Stopkey 가능
```

12c에서는 상관 서브쿼리 안 인라인 뷰 패턴도 더 자연스럽게 사용할 수 있습니다.

> ⚠️ **윈도우 함수 / Row Limiting 절은 아직 이르다.** 12c부터 `FETCH FIRST n ROWS ONLY`(Row Limiting 절)를 지원하지만, 인라인 뷰 안에서 `row_number()`나 Row Limiting 절을 쓰면 `Top N Stopkey`가 작동하지 않습니다. 옵티마이저가 Row Limiting 절을 윈도우 함수 형태로 변환하기 때문에 둘은 같은 실행계획·성능을 보입니다. **인덱스 활용이 중요한 온라인성 쿼리에서는 여전히 `rownum <= 1` Top N 쿼리를 쓰는 것이 정답입니다.**

### 윈도우 함수와 대량 이력 조회

소수 장비의 최신 이력을 찾을 때는 인덱스 + Stopkey가 유리합니다. 그러나 전체 또는 대량 장비의 최신 이력을 조회할 때는 장비별 랜덤 I/O가 폭증하므로 윈도우 함수나 `KEEP` 절이 더 유리할 수 있습니다.

```sql
select p.장비번호, p.장비명,
       h.변경일자, h.변경순번, h.상태코드
from 장비 p,
     (select 장비번호, 변경일자, 변경순번, 상태코드,
             row_number() over (
               partition by 장비번호
               order by 변경일자 desc, 변경순번 desc
             ) rnum
      from 상태변경이력) h
where h.장비번호 = p.장비번호
and h.rnum = 1;
```

전체 장비를 대상으로 하면 인덱스 + Stopkey가 오히려 독이 됩니다. 장비마다 인덱스를 타고 테이블로 랜덤 액세스하는 비용이 장비 수만큼 누적되기 때문입니다. 반면 위 방식은 상태변경이력을 `TABLE ACCESS FULL`로 한 번에 훑고 `WINDOW SORT PUSHED RANK` + 해시 조인으로 처리하므로, 대량일수록 유리합니다.

또 하나의 대안은 `MAX ... KEEP (DENSE_RANK LAST ...)` 절입니다. 윈도우 함수가 인라인 뷰 + 필터(`rnum = 1`)를 거치는 것과 달리, `KEEP`은 `GROUP BY` 집계 한 번으로 "그룹 내 정렬 마지막 행의 값"을 바로 뽑아냅니다.

```sql
select p.장비번호, p.장비명
     , h.변경일자   as 최종변경일자
     , h.변경순번   as 최종변경순번
     , h.상태코드   as 최종상태코드
from   장비 p
     , (select 장비번호
             , max(변경일자) 변경일자
             , max(변경순번) keep (dense_rank last order by 변경일자, 변경순번) 변경순번
             , max(상태코드) keep (dense_rank last order by 변경일자, 변경순번) 상태코드
        from   상태변경이력
        group by 장비번호) h
where  h.장비번호 = p.장비번호;
```

실행계획은 `WINDOW SORT PUSHED RANK` 대신 `SORT GROUP BY`로 나타나며, 책의 실측에서는 윈도우 함수 방식보다 빠릅니다(같은 Full Scan에 Buffers는 비슷하나 정렬 부담이 작음).

| 조회 범위 | 유리한 패턴 |
|---|---|
| 소수 장비 | 인덱스 + Top N Stopkey (`rownum <= 1`) |
| 전체 장비 | Full Scan + 윈도우 함수 또는 `KEEP (DENSE_RANK LAST)` |
| 현재 시점 조회가 잦음 | 선분이력 모델 고려 |

### 선분이력

선분이력 모델은 `유효시작일자`, `유효종료일자`를 둬서 시점 조회를 정렬·Stopkey 없이 **단순 범위 조건**으로 처리하게 합니다. 최신 이력을 찾는 문제 자체를 모델 단계에서 없애는 접근입니다.

**현재(최신) 상태 조회** — 현재 이력의 종료일자를 `'99991231'` 같은 무한대 값으로 채워 두면, 등치/상수 조건만으로 최신 행을 집어낼 수 있습니다.

```sql
select p.장비번호, p.장비명, h.상태코드, h.유효시작일자, h.유효종료일자
from 장비 p, 상태변경이력 h
where p.장비구분코드 = 'A001'
and h.장비번호 = p.장비번호
and h.유효종료일자 = '99991231';
```

**특정 시점 상태 조회** — 임의 기준일자의 상태는 `between` 한 줄로 찾습니다.

```sql
select p.장비번호, p.장비명, h.상태코드
from 장비 p, 상태변경이력 h
where p.장비구분코드 = 'A001'
and h.장비번호 = p.장비번호
and :base_dt between h.유효시작일자 and h.유효종료일자;
```

쿼리가 단순해지는 만큼 성능에서도 이점이 생깁니다. 다만 상태가 바뀔 때마다 직전 이력의 `유효종료일자`를 갱신해 주는 입력/수정 부담이 따릅니다(레이어 이동 트레이드오프 — 조회 비용을 입력 시점으로 미루는 것).

## ✅ 이 절(5.3.4)의 핵심 통찰

1. 이력 조회는 최신 행을 빠르게 찾는 문제입니다.
2. 소수 대상 조회는 인덱스 역순 스캔 + `rownum <= 1`이 강력합니다.
3. 인덱스 컬럼을 가공하면 `First Row Stopkey`가 깨집니다(Buffers 22 → 6,380).
4. 여러 컬럼이 필요하면 가공하지 말고 스칼라 서브쿼리를 컬럼별로 분리해 Stopkey를 유지합니다.
5. 윈도우 함수·Row Limiting 절은 인라인 뷰 안에서 Stopkey가 깨지므로 온라인성 소수 조회에는 부적합합니다.
6. 전체 대상 조회는 인덱스 랜덤 I/O보다 Full Scan + 윈도우 함수 또는 `KEEP (DENSE_RANK LAST)`가 유리합니다.
7. 이력 조회 패턴은 업무 범위와 모델(선분이력 포함)에 따라 달라져야 합니다.

## 🔗 흐름 속 위치

5.3.3의 최소/최대값 원리를 실제 업무에서 가장 자주 만나는 이력 조회 문제에 적용한 절입니다.
