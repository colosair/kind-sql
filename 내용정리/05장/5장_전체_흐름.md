## 5장 소트 튜닝 전체 흐름

### 핵심 개념

5장의 통일 원리는 **소트로 인한 I/O를 줄이는 것**입니다. 먼저 소트가 어떻게 수행되는지 이해하고, 그다음 소트가 발생하지 않도록 SQL을 작성하며, 불가피한 경우에는 인덱스를 이용해 소트를 생략하거나 `Sort Area` 사용량을 줄입니다.

```text
소트 수행 과정 이해
   ↓
소트 오퍼레이션 식별
   ↓
불필요한 소트 제거
   ↓
인덱스로 소트 생략
   ↓
불가피한 소트는 작게 수행
   ↓
Temp I/O 감소
```

| 절 | 핵심 질문 | 튜닝 방향 |
|---|---|---|
| 5.1.1 | 소트는 어디서 수행되는가 | PGA에서 끝내고 Temp 사용을 피합니다. |
| 5.1.2 | 어떤 오퍼레이션이 소트를 유발하는가 | `SORT`의 의미를 실행계획별로 구분합니다. |
| 5.2.1 | `UNION`이 꼭 필요한가 | 상호배타적이면 `UNION ALL`로 바꿉니다. |
| 5.2.2 | `DISTINCT`가 존재성 확인 때문에 쓰였는가 | `EXISTS`/`NOT EXISTS`로 바꿉니다. |
| 5.2.3 | 조인 방식이 정렬 순서를 깨는가 | NL 조인 등으로 인덱스 순서를 유지합니다. |
| 5.3.1 | `ORDER BY`를 인덱스로 대체할 수 있는가 | 정렬 기준과 인덱스 순서를 맞춥니다. |
| 5.3.2 | 상위 N건만 필요한가 | `Top N Stopkey`로 N건에서 멈춥니다. |
| 5.3.3 | 최소/최대값을 전체 스캔하고 있지 않은가 | 인덱스 끝 또는 Top 1로 찾습니다. |
| 5.3.4 | 최신 이력을 어떻게 찾는가 | 소수 조회는 Stopkey, 대량 조회는 집합 처리로 접근합니다. |
| 5.3.5 | `GROUP BY`도 인덱스로 처리 가능한가 | `SORT GROUP BY NOSORT`를 활용합니다. |
| 5.4.1 | 정렬 대상 로우 폭이 너무 크지 않은가 | 필요한 컬럼만 정렬하고 가공은 뒤로 미룹니다. |
| 5.4.2 | Top N 소트가 작동하는가 | N개 후보만 유지해 소트 부하를 줄입니다. |
| 5.4.3 | 페이징에서 Stopkey를 잃지 않았는가 | 안쪽 `rownum <= N`을 유지합니다. |
| 5.4.4 | 분석함수의 소트 부하가 큰가 | `rank`/`row_number`로 Top N 소트를 유도합니다. |

### 소트 최적화 메커니즘 한눈에 구분 (가장 헷갈리는 부분)

5장에는 이름이 닮은 최적화가 여럿 나옵니다. 핵심은 **두 갈래**입니다. 5.3은 "정렬을 아예 안 한다"(인덱스가 이미 정렬돼 있으니까), 5.4는 "정렬은 하되 작게 한다"입니다.

```text
인덱스 정렬 순서로 ORDER BY/GROUP BY를 대체할 수 있는가?
   ├─ 예 → 5.3 소트 생략 (정렬 연산 자체가 없음)
   └─ 아니오 → 5.4 소트 경감 (정렬은 하되 대상·메모리를 줄임)
```

| 개념 | 절 | 정렬 수행? | 메커니즘 | 실행계획 신호 |
|------|----|:---:|---------|--------------|
| Sort Order By 생략 | 5.3.1 | ✗ | 인덱스 순서 = `ORDER BY` 순서 | `SORT ORDER BY`가 **사라짐** |
| Top N Stopkey | 5.3.2 | ✗ | 정렬 생략 + `ROWNUM≤N` 도달 시 중단 | `COUNT STOPKEY` |
| First Row Stopkey | 5.3.3 | ✗ | `MIN/MAX`·조건 구간의 첫 행에서 중단 | `FIRST ROW` + `INDEX (MIN/MAX)` |
| Sort Group By 생략 | 5.3.5 | ✗ | 그룹핑 컬럼이 인덱스 선두 | `SORT GROUP BY NOSORT` |
| **Top N 소트** | 5.4.2 | ✓(부분) | 정렬은 하되 **N개 후보만** 유지 | `SORT ORDER BY STOPKEY` |
| 분석함수 Top N 소트 | 5.4.4 | ✓(부분) | 파티션별 **상위 순위만** 유지 | `WINDOW SORT PUSHED RANK` |

**가장 중요한 대비 — `Top N Stopkey`(5.3.2) vs `Top N 소트`(5.4.2):**

```text
Top N Stopkey (5.3.2)
   정렬 순서에 맞는 인덱스 있음
   → 정렬 안 함, 앞에서부터 읽다가 N건에서 멈춤
   → COUNT STOPKEY

Top N 소트 (5.4.2)
   정렬 순서에 맞는 인덱스 없음
   → 전부 읽되, 전체 정렬 대신 N개짜리 후보 배열만 유지
   → SORT ORDER BY STOPKEY
```

둘 다 "Top N"이고 이름에 `STOPKEY`가 들어가지만, 하나는 **읽기를 멈추는 것**(인덱스 활용)이고 다른 하나는 **정렬 메모리를 줄이는 것**(인덱스 없을 때의 차선책)입니다. 페이징 쿼리는 인덱스 유무에 따라 둘 중 하나로 작동하며, 안쪽 `rownum <= N`이 있어야 둘 다 발동합니다(없으면 5.4.3의 전체 정렬로 추락).

### 5장의 핵심 실행계획 키워드

| 실행계획 | 의미 | 확인 포인트 |
|---|---|---|
| `SORT ORDER BY` | 실제 정렬 | 인덱스로 생략 가능한지 확인합니다. |
| `SORT ORDER BY STOPKEY` | Top N 소트 | 전체 정렬이 아니라 N개 후보만 유지합니다. |
| `COUNT STOPKEY` | N건 도달 시 중단 | 페이징/Top N에서 가장 중요한 신호입니다. |
| `SORT UNIQUE` | 중복 제거 | `UNION`, `DISTINCT`, `MINUS` 필요성을 점검합니다. |
| `HASH UNIQUE` | 해시 기반 중복 제거 | 정렬 순서가 필요 없다면 가능성이 있습니다. |
| `SORT GROUP BY NOSORT` | 인덱스 기반 그룹핑 | 그룹핑 컬럼이 인덱스 선두인지 봅니다. |
| `INDEX RANGE SCAN (MIN/MAX)` | 인덱스 끝값 탐색 | 최소/최대값 최적화입니다. |
| `FIRST ROW` | 첫 행에서 중단 | MIN/MAX Stopkey 최적화 신호입니다. |
| `WINDOW SORT PUSHED RANK` | 분석함수 Top N 소트 | `rank`/`row_number` 최적화 신호입니다. |

### 통합 튜닝 흐름

```text
실행계획에 SORT 계열 오퍼레이션 발견
   ↓
1. 이 소트가 실제 정렬인지, 집계/중복제거/윈도우 처리인지 구분
   ↓
2. SQL 문법상 제거 가능한 소트인지 확인
      - UNION → UNION ALL
      - DISTINCT → EXISTS
      - MINUS → NOT EXISTS
   ↓
3. 인덱스로 소트 생략 가능한지 확인
      - ORDER BY
      - GROUP BY
      - MIN/MAX
      - Top N
   ↓
4. 조인 방식이 정렬 순서를 깨는지 확인
   ↓
5. 소트가 불가피하면 소트 대상 컬럼과 로우 수를 줄임
```

## ✅ 5장의 핵심 통찰

1. 소트는 가능하면 없애야 하고, 불가피하면 메모리 안에서 끝내야 합니다.
2. 실행계획의 `SORT`는 정렬, 집계, 중복 제거, 조인, 분석함수 등 여러 의미를 가집니다.
3. `UNION`, `DISTINCT`, `MINUS`는 중복 제거 소트를 유발하므로 필요성을 먼저 의심해야 합니다.
4. 인덱스는 이미 정렬된 자료구조이므로 `ORDER BY`, `GROUP BY`, `MIN/MAX`, Top N에 적극 활용할 수 있습니다.
5. 페이징에서 `STOPKEY`를 잃으면 작은 페이지 조회가 전체 정렬로 변합니다.
6. 분석함수도 함수 선택에 따라 소트 부하가 크게 달라집니다.

📌 한 줄 요약: 5장 소트 튜닝의 본질은 정렬을 수행하는 기술이 아니라 정렬하지 않아도 되는 실행 흐름을 만드는 기술입니다.
