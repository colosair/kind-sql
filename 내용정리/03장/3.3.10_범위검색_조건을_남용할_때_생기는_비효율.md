## 3.3.10 범위검색 조건을 남용할 때 생기는 비효율

### 핵심 개념
범위검색 조건을 습관적으로 사용하면 **인덱스 시작점과 종료점이 넓어져 많이 읽고 버리는 구조**가 됩니다.

```text
정확한 조건 대신 넓은 범위 조건 사용
   ↓
인덱스 스캔 범위 확대
   ↓
후행 조건 필터화
   ↓
논리적 I/O 증가
```

| 패턴 | 문제 |
|---|---|
| 모든 조건을 `LIKE :v || '%'`로 처리 | 등치 가능 조건도 범위화 |
| 날짜 조건을 과도하게 넓게 설정 | 불필요한 기간 스캔 |
| 옵션 조건을 범위로 우회 | 인덱스 효율 저하 |

### 구체 예제 — 옵션 조건을 LIKE '%'로 일괄 처리

인덱스 `(회사코드 + 지역코드 + 상품명)`. 지역코드는 **옵션 조건**(입력 안 될 수도 있음)이라, 개발자가 편의상 두 상황을 SQL 하나로 처리하려고 LIKE를 씁니다.

```sql
select 고객ID, 상품명, 지역코드, ...
from   가입상품
where  회사코드 = :com
and    지역코드 like :reg || '%'      -- ⚠️ 옵션 처리용 LIKE
and    상품명   like :prod || '%';
```

문제: 지역코드를 **`LIKE`로 처리하면, 값이 들어와도(예: '02') 등치(`=`)가 아닌 범위검색**이 됩니다. 그러면 인덱스 중간 컬럼이 범위가 되어 **후행 상품명이 필터 조건으로 전락**합니다.

```text
지역코드 = '02' (등치)     → 상품명까지 액세스 조건 (그림 3-43 좌, 좁은 스캔)
지역코드 like '02%' (범위) → 상품명이 필터로 전락 (그림 3-44, 넓은 스캔)
지역코드 미입력 → like '%' → 회사코드 구간 전체 스캔 (그림 3-43 우, 어쩔 수 없음)
```

지역코드가 **미입력일 때 `like '%'` 로 전체를 읽는 것은 어쩔 수 없지만**(중간 컬럼 누락), **입력될 때조차 LIKE라서 등치로 못 쓰는 것**이 남용의 비효율입니다. → 옵션 조건은 LIKE로 뭉개지 말고 별도 처리(3.3.11)해야 합니다.

### 정밀화
범위 조건 자체가 나쁜 것은 아닙니다. 문제는 **등치로 고정할 수 있는 조건까지 범위(LIKE '%')로 표현**해 인덱스의 군집성을 깨뜨리는 것입니다.

---

## ✅ 이 절(3.3.10)의 핵심 통찰
1. 범위 조건은 후행 컬럼의 액세스 효과를 약화시킵니다.
2. 등치로 표현 가능한 조건은 등치로 유지하는 것이 좋습니다.
3. 편의성 중심의 범용 SQL은 인덱스 효율을 희생할 수 있습니다.

📌 한 줄 요약: 범위 조건은 필요한 곳에만 써야 하며, 남용하면 인덱스를 타도 넓게 훑게 됩니다.
