## 3.3.12 함수호출부하 해소를 위한 인덱스 구성

### 핵심 개념 — PL/SQL 함수 호출은 비싸다

PL/SQL 함수는 인터프리터 언어라 Native로 컴파일된 내장 함수보다 느립니다. 게다가 실행 시 **SQL 실행엔진 ↔ PL/SQL 가상머신 사이의 컨텍스트 스위칭(Context Switching)** 이 발생합니다. C/Java/VB은 모듈화·공용화를 권장하지만, **PL/SQL 함수를 그렇게 쓰면 안 됩니다.**

> PL/SQL 사용자 정의 함수의 성능을 떨어뜨리는 가장 결정적 요소는 **Recursive Call(1.1.2/6.1.2)** 입니다. 조건을 만족하는 회원이 100만 명이면 `GET_ADDR(우편번호)` 함수가 **100만 번** 실행되고, 함수에 SQL이 내장돼 있으면 **그 SQL도 100만 번** 수행됩니다. 인터프리팅·컨텍스트 스위칭보다 **Recursive Call 부하가 가장 큽니다.**

```sql
-- ⚠️ PL/SQL 함수: 회원 수만큼 함수(+내장 SQL) Recursive Call
select 회원번호, 회원명, 생년, 생월일, GET_ADDR(우편번호) as 기본주소
from   회원 where 생월일 like '01%';

-- ✅ 조인(스칼라 서브쿼리)으로 대체 → 훨씬 빠름
select a.회원번호, a.회원명, a.생년, a.생월일
     , (select b.시도||' '||b.구군||' '||b.읍면동
        from 기본주소 b where b.우편번호 = a.우편번호 and b.순번 = 1) 기본주소
from   회원 a where a.생월일 like '01%';
```

### 함수 위치가 호출 횟수를 결정한다 ★

함수를 조건절 비교에 쓸 때(예: `암호화된_전화번호 = encryption(:phone_no)`), **그 함수가 인덱스 구성상 어디서 평가되느냐**에 따라 호출 횟수가 극적으로 달라집니다.

```sql
select 회원번호, 회원명, 생년, 생월일, 등록일자
from   회원
where  생년 = '1987'
and    암호화된_전화번호 = encryption(:phone_no);   -- 함수 호출
```

| 인덱스 | 암호화된_전화번호 조건의 역할 | `encryption()` 호출 횟수 |
|------|------|------|
| `회원_X01 (생년)` | **테이블 필터** 조건 | 생년='1987' 건수만큼 (많음) |
| `회원_X02 (생년 + 생월일 + 암호화된_전화번호)` | **인덱스 필터** (생월일 '=' 없어서) | 생년='1987' 건수만큼 |
| `회원_X03 (생년 + 암호화된_전화번호)` | **인덱스 액세스** 조건 | **단 1번!** |

```text
회원_X03: 생년='1987' AND 암호화된_전화번호=encryption(:phone_no)
   → 둘 다 연속 '=' 액세스 조건 → 인덱스 시작점을 한 번에 잡음
   → encryption(:phone_no)를 '시작점 계산'에 딱 1번만 호출 ✅
```

**핵심**: 함수 결과와 비교하는 컬럼을 **인덱스 액세스 조건**이 되도록 배치하면(선행 `=` 컬럼 바로 뒤), 함수는 **스캔 시작점을 구할 때 1번만** 호출됩니다. 필터 조건이면 후보 행마다 반복 호출됩니다(3.3.3).

> 즉 3.3.4(군집성)·3.3.5(선행 등치)의 원리가 **함수 호출 횟수**에도 그대로 적용됩니다 — 액세스 조건이면 1번, 필터면 N번.

## ✅ 이 절(3.3.12)의 핵심 통찰

1. **PL/SQL 함수**의 가장 큰 부하는 **Recursive Call** — 행 수만큼 호출(+내장 SQL도). 가능하면 **조인/스칼라 서브쿼리로 대체**한다.
2. 함수를 조건절 비교에 쓸 때, **인덱스 구성상 함수의 위치(액세스 vs 필터)가 호출 횟수를 결정**한다.
3. 함수 결과 비교 컬럼을 **인덱스 액세스 조건**으로 배치하면(선행 `=` 뒤) 함수는 **1번만** 호출된다.
4. 필터 조건(테이블/인덱스)이면 함수가 **후보 행마다 반복** 호출되어 CPU 부하가 커진다.
5. 인덱스는 I/O뿐 아니라 **반복 함수 호출(CPU)** 도 줄이는 도구다.

## 🔗 흐름 속 위치

3.3.3(액세스/필터 조건)·3.3.4(군집성)의 원리를 **함수 호출 횟수**라는 CPU 비용에 적용한 절입니다. PL/SQL 함수의 Recursive Call은 6.1.2(Call과 성능)와, 스칼라 서브쿼리 대체는 4.4.4와 연결됩니다.

📌 한 줄 요약: 함수호출부하 튜닝은 PL/SQL 함수를 조인으로 대체하고, 함수 비교 컬럼을 **인덱스 액세스 조건**으로 배치해 함수가 **1번만** 호출되게 만드는 것입니다.
