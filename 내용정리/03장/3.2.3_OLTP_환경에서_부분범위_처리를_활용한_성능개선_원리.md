## 3.2.3 OLTP 환경에서 부분범위 처리를 활용한 성능개선 원리

### 핵심 개념

OLTP는 보통 소량 데이터를 읽고 갱신하지만, 항상 그런 것은 아닙니다 — 수천만 건을 조회하는 경우도 있어 인덱스만으로 만족스러운 성능을 내기 어려울 수 있습니다(랜덤 액세스 多). **다행히 OLTP성 업무에서 쿼리 결과집합이 아무리 많아도 사용자가 모든 데이터를 일일이 다 확인하지는 않습니다.** 특정 정렬 순서로 상위 일부만 확인하는 경우(은행 입출금 조회, 뉴스·게시판 목록 등)가 대부분입니다.

→ 이때 **정렬 인덱스 + 부분범위 처리**를 결합하면 극적인 성능개선을 얻습니다.

---

### 게시판 예제 — 정렬 인덱스로 SORT 생략

```sql
select 게시글ID, 제목, 작성자, 등록일시
from   게시판
where  게시판구분코드 = 'A'
order by 등록일시 desc;
```

**① 인덱스가 (게시판구분코드)뿐이면 — SORT ORDER BY 발생**
```text
| Id | Operation                   | Name      | Cost |
|  1 |  SORT ORDER BY              |           | 2041 |   ← 모든 레코드 읽어 정렬
|  2 |   TABLE ACCESS BY INDEX ROWID| 게시판    | 1210 |
|* 3 |    INDEX RANGE SCAN         | 게시판_X01| 96   |
```
'A' 조건 전체를 읽어 등록일시 역순으로 정렬을 마쳐야 출력 시작 → 빠른 응답 불가.

**② 인덱스를 (게시판구분코드 + 등록일시)로 — SORT 생략**
```text
| Id | Operation                          | Name      | Cost |
|  1 |  TABLE ACCESS BY INDEX ROWID       | 게시판    | 1372 |   ← SORT 사라짐
|* 2 |   INDEX RANGE SCAN DESCENDING      | 게시판_X02| 258  |
```
인덱스가 이미 등록일시 순으로 정렬돼 있어, **전체 로우를 읽지 않고도 바로 결과 출력을 시작**할 수 있습니다(2.2.4, 2.3.6). 부분범위 처리와 만나면 첫 페이지가 즉시 나옵니다.

---

### ⚠️ "멈출 수 있어야" 의미있다 — n-Tier 환경의 제약

부분범위 처리의 핵심은 **앞 일부만 출력하고 멈출 수 있는가**입니다.

| 환경 | 부분범위 처리 |
|------|------|
| **2-Tier** (클라이언트가 DB에 직접 접속, 토드/오렌지) | 구현 가능 (커넥션 독점) |
| **n-Tier** (클라이언트 ↔ WAS/AP 서버 ↔ DB) | **구현 어려움** |

> n-Tier 아키텍처에서는 클라이언트가 특정 **DB 커넥션을 독점할 수 없습니다.** 단위 작업을 마치면 DB 커넥션을 곧바로 **커넥션 풀에 반환**해야 하므로, 그 전에 SQL 조회 결과를 클라이언트에게 **'모두' 전송하고 커서를 닫아야** 합니다. 따라서 SQL 결과집합을 조금씩 나눠 전송하도록 구현하기 어렵습니다.

**그렇다면 n-Tier에서 부분범위 처리는 무의미할까?** 그렇지 않습니다. 여전히 유효하며, 그 구현이 바로 **Top N 쿼리**(`where rownum <= N`, `FETCH FIRST N ROWS`)입니다 — "DB가 처음부터 N건만 만들고 멈추게" 하는 것. 자세한 내용은 5.3(Top N 쿼리)에서 다룹니다.

```text
2-Tier   : 클라이언트가 arraysize씩 Fetch하다 멈춤 (커서 유지)
n-Tier    : Top N 쿼리로 'DB가 N건만 만들고 멈추게' → 결과는 한 번에 전송
            (둘 다 "전체를 만들지 않는다"는 부분범위 원리)
```

---

### 핵심 원리

부분범위 처리는 `reduce I/O`를 **시간 축**으로 적용한 것입니다 — 총량을 줄일 뿐 아니라 **지금 당장 필요한 I/O만 먼저** 수행합니다. 정렬 인덱스로 SORT를 생략하면 "전체를 만들 필요 없이 앞부분부터" 가 성립합니다.

---

## ✅ 이 절(3.2.3)의 핵심 통찰

1. OLTP는 대량 결과라도 사용자가 **상위 일부만 확인**(게시판·뉴스·입출금)하므로 부분범위 처리가 유효하다.
2. **정렬 인덱스(구분코드+등록일시)** 가 있으면 SORT ORDER BY가 사라져(Cost 2041→1372) 전체를 안 읽고 첫 행을 즉시 반환한다.
3. 부분범위 처리는 **"멈출 수 있어야"** 의미가 있다.
4. **n-Tier 환경**은 커넥션 풀 반환 때문에 "결과를 모두 전송"해야 해 전통적 부분범위 처리가 어렵다 — 대신 **Top N 쿼리**로 구현한다(5.3).
5. 부분범위 처리는 `reduce I/O`의 시간 축 적용 — 지금 필요한 I/O만 먼저 한다.

## 🔗 흐름 속 위치

3.2.1~3.2.2의 부분범위 처리 원리·구현을 OLTP 성능개선으로 종합하는 절입니다. 정렬 인덱스(2.2.4·3.4.5)와 결합되며, n-Tier에서의 구현인 **Top N 쿼리는 5.3**에서 본격적으로 다룹니다. 옵티마이저 모드 `FIRST_ROWS_N`(7.2.2)이 이를 뒷받침합니다.

📌 한 줄 요약: OLTP의 빠른 화면 응답은 **정렬 인덱스로 SORT를 생략해 앞부분만 읽는** 부분범위 처리에서 나오며, n-Tier에서는 Top N 쿼리가 그 역할을 합니다.
