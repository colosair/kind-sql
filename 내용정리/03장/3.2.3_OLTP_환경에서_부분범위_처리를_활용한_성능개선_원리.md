## 3.2.3 OLTP 환경에서 부분범위 처리를 활용한 성능개선 원리

### 핵심 개념
OLTP에서 부분범위 처리가 중요한 이유는 **사용자가 실제로 소비하는 데이터가 전체 결과의 일부인 경우가 많기 때문**입니다.

```text
사용자 화면 조회
   ↓
첫 페이지 20~50건 필요
   ↓
인덱스 순서로 필요한 만큼 읽음
   ↓
응답시간 단축
```

| 상황 | 비효율 | 개선 방향 |
|---|---|---|
| 전체 정렬 후 첫 페이지 | 전체범위 처리 | ORDER BY 인덱스 구성 |
| 조건이 느슨함 | 많은 후보 스캔 | 필수 조건 선두 배치 |
| 화면은 일부만 필요 | 전체 fetch | 페이징/부분 fetch |

### 실행계획 관점
```text
좋은 흐름
INDEX RANGE SCAN
  → COUNT STOPKEY 또는 부분 fetch

나쁜 흐름
TABLE ACCESS FULL
  → SORT ORDER BY
  → 상위 일부 반환
```

### 핵심 원리
부분범위 처리는 `reduce I/O`를 시간 축으로 적용한 것입니다. 총량을 줄이는 것뿐 아니라, **지금 당장 필요한 I/O만 먼저 수행**합니다.

---

## ✅ 이 절(3.2.3)의 핵심 통찰
1. OLTP 성능은 총 수행시간보다 첫 응답시간이 더 중요할 수 있습니다.
2. 부분범위 처리는 화면 사용 패턴을 SQL 물리 설계에 반영하는 방법입니다.
3. 정렬 생략 인덱스와 결합하면 효과가 매우 큽니다.

📌 한 줄 요약: OLTP의 빠른 화면 응답은 필요한 앞부분만 읽게 만드는 인덱스 설계에서 나옵니다.
