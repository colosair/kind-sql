## 3.4.7 중복 인덱스 제거

### 핵심 개념
중복 인덱스 제거는 **기능이 겹치는 인덱스를 정리해 DML 비용과 관리 복잡도를 줄이는 작업**입니다.

```text
비슷한 인덱스 다수 존재
   ↓
INSERT/UPDATE/DELETE마다 모두 유지
   ↓
DML 지연 + 저장공간 증가
   ↓
중복 인덱스 통합/제거
```

### ① 완전 중복 — 한 인덱스가 다른 인덱스를 선두부터 완전 포함

```text
X01 : 계약ID + 청약일자
X02 : 계약ID + 청약일자 + 보험개시일자
X03 : 계약ID + 청약일자 + 보험개시일자 + 보험종료일자
```
X03 선두가 X01·X02를 **완전히 포함**합니다(완전 중복). → **X03만 남기고 X01·X02 삭제** 가능.

### ② 불완전 중복 — 선두 같지만 뒤가 다름 (통합 가능할 때)

```text
N3 : 거래일자 + 종목코드 + 계좌번호
N4 : 거래일자 + 계좌번호
```
선두는 같지만 두 번째 컬럼이 달라 얼핏 중복이 아닙니다. **하지만 거래일자가 항상 BETWEEN/부등호(범위)** 조건이면, 둘 다 거래일자만 액세스 조건이고 나머지는 필터로 전락하므로 **효율이 비슷** → 통합 가능:
- 방안 1: N4 제거 + N3 뒤에 계좌번호 추가
- 방안 2: 그냥 N4만 제거 (계좌번호+거래일자 조회는 다른 인덱스 N1으로, 거래일자만 조회는 N3로)

| 예 | 판단 |
|---|---|
| `(A)`와 `(A, B)` | `(A, B)`가 `(A)`를 완전 포함 → `(A)` 삭제(완전 중복) |
| `(A, B)`와 `(B, A)` | 선두가 달라 용도 다름 → 단순 중복 아님 |
| `(A, B)`와 `(A, C)` | A가 항상 범위 조건이면 통합 가능(불완전 중복) |

### 주의점
선두 컬럼이 같다고 항상 제거 가능한 것은 아닙니다. 짧은 인덱스가 더 작아 효율적일 수 있고, 정렬 순서나 Unique 제약, 특정 실행계획 안정성에 기여할 수도 있습니다.

---

## ✅ 이 절(3.4.7)의 핵심 통찰
1. 인덱스가 많으면 조회 선택지는 늘지만 DML 비용도 증가합니다.
2. 중복 여부는 컬럼 포함 관계뿐 아니라 실제 SQL 사용 패턴으로 판단해야 합니다.
3. 제거 전에는 실행계획, 제약조건, 정렬 생략 용도를 확인해야 합니다.

📌 한 줄 요약: 중복 인덱스 정리는 읽기 성능보다 쓰기 비용과 운영 안정성을 되찾는 작업입니다.
