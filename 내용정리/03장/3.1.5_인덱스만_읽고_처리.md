## 3.1.5 인덱스만 읽고 처리

### 핵심 개념
인덱스만 읽고 처리한다는 것은 **쿼리에 필요한 컬럼이 모두 인덱스에 있어 테이블을 방문하지 않는 상태**입니다.

```text
조건 컬럼 + 출력 컬럼이 모두 인덱스에 존재
   ↓
인덱스 리프 블록만 스캔
   ↓
TABLE ACCESS BY INDEX ROWID 생략
   ↓
랜덤 액세스 제거
```

| 방식 | 테이블 방문 | 특징 |
|---|---:|---|
| 일반 인덱스 조회 | 있음 | ROWID로 테이블 액세스 |
| Covered 조회 | 없음 | 인덱스만으로 결과 생성 |

### 실행계획
```sql
SELECT 고객번호, 주문일자
FROM 주문
WHERE 고객번호 = :cust_no;
```

```text
INDEX RANGE SCAN 주문_X1
```

테이블 액세스 단계가 없다면 인덱스만 읽고 처리한 것입니다.

> **용어**: 쿼리에 사용된 컬럼을 모두 인덱스에 포함해 테이블 액세스를 아예 없앤 쿼리를 **Covered Query**, 그 인덱스를 **Covered Index**라 합니다. 예: `select 부서번호, sum(수량) from 판매집계 where 부서번호 like '12%' group by 부서번호` 는 `(부서번호 + 수량)` 인덱스면 테이블에 안 갑니다.

### Include 인덱스 (SQL Server 2005~, Oracle엔 없음)

인덱스 키 외에 미리 지정한 컬럼을 **리프 레벨에만** 저장하는 기능입니다(컬럼 최대 1,023개).

```sql
create index emp_x01 on emp(deptno) include (sal)   -- SAL은 리프에만
create index emp_x02 on emp(deptno, sal)            -- DEPTNO+SAL 모두 루트/브랜치에도
```

| 비교 | EMP_X01 `(deptno) include (sal)` | EMP_X02 `(deptno, sal)` |
|------|------|------|
| SAL 저장 위치 | **리프 블록만** | 루트·브랜치·리프 모두 |
| `where deptno=20 and sal>=2000` | Covered (테이블 생략), 단 **인덱스 스캔량 많음**(SAL은 필터) | Covered + **스캔량 적음**(SAL이 액세스 조건) |
| `where deptno=20 order by sal` | 소트 생략 **불가** | 소트 **생략** 가능 |

→ **Include 인덱스는 순전히 테이블 랜덤 액세스를 줄이는 용도**입니다(SAL을 수직 탐색·정렬엔 못 쓰고, 필터·Covered용으로만). 인덱스 크기를 덜 키우면서 Covered 효과를 얻는 절충안입니다.

### CS 대응
HashMap의 value 전체를 가져오지 않고, 인덱스 구조 안에 필요한 컬럼이 이미 들어 있어 lookup 결과만으로 처리가 끝나는 구조입니다. Include 인덱스는 "정렬 키(브랜치)에는 안 넣고 페이로드(리프)에만 얹는" payload-only column에 해당합니다.

---

## ✅ 이 절(3.1.5)의 핵심 통찰
1. 테이블 랜덤 액세스를 완전히 제거하는 가장 강한 방법은 인덱스만 읽고 처리하는 것입니다.
2. 출력 컬럼까지 인덱스에 포함하면 Covered Query가 가능합니다.
3. 단, 인덱스 비대화와 DML 부하 증가를 함께 고려해야 합니다.

📌 한 줄 요약: 가장 빠른 테이블 액세스는 테이블에 아예 가지 않는 것입니다.
