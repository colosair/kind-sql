## 3.1.3 인덱스 손익분기점

### 핵심 개념
인덱스 손익분기점은 **인덱스로 읽는 것이 테이블 전체 스캔보다 유리한 경계**입니다.

```text
선택 결과가 적음
   ↓
인덱스 스캔 + 소량 ROWID 액세스
   ↓
인덱스 유리

선택 결과가 많음
   ↓
대량 ROWID 랜덤 액세스
   ↓
Full Table Scan 유리
```

인덱스 액세스가 Full Scan보다 느려지게 만드는 **가장 핵심적인 두 요인**:

| 구분 | Table Full Scan | 인덱스 ROWID 테이블 액세스 |
|------|------|------|
| 액세스 방식 | **시퀀셜** 액세스 | **랜덤** 액세스 |
| I/O 단위 | **Multiblock** I/O | **Single Block** I/O |

| 방식 | 장점 | 불리한 경우 |
|---|---|---|
| Index Range Scan | 필요한 범위만 접근 | 테이블 액세스가 많을 때 |
| Full Table Scan | 멀티블록 I/O 가능 | 읽을 필요 없는 블록이 많을 때 |

인덱스는 소량 조회에 강합니다. 하지만 결과 건수가 많아지면 ROWID를 따라 테이블 블록을 반복 방문(랜덤·Single Block)해야 하므로, 차라리 테이블을 순차적으로(Multiblock) 크게 읽는 편이 더 싸질 수 있습니다.

> Table Full Scan은 성능이 **일정**합니다 — 전체 1,000만 건 중 1건을 조회하든 1,000만 건을 다 조회하든 시간 차이가 거의 없습니다(`select /*+ full(t) */ count(*) from big_table t where ...`로 확인). 반면 인덱스 액세스는 추출 건수에 비례해 느려집니다.

### 손익분기점의 실제 수치

그림으로 보면, 인덱스 경유(Table Access By Index Scan)는 추출 건수에 비례해 소요시간이 **선형 증가**하는 반면, Full Table Scan은 추출 건수와 거의 무관하게 **수평선**에 가깝습니다. 두 선이 만나는 지점이 손익분기점입니다.

```text
소요시간
  │            Index Scan (선형 증가) ╱
  │                                 ╱
  │ ─────────────────────────────╳──── Full Scan (거의 수평)
  │                             ╱  └ 손익분기점
  └──────────────────────────────── 추출 건수
```

| 상황 | 손익분기점 |
|---|---|
| 일반적 | **전체의 5~20%** |
| CF가 나쁨 | 5% 미만 (심하면 1% 미만) |
| CF가 아주 좋음(인덱스 컬럼 순으로 정렬·재적재) | **90% 수준까지 상승** |

같은 5~20%라도 **테이블이 클수록 손익분기점 비율은 더 낮아집니다.** 10만 건 테이블의 10%(1만 건)는 버퍼캐시 도움을 받기 쉽지만, 1,000만 건 테이블의 10%(100만 건)를 인덱스로 추출하는 것은 비현실적입니다. 즉 손익분기점은 고정 비율이 아니라 CF·캐시·테이블 크기에 따라 달라집니다.

### 실행계획 관점
```text
소량 조회
TABLE ACCESS BY INDEX ROWID
  INDEX RANGE SCAN

대량 조회
TABLE ACCESS FULL
```

### 왜 이 개념이 중요한가 — OLTP vs 배치

> "인덱스가 항상 좋을 수 없음을 설명하려고 손익분기점 개념을 사용했을 뿐, 이를 높이려 어떤 조치를 해야 한다고 오해하면 곤란하다. ... 이 개념이 시사하는 바는, **온라인 트랜잭션 프로그램(OLTP)과 DW/OLAP/배치 프로그램 튜닝의 특징을 구분 짓는 핵심 개념**이기 때문이다."

- **OLTP(온라인)**: 소량 조회 위주 → 손익분기점 아래 → **인덱스** 튜닝이 중심.
- **DW/배치**: 대량 조회 위주 → 손익분기점 위 → **Full Scan + 해시 조인 + 병렬** 이 중심.

### CS 대응
정렬된 배열에서 이진 탐색으로 시작점을 찾아 여러 객체를 포인터로 따라가는 방식 vs 배열 전체를 순차 스캔하는 방식의 비교입니다. 찾을 대상이 많으면 **순차 스캔의 locality(Multiblock)** 가 포인터 추적(랜덤)을 이깁니다.

---

## ✅ 이 절(3.1.3)의 핵심 통찰
1. 인덱스는 항상 빠른 것이 아니라 소량 랜덤 액세스에 강합니다.
2. 일반적 손익분기점은 전체의 5~20%, CF가 나쁘면 1~5% 미만, CF가 아주 좋으면 90%까지 올라갑니다.
3. 테이블이 클수록 손익분기점 비율은 낮아집니다(1,000만 건의 10%는 인덱스로 비현실적).
4. 손익분기점은 선택도뿐 아니라 클러스터링 팩터, 캐시 상태, 멀티블록 I/O에 영향을 받습니다.

📌 한 줄 요약: 인덱스의 적은 방문 비용이 대량 랜덤 액세스로 변하는 순간 Full Scan이 더 유리해집니다.
