## 3.4.5 소트 연산을 생략하기 위한 컬럼 추가

### 핵심 개념
인덱스에 정렬 기준 컬럼을 추가하면 **ORDER BY를 위한 별도 Sort 연산을 생략**할 수 있습니다.

```text
조건으로 범위 축소
   ↓
인덱스 리프가 이미 정렬된 순서
   ↓
ORDER BY 생략
   ↓
부분범위 처리까지 가능
```

| 인덱스 구성 | 효과 |
|---|---|
| 조건 컬럼만 포함 | 조회 후 정렬 필요 가능 |
| 조건 컬럼 + 정렬 컬럼 | 인덱스 순서로 결과 반환 |

### 핵심 규칙 — '=' 컬럼은 위치 자유, 범위 컬럼은 ORDER BY 뒤

```sql
select 계약ID, 청약일자, 입력자ID, 계약상태코드, 보험시작일자, 보험종료일자
from   계약
where  취급지점ID = :br_id                          -- '=' 조건
and    청약일자 between :dt1 and :dt2                -- 범위 조건
and    입력일자 >= trunc(sysdate - 3)               -- 범위 조건
and    계약상태코드 in (:c1, :c2, :c3)
order by 청약일자, 입력자ID;
```

| 컬럼 종류 | 인덱스 배치 규칙 |
|------|------|
| **`=` 조건 컬럼** (취급지점ID) | **ORDER BY 절에 없어도** 인덱스에 포함 가능, **위치는 앞·중간 어디든 자유** |
| **ORDER BY 컬럼** (청약일자, 입력자ID) | 정렬 순서대로 |
| **범위(`=` 아닌) 조건 컬럼** (계약상태코드 등) | **반드시 ORDER BY 컬럼보다 뒤쪽**에 둬야 소트 생략 |

```text
✅ 소트 생략 가능: (취급지점ID + 청약일자 + 입력자ID)
   → '=' 컬럼(취급지점ID)이 선두여도 그 구간 안에서 청약일자·입력자ID 순 정렬 유지

✅ 소트 생략 가능: (청약일자 + 입력자ID + 계약상태코드)
   → 범위 컬럼(계약상태코드)을 ORDER BY 뒤에 배치

⚠️ 왜? '=' 컬럼은 한 값으로 고정돼 정렬을 안 흩뜨리지만,
   범위 컬럼이 ORDER BY 앞에 오면 그 컬럼 변화가 정렬을 깨뜨림 → SORT 발생
```

> 핵심: `=` 조건 컬럼은 단일 값이라 인덱스 정렬을 흩뜨리지 않으므로 어디 둬도 되지만, **범위 조건 컬럼이 ORDER BY 컬럼 앞에 오면 정렬이 깨져 소트가 발생**합니다.

---

## ✅ 이 절(3.4.5)의 핵심 통찰
1. 인덱스는 검색 도구이면서 정렬된 결과 제공 도구입니다.
2. ORDER BY 컬럼을 적절히 추가하면 Sort I/O와 CPU 비용을 줄일 수 있습니다.
3. 페이징 SQL에서는 정렬 생략과 부분범위 처리가 결합되어 효과가 큽니다.

📌 한 줄 요약: 정렬 기준까지 인덱스에 담으면 DB는 정렬하지 않고 이미 정렬된 길을 따라 읽을 수 있습니다.
