## 3.4.1 인덱스 설계가 어려운 이유

### 핵심 개념
인덱스 설계가 어려운 이유는 **조회 성능, DML 성능, 저장공간, 업무 패턴이 서로 충돌**하기 때문입니다.

```text
조회 SQL마다 최적 인덱스 생성
   ↓
조회는 빨라질 수 있음
   ↓
DML 비용 증가 + 저장공간 증가 + 관리 복잡도 증가
```

| 요소 | 인덱스 증가 시 영향 |
|---|---|
| SELECT | 대체로 유리 |
| INSERT/UPDATE/DELETE | 불리 |
| 저장공간 | 증가 |
| 옵티마이저 선택지 | 증가하지만 불안정 가능 |

인덱스는 공짜 캐시가 아니라 별도 자료구조입니다. 데이터를 바꿀 때마다 인덱스도 함께 유지해야 하므로, 많이 만들수록 쓰기 비용이 증가합니다.

---

## ✅ 이 절(3.4.1)의 핵심 통찰
1. 인덱스 설계는 개별 SQL 최적화가 아니라 시스템 전체 균형 문제입니다.
2. 인덱스는 조회 비용을 줄이는 대신 DML 비용을 늘립니다.
3. 업무 중요도와 실행 빈도를 기준으로 전략적 선택이 필요합니다.

📌 한 줄 요약: 인덱스 설계는 빠른 SELECT와 무거워지는 DML 사이에서 시스템 전체 비용을 배분하는 작업입니다.
