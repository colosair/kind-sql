## 3.2.1 부분범위 처리

### 핵심 개념 — 미장공/운반공 비유 (그림 3-15)

**부분범위 처리**는 전체 결과를 다 만들기 전에 **일정량(Array Size)씩 나누어 먼저 반환**하고, 사용자(클라이언트)로부터 **추가 Fetch Call이 있을 때만** 그다음을 전송하는 방식입니다.

```text
DB 클라이언트(미장공) ──Fetch Call("100개 더 가져오세요")──► DB 서버(운반공)
   ◄── arraysize만큼 데이터(벽돌)를 손수레에 실어 전송 ──
   추가 Fetch Call이 없으면 서버 프로세스는 멈추고 대기 큐에서 잠(1.3.1)
```

운반공(서버 프로세스)은 Fetch Call을 받을 때만 arraysize만큼 데이터를 나르고, 요청이 없으면 거기서 멈춥니다. 전체 결과집합을 다 보내지 않고 남겨둔 채 대기합니다.

| 처리 방식 | 특징 | 초기 응답 |
|------|------|------|
| 전체범위 처리 | 결과 전체를 만든 뒤 반환 | 느림 |
| **부분범위 처리** | arraysize씩 먼저 반환 | **빠름** |

---

### 1억 건이어도 즉시 출력되는 이유

```java
// BIG_TABLE이 1억 건인데도 버튼 클릭하자마자 화면 출력
ResultSet rs = stmt.executeQuery("select name from big_table");
for (int i = 0; i < 100; i++) {
  if (rs.next()) System.out.println(rs.getString(1));
}
rs.close(); stmt.close();
```
DBMS가 데이터를 **모두 읽어 한 번에 전송하지 않고, 먼저 읽은 arraysize만큼 전송하고 멈추기** 때문입니다. 전송 후 서버 프로세스는 CPU를 OS에 반환하고 대기 큐에서 잠을 자다, 다음 Fetch Call을 받으면 그다음을 읽어 전송합니다.

### Array Size와 클라이언트 캐싱

**Array Size**는 클라이언트에서 설정합니다(Java 기본값 **10**, `Statement.setFetchSize()`로 변경). Array Size가 10일 때 100건을 읽는 메커니즘:

```text
1. 최초 rs.next()  → Fetch Call로 10건 받아 클라이언트 캐시에 저장
2. 이후 rs.next()  → Fetch Call 없이 캐시에서 읽음 (9번)
3. 캐시 10건 소진 후 rs.next() → 추가 Fetch Call로 10건
4. 100건 다 읽을 때까지 2~3 반복  → 총 Fetch Call 10번
```
100건을 받아 출력하고 곧바로 `close`하면, 1억 건 테이블이어도 **나머지 9999만 9900건은 읽지도 않습니다.** (전체범위 처리라면 절대 이렇게 빠를 수 없습니다.)

### CS 대응

**Lazy evaluation / iterator(pull 모델)** 와 동일합니다. 전체 리스트를 materialize하지 않고 필요한 만큼만 pull하며, Array Size는 iterator의 **batch/buffer 크기**에 해당합니다. (6.1.3 Array Processing은 같은 arraysize를 **쓰기/Call 축소** 방향으로 쓰는 것)

---

## ✅ 이 절(3.2.1)의 핵심 통찰

1. 부분범위 처리는 전체 처리량이 아니라 **초기 응답시간**을 줄이는 전략 — Fetch Call이 있을 때만 arraysize씩 전송한다(미장공/운반공).
2. 그래서 **1억 건 테이블도 앞 100건만 보면 즉시** 반환된다(나머지는 읽지도 않음).
3. **Array Size**(Java 기본 10, `setFetchSize`)만큼 받아 클라이언트 캐시에 두고, 소진되면 다음 Fetch Call을 보낸다.
4. 인덱스 정렬 순서를 활용하면 정렬 없이 앞부분만 빠르게 읽어 부분범위 처리가 극대화된다(2.2.4).
5. OLTP 화면 조회에서는 전체 결과보다 **첫 페이지 응답성**이 더 중요할 때가 많다.

## 🔗 흐름 속 위치

3.1이 "테이블 액세스를 줄이는" 공간적 튜닝이었다면, 3.2는 "결과를 다 만들지 않고 앞부분만 반환하는" 시간적 튜닝입니다. 다음 3.2.2는 구현 방법을, 3.2.3은 OLTP에서의 성능개선 원리를 다룹니다. 옵티마이저 모드 `FIRST_ROWS_N`(7.2.2)과 직결됩니다.

📌 한 줄 요약: 부분범위 처리는 전체를 다 처리하지 않고 **Array Size씩 Fetch Call에 응답**해 사용자가 필요한 앞부분만 빠르게 보여주는 전략입니다.
