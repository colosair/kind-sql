## 3.1.4 인덱스 컬럼 추가

### 핵심 개념

테이블 액세스 최소화를 위해 가장 일반적으로 사용하는 튜닝 기법은 **인덱스에 컬럼을 추가**하는 것입니다. 목적은 스캔량 감소가 아니라 **테이블 랜덤 액세스 횟수 감소**입니다.

---

### 예제 — 1명인데 테이블을 6번 방문

`EMP_X01 = (DEPTNO + JOB)` 인덱스만 있는 상태에서:
```sql
select /*+ index(emp emp_x01) */ * from emp
where deptno = 30 and sal >= 2000;
```
조건을 만족하는 사원은 **단 1명**(BLAKE, sal 2850)인데, 인덱스에는 SAL이 없으므로:
```text
인덱스(DEPTNO=30): CLERK·MANAGER·SALESMAN... 6건의 ROWID
   ↓ 각 ROWID로 테이블 방문 (6번)
테이블에서 sal >= 2000 필터 → 최종 1건  (그림 3-9)
   → 1명 찾으려고 테이블을 6번 랜덤 액세스 (5번은 헛걸음)
```

**이상적 해결**은 인덱스를 `(DEPTNO + SAL)`로 바꾸는 것이지만, 실 운영에선 **인덱스 변경이 어렵습니다** — 기존 SQL(`where deptno=30 and job='CLERK'`)이 그 인덱스를 쓰고 있을 수 있기 때문입니다.

### 차선책 — 기존 인덱스에 컬럼 추가

`EMP_X01`에 SAL을 추가해 `(DEPTNO + JOB + SAL)`로 만듭니다.

```text
인덱스(DEPTNO=30): 6건 스캔  ← 인덱스 스캔량은 그대로!
   ↓ 인덱스 단계에서 SAL>=2000 필터링
테이블 방문: 1번  ← 테이블 랜덤 액세스 6→1 (그림 3-10)
```

> **핵심 nuance**: 인덱스에 JOB이 있어도 조건엔 없으므로 **인덱스 스캔량은 줄지 않습니다(deptno=30 6건 그대로)**. 하지만 SAL을 인덱스에 넣어 **테이블에 가기 전에 걸러내므로 테이블 랜덤 액세스가 6→1로 줄어듭니다.** 이것이 컬럼 추가의 본질입니다.

| 구성 | 필터 위치 | 테이블 액세스 |
|------|------|------|
| `(DEPTNO + JOB)` | 테이블 방문 후 SAL 필터 | 6번 (랜덤 多) |
| `(DEPTNO + JOB + SAL)` | **인덱스에서 SAL 필터** | 1번 |

---

### 실제 사례 — 99.6%가 헛걸음

`로밍렌탈_N2 = (서비스번호)` 인덱스로:
```sql
select 렌탈관리번호, 고객명, 서비스번호, 예약접수일시, ...
from   로밍렌탈
where  서비스번호 like '010%' and 사용여부 = 'Y';
```
```text
SQL 트레이스: Fetch 78 calls, 49.2초, Query(논리 I/O) 266,968
  인덱스 스캔 266,476건 → 테이블 266,476번 방문 → 265,957 블록(전체 99.6%!)
  그런데 최종 결과는 1,909건뿐
  → 테이블 방문 후 '사용여부=Y'를 체크하는 과정에서 대부분 걸러짐 (헛걸음 99.6%)
```
**해결**: 로밍렌탈_N2에 `사용여부` 컬럼을 추가 → 테이블에 가기 전 인덱스에서 `사용여부='Y'`를 필터링 → 테이블 랜덤 액세스가 급감 → 49초가 극적으로 단축.

---

### 주의점

컬럼을 무조건 많이 붙이면 **DML 비용(6.1.1)과 저장공간**이 증가합니다. 추가 컬럼은 "테이블 액세스를 실질적으로 줄이는가"를 기준으로 판단합니다.

---

## ✅ 이 절(3.1.4)의 핵심 통찰

1. 인덱스 컬럼 추가의 목적은 **스캔량 감소가 아니라 테이블 랜덤 액세스 횟수 감소**다.
2. 필터 컬럼을 인덱스에 넣으면 **인덱스 스캔량은 그대로여도** 테이블에 가기 전에 걸러내 액세스가 준다(EMP: 6→1).
3. 인덱스 순서 변경이 이상적이지만 기존 SQL 영향으로 어려울 때, **기존 인덱스에 컬럼 추가**가 현실적 차선책이다.
4. 실제 사례: 266,476번 방문 중 **99.6%가 헛걸음**(결과 1,909건) → 필터 컬럼 추가로 해결.
5. 컬럼 추가는 조회 성능과 **DML 비용·저장공간의 교환**이므로 실효성으로 판단한다.

## 🔗 흐름 속 위치

3.1.1~3.1.3에서 본 "랜덤 액세스가 비싸다·손익분기점"의 직접적 해결책입니다. "인덱스 단계에서 거른다 vs 테이블에서 거른다"는 구분은 3.3.3(액세스 조건 vs 필터 조건)에서 이론화되고, 더 나아가 테이블을 아예 안 가는 3.1.5(인덱스만 읽고 처리)로 이어집니다.

📌 한 줄 요약: 인덱스 컬럼 추가는 인덱스 스캔량은 그대로 두되 **테이블에 가기 전에 필터링**해 랜덤 액세스를 줄이는, 가장 일반적인 테이블 액세스 최소화 기법입니다.
