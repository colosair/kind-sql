## 3.3.8 IN 조건은 '='인가

### 핵심 명제

SQL 튜닝 입문자에게 흔한 오해가 **"IN 조건 = '=' 조건"** 으로 동일시하는 것입니다. 결론:

> **IN 조건은 '='이 아니다. IN 조건이 '='이 되려면 IN-List Iterator 방식으로 풀려야 한다. 그렇게 풀리지 않으면 IN은 필터 조건이다.**

```text
col IN (1,2,3)
 ├─ IN-List Iterator로 풀림  → col=1, col=2, col=3 각각 수직 탐색 → '='처럼 작동 ✅
 └─ 필터로 처리됨           → 스캔하며 검사만 → '=' 아님
```

---

### 인덱스 구성에 따라 달라진다 — 상품ID/고객번호

```sql
select * from 고객별가입상품
where 고객번호 = :cust_no
and   상품ID in ('NH00037', 'NH00041', 'NH00050');   -- 고객별 평균 3건 가입
```

**① 인덱스 `(상품ID + 고객번호)`** — 같은 고객번호(1234)가 상품ID에 따라 **흩어짐**(그림 3-39, 연속되지 않은 3개 리프 블록).
→ 상품ID IN을 **IN-List Iterator로 풀어야** 효과적. 이는 SQL을 아래처럼 `union all`로 분해하는 것과 같습니다 — **IN이 '='이 됨.**
```sql
select * from 고객별가입상품 where 고객번호=:cust_no and 상품ID='NH00037'
union all select * ... and 상품ID='NH00041'
union all select * ... and 상품ID='NH00050';
-- → 고객번호·상품ID 둘 다 인덱스 액세스 조건, 수직 탐색 3번 (약 9블록)
```
이 구성에선 IN-List Iterator로 풀려야 하고 **반드시 그렇게 풀려야** 합니다(필터로 처리되면 고객번호만 액세스 조건이라 흩어진 블록을 다 읽음).

**② 인덱스 `(고객번호 + 상품ID)`** — 같은 고객번호가 상품ID 순으로 정렬돼 **같은 리프 블록에 모임**(그림 3-40).
→ 상품ID IN을 IN-List Iterator로 풀어도 되고(수직 3번, 9블록), **필터로 처리해도 됨**(고객번호로 액세스, 같은 블록이라 1~3블록). **오히려 필터가 나을 수 있음.**

---

### ⚠️ 반전 — IN-List Iterator가 항상 좋은 건 아니다

> "IN 조건을 '=' 조건으로 만들기 위해(즉 액세스 조건으로 만들기 위해) IN-List Iterator 방식으로 푸는 것이 **항상 효과적인가? 아니다.** 상품ID가 액세스 조건으로 의미 있는 역할을 하려면 **고객별 상품 데이터가 아주 많아야** 한다. 그렇지 않으면 상품ID는 **필터 방식으로 처리되는 게 오히려 낫다.**"

데이터가 흩어져 있고 많으면 IN-List Iterator(='), 모여 있고 적으면 필터가 유리합니다.

**더 쉬운 예** — `상품_PK(상품ID)`, `상품_X01(상품ID + 상품구분코드)`:
```sql
select * from 상품 where 상품ID = :prod_id and 상품구분코드 in ('GX', 'KR');
```
`상품ID=:prod_id`면 이미 1건(PK)이므로, 상품구분코드 IN은 **X01 인덱스의 필터 조건**으로 처리됩니다(Predicate Information에서 `filter`로 확인). IN-List Iterator로 풀 이유가 없습니다.

---

## ✅ 이 절(3.3.8)의 핵심 통찰

1. **IN ≠ '='.** IN이 '='처럼 작동하려면 **IN-List Iterator**로 풀려야 하고, 안 풀리면 **필터 조건**이다.
2. IN-List Iterator = `union all` 분해 = 각 값마다 수직 탐색(액세스 조건).
3. 인덱스 구성이 핵심: **데이터가 흩어진 구성**이면 IN-List Iterator로 풀어야 하고, **모여 있는 구성**이면 필터가 오히려 낫다.
4. IN-List Iterator가 **항상 좋은 건 아니다** — 값별 데이터가 많아야 액세스 조건으로서 의미가 있다.
5. 실행계획 Predicate Information의 `access`/`filter`로 IN이 어떻게 처리됐는지 확인한다.

## 🔗 흐름 속 위치

3.3.6(BETWEEN→IN-List)·3.3.7(Skip Scan)과 함께 "범위/흩어진 조건을 어떻게 등치 반복으로 바꿀까"의 일부이며, 3.3.3(액세스/필터 조건)의 구체 적용입니다. "항상 좋은 건 아니다"라는 균형 감각은 3.4(인덱스 설계)로 이어집니다.

📌 한 줄 요약: IN 조건은 IN-List Iterator로 풀릴 때만 '='처럼 작동하며, 데이터가 흩어져 많을 때 효과적이고 모여 있으면 필터가 오히려 낫습니다.
