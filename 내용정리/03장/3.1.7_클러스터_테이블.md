## 3.1.7 클러스터 테이블

### 핵심 개념
클러스터 테이블은 **같은 클러스터 키 값을 갖는 행들을 같은 블록 근처에 모아 저장하는 구조**입니다.

```text
클러스터 키 기준 저장
   ↓
같은 키의 여러 행이 가까운 블록에 위치
   ↓
조인/범위 조회 시 블록 방문 감소
```

| 구분 | 특징 | 유리한 경우 |
|---|---|---|
| 인덱스 클러스터 | 클러스터 키 인덱스로 위치 탐색 | 키 기반 반복 조회 |
| 해시 클러스터 | 해시 함수로 저장 위치 결정 | 등치 검색 |

**인덱스 클러스터**는 클러스터 키(예: deptno) 값이 같은 레코드를 한 블록에 모아 저장합니다(그림 3-12). 한 블록에 다 못 담으면 새 블록을 할당해 클러스터 체인으로 연결합니다. 심지어 **여러 테이블 레코드를 같은 블록에 저장**할 수도 있는데, 이를 **다중 테이블 클러스터**라 합니다(일반 테이블은 한 블록을 여러 테이블이 공유 못 함, 1.3.2).

```sql
-- ① 클러스터 생성 → ② 클러스터 인덱스(필수) → ③ 테이블 생성
create cluster c_dept# (deptno number(2)) index;
create index c_dept#_idx on cluster c_dept#;
create table dept (deptno number(2) not null, dname varchar2(14), loc varchar2(13))
cluster c_dept#(deptno);
```

> ⚠️ **클러스터 인덱스의 특징**: 일반 B*Tree 인덱스는 테이블 레코드와 **1:1**(레코드마다 ROWID)이지만, 클러스터 인덱스는 **키 값이 저장된 첫 데이터 블록만 가리켜 1:M** 관계입니다. 따라서 **클러스터 인덱스 키는 항상 Unique**(중복 값 없음)입니다.
> 또한 MS-SQL/Sybase의 '클러스터형 인덱스'는 오라클 클러스터가 아니라 **IOT에 가깝습니다** — 오라클 클러스터는 같은 키를 같은 공간에 모으기만 할 뿐 IOT처럼 정렬하지는 않습니다.

클러스터 테이블은 논리적으로 관련 있는 행을 물리적으로 가깝게 배치합니다. 이는 DB 블록 레벨의 locality를 높여 랜덤 액세스를 줄이는 전략입니다.

### CS 대응
Separate Chaining HashMap에서 같은 해시 버킷에 관련 데이터를 모아두는 발상과 유사합니다. 탐색 시작점을 빨리 찾고, 관련 데이터를 한 번에 훑게 만드는 구조입니다.

---

## ✅ 이 절(3.1.7)의 핵심 통찰
1. 클러스터 테이블은 물리 저장 순서를 액세스 패턴에 맞추는 설계입니다.
2. 관련 행을 같은 블록 근처에 모아 랜덤 액세스를 줄입니다.
3. 조회 패턴이 안정적일 때 효과가 크고, 범용 테이블에는 신중해야 합니다.

📌 한 줄 요약: 클러스터 테이블은 자주 함께 읽는 데이터를 물리적으로 함께 두어 I/O를 줄이는 구조입니다.
