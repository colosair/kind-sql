## 3.3.11 다양한 옵션 조건 처리 방식의 장단점 비교

### 핵심 개념
옵션 조건 처리의 핵심은 **조건이 있을 때와 없을 때 서로 다른 최적 경로를 선택하게 만드는 것**입니다.

```text
옵션 조건 존재
   ↓
해당 조건을 액세스 조건으로 활용

옵션 조건 없음
   ↓
불필요한 조건 없이 다른 인덱스/경로 선택
```

| 방식 | 장점 | 단점 |
|---|---|---|
| `(:v IS NULL OR col = :v)` | SQL 하나로 처리 | 인덱스 액세스 어려움 |
| `col = NVL(:v, col)` | 작성 편함 | 컬럼 가공/필터화 위험 |
| 동적 SQL | 조건별 최적화 가능 | SQL 관리 복잡 |
| UNION ALL 분기 | 경로 분리 가능 | SQL 길어짐 |

### 예시
```sql
-- 범용 조건
WHERE (:cust_no IS NULL OR 고객번호 = :cust_no)

-- 조건이 있을 때만 추가하는 동적 SQL
WHERE 1 = 1
  AND 고객번호 = :cust_no
```

### 레이어 이동 관점
옵션 조건을 하나의 SQL에서 모두 처리하면 DB 옵티마이저가 애매한 경로를 선택할 수 있습니다. 동적 SQL은 조건 조합에 대한 결정을 애플리케이션 레이어에서 명확히 하여 DB가 더 정확한 실행계획을 세우게 합니다.

---

## ✅ 이 절(3.3.11)의 핵심 통찰
1. 옵션 조건은 편하게 작성할수록 인덱스 사용성이 나빠질 수 있습니다.
2. 동적 SQL은 조건별 최적 경로를 만들 수 있어 OLTP에서 강력합니다.
3. 하나의 정적 SQL로 모든 경우를 해결하려는 설계는 `reduce I/O`와 충돌할 수 있습니다.

📌 한 줄 요약: 옵션 조건 튜닝은 범용 SQL의 편의성과 조건별 최적 실행계획 사이의 균형입니다.
