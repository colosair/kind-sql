## 3.3.11 다양한 옵션 조건 처리 방식의 장단점 비교

### 핵심 개념

**옵션 조건**(입력될 수도, 안 될 수도 있는 조건)을 처리하는 방식은 여러 가지이며, 각각 인덱스 사용성이 다릅니다. 목표는 **조건이 있을 때는 액세스 조건으로 쓰고, 없을 때는 불필요한 제약 없이** 동작하게 하는 것입니다. 거래 테이블(`고객ID`, `거래일자` 조건, `고객ID`가 옵션)로 5가지 방식을 비교합니다.

---

### ① OR 조건 — 권장하지 않음

```sql
select * from 거래
where (:cust_id is null or 고객ID = :cust_id)
and   거래일자 between :dt1 and :dt2;
```
```text
Execution Plan
1   TABLE ACCESS (FULL) OF '거래'        ← Full Scan! (OR Expansion 안 됨)
```
- **문제**: 옵티마이저가 OR Expansion을 못 해 **Full Scan**으로 풀립니다. (18c부터는 인덱스 구성 컬럼 중 하나 이상이 Not Null이면 인덱스 필터 조건으로 사용 가능)
- **유일한 장점**: 옵션 컬럼이 **NULL 허용 컬럼이어도 결과집합을 보장**합니다(다른 방식은 union all 빼고 NULL 허용 컬럼에 못 씀).
- → 가급적 사용하지 않습니다.

### ② LIKE / BETWEEN — 선두는 필수 컬럼, 옵션은 필터

```sql
where 거래일자 between :dt1 and :dt2          -- 필수 컬럼을 선두 액세스 조건으로
and   고객ID like :cust_id || '%';            -- 옵션은 인덱스 필터로 (미입력 시 '%')
```
- 필수 조건 컬럼(거래일자)을 인덱스 선두 액세스 조건으로 두면, 옵션 컬럼(고객ID)이 LIKE/BETWEEN 필터여도 충분히 좋은 성능을 냅니다(고객별 등록 상품 소수 등).
- ⚠️ **주의 1**: 옵션 컬럼을 **선두에** LIKE/BETWEEN으로 두면 범위검색이라 약간 비효율.
- ⚠️ **주의 2 (결과 오류)**: **NULL 허용 컬럼에 LIKE는 금물** — `고객ID like '%'`는 NULL 값을 누락시킵니다(성능 아닌 품질 문제).
- ⚠️ **주의 3 (가변 길이)**: `고객명 like :nm||'%'` 는 '김훈' 입력 시 '김훈남'도 조회됨 → `and length(고객명)=length(nvl(:nm, 고객명))` 추가. (또는 `like :nm` 처럼 '%' 없이 쓰고 미입력 시 '%' 입력)

### ③ UNION ALL — 분기별 최적 인덱스

```sql
select * from 거래
where :cust_id is null  and 거래일자 between :dt1 and :dt2     -- 거래일자 인덱스
union all
select * from 거래
where :cust_id is not null and 고객ID = :cust_id and 거래일자 between :dt1 and :dt2;  -- 고객ID+거래일자 인덱스
```
- 값 입력 여부에 따라 **두 SQL 중 하나만** 실행되어, 각 분기가 **최적 인덱스**를 씁니다.
- NULL 허용 컬럼에도 안전. 단 SQL이 길어집니다.

### ④ NVL / DECODE + OR Expansion ★ — 옵션을 액세스 조건으로

```sql
where 고객ID = nvl(:cust_id, 고객ID) and 거래일자 between :dt1 and :dt2
-- 또는
where 고객ID = decode(:cust_id, null, 고객ID, :cust_id) and 거래일자 between :dt1 and :dt2;
```
```text
Execution Plan
1   CONCATENATION                          ← OR Expansion으로 두 브랜치 분리
2    FILTER (:cust_id is null)
3     INDEX RANGE SCAN 거래_IDX1            -- 거래일자
5    FILTER (:cust_id is not null)
6     INDEX RANGE SCAN 거래_IDX2            -- 고객ID + 거래일자
```
- 함수 인자로 **컬럼을 가공(`nvl(:cust_id, 고객ID)`)했는데도 인덱스를 쓸 수 있는** 이유는, 옵티마이저가 **OR Expansion**(`CONCATENATION`)으로 쿼리를 변환하기 때문입니다 → **옵션 컬럼을 인덱스 액세스 조건으로** 사용!
- ⚠️ **NVL·DECODE만** OR Expansion이 작동합니다. **Coalesce·CASE는 안 됩니다.** (파라미터: `_or_expand_nvl_predicate`)

### ⑤ Dynamic SQL — 조건절을 동적 구성

```xml
<!-- iBatis/MyBatis 예: 값이 있을 때만 조건절 추가 -->
select * from 거래
where 거래일자 between #dt1# and #dt2#
<isNotEmpty prepend="and" property="cust_id">
  고객ID = #cust_id#                        <!-- 값 있을 때만 '=' 조건 -->
</isNotEmpty>
```
- 조건절을 **동적으로 구성**하면 옵션 조건에 **`=` 연산자를 그대로** 쓸 수 있어 가장 깔끔합니다. 변별력 좋은 컬럼을 액세스 조건으로 쓸 인덱스만 잘 구성하면 됩니다.
- ⚠️ **하드파싱 주의**: 조건절은 동적으로 구성하되 **입력 값은 반드시 바인드 변수**로 처리해야 합니다(1.2.2). "조건절을 동적 구성한다고 입력 값까지 동적으로 바꿀 이유는 없습니다."
- Dynamic SQL을 허용 안 하는 시스템(금융권 등)이면 힌트로 액세스 경로를 고정해야 할 수도 있습니다.

---

### 방식 비교 요약

| 방식 | 옵션 컬럼 역할 | NULL 허용 컬럼 | 비고 |
|------|------|:---:|------|
| ① OR 조건 | Full Scan(18c~ 필터) | **O** | 권장 안 함 |
| ② LIKE/BETWEEN | 인덱스 필터 | X | 필수 컬럼 선두 + 가변길이 주의 |
| ③ UNION ALL | 분기별 액세스 조건 | O | SQL 길어짐 |
| ④ NVL/DECODE | **액세스 조건**(OR Expansion) | X | NVL/DECODE만(Coalesce/CASE X) |
| ⑤ Dynamic SQL | **액세스 조건**(`=`) | O | 가장 깔끔, 바인드 변수 필수 |

### 레이어 이동 관점

①·②는 "하나의 정적 SQL로 모든 경우"를 처리하려다 인덱스 효율을 희생합니다. ④·⑤는 **조건 조합에 대한 결정을 옵티마이저(OR Expansion)나 애플리케이션(Dynamic SQL) 레이어에서 명확히** 하여 DB가 정확한 실행계획을 세우게 합니다 — "비싼 결정을 더 적합한 레이어로 올린다"는 통일 패턴입니다.

---

## ✅ 이 절(3.3.11)의 핵심 통찰

1. 옵션 조건 처리는 **인덱스 사용성과 SQL 편의성의 균형** 문제다.
2. **OR 조건(①)** 은 OR Expansion이 안 돼 Full Scan — 권장 안 함(장점은 NULL 허용 컬럼 보장뿐).
3. **LIKE/BETWEEN(②)** 은 옵션을 인덱스 필터로 — NULL 허용 컬럼 결과 오류·가변 길이(`length`)에 주의.
4. **NVL/DECODE(④)** 는 OR Expansion(CONCATENATION)으로 옵션을 **액세스 조건**으로 만든다(단 Coalesce/CASE는 안 됨).
5. **Dynamic SQL(⑤)** 이 가장 깔끔하게 `=`를 쓰지만, **입력 값은 반드시 바인드 변수**로 처리해 하드파싱을 막아야 한다.

## 🔗 흐름 속 위치

3.3.9~3.3.10에서 본 "범위검색·LIKE 남용의 비효율"에 대한 종합 해법입니다. OR Expansion은 2.2.2(OR 조건)와, Dynamic SQL의 바인드 변수 원칙은 1.2.2와, 레이어 이동은 6.1.2와 연결됩니다.

📌 한 줄 요약: 옵션 조건은 OR/LIKE로 뭉개지 말고, **NVL/DECODE(OR Expansion)나 Dynamic SQL(=)** 로 옵션 컬럼을 액세스 조건으로 살리는 것이 핵심이며, 후자는 바인드 변수를 지켜야 합니다.
