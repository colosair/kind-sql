## 3.4.3 스캔 효율성 이외의 판단 기준

### 핵심 개념
인덱스 설계는 스캔 효율만으로 결정하지 않습니다. **정렬 생략, 테이블 액세스 감소, DML 부담, 업무 중요도**도 함께 판단해야 합니다.

```text
스캔 효율 좋은 인덱스
   ↓
하지만 정렬 필요 / 테이블 방문 많음 / DML 부담 큼
   ↓
전체 비용 기준으로 재평가
```

> 3.4.2의 두 공식(① 항상/자주 사용 ② 등치)은 **인덱스 스캔 효율성** 기준입니다. 하지만 공식대로만 하면 오히려 인덱스 개수가 더 늘어나는 현상도 생깁니다. 책이 추가로 제시하는 판단 기준은 다음과 같습니다.

```text
스캔 효율성 외 판단 기준
 ├─ 수행 빈도          (자주 쓰는 SQL인가)
 ├─ 업무상 중요도       (핵심 거래인가)
 ├─ 클러스터링 팩터     (테이블 액세스 비용, 3.1.2)
 ├─ 데이터량           (테이블 크기, 손익분기점)
 ├─ DML 부하          (기존 인덱스 개수, 초당 DML 발생량, 자주 갱신하는 컬럼 포함 여부)
 ├─ 저장 공간
 └─ 인덱스 관리 비용
```

| 판단 기준 | 질문 |
|---|---|
| 스캔 효율 | 읽는 인덱스 엔트리가 충분히 적은가 |
| 테이블 액세스(CF) | 인덱스만으로 처리하거나 군집성이 좋은가 |
| 정렬 생략 | ORDER BY/GROUP BY를 대체할 수 있는가 |
| DML 부하 | 쓰기 비용 증가를 감당할 수 있는가 |
| 수행 빈도·중요도 | 핵심 거래 SQL인가 |
| 저장 공간·관리 비용 | 운영 부담은 적정한가 |

### 핵심 정밀화
가장 좋은 인덱스는 특정 SQL 하나만 빠르게 하는 인덱스가 아니라, 중요한 SQL 묶음에서 총 I/O와 운영 비용을 줄이는 인덱스입니다.

---

## ✅ 이 절(3.4.3)의 핵심 통찰
1. 스캔 효율은 중요하지만 유일한 기준은 아닙니다.
2. 정렬 생략과 Covered 처리까지 고려하면 컬럼 순서가 달라질 수 있습니다.
3. 인덱스 설계는 SQL 단건 최적화가 아니라 workload 최적화입니다.

📌 한 줄 요약: 인덱스는 스캔량뿐 아니라 정렬, 테이블 방문, DML 비용까지 포함해 설계해야 합니다.
