## 3.1.1 테이블 랜덤 액세스

### 핵심 개념
2.1.1에서 본 인덱스 튜닝의 두 핵심요소 중, 3장은 **② 테이블 랜덤 액세스 최소화**(더 중요한 축)를 본격적으로 다룹니다. 책은 이 장을 다음 문장으로 엽니다.

> "SQL 튜닝은 랜덤 I/O와의 전쟁이다."

조인 메소드의 발전을 비롯한 수많은 튜닝 기법이 모두 랜덤 I/O 최소화에 맞춰져 있을 만큼, 랜덤 액세스 감소가 인덱스 튜닝의 핵심입니다.

```text
인덱스에서 조건에 맞는 엔트리 탐색
   ↓
ROWID 획득
   ↓
ROWID가 가리키는 테이블 블록 방문   ← 랜덤 액세스 (Single Block I/O)
   ↓
필요 컬럼 확인 또는 필터링
```

인덱스는 정렬된 자료구조라 원하는 범위를 빠르게 찾을 수 있지만, 실제 행 데이터가 모두 인덱스에 있는 것은 아닙니다. 따라서 인덱스 엔트리마다 테이블 블록을 다시 방문해야 할 수 있습니다. 이 과정이 테이블 랜덤 액세스이며, ROWID는 물리적 주소에 가깝지만 비용은 여전히 블록 단위 Single Block I/O로 발생합니다.

| 구분 | 설명 | 튜닝 관점 |
|---|---|---|
| 인덱스 스캔 | 인덱스 리프 블록을 순서대로 읽음 | 범위 축소가 중요 |
| 테이블 액세스 | ROWID로 테이블 블록 방문 | 횟수 축소가 중요 |
| 랜덤 액세스 | 행마다 다른 블록을 방문할 수 있음 | 대량 처리에서 치명적 |

### CS 대응
B*Tree 인덱스는 `Map<Key, RowPointer>`에 가깝습니다. 문제는 `RowPointer`를 따라가면 실제 객체가 메모리에 연속 배치되어 있지 않을 수 있다는 점입니다. 포인터 추적이 캐시 미스처럼 비싸지는 구조입니다.

### 실행계획
```sql
SELECT *
FROM emp
WHERE deptno = 10;
```

```text
TABLE ACCESS BY INDEX ROWID EMP
  INDEX RANGE SCAN EMP_DEPTNO_IDX
```

## ✅ 이 절(3.1.1)의 핵심 통찰
1. "SQL 튜닝은 랜덤 I/O와의 전쟁" — 3장은 두 핵심요소 중 더 중요한 랜덤 액세스 최소화를 다룹니다.
2. 인덱스를 사용해도 테이블을 많이 방문하면 느릴 수 있습니다.
3. ROWID 액세스는 빠른 길찾기이지만, 방문 횟수가 많으면 Single Block I/O가 누적됩니다.
4. 인덱스 튜닝은 인덱스 스캔량을 줄이되, 궁극적으로 테이블 랜덤 액세스를 줄이는 작업입니다.

## 🔗 흐름 속 위치
2장이 인덱스 탐색 구조를 설명했다면, 3장은 그 탐색 이후 발생하는 테이블 액세스 비용을 본격적으로 다룹니다.

📌 한 줄 요약: 인덱스가 빠른 이유는 찾는 범위를 줄이기 때문이지, 테이블 방문 비용을 없애기 때문은 아닙니다.
