## 3장 인덱스 튜닝 전체 흐름

### 전체 축
3장은 2.1.1에서 제시한 인덱스 튜닝의 두 핵심요소 중 **② 테이블 랜덤 액세스 최소화**(더 중요한 축)를 본격적으로 다룹니다. 책의 선언처럼 **"SQL 튜닝은 랜덤 I/O와의 전쟁"**이며, 핵심은 인덱스를 사용하느냐가 아니라 인덱스를 얼마나 효율적으로 사용해 테이블 방문을 줄이느냐입니다.

```text
2장: 인덱스는 정렬된 B*Tree다
   ↓
3장: 그래서 어떤 비용이 생기는가?
   ↓
인덱스 스캔량 + 테이블 랜덤 액세스
   ↓
부분범위 처리로 초기 응답시간 개선
   ↓
액세스 조건/필터 조건 분리
   ↓
결합 인덱스 컬럼 순서 설계
   ↓
중복 제거와 설계도 작성
```

## 3.1 테이블 액세스 최소화

### 핵심 개념
인덱스를 타도 테이블 랜덤 액세스가 많으면 느립니다.

| 절 | 핵심 |
|---|---|
| 3.1.1 | ROWID 기반 테이블 랜덤 액세스 비용 |
| 3.1.2 | 클러스터링 팩터는 DB 블록 locality |
| 3.1.3 | 인덱스와 Full Scan의 손익분기점 |
| 3.1.4 | 인덱스 컬럼 추가로 테이블 방문 전 후보 축소 |
| 3.1.5 | 인덱스만 읽고 처리해 테이블 방문 제거 |
| 3.1.6 | IOT로 테이블 자체를 인덱스 구조화 |
| 3.1.7 | 클러스터 테이블로 관련 행을 물리적으로 모음 |

```text
인덱스 엔트리 탐색
   ↓
ROWID 획득
   ↓
테이블 블록 방문
   ↓
랜덤 액세스 비용 발생
```

## 3.2 부분범위 처리 활용

### 핵심 개념
부분범위 처리는 전체 결과를 다 만들지 않고, 사용자가 필요한 앞부분부터 반환하는 전략입니다.

| 절 | 핵심 |
|---|---|
| 3.2.1 | 전체범위 처리와 부분범위 처리 구분 |
| 3.2.2 | DB, fetch, 화면이 함께 맞아야 구현 |
| 3.2.3 | OLTP 첫 화면 응답속도 개선 원리 |

```text
ORDER BY를 만족하는 인덱스
   ↓
정렬 없이 앞부분부터 읽음
   ↓
첫 화면 빠른 응답
```

## 3.3 인덱스 스캔 효율화

### 핵심 개념
Index Range Scan이라는 이름보다 중요한 것은 **얼마나 좁게 스캔하는가**입니다.

| 절 | 핵심 |
|---|---|
| 3.3.1 | 수직 탐색보다 수평 스캔량이 중요 |
| 3.3.2 | 읽은 양 대비 결과 비율이 스캔 효율 |
| 3.3.3 | 액세스 조건은 읽는 양을 줄이고 필터 조건은 읽은 뒤 버림 |
| 3.3.4 | 등치 조건 선두 배치가 군집성 유지 |
| 3.3.5 | 선행 컬럼이 범위면 후행 컬럼 효율 저하 |
| 3.3.6 | BETWEEN을 IN-List로 바꿔 좁은 탐색 반복 |
| 3.3.7 | Skip Scan은 선두 컬럼 NDV가 작을 때 유효 |
| 3.3.8 | IN은 논리상 등치 묶음이나 실행 방식 확인 필요 |
| 3.3.9 | LIKE는 왼쪽 prefix가 고정될 때 Range Scan 가능 |
| 3.3.10 | 범위검색 남용은 넓은 스캔을 유발 |
| 3.3.11 | 옵션 조건은 동적 SQL/분기로 최적 경로를 분리 |
| 3.3.12 | 함수 호출은 후보 축소 또는 함수 기반 인덱스로 줄임 |

```text
조건절 존재
   ↓
액세스 조건인가, 필터 조건인가
   ↓
선두 컬럼이 등치인가, 범위인가
   ↓
인덱스 스캔량 결정
```

## 3.4 인덱스 설계

### 핵심 개념
인덱스 설계는 공식 암기가 아니라 workload 전체의 I/O 비용을 줄이는 의사결정입니다.

| 절 | 핵심 |
|---|---|
| 3.4.1 | 조회 성능과 DML 비용의 충돌 |
| 3.4.2 | 가장 중요한 기준은 필수 조건 여부와 등치 여부 |
| 3.4.3 | 스캔 효율 외에 정렬, 테이블 액세스, DML 고려 |
| 3.4.4 | 공식보다 업무 중요도와 호출 빈도 |
| 3.4.5 | 정렬 생략을 위한 컬럼 추가 |
| 3.4.6 | 선택도는 중요하지만 유일한 기준이 아님 |
| 3.4.7 | 중복 인덱스 제거로 DML 비용 감소 |
| 3.4.8 | 인덱스 설계도로 전체 SQL 패턴 관리 |

## ✅ 3장의 핵심 통찰
1. 인덱스를 사용한다는 사실보다 중요한 것은 인덱스 스캔량과 테이블 랜덤 액세스량입니다.
2. 클러스터링 팩터는 DB 블록 레벨의 locality이며, 인덱스 손익분기점에 큰 영향을 줍니다.
3. 부분범위 처리는 OLTP 화면 응답성을 높이는 핵심 전략입니다.
4. 액세스 조건과 필터 조건을 구분해야 인덱스 비효율을 정확히 진단할 수 있습니다.
5. 결합 인덱스 컬럼 순서는 선택도보다 필수 여부와 연산자 형태가 우선입니다.
6. 옵션 조건, 함수 조건, 범위 조건은 편의적으로 작성하면 인덱스 효율을 망가뜨릴 수 있습니다.
7. 인덱스 설계는 개별 SQL 최적화가 아니라 전체 workload의 `reduce I/O` 문제입니다.

## 🔗 흐름 속 위치
1장은 SQL 성능의 본질을 I/O로 설명했고, 2장은 인덱스가 B*Tree로 범위를 줄이는 방식을 설명했습니다. 3장은 그 인덱스를 실제 업무 SQL에서 어떻게 설계하고 조정해야 하는지를 다룹니다. 이후 4장은 조인으로 넘어가, 여러 테이블 사이에서 이 I/O 절감 원리가 어떻게 확장되는지 설명합니다.

📌 한 줄 요약: 3장은 인덱스를 “타게 만드는 장”이 아니라 “적게 읽고, 적게 방문하고, 필요한 순서로 읽게 만드는 장”입니다.
