## 2.2.5 ORDER BY 절에서 컬럼 가공

### 핵심 개념

"인덱스 컬럼을 가공하면 인덱스를 정상적으로 사용할 수 없다"에서 말하는 '인덱스 컬럼'은 대개 조건절(WHERE) 컬럼이지만, **조건절이 아닌 `ORDER BY`나 `SELECT-LIST`에서 컬럼을 가공해도** 인덱스를 정상적으로 쓰지 못하는(소트가 생기는) 경우가 있습니다.

```text
ORDER BY col            → 인덱스 정렬 순서 사용 → SORT 생략 ✅
ORDER BY function(col)  → 가공값 기준 정렬     → SORT 발생 ⚠️
```

상태변경이력 PK `(장비번호 + 변경일자 + 변경순번)`에서:
```sql
-- ✅ 소트 생략: order by 컬럼이 인덱스 정렬 순서와 일치
where 장비번호 = 'C' order by 변경일자, 변경순번

-- ⚠️ 소트 발생: order by 컬럼을 가공
where 장비번호 = 'C' order by 변경일자 || 변경순번
```

---

### ⚠️ 재미있는 함정 — ORDER BY가 가공된 alias를 가리킬 때

`주문_PK = (주문일자 + 주문번호)` 인덱스가 있고, 선두 컬럼 `주문일자`가 `=` 조건이므로 데이터는 주문번호 순으로 출력됩니다. `ORDER BY`가 있어도 정렬을 생략할 수 있을 것 같은데, **실행계획에 SORT ORDER BY가 나타납니다.** 왜일까요?

```sql
select *
from (
  select TO_CHAR(A.주문번호, 'FM000000') AS 주문번호, A.업체번호, A.주문금액
  from   주문 A
  where  A.주문일자 = :dt
  and    A.주문번호 > NVL(:next_ord_no, 0)
  order by 주문번호                      -- ⚠️ 이 '주문번호'가 함정
)
where rownum <= 30;
```
```text
Execution Plan
0  SELECT STATEMENT
1   COUNT STOPKEY
2    VIEW
3     SORT (ORDER BY STOPKEY)            ← 소트가 발생!
4      TABLE ACCESS BY INDEX ROWID  주문
5       INDEX RANGE SCAN            주문_PK
```

**원인**: `ORDER BY`에 기술한 `주문번호`는 순수 컬럼이 아니라 **SELECT-LIST의 `TO_CHAR(A.주문번호,'FM000000')` 가공 결과의 alias**를 가리킵니다. `TO_CHAR(주문번호,'FM000000')`은 숫자 1234를 문자 `'001234'`로 변환하므로(여섯 자리 리딩 제로), **가공된 값 기준으로 정렬** → 소트 발생.

**해결**: `ORDER BY`에 **테이블 별칭(A.)을 붙여 원본 컬럼을 지정**합니다.

```sql
  ...
  order by A.주문번호                     -- ✅ 원본 컬럼 → 인덱스 정렬 사용
```
```text
Execution Plan
0  SELECT STATEMENT
1   COUNT STOPKEY
2    VIEW
4     TABLE ACCESS BY INDEX ROWID  주문    ← SORT ORDER BY 사라짐!
5      INDEX RANGE SCAN            주문_PK
```

| `ORDER BY` 표기 | 가리키는 대상 | 결과 |
|------|------|------|
| `주문번호` | SELECT-LIST의 첫 컬럼(= 가공된 `TO_CHAR(...)`) | 가공값 정렬 → **SORT 발생** |
| `A.주문번호` | 테이블 `주문 A`의 **원본 컬럼** | 인덱스 정렬 사용 → **SORT 생략** |

> 즉 `ORDER BY`에 컬럼명만 쓰면 SELECT-LIST의 alias를 참조할 수 있으므로, **소트를 생략하려면 테이블 별칭을 붙여 원본 컬럼을 명시**해야 합니다.

---

### 튜닝 방향

```sql
-- ✅ 원본 컬럼 그대로
order by 주문일시
order by A.주문번호

-- ⚠️ 가공 / 가공된 alias 참조
order by to_char(주문일시, 'yyyymmdd')
order by 주문번호    -- (SELECT-LIST에서 가공된 경우)
```

---

## ✅ 이 절(2.2.5)의 핵심 통찰

1. 컬럼 가공은 WHERE뿐 아니라 **ORDER BY·SELECT-LIST에서도** 인덱스 정렬성을 깨 소트를 유발한다.
2. 인덱스는 **원본 컬럼 값 순서**로 정렬돼 있으므로, 소트 생략을 원하면 정렬 컬럼을 **원형 그대로** 써야 한다.
3. **함정**: `ORDER BY 주문번호`가 SELECT-LIST의 **가공된 alias**(`TO_CHAR(...)`)를 가리키면 소트가 발생한다.
4. **해결**: `ORDER BY A.주문번호`처럼 **테이블 별칭을 붙여 원본 컬럼을 명시**하면 소트가 생략된다.
5. 실행계획의 `SORT (ORDER BY)` 유무로 가공 여부를 확인할 수 있다.
