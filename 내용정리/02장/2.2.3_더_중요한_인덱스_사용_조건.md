## 2.2.3 더 중요한 인덱스 사용 조건

### 핵심 개념

2.2.2에서 "컬럼을 가공하면 Range Scan을 못 한다"고 했지만, 그보다 **더 중요한 선행 조건**이 있습니다 (의외로 모르는 분이 많습니다):

> **"인덱스를 Range Scan 하기 위한 가장 첫 번째 조건은, 인덱스 선두 컬럼이 (가공하지 않은 상태로) 조건절에 있어야 한다는 것이다."**

---

### 선두 컬럼이 빠지면 — 흩어져서 못 찾는다

인덱스를 `(소속팀 + 사원명 + 연령)` 순으로 구성했다고 합시다(그림 2-12). 이는 "데이터를 **소속팀 순으로 정렬**, 같으면 사원명 순, 같으면 연령 순"이라는 뜻입니다.

```sql
select 사원번호, 소속팀, 연령, 입사일자, 전화번호
from   사원
where  사원명 = '홍길동';     -- ⚠️ 선두 컬럼(소속팀)이 조건에 없음
```

```text
인덱스 정렬: 소속팀 → 사원명 → 연령
   경영지원/홍길동   마케팅/홍길동   생산/홍길동   영업/홍길동   인사/홍길동 ...
   '홍길동'이 소속팀마다 흩어져 리프 전 구간에 분포 (그림 2-13)
   ↓
시작점 못 찾음 + 어디서 멈출지 모름 → 리프 블록 전체 스캔 ⚠️
```

사원명이 같아도 소속팀이 다르면 서로 멀리 떨어지므로, 선두 컬럼(소속팀) 없이 `사원명='홍길동'`만으로는 Range Scan을 할 수 없습니다.

---

### ⚠️ 반전 — 선두 컬럼만 살아있으면 후행은 가공해도 된다

여기서 자주 받는 질문입니다.

> "아래 SQL은 인덱스 컬럼(과세구분코드)을 **가공**했는데, 어떻게 Index Range Scan이 되나요?"

`TXA1234_IX02` 인덱스 = `기준연도 + 과세구분코드 + 보고회차 + 실명확인번호`

```sql
select * from TXA1234
where 기준연도 = :stdr_year                    -- ① 선두 컬럼, 미가공 ✅
and   substr(과세구분코드,1,4) = :txtn_dcd      -- ② 후행 컬럼, 가공됨
and   보고회차 = :rpt_tmrd
and   실명확인번호 = :rnm_cnfm_no;
```
```text
Execution Plan
0  SELECT STATEMENT
1   TABLE ACCESS (BY INDEX ROWID) OF 'TXA1234'
2    INDEX (RANGE SCAN) OF 'TXA1234_IX02'      ← Range Scan 작동!
```

**답: 선두 컬럼인 `기준연도`가 가공되지 않았기 때문**입니다. 선두 컬럼으로 시작점·종료점(`기준연도 = :stdr_year` 구간)을 잡을 수 있으므로 Range Scan이 됩니다. 가공된 `과세구분코드`는 그 범위 안에서 **필터 조건**으로 처리될 뿐입니다(3.3.3 액세스/필터).

핵심 정리:

| 상황 | Range Scan |
|------|:---:|
| 선두 컬럼 미가공 + 조건절에 존재 | **가능** (후행 컬럼이 가공돼도 OK) |
| 선두 컬럼 가공됨 | 불가 (2.2.2) |
| 선두 컬럼이 조건절에서 누락 | 불가 (위 홍길동 사례) |

→ "컬럼 가공이 Range Scan을 막는다"의 **정확한 의미는 '선두 컬럼 가공'** 입니다. 후행 컬럼 가공은 Range Scan을 막지 않고 스캔 효율(필터로 전락)만 떨어뜨립니다.

---

### 컬럼 순서 판단 기준 (예고: 3.4)

선택도가 아니라 다음 순으로 판단합니다.

```text
① 항상 조건절에 쓰이는(필수) 컬럼인가?   → 선두에
② 등치(=) 조건인가?                      → 앞쪽에
③ 범위 조건인가?                         → 맨 뒤에
④ 정렬/그룹핑 생략이 필요한가?            → 해당 순서 반영
```

---

## ✅ 이 절(2.2.3)의 핵심 통찰

1. Range Scan의 **가장 중요한 조건**은 컬럼 가공 여부보다 **"선두 컬럼이 미가공 상태로 조건절에 있어야 한다"** 이다.
2. 선두 컬럼이 빠지면 값이 리프 전 구간에 흩어져 시작점을 못 잡는다(홍길동 사례).
3. **반대로 선두 컬럼만 살아있으면 후행 컬럼을 가공해도 Range Scan은 작동**한다(TXA1234 사례) — 가공된 후행 컬럼은 필터로 처리된다.
4. 따라서 "컬럼 가공이 Range Scan을 막는다"의 정확한 의미는 **'선두 컬럼 가공'** 이다.
5. 컬럼 순서는 선택도가 아니라 **필수 여부·연산자 형태(= 앞, 범위 뒤)** 로 정한다.

## 🔗 흐름 속 위치

2.2.2가 "컬럼 가공이 Range Scan을 막는다"였다면, 2.2.3은 그것을 정밀화합니다 — **막는 것은 '선두 컬럼' 가공/누락**이고, 후행 컬럼 가공은 Range Scan을 막지 않습니다. 이 "선두는 액세스 조건, 후행은 필터"라는 구분은 3.3.3에서 본격화됩니다.

📌 한 줄 요약: 인덱스 Range Scan의 제1 조건은 **선두 컬럼이 미가공 상태로 조건절에 존재**하는 것이며, 이것만 지켜지면 후행 컬럼은 가공돼도 Range Scan이 작동합니다.
