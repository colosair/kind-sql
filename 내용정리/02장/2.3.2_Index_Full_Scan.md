## 2.3.2 Index Full Scan

### 핵심 개념

`Index Full Scan`은 인덱스 리프 블록 전체를 키 순서대로 읽는 방식입니다. 테이블 전체가 아니라 인덱스 전체를 읽습니다.

```text
인덱스 왼쪽 끝
   ↓
리프 블록 전체 순차 스캔
   ↓
키 순서 보장
```

Index Full Scan은 **수직적 탐색 없이** 인덱스 리프 블록을 처음부터 끝까지 수평으로 스캔합니다(그림 2-16).

### 언제 사용되는가 — Range Scan의 '차선책'

```sql
SQL> create index emp_ename_sal_idx on emp(ename, sal);
SQL> select * from emp where sal > 2000 order by ename;
```
```text
Execution Plan
1   TABLE ACCESS (BY INDEX ROWID) OF 'EMP'
2    INDEX (FULL SCAN) OF 'EMP_ENAME_SAL_IDX'   ← Full Scan
```
선두 컬럼 `ENAME`이 조건절에 없으므로 **Index Range Scan은 불가능**합니다. 하지만 뒤쪽 `SAL`이 인덱스에 있으므로, 인덱스 전체를 훑으며(`sal > 2000`을 필터링하며) 레코드를 찾을 수 있습니다.

### Index Full Scan의 효용성

선두 컬럼이 조건에 없으면 먼저 **Table Full Scan**을 고려하는데, 대용량 테이블이면 부담이 큽니다. 이때 인덱스를 다시 봅니다.

```text
저장공간 = 컬럼 길이 × 레코드 수
   → 인덱스는 일부 컬럼만 담으므로 테이블보다 면적이 훨씬 작음
   ↓
인덱스 스캔 단계에서 대부분 필터링하고 일부만 테이블 액세스한다면
   → 테이블 전체보다 인덱스 전체를 스캔하는 쪽이 유리 (그림 2-17)
```
예: `sal > 9000`인 사원이 극히 일부라면, Table Full Scan보다 Index Full Scan으로 필터링하는 편이 효과적입니다.

> ⚠️ 단, 이는 **적절한 인덱스가 없어 Range Scan의 차선책**으로 선택된 것입니다. 수행빈도가 낮으면 무방하나, 잦으면 **SAL 컬럼이 선두인 인덱스를 생성**해 Range Scan으로 바꾸는 게 좋습니다.

| 상황 | 설명 |
|---|---|
| 선두 컬럼 조건이 없어 Range Scan 불가 | 테이블보다 작은 인덱스를 전체 스캔(차선책) |
| ORDER BY 생략 | 인덱스가 키 순서를 보장 |
| 인덱스만으로 처리 가능 | 테이블 액세스 없이 결과 생성(Covering, 3.1.5) |

### Range Scan과 차이

| 구분 | Index Range Scan | Index Full Scan |
|---|---|---|
| 시작점 | 조건에 맞는 지점 | 인덱스 시작 또는 끝 |
| 읽는 범위 | 조건 범위 | 인덱스 전체 |
| 정렬성 | 조건 범위 내 정렬 | 전체 인덱스 키 순서 |

### 주의점

Index Full Scan은 인덱스를 전체 읽는 방식이므로 데이터량이 많으면 비용이 큽니다. 다만 테이블보다 인덱스가 훨씬 작거나 정렬 생략 효과가 크면 선택될 수 있습니다.

---

## ✅ 이 절(2.3.2)의 핵심 통찰

1. Index Full Scan은 인덱스 전체를 키 순서대로 읽습니다.
2. 테이블 Full Scan과 다르며, 인덱스 정렬 순서를 보장합니다.
3. ORDER BY 생략이나 인덱스만 읽고 처리할 때 유용할 수 있습니다.
4. 전체 인덱스를 읽으므로 무조건 빠른 것은 아닙니다.

## 🔗 흐름 속 위치

Range Scan이 특정 범위를 읽는 방식이라면, Full Scan은 인덱스 전체를 정렬 순서대로 읽는 방식입니다.

📌 한 줄 요약: Index Full Scan은 테이블이 아니라 인덱스 전체를 키 순서대로 읽어 정렬성과 작은 크기를 활용하는 방식입니다.
