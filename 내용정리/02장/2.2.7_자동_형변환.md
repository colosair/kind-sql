## 2.2.7 자동 형변환

### 핵심 개념

자동 형변환은 SQL 작성자가 명시하지 않아도 DBMS가 데이터 타입을 맞추기 위해 내부적으로 변환 함수를 적용하는 현상입니다. 문제는 변환이 컬럼 쪽에 적용되면 인덱스 사용을 방해할 수 있다는 점입니다.

```text
컬럼 타입 ≠ 비교 값 타입
   ↓
DBMS가 자동 형변환
   ↓
컬럼 쪽 변환 발생 가능
   ↓
인덱스 Range Scan 방해
```

### 위험한 패턴

```sql
where 문자컬럼 = 123
```

문자 컬럼과 숫자 상수를 비교하면 DBMS가 타입을 맞춥니다. 이때 내부적으로 다음처럼 컬럼을 변환하면 인덱스 정렬성을 사용할 수 없습니다.

```sql
where to_number(문자컬럼) = 123
```

### 변환 우선순위 규칙 — 누가 변환되는가

어느 쪽이 변환되는지는 타입 우선순위가 결정합니다. 컬럼이 변환되면 위험하고, 상수가 변환되면 안전합니다.

| 비교 | 우선순위(이기는 쪽) | 변환되는 쪽 | 인덱스 영향 |
|---|---|---|---|
| 문자컬럼 = 숫자상수 | **숫자형이 이긴다** | 문자 **컬럼** → `to_number(컬럼)` | ✗ 깨짐 |
| 날짜컬럼 = 문자상수 | **날짜형이 이긴다** | 문자 **상수** → `to_date(상수)` | ✅ 안전 |
| 숫자컬럼 = 문자상수 | 숫자형이 이긴다 | 문자 **상수** → `to_number(상수)` | ✅ 안전 |

즉 "숫자형 > 문자형, 날짜형 > 문자형" 우선순위 덕분에 **숫자/날짜 컬럼**은 문자 상수와 비교해도 대개 안전하지만, **문자 컬럼**을 숫자 상수와 비교하면 컬럼이 변환되어 인덱스가 깨집니다.

### LIKE는 예외 — 숫자 컬럼도 위험

`LIKE`는 문자열 비교 연산자이므로 위 우선순위가 뒤집힙니다. 숫자 컬럼을 `LIKE`로 비교하면 **숫자 컬럼이 문자형으로 변환**됩니다.

```sql
where 고객번호 like '9410%'
   ↓  (Predicate Information)
filter(TO_CHAR(고객번호) LIKE '9410%')   -- 컬럼 가공 → 인덱스 Range Scan 불가
```

평소 "숫자형이 이긴다"고 외워 두면 `LIKE`에서 방심하기 쉽습니다. LIKE를 옵션 조건 처리에 쓰는 경우가 많으므로 특히 주의해야 합니다.

### 날짜형도 NLS 의존성 주의

`날짜컬럼 = 문자상수`는 날짜형이 이겨 상수가 변환되므로 인덱스는 안전하지만, **NLS_DATE_FORMAT 파라미터에 의존**해 환경이 다르면 컴파일 오류·결과 오류가 날 수 있습니다.

```sql
-- ⚠️ NLS_DATE_FORMAT 의존
where 가입일자 = '01-JAN-2018'
-- ✅ 포맷 명시
where 가입일자 = TO_DATE('01-JAN-2018', 'DD-MON-YYYY')
```

### 데이터 품질 측면 — 에러와 잘못된 결과

자동 형변환은 성능뿐 아니라 **애플리케이션 품질**도 해칩니다.

**① 실행 에러** — `숫자컬럼 = 문자컬럼`에서 문자열이 숫자로 변환되는데, 변환 불가한 문자열이 입력되면:
```sql
where n_col = v_col   -- → ORA-01722: 수치가 부적합합니다
```

**② 잘못된 결과** — EMP에서 PRESIDENT를 제외한 최고 급여를 구하려는데 950이 나옵니다(실제 최고는 ANALYST 3000).
```sql
-- ⚠️ 결과 오류
select max(decode(job, 'PRESIDENT', NULL, sal)) max_sal2 from emp;   -- → 950 (틀림)
-- ✅ 타입 명시
select max(decode(job, 'PRESIDENT', to_number(NULL), sal)) max_sal2 from emp;  -- → 3000
```
원인: `decode(a,b,c,d)`의 반환 타입은 **세 번째 인자 c로 결정**되고, `c`가 문자면 `d`(sal)도 문자로 변환됩니다. 게다가 **세 번째 인자 NULL은 varchar2로 취급**되므로 sal이 문자열로 변환 → 문자열 기준 max → '950'. `to_number(NULL)`로 타입을 맞추면 해결됩니다.

### 안전한 방향

상수 또는 바인드 변수 타입을 컬럼 타입에 맞추고, **형변환 함수를 의도적으로 생략하지 않습니다.**

```sql
where 문자컬럼 = '123'                       -- 문자 컬럼이면 문자로
where 주문일시 >= date '2024-06-01'          -- 날짜 컬럼이면 날짜로
```

> SQL 성능을 모르는 개발자는 "TO_CHAR/TO_DATE/TO_NUMBER를 생략하면 연산 횟수가 줄어 빠를 것"이라 오해해 생략하곤 합니다. 하지만 **SQL 성능은 블록 I/O로 결정**되며, 형변환 함수를 생략해도 **옵티마이저가 자동으로 생성**하므로 연산 횟수는 줄지 않습니다 — 오히려 컬럼 쪽에 몰래 씌워져 인덱스를 깰 뿐입니다.

### 튜닝 포인트

자동 형변환은 실행계획만 봐서는 놓치기 쉽습니다. Predicate Information에서 컬럼 쪽에 함수가 씌워졌는지 확인해야 합니다.

```text
Predicate Information
   ↓
TO_NUMBER(컬럼)
TO_CHAR(컬럼)
INTERNAL_FUNCTION(컬럼)
   ↓
인덱스 비효율 의심
```

## ✅ 이 절(2.2.7)의 핵심 통찰

1. 자동 형변환은 컬럼 가공을 몰래 만들 수 있습니다.
2. 우선순위는 "숫자형 > 문자형, 날짜형 > 문자형" — 이기는 타입의 반대쪽이 변환됩니다.
3. 문자 컬럼 = 숫자 상수는 컬럼이 변환되어 위험하고, 숫자/날짜 컬럼 = 문자 상수는 상수가 변환되어 안전합니다.
4. `LIKE`는 예외 — 숫자 컬럼도 `TO_CHAR(컬럼)`으로 변환되어 인덱스가 깨집니다.
5. Predicate Information에서 컬럼 쪽 내부 변환을 확인해야 합니다.
