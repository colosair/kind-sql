## 2.2.6 SELECT-LIST에서 컬럼 가공

### 핵심 개념 — MIN/MAX는 인덱스로 한 건만 읽는다

인덱스 `(장비번호 + 변경일자 + 변경순번)`에서 **MIN/MAX 값을 구할 때 옵티마이저는 정렬 연산 없이 인덱스 리프의 끝에서 한 건만 읽고 멈춥니다.**

```sql
select MIN(변경순번) from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316';   -- 가장 왼쪽 첫 레코드 = 최소값
select MAX(변경순번) from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316';   -- 가장 오른쪽 첫 레코드 = 최대값
```
```text
Row Source Operation
  SORT AGGREGATE
   FIRST ROW                                  ← 한 건만!
    INDEX RANGE SCAN (MIN/MAX) 상태변경이력_PK   (cr=6 pr=0)
```
- MIN: 수직 탐색으로 조건을 만족하는 **가장 왼쪽** 지점 → 첫 레코드가 최소값
- MAX: 수직 탐색으로 **가장 오른쪽** 지점 → 첫 레코드가 최대값
- 리프 끝에서 **FIRST ROW(레코드 하나)** 만 읽고 멈추므로 `cr`(논리 I/O)이 극히 작습니다.

---

### ⚠️ MIN/MAX 안에서 컬럼을 가공하면 최적화가 깨진다

```sql
-- ⚠️ MAX 안에서 TO_NUMBER로 가공
select NVL(MAX(TO_NUMBER(변경순번)), 0) from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316';
```
```text
Row Source Operation
  SORT AGGREGATE
   INDEX RANGE SCAN 상태변경이력_PK   (cr=1670, 131,577 rows)   ← 전체 스캔!
```
인덱스는 **문자열 기준**으로 정렬돼 있는데, `TO_NUMBER(변경순번)`은 **숫자값 기준** 최대값을 요구합니다. 정렬 기준이 달라 FIRST ROW 최적화가 깨지고 **조건 범위 전체(131,577건)를 스캔**합니다.

**해결 — 가공을 MAX 바깥으로 빼서 결과 1건에만 적용**
```sql
-- ✅ MAX를 먼저, TO_NUMBER는 그 결과 1건에만
select NVL(TO_NUMBER(MAX(변경순번)), 0) from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316';
```
```text
Row Source Operation
  SORT AGGREGATE
   FIRST ROW
    INDEX RANGE SCAN (MIN/MAX) 상태변경이력_PK   (cr=4)   ← 다시 한 건만!
```
> 애초에 변경순번을 숫자형으로 설계했다면 이런 튜닝이 필요 없습니다. **`MAX(가공(컬럼))`이 아니라 `가공(MAX(컬럼))`** 으로, 가공을 집계함수 바깥에 두는 것이 핵심입니다.

---

### 응용 — 스칼라 서브쿼리의 MIN/MAX

장비구분코드='A001' 장비들의 최종 변경일자를 스칼라 서브쿼리로 구할 때도 MIN/MAX·FIRST ROW로 최적화됩니다.
```sql
select 장비번호, 장비명, 상태코드
     , (select MAX(변경일자) from 상태변경이력 where 장비번호 = P.장비번호) 최종변경일자
from   장비 P
where  장비구분코드 = 'A001';        -- FIRST ROW (MIN/MAX)로 동작
```

**최종 변경일자 + 최종 변경순번을 둘 다** 출력하려면? 스칼라 서브쿼리를 2개 쓰면 **상태변경이력을 두 번** 읽어 비효율적입니다(PK 컬럼이 많아질수록 더 복잡·느림).

```sql
-- ⚠️ 하나로 묶으려 가공(|| 결합 + SUBSTR)하면 — 이력이 많을 때 성능 문제
select 장비번호, 장비명, 상태코드
     , SUBSTR(최종이력, 1, 8) 최종변경일자
     , SUBSTR(최종이력, 9)    최종변경순번
from ( select 장비번호, 장비명, 상태코드
            , (select MAX(변경일자 || 변경순번) from 상태변경이력
               where 장비번호 = P.장비번호) 최종이력
       from 장비 P where 장비구분코드 = 'A001' );
```
`MAX(변경일자 || 변경순번)`도 결국 인덱스 컬럼을 가공한 것이라, 이력이 많으면 각 장비의 **과거 이력을 모두 읽어** 비효율입니다(트레이스에서 상태변경이력 1,825,000 레코드·6,380 블록을 읽음 — 장비 테이블은 10건밖에 안 되는데도).

> 이런 상황의 효과적 해법은 **Top N 알고리즘**이 필요하므로 5.3.4 '이력 조회'에서 소개합니다.

---

## ✅ 이 절(2.2.6)의 핵심 통찰

1. **MIN/MAX는 인덱스 리프 끝에서 FIRST ROW 한 건만 읽고 멈춘다**(SORT AGGREGATE → FIRST ROW → INDEX RANGE SCAN MIN/MAX) — 정렬 연산 없이.
2. **MIN/MAX 안에서 컬럼을 가공(`MAX(TO_NUMBER(...))`)하면** 정렬 기준이 달라져 이 최적화가 깨지고 범위 전체를 스캔한다.
3. 해결은 **가공을 집계함수 바깥으로**(`TO_NUMBER(MAX(...))`) 빼서 결과 1건에만 적용하는 것.
4. 두 값을 한 번에 구하려 `MAX(A||B)`로 묶는 것도 가공이라, 이력이 많으면 과거 전체를 읽어 느리다.
5. 근본 해법(이력 조회)은 Top N 알고리즘(5.3.4)으로 이어진다.

## 🔗 흐름 속 위치

2.2.5(ORDER BY 가공)에 이어 **SELECT-LIST/집계함수 안의 가공**이 인덱스 최적화(MIN/MAX FIRST ROW)를 깨는 사례를 다룹니다. 최소/최대값 구하기는 5.3.3, 이력 조회는 5.3.4로 확장됩니다.

📌 한 줄 요약: MIN/MAX는 인덱스로 한 건만 읽는 강력한 최적화가 되지만, **집계함수 안에서 컬럼을 가공하면 깨지므로** 가공은 `가공(MAX(컬럼))` 형태로 바깥에 둬야 합니다.
