## 2.3.5 Index Fast Full Scan

### 핵심 개념

말 그대로 `Index Fast Full Scan`은 `Index Full Scan`보다 **빠릅니다.** 빠른 이유는 **논리적인 인덱스 트리 구조(리프 연결 순서)를 무시하고, 인덱스 세그먼트 전체를 Multiblock I/O 방식으로 스캔**하기 때문입니다. 힌트는 **`index_ffs`**(유도) / **`no_index_ffs`**(방지).

```text
Index Full Scan      : 논리적 순서(리프 연결)대로  1→2→3→…→10
Index Fast Full Scan : 물리적 순서(익스텐트 맵)대로 1→2→10→3→9 …  (Multiblock)
```

- 리프 블록을 **디스크에 저장된 물리적 순서**대로 읽습니다(그림 2-22 논리 순서 vs 그림 2-23 물리 순서). 그래서 **결과집합이 인덱스 키 순서로 정렬되지 않습니다.**
- 루트·브랜치 블록도 (세그먼트 전체를 읽으므로) 함께 읽지만 **필요 없는 블록이라 버립니다.**
- 인덱스를 Table Full Scan처럼 읽으며, 읽을 익스텐트 목록은 익스텐트 맵에서 얻습니다(1.3.4).

### Index Full Scan과 차이

| 구분 | Index Full Scan | Index Fast Full Scan |
|---|---|---|
| 읽는 방식 | 키 순서대로 리프 스캔 | 인덱스 세그먼트 전체 스캔 |
| 정렬 순서 | 보장 | 보장 안 함 |
| I/O | 주로 Single Block | Multiblock 가능 |
| 병렬 처리 | 제한적 | 가능 |
| ORDER BY 생략 | 가능 | 불가능 |

### 언제 유용한가

인덱스만 읽고 결과를 만들 수 있고, 정렬 순서가 필요 없을 때 유리합니다.

```text
SELECT 컬럼이 모두 인덱스에 포함
   ↓
테이블 액세스 불필요
   ↓
인덱스가 테이블보다 작음
   ↓
Fast Full Scan 유리
```

---

## ✅ 이 절(2.3.5)의 핵심 통찰

1. Index Fast Full Scan은 인덱스 전체를 빠르게 읽는 방식입니다.
2. 키 순서를 보장하지 않습니다.
3. Multiblock I/O와 병렬 처리가 가능합니다.
4. 인덱스만으로 처리 가능하고 정렬이 필요 없을 때 유리합니다.

## 🔗 흐름 속 위치

Index Full Scan과 혼동하기 쉬운 확장 기능입니다. 정렬성 보장 여부가 핵심 차이입니다.

📌 한 줄 요약: Index Fast Full Scan은 정렬 순서를 포기하고 인덱스 전체를 빠르게 읽는 방식입니다.
