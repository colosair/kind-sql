## 2.1.1 미리 보는 인덱스 튜닝

### 핵심 개념

인덱스 튜닝의 출발점은 **인덱스가 테이블 전체를 읽지 않고 필요한 범위로 접근하게 해주는 정렬된 탐색 구조**라는 점입니다. 하지만 인덱스가 있다고 항상 빠른 것은 아닙니다.

```text
조건절
   ↓
인덱스 탐색 가능 여부 판단
   ↓
가능하면 필요한 범위만 스캔
   ↓
ROWID로 테이블 액세스
```

1장에서 본 통일 원리와 연결하면 다음과 같습니다.

```text
좋은 인덱스 사용
   ↓
읽을 블록 범위 축소
   ↓
논리적 I/O 감소
   ↓
물리적 I/O 가능성 감소
```

### 데이터를 찾는 두 가지 방법 — 학생명부 비유

책은 인덱스를 학생명부에 비유합니다. 어떤 초등학교에서 '홍길동' 학생을 찾는 방법은 두 가지입니다.

| 방법 | DB 대응 | 언제 빠른가 |
|---|---|---|
| 1학년 1반부터 모든 교실을 도는 것 | Table Full Scan | 홍길동이 아주 많을 때 |
| 이름순 정렬된 학생명부를 보는 것 | 인덱스 | 홍길동이 적을 때 |

이름순으로 정렬해 둔 학생명부가 곧 **인덱스**이고, 명부의 `학년-반-번호` 컬럼이 곧 **ROWID**(실제 교실 위치)입니다. 수십 년간 DBMS가 발전했어도 데이터를 찾는 방법은 이 두 가지에서 벗어나지 않습니다.

### 인덱스 튜닝의 두 가지 핵심요소 (2장·3장의 뼈대)

이 절에서 가장 중요한 것은 인덱스 튜닝이 **두 축**으로 나뉜다는 사실입니다. 이 프레임이 2장과 3장의 구성을 결정합니다.

```text
인덱스 튜닝
   ├─ ① 인덱스 스캔 효율화 튜닝  → 2장 (인덱스 스캔량 자체를 줄임)
   └─ ② 랜덤 액세스 최소화 튜닝   → 3장 (테이블 액세스 횟수를 줄임) ★ 더 중요
```

**① 인덱스 스캔 효율화** — 시력이 1.0~1.5인 홍길동을 찾을 때, 학생명부가 `이름+시력`순이면 소량만 스캔하지만 `시력+이름`순이면 많은 양을 스캔해야 합니다. 같은 인덱스라도 **컬럼 순서**에 따라 스캔량이 달라집니다.

**② 랜덤 액세스 최소화** — 인덱스에서 찾은 학생을 실제로 만나려면 교실(테이블 블록)로 찾아가야 합니다. 시력 1.0~1.5 홍길동이 두 명일 때:

```text
이름순 명부  → 교실 5번 방문   (홍길동만 모여 있음)
시력순 명부  → 교실 50번 방문  (시력 1.0~1.5가 50명, 그중 홍길동 골라냄)
```

두 튜닝 모두 중요하지만, **더 중요한 것은 랜덤 액세스 최소화**입니다. 테이블 랜덤 액세스(Single Block I/O)가 성능에 미치는 영향이 더 크기 때문입니다(1.3.7 참고). 그래서 3장 전체가 이 주제에 할애됩니다.

### 튜닝 관점

인덱스 튜닝은 단순히 인덱스를 만드는 작업이 아닙니다. SQL 조건과 인덱스 컬럼 순서가 맞아야 합니다.

```text
조건절 컬럼
   +
연산자 형태
   +
인덱스 컬럼 순서
   ↓
① 인덱스 스캔 효율 + ② 테이블 액세스량 결정
```

---

## ✅ 이 절(2.1.1)의 핵심 통찰

1. 데이터를 찾는 방법은 Table Full Scan과 인덱스 두 가지뿐입니다(홍길동 학생명부 비유).
2. 인덱스 튜닝은 ① 인덱스 스캔 효율화(2장)와 ② 랜덤 액세스 최소화(3장) 두 축으로 나뉩니다.
3. 둘 중 **랜덤 액세스 최소화가 더 중요**합니다 — 테이블 랜덤 액세스 비용이 더 크기 때문입니다.
4. 인덱스 컬럼 순서가 스캔량과 테이블 액세스량을 함께 좌우합니다.

## 🔗 흐름 속 위치

1장에서 블록 I/O와 랜덤 액세스 비용을 봤습니다. 2장은 그 비용을 줄이는 핵심 도구인 인덱스를 본격적으로 다룹니다. 이 절의 두 핵심요소 프레임에서 ①은 2장 나머지 절들로, ②는 3장 전체로 이어집니다.

📌 한 줄 요약: 인덱스 튜닝은 ① 인덱스 스캔량과 ② 테이블 랜덤 액세스를 줄이는 두 축이며, 후자가 더 중요합니다.
