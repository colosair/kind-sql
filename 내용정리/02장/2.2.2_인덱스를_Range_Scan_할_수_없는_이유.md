## 2.2.2 인덱스를 Range Scan 할 수 없는 이유

### 핵심 개념

인덱스를 Range Scan 하려면 인덱스 키의 정렬 순서를 이용해 시작 지점과 끝 지점을 찾을 수 있어야 합니다. 컬럼을 가공하거나 선두 컬럼이 빠지면 정렬 순서를 사용할 수 없습니다.

```text
인덱스 Range Scan 가능
   ↓
정렬된 키 기준으로 시작/끝 지점 결정 가능

Range Scan 불가능
   ↓
정렬된 원본 키와 조건 비교 기준이 달라짐
```

### 대표 원인

| 원인 | 예 |
|---|---|
| 인덱스 컬럼 가공 | `where substr(생년월일,1,6) = '202401'` |
| 선두 컬럼 누락 | 인덱스 `(A,B)`에서 `where B = :b` |
| 부정 조건 | `where col <> :v` |
| LIKE 선두 와일드카드 | `where name like '%길동'` |
| 묵시적 형변환 | 컬럼 쪽 변환 발생 |

### 컬럼 가공

```sql
where substr(전화번호, 1, 3) = '010'
```

인덱스는 원본 `전화번호` 값 순서로 정렬되어 있습니다. 그런데 조건은 `substr(전화번호,1,3)` 결과를 기준으로 비교합니다. 원본 정렬 순서와 조건 비교 순서가 다르므로 시작/끝 범위를 찾기 어렵습니다.

```text
원본 컬럼 정렬
   ↓
가공 결과 기준 검색
   ↓
정렬 순서 불일치
   ↓
Range Scan 어려움
```

### 튜닝 방향

가능하면 컬럼을 가공하지 않고 상수 쪽을 변형해야 합니다.

```sql
-- 비효율
where to_char(주문일시, 'yyyymmdd') = '20240601'

-- 효율
where 주문일시 >= date '2024-06-01'
and 주문일시 <  date '2024-06-02'
```

---

## ✅ 이 절(2.2.2)의 핵심 통찰

1. Range Scan은 인덱스 정렬 순서로 시작/끝 지점을 찾을 수 있어야 합니다.
2. 컬럼을 가공하면 원본 인덱스 정렬성을 잃습니다.
3. 선두 컬럼이 빠지면 결합 인덱스의 정렬성을 활용하기 어렵습니다.
4. 조건을 인덱스 컬럼 원형에 맞게 작성해야 합니다.

## 🔗 흐름 속 위치

2.1의 B*Tree 정렬 구조가 실제 SQL 조건에서 깨지는 대표 사례를 설명하는 절입니다.
