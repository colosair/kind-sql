## 2.2.4 인덱스를 이용한 소트 연산 생략

### 핵심 개념

인덱스는 키 순서로 **정렬**되어 있습니다. 그래서 `ORDER BY` 컬럼과 인덱스 정렬 순서가 일치하면, **별도 소트 연산(SORT ORDER BY)을 생략**할 수 있습니다 — 인덱스를 Range Scan 하면 부수적으로 따라오는 효과입니다.

```text
인덱스 정렬 순서 = ORDER BY 순서
   ↓
인덱스 순서대로 스캔하면 이미 정렬됨
   ↓
SORT ORDER BY 생략 → 부분범위 처리 가능
```

---

### 실험 — 상태변경이력 테이블

PK를 `(장비번호 + 변경일자 + 변경순번)` 순으로 구성합니다. 장비번호·변경일자가 같은 레코드는 **변경순번 순으로 정렬**돼 있습니다(그림 2-14).

**① ORDER BY가 없어도 변경순번 순으로 출력**
```sql
select * from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316';
```
```text
Execution Plan
0  SELECT STATEMENT
1   TABLE ACCESS (BY INDEX ROWID) OF '상태변경이력'
2    INDEX (RANGE SCAN) OF '상태변경이력_PK'      ← SORT 없음
```

**② ORDER BY 변경순번을 붙여도 실행계획 동일 (SORT 생략)**
```sql
select * from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316'
order by 변경순번;
```
```text
Execution Plan
0  SELECT STATEMENT  (Cost=85)
1   TABLE ACCESS (BY INDEX ROWID) OF '상태변경이력'
2    INDEX (RANGE SCAN) OF '상태변경이력_PK'      ← SORT ORDER BY 없음! ①과 100% 동일
```
인덱스 스캔 순서가 이미 변경순번 순이므로 정렬할 필요가 없습니다.

**③ 인덱스가 없었다면 — SORT ORDER BY 단계 추가**
```text
Execution Plan
0  SELECT STATEMENT  (Cost=86)          ← Cost 85→86 증가
1   SORT (ORDER BY)                      ← 정렬 단계 추가됨
2    TABLE ACCESS (BY INDEX ROWID) OF '상태변경이력'
3     INDEX (RANGE SCAN) OF '상태변경이력_PK'
```

---

### 내림차순(Descending) 정렬도 인덱스로

인덱스 리프 블록은 **양방향 연결 리스트**이므로(2.1.4), 내림차순 정렬에도 인덱스를 활용할 수 있습니다.

| 정렬 | 수직 탐색으로 찾는 값 | 수평 탐색 방향 |
|------|------|------|
| 오름차순(Asc) | 조건을 만족하는 **가장 작은** 값 | 좌 → 우 |
| 내림차순(Desc) | 조건을 만족하는 **가장 큰** 값 | 우 → 좌 |

```sql
select * from 상태변경이력
where 장비번호 = 'C' and 변경일자 = '20180316'
order by 변경순번 DESC;
```
```text
Execution Plan
1   TABLE ACCESS (BY INDEX ROWID) OF '상태변경이력'
2    INDEX (RANGE SCAN DESCENDING) OF '상태변경이력_PK'   ← SORT 없음 + DESCENDING 표시
```
SORT ORDER BY가 없고, INDEX RANGE SCAN 단계에 **DESCENDING**이 표시됩니다(2.3.6).

---

### 부분범위 처리와의 연결

소트가 생략되면 전체 결과를 모아 정렬할 필요가 없으므로 **앞쪽 결과부터 바로 반환**할 수 있습니다(부분범위 처리, 3.2).

```text
[소트 필요]   전체 후보 수집 → 정렬 → 반환   (다 모아야 첫 행 반환)
[소트 생략]   인덱스 순서대로 읽으며 즉시 반환 (첫 행 빨리 반환)
```

---

## ✅ 이 절(2.2.4)의 핵심 통찰

1. 인덱스는 정렬된 구조라 **ORDER BY 순서와 일치하면 SORT ORDER BY를 생략**한다 — 실행계획에서 SORT 단계가 사라진다.
2. 소트 생략 여부는 **Cost에도 반영**된다(인덱스 없으면 SORT 추가 + Cost 증가).
3. 리프가 양방향 연결이라 **내림차순(DESC)도 인덱스로 처리**된다(RANGE SCAN DESCENDING).
4. 오름차순은 가장 작은 값 찾아 좌→우, 내림차순은 가장 큰 값 찾아 우→좌로 수평 탐색한다.
5. 소트 생략은 **부분범위 처리(앞쪽부터 즉시 반환)** 를 가능하게 한다.

## 🔗 흐름 속 위치

2장에서 미리 소개하는 소트 생략 원리로, 정렬된 인덱스의 또 다른 가치입니다. 내림차순 스캔은 2.3.6에서, 부분범위 처리는 3.2에서, Top N·Group By·이력 조회로의 확장은 5장에서 다룹니다.

📌 한 줄 요약: 정렬된 인덱스를 ORDER BY와 같은 순서로 읽으면 별도 소트가 필요 없고(실행계획에서 SORT 사라짐), 양방향 연결 덕에 내림차순도 인덱스로 처리됩니다.
