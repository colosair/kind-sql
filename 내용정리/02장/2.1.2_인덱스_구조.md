## 2.1.2 인덱스 구조

### 핵심 개념 — 책 뒤쪽 '찾아보기'

인덱스는 대용량 테이블에서 필요한 데이터만 빠르고 효율적으로 액세스하기 위한 오브젝트로, **모든 책 뒤쪽의 색인(찾아보기)** 과 같은 역할을 합니다(그림 2-2). 색인 없이 특정 용어를 찾으려면 첫 페이지부터 끝까지 다 뒤져야 하지만(= Table Full Scan), 색인을 이용하면 일부만 읽고 멈출 수 있습니다(= Range Scan). **범위 스캔이 가능한 이유는 인덱스가 정렬돼 있기 때문**입니다.

---

### B*Tree 구조 — 거꾸로 뒤집은 나무 (그림 2-3)

DBMS는 일반적으로 **B*Tree(Balanced Tree)** 인덱스를 사용합니다. 나무(Tree)를 거꾸로 뒤집은 모양이어서 **뿌리(Root)가 위**에 있고, 가지(Branch)를 거쳐 맨 아래에 **잎사귀(Leaf)** 가 있습니다.

```text
                  [Root]  서
                 /        \  (>= 서)
          [Branch]        [Branch]
          강덕승           이재훈 / 송재훈
         /   |   \        /    |    \
     [Leaf]→[Leaf]→ ... →[Leaf]→[Leaf]→ ...   (리프끼리 double linked list)
     키+ROWID  (키값 순 정렬)
```
예: `create index 고객_N1 on 고객(고객명)` 인덱스에서 `select * from 고객 where 고객명 = '이재훈'`을 탐색.

| 구성요소 | 역할 |
|------|------|
| **루트 블록(Root)** | 탐색의 시작점 |
| **브랜치 블록(Branch)** | 하위 블록으로 내려가는 경로 안내 |
| **리프 블록(Leaf)** | 실제 인덱스 키 + ROWID 저장, 키값 순 정렬, 서로 연결(double linked list) |

**루트·브랜치 블록의 레코드는 하위 블록의 주소값을 가지며, 키값은 하위 블록에 저장된 키값의 범위**를 나타냅니다.
- 예: 루트 블록의 `'서'` 레코드가 가리키는 하위 블록에는 `고객명 >= '서'` 레코드가 저장돼 있습니다.
- 예: 오른쪽 브랜치의 `'이재훈'` 레코드가 가리키는 하위 블록에는 `고객명 >= '이재훈'` 레코드가 저장돼 있습니다.

---

### LMC (Leftmost Child) — 키값 없는 특별 레코드

루트·브랜치 블록에는 **키값을 갖지 않는 특별한 레코드**가 가장 왼쪽에 하나 있습니다 — **LMC(Leftmost Child)** 입니다.

- LMC는 자식 노드 중 **가장 왼쪽 끝 블록**을 가리킵니다.
- LMC가 가리키는 블록에는 **그 블록의 첫 키값 레코드보다 "작거나 같은"** 레코드가 저장됩니다.
  - 예: 루트의 첫 키값이 `'서'`라면, LMC가 가리키는 블록엔 `'서'`보다 작은(`< '서'`) 값들이 들어갑니다.

```text
[Root]  (LMC) | 서 | ...
         │      │
         ▼      ▼
   '서'보다 작은   '서' 이상
```
LMC가 없으면 첫 키값보다 작은 데이터를 가리킬 방법이 없으므로, B*Tree의 빈틈없는 탐색을 위해 반드시 필요합니다.

---

### 인덱스 엔트리와 ROWID

리프 블록의 각 레코드는 키값 순으로 정렬되며 **ROWID**(테이블 로우 위치)를 갖습니다. **인덱스 키값이 같으면 ROWID 순으로** 정렬됩니다.

```text
ROWID = 데이터 블록 주소(DBA) + 로우 번호
        ├─ DBA = 데이터파일 번호 + 블록 번호
        └─ 로우 번호 = 블록 내 순번
```

이 ROWID 값을 알면 테이블 레코드를 바로 찾아갈 수 있습니다. **인덱스를 스캔하는 이유**가 바로 "조건을 만족하는 소량 데이터를 빨리 찾고 거기서 ROWID를 얻기 위해"입니다.

### 인덱스 탐색의 두 단계 (예고)

```text
인덱스 탐색
 ├─ 수직적 탐색(2.1.3): 인덱스 스캔 시작점(리프)을 찾는 과정
 └─ 수평적 탐색(2.1.4): 리프를 옆으로 훑으며 데이터를 찾는 과정
```

---

## ✅ 이 절(2.1.2)의 핵심 통찰

1. 인덱스는 책 뒤 '찾아보기'와 같고, **정렬돼 있어 Range Scan(일부만 읽고 멈춤)** 이 가능하다.
2. B*Tree는 거꾸로 뒤집은 나무 — **Root(위) → Branch → Leaf(아래)**, 리프는 키값 순 정렬 + double linked list 연결.
3. 루트·브랜치 레코드의 **키값은 하위 블록의 키값 범위**를 나타내고 하위 블록 주소를 가리킨다.
4. **LMC(Leftmost Child)** 는 키값 없는 맨 왼쪽 레코드로, 첫 키값보다 작은 값을 담은 가장 왼쪽 블록을 가리킨다.
5. 리프 엔트리는 **키 + ROWID(DBA + 로우 번호)** 이며, 인덱스 스캔의 목적은 ROWID 획득이다. 탐색은 수직(시작점) + 수평(데이터 찾기) 두 단계다.

## 🔗 흐름 속 위치

인덱스 튜닝을 이해하려면 먼저 이 B*Tree 구조를 알아야 합니다. 다음 2.1.3(수직적 탐색)·2.1.4(수평적 탐색)는 이 구조 위에서 작동하며, ROWID로 테이블을 찾는 랜덤 액세스(3장)의 출발점입니다.

📌 한 줄 요약: 인덱스는 키값 순으로 정렬된 거꾸로 된 나무(B*Tree)이며, 브랜치 키값은 하위 범위를, LMC는 첫 키보다 작은 영역을 가리키고, 리프 엔트리의 ROWID로 테이블을 찾아갑니다.
