## 부록.4 DBMS_XPLAN 패키지

### 핵심 개념

오라클 9.2에 소개된 `dbms_xplan` 패키지로, **plan_table에 저장된 예상 실행계획**을 쉽게 출력할 수 있고, 10g부터는 **라이브러리 캐시에 캐싱된 SQL의 실측 실행계획**과 **오퍼레이션 단계(Row Source)별 수행통계**까지 손쉽게 볼 수 있습니다.

| 함수 | 대상 | 보여주는 것 |
|------|------|------|
| `DISPLAY` | `plan_table` (EXPLAIN PLAN 결과) | **예상** 실행계획 (E-Rows) |
| `DISPLAY_CURSOR` | 캐시에 있는 실제 커서 | **실측** 실행계획 (A-Rows 등) |

---

### ① 예상 실행계획 출력 — DISPLAY

```sql
SQL> explain plan set statement_id = 'SQL1' for
  2  select * from emp e, dept d
  3  where d.deptno = e.deptno and e.sal >= 1000;

-- display('plan_table명', statement_id, format)
SQL> select * from table(dbms_xplan.display('PLAN_TABLE', 'SQL1', 'BASIC'));     -- ID/Operation/Name만
SQL> select * from table(dbms_xplan.display('PLAN_TABLE', 'SQL1', 'BASIC ROWS BYTES COST'));  -- + Rows/Bytes/Cost
```
- 첫 인자: plan_table명 / 둘째 인자: statement_id(NULL이면 가장 마지막 explain plan) / 셋째 인자: format

**format 옵션** (단독·조합 사용): `BASIC`, `TYPICAL`(기본), `SERIAL`, `PARTITION`, `PARALLEL`, `PREDICATE`, `PROJECTION`, `ALIAS`, `REMOTE`, `NOTE`, `ALL`, `OUTLINE`, `ADVANCED`

---

### ② 캐싱된 커서의 실측 실행계획 — DISPLAY_CURSOR

**'SQL 커서'** 란 하드파싱을 거쳐 메모리에 적재된 SQL·파싱트리·실행계획과 SQL Area를 말합니다. 수행통계는 `v$sql`, 실행계획은 `v$sql_plan` 뷰에 있으며, 조회하려면 **`sql_id`와 `child_number`** 가 필요합니다.

```sql
-- 직전에 수행한 SQL의 sql_id / child_number 찾기
SQL> select prev_sql_id as sql_id, prev_child_number as child_no
  2  from v$session
  3  where sid = userenv('sid') and username is not null and prev_hash_value <> 0;

-- 더 이전 SQL은 텍스트로 검색
SQL> select sql_id, child_number, sql_fulltext, last_active_time
  2  from v$sql where sql_text like '%select/* comment */%from%emp%dept%';
```

```sql
-- display_cursor('sql_id', child_no, 'format')
-- sql_id/child_no에 null을 주면 '직전에 수행한 SQL'을 보여줌
SQL> select * from table(
  2    dbms_xplan.display_cursor(null, null, 'BASIC ROWS BYTES COST PREDICATE'));
```
```text
| Id | Operation                    | Name        | Rows | Bytes | Cost |
|  0 | SELECT STATEMENT             |             |      |       |  3   |
|  1 |  TABLE ACCESS BY INDEX ROWID | DEPT        |   1  |  18   |  1   |
|  2 |   NESTED LOOPS               |             |  12  | 660   |  3   |
|  3 |    TABLE ACCESS BY INDEX ROWID| EMP         |  12  | 444   |  2   |
|* 4 |     INDEX RANGE SCAN         | EMP_SAL_IDX |  12  |       |  1   |
|* 5 |    INDEX RANGE SCAN          | DEPT_PK     |   1  |       |  0   |
Predicate Information:
   4 - access("E"."SAL">=1000)
   5 - access("D"."DEPTNO"="E"."DEPTNO")
```

> ⚠️ **serveroutput off 주의**: SQL*Plus에서 `null, null`로 직전 SQL을 잡으려면 `set serveroutput off` 상태여야 합니다(on이면 내부 SQL이 끼어들어 직전 SQL이 바뀜).
> ```sql
> SQL> set serveroutput off;
> SQL> select * from emp;
> SQL> select * from table(dbms_xplan.display_cursor(null, null, 'allstats last'));
> ```

### ③ Row Source별 수행통계 — A-Rows를 보려면 통계 수집부터

`DISPLAY_CURSOR`는 SQL 트레이스처럼 **각 오퍼레이션 단계별 실측 수행통계**를 보여줍니다. 단, 그냥 실행하면 통계가 안 모여 A-Rows가 비어 나오므로 **둘 중 하나를 먼저** 해야 합니다.

```sql
-- 방법 1: 분석 대상 SQL에 힌트 (가장 간편)
select /*+ gather_plan_statistics */ ...

-- 방법 2: 세션 레벨 설정
alter session set statistics_level = all;
```

```sql
-- 통계 종류별 format
display_cursor('sql_id', 0, 'IOSTATS')    -- I/O 통계
display_cursor('sql_id', 0, 'MEMSTATS')   -- 메모리(PGA) 통계
display_cursor('sql_id', 0, 'ALLSTATS')   -- I/O + 메모리 (누적)
display_cursor(null, null, 'ALLSTATS LAST')  -- 마지막 수행 한 번의 통계 ★기본 조합
```
```text
| Id | Operation        | Name    | Starts | E-Rows | A-Rows | A-Time      | Buffers | Reads |
|  1 | TABLE ACC  DEPT  |         |   1    |   1    |  12    | 00:00:00.04 |   20    |  18   |
|  2 |  NESTED L        |         |   1    |   1    |  12    | 00:00:00.93 |    8    |  17   |
|  3 |   TABLE A  EMP    |         |   1    |  12    |  12    | 00:00:00.02 |    4    |  16   |
|* 4 |    INDEX   EMP_SAL|         |   1    |  12    |  12    | 00:00:00.01 |    2    |   6   |
|* 5 |   INDEX R  PK_DEPT|        |  12    |   1    |  12    | 00:00:00.01 |    4    |   0   |
```

| 컬럼 | 의미 | 연결 |
|------|------|------|
| **Starts** | 해당 단계 **반복 수행 횟수** | NL 조인 Inner 반복(4.1.6) |
| **E-Rows** | 옵티마이저 **예상** 행 수 (= DISPLAY의 Rows/Card) | 7장 카디널리티 |
| **A-Rows** | **실제** 행 수 | E-Rows와 비교 |
| A-Time | 실제 누적 수행 시간 | 병목 단계 식별 |
| Buffers | 단계별 논리적 I/O | `reduce I/O` 검증 |
| Reads | 단계별 물리적 I/O | |

> **Starts 해석 예**: PK_DEPT 인덱스 단계(ID=5) `Starts=12`는 그 단계를 12번 실행했다는 뜻 — EMP 테이블 액세스(ID=3) 결과건수 12와 일치합니다. **NL 조인이 드라이빙 집합의 건수만큼 반대쪽을 반복 액세스**하기 때문입니다(4장).

**SQL 트레이스와의 매핑** [표 3]

| DBMS_XPLAN | SQL 트레이스 | 의미 |
|------|------|------|
| A-Rows | rows | 각 단계에서 읽거나 갱신한 건수 |
| A-Time | time | 단계별 소요시간 |
| **Buffers** | **cr(=query) + current** | SQL 수행 중 읽은 총 블록 수(논리적 I/O) |
| Reads | pr | 디스크에서 읽은 총 블록 수 |

(각주 1: 일반적으로 Buffers는 버퍼캐시 경유 블록이지만, Direct Path Read로 읽은 블록도 여기 포함됩니다.) **각 통계는 누적값**이며(부모가 자식 포함, 부록.3), `LAST` 옵션을 붙이면 마지막 수행 한 번의 일량만 보여줍니다.

---

### 가장 중요한 해석 포인트 — E-Rows vs A-Rows

```text
E-Rows(예상) ≠ A-Rows(실측)  →  차이가 크면:
   통계정보 노후 / 선택도 오류 / 히스토그램 부재 / 바인드 변수 문제 의심 (7장)
```
이것이 옵티마이저가 왜 나쁜 계획을 골랐는지 진단하는 **1번 단서**입니다.

### 권한

`DISPLAY_CURSOR`가 권한 오류면 DBA에게 요청합니다.
```sql
grant select on v_$session to scott;
grant select on v_$sql to scott;
grant select on v_$sql_plan to scott;
grant select on v_$sql_plan_statistics_all to scott;
```

## ✅ 이 항목의 핵심 통찰

1. `DBMS_XPLAN`은 실행계획을 가장 실전적으로 읽는 핵심 도구 — **`DISPLAY`(예상, plan_table)** 와 **`DISPLAY_CURSOR`(실측, 캐시 커서)** 로 나뉜다.
2. A-Rows를 보려면 **먼저** `/*+ gather_plan_statistics */` 힌트나 `statistics_level=all`이 필요하다(없으면 비어 나옴).
3. **`E-Rows` vs `A-Rows` 차이**가 옵티마이저 추정 오류의 1번 신호다(통계·선택도·히스토그램·바인드 의심).
4. **`Starts`** 는 단계 반복 횟수 — NL 조인 Inner 반복 진단의 핵심(Starts=드라이빙 건수).
5. **`Buffers` = SQL 트레이스의 query + current**(논리적 I/O), `Reads` = pr(물리적 I/O). 누적값이므로 자식을 빼서 순수 비용을 본다.
6. 기본 조합은 **`ALLSTATS LAST`**(+`PREDICATE`), `null,null`로 직전 SQL을 잡을 땐 **`set serveroutput off`** 가 필요하다.

## 🔗 흐름 속 위치

부록.3(SQL 트레이스)이 별도 파일·TKProf를 거쳤다면, 부록.4는 **트레이스 없이 SQL*Plus에서 바로** 캐시 커서의 실측 통계(A-Rows·Buffers·Starts)를 봅니다 — 실무 튜닝에서 가장 자주 쓰는 방식입니다. 다음 부록.5·6은 운영 중 SQL을 모니터링·조회하는 방법으로 확장됩니다.

📌 한 줄 요약: DBMS_XPLAN은 실행계획을 추정치가 아니라 **실제 Rows(A-Rows)·반복(Starts)·논리적 I/O(Buffers)·Predicate** 까지 포함한 진단 자료로 바꿔주며, `ALLSTATS LAST`가 그 기본 조합입니다.
