## 부록 SQL 분석 도구 전체 흐름

### 전체 축
부록은 1~7장의 이론을 실제 진단 도구로 연결합니다. 핵심은 **실행계획을 확인하고, 실제 I/O와 Rows를 검증하고, 시스템에서 튜닝 우선순위를 찾는 것**입니다.

```text
튜닝 대상 SQL 찾기
   ↓
V$SQL
   ↓
실행계획 확인
   ↓
DBMS_XPLAN / AutoTrace
   ↓
실제 수행 상세 분석
   ↓
SQL Trace / SQL Monitoring
   ↓
원인 진단과 개선
```

## 도구별 위치

| 도구 | 주용도 | 핵심 지표 |
|---|---|---|
| 실행계획 확인 | 액세스 경로와 조인 방식 파악 | Operation, Rows, Cost |
| AutoTrace | 간단한 실행계획과 통계 확인 | consistent gets |
| SQL Trace | 실제 수행 프로파일링 | Parse/Execute/Fetch, CPU, Elapsed |
| DBMS_XPLAN | 실제 Rows와 Predicate 확인 | E-Rows, A-Rows, access/filter |
| SQL Monitoring | 장시간 SQL 실시간 추적 | Plan Line별 Activity/I/O/TEMP |
| V$SQL | 튜닝 대상 SQL 선별 | buffer_gets, executions, elapsed_time |

## 진단 흐름

### 1. 튜닝 대상 찾기
```text
V$SQL
   ↓
BUFFER_GETS 큰 SQL
   ↓
EXECUTIONS 많은 SQL
   ↓
실행당 비용 큰 SQL
```

### 2. 실행계획 확인
```text
DBMS_XPLAN
   ↓
인덱스/Full Scan 확인
   ↓
조인 방식 확인
   ↓
정렬/해시/TEMP 여부 확인
```

### 3. 추정과 실제 비교
```text
E-Rows
   ↓
옵티마이저 예상

A-Rows
   ↓
실제 실행 결과

차이가 큼
   ↓
통계정보, 선택도, 히스토그램, 바인드 문제 의심
```

### 4. I/O 원인 추적
```text
consistent gets 큼
   ↓
논리적 I/O 과다
   ↓
인덱스 스캔량 / 테이블 랜덤 액세스 / 조인 반복 / 정렬 확인
```

## ✅ 부록의 핵심 통찰
1. 실행계획은 SQL의 자료구조 연산 흐름을 보여줍니다.
2. AutoTrace는 빠른 1차 진단에 유용합니다.
3. SQL Trace는 실제 수행 프로파일을 보여줍니다.
4. DBMS_XPLAN은 `E-Rows`와 `A-Rows`, access/filter Predicate 확인에 핵심입니다.
5. SQL Monitoring은 장시간 SQL의 현재 병목 지점을 보여줍니다.
6. V$SQL은 시스템 전체에서 튜닝 우선순위를 찾는 출발점입니다.
7. 도구 사용의 목적은 결국 `reduce I/O`를 검증하는 것입니다.

## 🔗 흐름 속 위치
7장에서 옵티마이저가 비용을 추정해 실행계획을 고른다고 배웠습니다. 부록은 그 추정이 실제 실행에서 맞았는지 확인하고, 틀렸다면 어디서 I/O가 증가했는지 찾아내는 실전 도구 모음입니다.

📌 한 줄 요약: 부록은 실행계획과 실제 수행 통계를 이용해 “왜 많이 읽었는가”를 확인하는 실전 진단 장입니다.
