## 부록.1 실행계획 확인

### 핵심 개념
실행계획 확인은 SQL 튜닝의 출발점입니다. SQL이 실제로 어떤 경로로 데이터를 읽고, 어떤 조인·정렬·필터 연산을 수행하는지 확인해야 원인을 진단할 수 있습니다.

```text
SQL 문장
   ↓
옵티마이저 실행계획 생성
   ↓
액세스 경로 / 조인 방식 / 정렬 여부 확인
   ↓
I/O 발생 지점 추적
```

| 확인 항목 | 의미 | 연결되는 장 |
|---|---|---|
| Access Path | Full Scan, Index Scan 등 | 1~3장 |
| Join Method | NL, Sort Merge, Hash | 4장 |
| Sort Operation | ORDER BY, GROUP BY, UNIQUE | 5장 |
| Partition Access | 어떤 파티션을 읽는가 | 6장 |
| Rows/Cost | 옵티마이저 추정값 | 7장 |

### 실행계획 생성·확인 방법

실제 수행하지 않고 **예상 실행계획만** 보려면 `EXPLAIN PLAN FOR`로 PLAN_TABLE에 저장한 뒤 `DBMS_XPLAN.DISPLAY`로 출력합니다.

```sql
explain plan for
select * from emp where empno = 7900;

select * from table(dbms_xplan.display);
```

- PLAN_TABLE은 10g부터 오라클이 `sys.plan_table$` + `PLAN_TABLE` public synonym으로 자동 제공하므로 따로 만들 필요가 없습니다(과거에는 `@?/rdbms/admin/utlxplan.sql` 실행).
- `EXPLAIN PLAN`은 **추정치(E-Rows)** 만 보여줍니다. 실제로 수행한 SQL의 **실측치(A-Rows)** 까지 보려면 [[부록.4_DBMS_XPLAN_패키지]]의 `DISPLAY_CURSOR` + `gather_plan_statistics`를 씁니다.

### 실행계획 예시
```text
--------------------------------------------------------------------------------
| Id | Operation                    | Name      | Rows | Cost |
--------------------------------------------------------------------------------
|  0 | SELECT STATEMENT             |           |      |      |
|  1 |  TABLE ACCESS BY INDEX ROWID | 주문      |   10 |    5 |
|  2 |   INDEX RANGE SCAN           | 주문_X1   |   10 |    2 |
--------------------------------------------------------------------------------
```

### 해석 흐름
```text
Operation 확인
   ↓
인덱스가 쓰였는가?
   ↓
액세스 조건인가, 필터 조건인가?
   ↓
Rows 추정이 현실적인가?
   ↓
논리적 I/O가 어디서 많이 발생하는가?
```

---

## ✅ 이 항목의 핵심 통찰
1. 실행계획은 SQL이 사용할 자료구조와 연산 순서를 보여줍니다.
2. 실행계획 이름만 보고 빠르다고 판단하면 안 됩니다.
3. Rows, Starts, Buffers 같은 실제 실행 통계와 함께 봐야 합니다.

## 🔗 흐름 속 위치
부록은 1~7장의 이론을 실제 SQL 진단으로 연결합니다. 실행계획 확인은 그 첫 단계입니다.

📌 한 줄 요약: 실행계획 확인은 SQL을 “선언문”에서 “실제 자료구조 연산 흐름”으로 번역하는 작업입니다.
