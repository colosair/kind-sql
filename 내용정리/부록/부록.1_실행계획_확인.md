## 부록.1 실행계획 확인

### 핵심 개념

실행계획 확인은 SQL 튜닝의 출발점입니다. SQL이 실제로 어떤 경로로 데이터를 읽고, 어떤 조인·정렬·필터 연산을 수행하는지 확인해야 원인을 진단할 수 있습니다.

| 확인 항목 | 의미 | 연결 장 |
|------|------|------|
| Access Path | Full Scan, Index Scan 등 | 1~3장 |
| Join Method | NL, Sort Merge, Hash | 4장 |
| Sort Operation | ORDER BY, GROUP BY, UNIQUE | 5장 |
| Partition Access | 어떤 파티션을 읽는가 | 6장 |
| Rows/Cost | 옵티마이저 추정값 | 7장 |

---

### PLAN_TABLE 생성

실행계획을 저장할 `plan_table`이 필요합니다.

```sql
SQL> @?/rdbms/admin/utlxplan.sql     -- '?'는 $ORACLE_HOME 디렉터리를 대체하는 기호
```

> **10g부터는** 오라클이 기본적으로 `sys.plan_table$` 테이블을 만들고 `PLAN_TABLE`이라는 public synonym도 생성하므로 **사용자가 별도로 만들 필요가 없습니다.**

```sql
SQL> select owner, synonym_name, table_owner, table_name
  2  from all_synonyms where synonym_name = 'PLAN_TABLE';
--  OWNER   SYNONYM_NAME  TABLE_OWNER  TABLE_NAME
--  ------  ------------  -----------  -----------
--  PUBLIC  PLAN_TABLE    SYS          PLAN_TABLE$
```

---

### SQL*Plus에서 실행계획 확인

**① `explain plan for`로 실행계획을 plan_table에 저장** (실제 수행 X, 예상 실행계획만)

```sql
SQL> explain plan for
  2  select * from emp where empno = 7900;
해석되었습니다.
```

**② `utlxpls.sql` 또는 `dbms_xplan.display`로 출력**

```sql
SQL> SET LINESIZE 200          -- 행이 밀려 보기 불편하지 않도록 넉넉히
SQL> @?/rdbms/admin/utlxpls
-- 또는
SQL> select * from table(dbms_xplan.display);
```
```text
Plan hash value: 4024650034
-----------------------------------------------------------------
| Id | Operation                    | Name   | Rows | Bytes | Cost (%CPU)|
-----------------------------------------------------------------
|  0 | SELECT STATEMENT             |        |   1  |  32   |  1   (0)  |
|  1 |  TABLE ACCESS BY INDEX ROWID | EMP    |   1  |  32   |  1   (0)  |
|* 2 |   INDEX UNIQUE SCAN          | PK_EMP |   1  |       |  0   (0)  |
-----------------------------------------------------------------
Predicate Information (identified by operation id):
   2 - access("EMPNO"=7900)
```

핵심 두 가지를 함께 봅니다.
- **`Plan hash value`**: 실행계획의 지문(같은 값이면 같은 계획).
- **`Predicate Information`**: 각 조건이 **`access`(액세스 조건)** 인지 **`filter`(필터 조건)** 인지 — 3.3.3의 구분이 여기서 드러납니다.

> ⚠️ 위 실행계획은 어디까지나 **예상(E-Rows)** 실행계획입니다. 대개 이대로 실행하지만 다른 방식으로 실행하는 경우도 있어, **실제 실행계획**은 부록.4(DBMS_XPLAN)의 `DISPLAY_CURSOR`로 확인해야 합니다.

### 상용 쿼리 툴에서 실행계획 확인

토드(Toad)·오렌지(Orange)·SQLGate 같은 상용 쿼리 툴이 더 편리합니다(그림 1).

| 툴 | 단축키 |
|------|------|
| 오렌지(Orange) | `Ctrl + E` (또는 Plan Tool: `Alt+F7` / Tools 메뉴 / 입력 후 `F5`) |
| SQLGate | `F7` |

---

### 더 많은 정보 확인하기 — display advanced 옵션

`plan_table`에는 쿼리 툴이 보여주는 항목 외에도 **수많은 정보**가 수집되어 있습니다. 이를 보려면 `dbms_xplan.display`에 옵션을 줍니다.

```sql
SQL> select * from table(dbms_xplan.display(null, null, 'advanced'));
```
세 번째 인자에는 `serial`, `parallel`, `outline`, `alias`, `projection`, `all` 같은 다양한 옵션을 쓸 수 있습니다(`advanced`는 거의 모든 정보 출력).

---

## ✅ 이 항목의 핵심 통찰

1. 10g부터 `PLAN_TABLE`은 자동 제공되므로 `utlxplan.sql`을 따로 실행할 필요가 없다.
2. `explain plan for` → `utlxpls`/`dbms_xplan.display`로 **예상 실행계획**을 본다.
3. 출력의 **`Predicate Information`** 으로 `access`(액세스)/`filter`(필터) 조건을 구분하는 것이 핵심이다.
4. `explain plan`은 **추정(E-Rows)** 만 보여준다 — 실측(A-Rows)은 부록.4 `DISPLAY_CURSOR`로.
5. `dbms_xplan.display(...,'advanced')`로 outline·alias·projection 등 숨은 정보까지 볼 수 있다.

## 🔗 흐름 속 위치

부록은 1~7장의 이론을 실제 SQL 진단으로 연결하는 도구 모음이며, 실행계획 확인이 그 첫 단계입니다. 다음 부록.2(AutoTrace)·부록.4(DBMS_XPLAN)는 여기서 본 예상 실행계획을 **실측치와 함께** 보는 방법으로 이어집니다.

📌 한 줄 요약: 실행계획 확인은 SQL을 "선언문"에서 "실제 자료구조 연산 흐름"으로 번역하는 작업이며, `Predicate Information`의 access/filter 구분이 진단의 출발점입니다.
