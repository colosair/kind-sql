## 부록.2 AutoTrace

### 핵심 개념

AutoTrace는 SQL 튜닝에 유용한 정보를 많이 포함하고 있어 **튜너들이 가장 즐겨 사용하는** 도구입니다. SQL을 실행하면 **쿼리 실행결과 + 예상 실행계획 + 실행통계**를 한 번에 보여줍니다.

```sql
SQL> set autotrace on
SQL> select * from scott.emp where empno = 7900;
```
```text
-- ① 쿼리 실행결과
EMPNO ENAME JOB    MGR  HIREDATE   SAL  COMM DEPTNO
7900  JAMES CLERK  7698 81/12/03   950        30

-- ② Execution Plan (예상 실행계획)
Plan hash value: 4024650034
| Id | Operation                    | Name   | Rows | Bytes | Cost |
|  0 | SELECT STATEMENT             |        |   1  |  32   |  1   |
|  1 |  TABLE ACCESS BY INDEX ROWID | EMP    |   1  |  32   |  1   |
|* 2 |   INDEX UNIQUE SCAN          | PK_EMP |   1  |       |  0   |
Predicate Information:  2 - access("EMPNO"=7900)

-- ③ Statistics (실행통계)
   1 recursive calls
   0 db block gets
   2 consistent gets        ← 논리적 I/O (가장 핵심 지표)
   0 physical reads         ← 물리적 I/O
   0 redo size
 743 bytes sent via SQL*Net to client
 374 bytes received via SQL*Net from client
   1 SQL*Net roundtrips to/from client
   0 sorts (memory)
   0 sorts (disk)
   1 rows processed
```

| Statistics 항목 | 의미 | 튜닝 관점 |
|------|------|------|
| **consistent gets** | 논리적 I/O (Consistent 모드 블록 읽기) | **가장 핵심 지표** |
| db block gets | 논리적 I/O (Current 모드 블록 읽기) | DML/현재값 읽기 |
| physical reads | 물리적 I/O | 캐시 미스/디스크 |
| sorts (memory/disk) | 정렬 발생(메모리/디스크) | 5장 소트 튜닝 |
| SQL*Net roundtrips | 네트워크 왕복 | 6.1.2 Call 튜닝 |
| rows processed | 처리 행 수 | 카디널리티 검증 |

---

### 6가지 옵션 조합

`on`(결과+α) vs `traceonly`(결과 생략), `explain`(실행계획만) vs `statistics`(통계만)의 조합입니다.

| 옵션 | 결과 | 예상 실행계획 | 실행통계 | 쿼리 **실제 수행** |
|------|:---:|:---:|:---:|:---:|
| ① `set autotrace on` | O | O | O | O |
| ② `set autotrace on explain` | O | O | | O |
| ③ `set autotrace on statistics` | O | | O | O |
| ④ `set autotrace traceonly` | | O | O | O |
| ⑤ `set autotrace traceonly explain` | | O | | **X** |
| ⑥ `set autotrace traceonly statistics` | | | O | O |

> ①~③은 결과를 출력해야 하므로, ④·⑥은 실행통계를 보여줘야 하므로 쿼리를 **실제 수행**합니다. **⑤만** 예상 실행계획만 출력하므로 **실제 수행하지 않습니다.** (대용량 쿼리의 실행계획만 빠르게 보고 싶을 때 ⑤가 유용)

---

### ⚠️ 주의 — AutoTrace의 Execution Plan은 '예상' 실행계획

AutoTrace가 보여주는 Execution Plan은 `EXPLAIN PLAN`과 동일한 **예상(E-Rows) 실행계획**입니다. 실제로 어떤 계획으로 수행됐는지(실측, A-Rows)와 **다를 수 있습니다.** 실측 실행계획·단계별 통계가 필요하면 부록.3(SQL 트레이스)나 부록.4(DBMS_XPLAN `DISPLAY_CURSOR`)를 씁니다.

---

### 권한 설정 — plustrace 롤

- **실행계획만** 보려면 `plan_table`만 있으면 됩니다.
- **실행통계까지** 보려면 `v_$sesstat`, `v_$statname`, `v_$mystat` 뷰에 대한 읽기 권한이 필요합니다. 일반 사용자에게 일일이 부여해도 되지만, **`plustrace` 롤**을 만들어 부여하면 편리합니다.

```sql
SQL> @?/sqlplus/admin/plustrce.sql      -- plustrace 롤 생성
SQL> grant plustrace to scott;
```

### 상용 쿼리 툴

- **토드(Toad)**: SQL 편집 창에서 우클릭 → 컨텍스트 메뉴 'Auto Trace' 활성화 → 실행하면 'Auto Trace' 탭에 실행통계 출력
- **오렌지(Orange)**: 상단 'Action' 메뉴 → 'Extract Statistics' (또는 `Ctrl+Shift+S`) → 'Statistics' 탭에 출력

---

## ✅ 이 항목의 핵심 통찰

1. AutoTrace는 **결과 + 예상 실행계획 + 실행통계**를 한 번에 보여주는, SQL*Plus에서 가장 쉽고 빠른 진단 도구다.
2. 가장 먼저 볼 지표는 **`consistent gets`(논리적 I/O)** — 실행계획이 좋아 보여도 이 값이 크면 비효율이 남아 있다.
3. `on`/`traceonly` × `explain`/`statistics` **6가지 조합**으로 필요한 부분만 출력하며, **`traceonly explain`(⑤)만 쿼리를 실제 수행하지 않는다.**
4. AutoTrace의 Execution Plan은 **예상 실행계획**이다 — 실측은 SQL 트레이스/DISPLAY_CURSOR로.
5. 실행통계를 보려면 **`plustrace` 롤** 권한이 필요하다.

## 🔗 흐름 속 위치

부록.1이 예상 실행계획만 봤다면, 부록.2는 거기에 **실행통계(consistent gets 등)** 를 더한 1차 진단 도구입니다. 더 정밀한 단계별 실측이 필요하면 부록.3(SQL 트레이스)·부록.4(DBMS_XPLAN)로 넘어갑니다.

📌 한 줄 요약: AutoTrace는 SQL의 실행경로와 전체 I/O 비용(`consistent gets`)을 빠르게 보는 1차 진단 도구이며, 보여주는 실행계획은 어디까지나 '예상'입니다.
