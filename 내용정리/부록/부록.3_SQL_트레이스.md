## 부록.3 SQL 트레이스

### 핵심 개념
SQL 트레이스는 SQL 수행 과정의 상세 통계를 수집해 **실제 수행 시간, Parse/Execute/Fetch, I/O, 대기 이벤트**를 분석하게 해줍니다.

```text
Trace 활성화
   ↓
SQL 실행
   ↓
Trace 파일 생성
   ↓
TKPROF 등으로 분석
   ↓
실제 병목 지점 확인
```

### 수집 방법

현재 세션에 트레이스를 켜고 끄는 가장 단순한 방법입니다.

```sql
alter session set sql_trace = true;
select * from emp where empno = 7900;   -- 분석 대상 SQL
alter session set sql_trace = false;
```

생성된 `.trc` 파일 위치는 `v$diag_info`로 찾습니다.

```sql
select value from v$diag_info where name = 'Default Trace File';
-- 예: /oracle/diag/rdbms/ora11g/trace/ora11g_ora_22827.trc
```

이 raw 트레이스 파일을 `tkprof`로 정렬·요약하면 SQL별 Parse/Execute/Fetch 통계를 읽기 쉽게 볼 수 있습니다.

```text
tkprof ora11g_ora_22827.trc report.txt sys=no sort=prsela,exeela,fchela
```

| 구분 | 의미 |
|---|---|
| Parse | SQL 파싱과 실행계획 준비 |
| Execute | SQL 실행 요청 |
| Fetch | 결과 행 가져오기 |
| CPU | CPU 사용 시간 |
| Elapsed | 경과 시간 |
| Disk | 물리적 읽기 |
| Query/Current | 논리적 읽기 |

### 분석 관점
```text
Elapsed가 큰가?
   ↓
CPU가 큰가, Wait가 큰가?
   ↓
논리적 I/O가 큰가, 물리적 I/O가 큰가?
   ↓
Fetch 단계에서 반복적으로 비용이 발생하는가?
```

### CS 대응
SQL 트레이스는 프로그램 프로파일러에 가깝습니다. 실행계획이 “어떤 알고리즘을 쓸 예정인가”라면, 트레이스는 “실제로 어디서 시간이 쓰였는가”를 보여줍니다.

---

## ✅ 이 항목의 핵심 통찰
1. SQL 트레이스는 추정이 아니라 실제 수행 통계를 봅니다.
2. Parse, Execute, Fetch 중 어디서 비용이 발생하는지 분리할 수 있습니다.
3. 논리적 I/O와 대기 이벤트를 함께 봐야 병목 원인을 정확히 잡을 수 있습니다.

📌 한 줄 요약: SQL 트레이스는 SQL 튜닝을 실행계획 해석에서 실제 수행 프로파일링으로 확장합니다.
