## 부록.3 SQL 트레이스

### 핵심 개념

SQL을 튜닝할 때 가장 강력한 도구가 SQL 트레이스입니다. 부록.1의 예상 실행계획과 부록.2의 AutoTrace만으로 문제점을 찾을 수 없을 때, 트레이스는 SQL 수행 과정의 상세 통계(**실제 수행 시간, Parse/Execute/Fetch, I/O, 대기 이벤트**)를 수집해 병목을 찾아냅니다.

```text
Trace 활성화 → SQL 실행 → .trc 파일 생성 → TKProf로 리포트 → 실제 병목 확인
```

---

### ① 트레이스 수집

현재 세션에 트레이스를 켜고 끕니다.

```sql
SQL> alter session set sql_trace = true;
SQL> select * from emp where empno = 7900;   -- 분석 대상 SQL
SQL> select * from dual;
SQL> alter session set sql_trace = false;
```

### ② 트레이스 파일 찾기

`v$diag_info` 뷰로 디렉터리와 파일명을 찾습니다.

```sql
-- 디렉터리
SQL> select value from v$diag_info where name = 'Diag Trace';
--  /oracle/diag/rdbms/ora11g/trace

-- 파일명까지
SQL> select value from v$diag_info where name = 'Default Trace File';
--  /oracle/diag/rdbms/ora11g/trace/ora11g_ora_22827.trc
```
(10g 이하는 `v$process`·`v$session`·`v$parameter(user_dump_dest)`를 조인하는 복잡한 쿼리로 찾습니다.)

### ③ 리포트 생성 — TKProf

raw 트레이스 파일은 그대로 보기 어려우므로 **TKProf** 유틸리티로 보기 쉽게 요약합니다.

```bash
$ tkprof tracefile outputfile [explain=] [table=] [print=] [insert=] [sys=] [sort=]

# 가장 일반적인 사용 (sys=no: 파싱 중 내부적으로 수행되는 SQL문 제외)
$ tkprof ora11g_ora_22827.trc report.prf sys=no
```

---

### TKProf 리포트 읽기

```text
select * from emp where empno = 7900

call     count   cpu   elapsed   disk   query  current   rows
------- ------ ----- --------- ------ ------- -------- ------
Parse        1  0.00      0.00      0       0        0      0
Execute      1  0.00      0.00      0       0        0      0
Fetch        2  0.00      0.00      0       2        0      1
------- ------ ----- --------- ------ ------- -------- ------
total        4  0.00      0.00      0       2        0      1

Misses in library cache during parse: 1     ← 1이면 하드파싱 발생(1.2.1)
Optimizer mode: ALL_ROWS
Parsing user id: 61

Rows   Row Source Operation
----   --------------------------------------------------
   1   TABLE ACCESS BY INDEX ROWID EMP (cr=2 pr=0 pw=0 time=80 us)
   1    INDEX UNIQUE SCAN PK_EMP (cr=1 pr=0 pw=0 time=44 us)(object id 5278)
```

**(1) Call 통계** — Parse/Execute/Fetch 세 Call별 통계 [표 1]

| 항목 | 의미 |
|------|------|
| call | Parse(파싱·실행계획 생성) / Execute(커서 실행) / Fetch(레코드 실제 Fetch) |
| count | 각 단계 수행 횟수 |
| cpu / elapsed | 각 단계 CPU 시간 / 경과 시간 |
| disk | 디스크에서 읽은 블록 수 |
| **query** | **Consistent 모드**로 읽은 블록 수 (MVCC, 6.1.1) |
| **current** | **Current 모드**로 읽은 블록 수 (6.1.1) |
| rows | 각 단계에서 읽거나 갱신한 건수 |

> **논리적 I/O = query + current** (1.3.5). 이는 DB 버퍼캐시 경유 블록 + Direct Path I/O 블록을 모두 포함합니다.

**(2) AutoTrace 항목과의 매핑** [표 2]

| SQL 트레이스 | AutoTrace | 의미 |
|------|------|------|
| current | db block gets | Current 모드로 읽은 블록 수 |
| query | consistent gets | Consistent 모드로 읽은 블록 수 |
| disk | physical reads | 디스크에서 읽은 블록 수 |
| fetch count | SQL*Net roundtrips to/from client | Fetch Call 횟수 |
| fetch rows | rows processed | 조회 건수 |

**(3) Row Source Operation (실행계획 통계)** — `Rows`와 단계별 `cr/pr/pw/time`

- **`Rows`**: 각 수행 단계에서 **출력(Flow-Out)된 로우 수**. (오라클 7까지는 처리(Processing) 건수였다가 8i부터 출력 건수로, 9i부터 확실히 출력 건수로 바뀜)
- **단계별 통계** (9.2.0.2~): `cr`=Consistent 모드 블록 읽기, `pr`=디스크 블록 읽기, `pw`=디스크 블록 쓰기, `time`=소요시간(us=microsecond)
- ⚠️ **꼭 기억 — 부모는 자식 노드의 값을 포함한다.** 위에서 EMP 테이블 액세스 단계는 `cr=2`, 자식인 PK_EMP 인덱스 액세스 단계는 `cr=1`이므로, **인덱스를 읽고 난 후 테이블을 액세스하는 단계의 순수 cr 개수는 1**입니다(2−1).

### CS 대응

SQL 트레이스는 **프로그램 프로파일러**에 가깝습니다. 실행계획이 "어떤 알고리즘을 쓸 예정인가"라면, 트레이스는 "실제로 어디서 시간(elapsed)과 I/O(cr/pr)가 쓰였는가"를 보여줍니다. "부모는 자식을 포함"하는 누적 통계는 프로파일러의 **inclusive vs self time** 구분과 똑같습니다.

---

## ✅ 이 항목의 핵심 통찰

1. SQL 트레이스는 **추정이 아니라 실제 수행 통계**를 본다 — `alter session set sql_trace=true` → `v$diag_info`로 파일 찾기 → **TKProf**(`sys=no`)로 리포트.
2. Call 통계로 **Parse/Execute/Fetch** 중 어디서 비용이 나는지 분리한다. `Misses in library cache during parse: 1`이면 하드파싱.
3. **논리적 I/O = query(Consistent) + current(Current)** — AutoTrace의 consistent gets/db block gets와 1:1 매핑된다.
4. Row Source의 **`Rows`는 출력(Flow-Out) 건수**, `cr/pr/pw/time`은 단계별 실측치다.
5. ⚠️ **부모 노드 통계는 자식 노드를 포함**하므로, 특정 단계의 순수 비용은 자식 값을 빼서 봐야 한다.

## 🔗 흐름 속 위치

부록.2(AutoTrace)가 SQL 단위 요약 통계였다면, 부록.3은 **Parse/Execute/Fetch와 단계별 cr/pr**까지 분해한 정밀 프로파일입니다. 다음 부록.4(DBMS_XPLAN)는 트레이스 없이도 **캐시에 있는 커서의 실측 실행계획**을 보는 더 가벼운 방법을 제공합니다.

📌 한 줄 요약: SQL 트레이스는 튜닝을 실행계획 해석에서 **실제 수행 프로파일링(query+current, 단계별 cr/time)** 으로 확장하며, "부모는 자식을 포함한다"가 해석의 핵심 규칙입니다.
