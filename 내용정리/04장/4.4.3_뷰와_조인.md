## 4.4.3 뷰와 조인

### 핵심 개념
뷰와 조인 튜닝의 핵심은 **뷰 경계가 조건 이동과 조인 순서 선택을 막는지**를 확인하는 것입니다.

```text
뷰 내부 SQL
   ↓
외부 조건과 분리
   ↓
먼저 큰 집합 생성 가능
   ↓
이후 필터링
```

| 최적화 | 힌트 | 설명 |
|---|---|---|
| **View Merging**(뷰 병합) | `merge` / `no_merge` | 뷰를 풀어 외부 SQL과 합쳐 최적화 |
| **조건절 Pushing**(Predicate Pushdown) | `push_pred` / `no_push_pred` | 외부 조건을 뷰 내부로 밀어넣음 |

### 구체 예제 — 조건이 뷰 바깥에 갇힌 비효율

```sql
select c.고객번호, c.고객명, t.평균거래, t.최소거래, t.최대거래
from   고객 c
     , (select 고객번호, avg(거래금액) 평균거래, min(거래금액) 최소거래, max(거래금액) 최대거래
        from   거래
        where  거래일시 >= trunc(sysdate, 'mm')      -- 당월 거래
        group by 고객번호) t
where  c.가입일시 >= trunc(add_months(sysdate, -1), 'mm')  -- ⚠️ 전월 이후 가입 고객(뷰 바깥)
and    c.고객번호 = t.고객번호;
```
```text
Execution Plan
2   NESTED LOOPS
3    VIEW                                       ← 뷰를 독립적으로 먼저 최적화
4     HASH (GROUP BY)
5      TABLE ACCESS BY INDEX ROWID 거래         ← 당월 '모든' 고객의 거래를 집계!
8    TABLE ACCESS BY INDEX ROWID 고객
```

**문제**: '전월 이후 가입 고객' 필터가 **뷰 바깥**에 있어, 뷰 안에서는 (그 필터와 무관하게) **당월 거래한 모든 고객**의 거래를 집계합니다(불필요한 대량 집계).

**해결**: View Merging이나 조건절 Pushing이 일어나면, 고객 필터가 거래 집계 전에 적용되어 **소수 고객의 거래만** 집계합니다. (옵티마이저가 자동으로 못 하면 `push_pred` 힌트 등으로 유도)

---

## ✅ 이 절(4.4.3)의 핵심 통찰
1. 뷰는 SQL을 보기 좋게 만들지만 최적화 경계를 만들 수 있습니다.
2. 조건이 뷰 내부로 들어가야 읽는 양이 줄어듭니다.
3. View Merging과 Predicate Pushing 여부를 실행계획에서 확인해야 합니다.

📌 한 줄 요약: 뷰 튜닝은 조건과 조인 순서가 뷰 경계에 갇히지 않게 하는 일입니다.
