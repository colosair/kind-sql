## 4.1.7 NL 조인 확장 메커니즘

### 핵심 개념
Oracle은 전통적인 NL 조인의 랜덤 액세스 비용을 줄이기 위해 여러 확장 메커니즘을 사용합니다.

```text
기본 NL
Outer 한 건
   ↓
Inner 한 번 탐색

확장 NL
Outer 여러 건을 모음
   ↓
Inner 접근을 묶거나 최적화
```

| 메커니즘 | 목적 |
|---|---|
| **테이블 Prefetch** | 인덱스로 테이블 액세스 시 **읽을 블록을 미리 읽어** 버퍼캐시에 적재 |
| **배치 I/O (Batch I/O)** | 디스크 I/O Call을 미뤘다 **일정량 쌓이면 한꺼번에** 처리 |

두 기능 모두 "읽는 블록마다 건건이 I/O Call을 발생시키는 비효율"을 줄이기 위한 것입니다.

### 실행계획 표현 변화 (익혀둘 것)

내부 원리는 몰라도, 튜닝 중 이 기능들이 실행계획에 나타나는 **표현 방식**은 알아야 합니다.

```text
① 전통적 표현
   NESTED LOOPS
     TABLE ACCESS BY INDEX ROWID OF 사원
       INDEX RANGE SCAN OF 사원_X1
     TABLE ACCESS BY INDEX ROWID OF 고객      ← Inner 테이블이 NL 아래
       INDEX RANGE SCAN OF 고객_X1

② 테이블 Prefetch (9i~)
   TABLE ACCESS BY INDEX ROWID OF 고객         ← Inner 테이블이 NL '위'로 올라감!
     NESTED LOOPS
       TABLE ACCESS BY INDEX ROWID OF 사원
         INDEX RANGE SCAN OF 사원_X1
       INDEX RANGE SCAN OF 고객_X1

③ 배치 I/O (11g~)
   NESTED LOOPS                                 ← NESTED LOOPS가 '두 번' 나타남
     NESTED LOOPS
       ...
   (TABLE ACCESS ... BATCHED 로 표시되기도)
```

### 정밀화
확장 메커니즘은 NL 조인의 **본질을 바꾸지 않습니다.** 여전히 Outer 기반 반복 탐색이며, 반복 중 발생하는 블록 접근을 더 효율적으로 배치할 뿐입니다. 근본 튜닝은 여전히 **Outer 축소 + Inner 인덱스 효율**입니다.

---

## ✅ 이 절(4.1.7)의 핵심 통찰
1. 확장 NL은 랜덤 액세스 비용을 줄이려는 최적화입니다.
2. 기본 구조는 여전히 Outer 반복 기반입니다.
3. 근본 튜닝은 Outer 축소와 Inner 인덱스 효율 개선입니다.

📌 한 줄 요약: NL 확장 메커니즘은 반복 탐색의 물리 I/O 부담을 줄이는 보조 장치입니다.
