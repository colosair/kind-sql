## 4.3.5 조인 메소드 선택 기준

### 핵심 개념
조인 메소드 선택은 **입력 크기, 인덱스 존재, 조인 조건, 응답성, 메모리**를 함께 보고 결정합니다.

| 기준 | NL 조인 | 소트 머지 조인 | 해시 조인 |
|---|---|---|---|
| 소량 Outer | 유리 | 보통 | 보통 |
| 대량 등치 조인 | 불리 가능 | 가능 | 유리 |
| 비등치 조인 | 가능 | 유리 | 제한적 |
| 첫 행 응답 | 유리 | 불리 | 불리 |
| 메모리 의존 | 낮음 | 정렬 PGA | 해시 PGA |

### 의사결정 트리 (그림 4-18)

```text
소량 데이터 조인? ──Y──► NL 조인
   │ N
   ▼
조인 조건식이 등치(=)? ──Y──► 해시 조인
   │ N
   ▼
소트 머지 조인  (비등치, 또는 조인 조건 아예 없는 카테시안 곱 포함)
```
- ① 소량 데이터 조인 → **NL 조인**
- ② 대량 데이터 + 등치 조인 → **해시 조인**
- ③ 대량 데이터 + 비등치(또는 카테시안 곱) → **소트 머지 조인**

> **소량/대량의 기준**은 단순 데이터량이 아닙니다. **"NL 조인으로 최적화했는데도 랜덤 액세스가 많아 만족스러운 성능이 안 나오면 대량 데이터 조인"** 에 해당합니다.

### ⚠️ 수행빈도 높은 쿼리(OLTP)는 해시 조인 남용 주의

해시 조인이 빠르다 보니 인덱스 설계 없이 해시 조인으로 처리하려는 유혹에 빠지기 쉽지만, **수행빈도가 매우 높은 쿼리에는 위험**합니다. 책의 기준:

```text
① 최적화된 NL 조인과 해시 조인 성능이 같으면  → NL 조인
② 해시 조인이 약간 더 빨라도                   → NL 조인
③ NL 조인보다 해시 조인이 '매우' 빠른 경우만    → 해시 조인
```
이유: 해시 조인은 **수행할 때마다 해시 테이블을 새로 Build**하므로, 동시다발로 수행되면 PGA·Temp 부담이 커집니다. OLTP는 웬만하면 NL 조인이 안전합니다.

---

## ✅ 이 절(4.3.5)의 핵심 통찰
1. 조인 메소드는 절대 우열이 아니라 상황별 선택입니다.
2. NL은 반복 B*Tree 탐색, 해시는 HashMap lookup, 소트 머지는 정렬 배열 merge입니다.
3. 모두 결국 `reduce I/O`와 메모리 활용 방식의 차이입니다.

📌 한 줄 요약: 조인 메소드 선택은 같은 조인 문제를 B*Tree, 정렬 배열, HashMap 중 무엇으로 풀지 고르는 일입니다.
