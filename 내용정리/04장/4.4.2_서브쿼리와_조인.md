## 4.4.2 서브쿼리와 조인

### 핵심 개념

메인쿼리와 서브쿼리는 **부모-자식의 종속·계층 관계**입니다. 서브쿼리는 단독 실행할 수 없고 메인쿼리에 종속됩니다. 옵티마이저는 이를 두 가지 방식으로 처리합니다.

```text
서브쿼리 처리
 ├─ ① 필터 오퍼레이션  : 풀지 않고 메인 건건이 필터 (no_unnest)
 └─ ② 서브쿼리 Unnesting: 메인과 같은 레벨로 풀어 조인 (unnest)
```

---

### ① 필터 오퍼레이션 (no_unnest)

서브쿼리를 풀지 않고, **메인쿼리 건수만큼 값을 받아 반복 필터링**합니다. `no_unnest` 힌트로 강제합니다.

```sql
select * from 고객 c
where c.가입일시 >= trunc(add_months(sysdate, -1), 'mm')
and   exists (select /*+ no_unnest */ 'x' from 거래 t
              where t.고객번호 = c.고객번호 and t.거래일시 >= trunc(sysdate, 'mm'));
```
- **메인쿼리가 항상 드라이빙**(Outer)이 되고, 조인 순서가 **고정**됩니다(일반 NL 조인과 다른 점).
- 비효율: 메인 건수만큼 서브쿼리를 반복 수행.

**💡 서브쿼리 캐싱** — 이 비효율을 완화하는 기능:
> 서브쿼리 **입력 값에 따른 반환 값(true/false)을 PGA 메모리에 캐싱**합니다. 캐시에서 true/false를 찾으면 서브쿼리를 재수행하지 않아 성능이 좋아집니다. 캐싱은 **쿼리 단위**로 이뤄집니다(쿼리 시작 시 PGA 공간 할당, 마치면 반환). → 입력 값 종류(NDV)가 적을수록 효과적입니다.

### ② 서브쿼리 Unnesting (unnest)

서브쿼리를 메인쿼리와 **같은 레벨로 풀어(flat) 조인**으로 변환합니다.

```sql
select c.고객번호, c.고객명 from 고객 c
where c.가입일시 >= trunc(add_months(sysdate, -1), 'mm')
and   exists (select /*+ unnest nl_sj */ 'x' from 거래
              where 고객번호 = c.고객번호 and 거래일시 >= trunc(sysdate, 'mm'));
```
```text
Execution Plan
1   NESTED LOOPS (SEMI)                       ← 세미 조인으로 풀림
2    TABLE ACCESS BY INDEX ROWID 고객
3     INDEX RANGE SCAN 고객_X01
4    INDEX RANGE SCAN 거래_X01
```
> **'un-nest'** = 'nest(중첩)'를 풀어낸다 → 메인·서브쿼리의 계층 구조를 풀어 같은 레벨(flat)로 만드는 것 = **'서브쿼리 Flattening'**. (`nl_sj` = NL 세미 조인, `hash_sj`/`merge_sj`도 가능)

Unnesting되면 옵티마이저가 **조인 순서·방식을 자유롭게** 최적화할 수 있습니다(필터 오퍼레이션은 메인이 고정 드라이빙).

---

### 세미 조인과 안티 조인

| 서브쿼리 | 변환되는 조인 | 의미 |
|------|------|------|
| `EXISTS` / `IN` | **세미 조인(SEMI)** | 매칭 **존재** 여부만 (중복 확장 없음) |
| `NOT EXISTS` / `NOT IN` | **안티 조인(ANTI)** | 매칭 **부재** 여부만 |

실행계획에 `NESTED LOOPS (SEMI/ANTI)`, `HASH JOIN (SEMI/ANTI)`, `MERGE JOIN (SEMI/ANTI)`로 나타납니다.

> **일반 조인과의 차이**: 일반 조인은 매칭되는 행 수만큼 결과가 **늘어날(M배)** 수 있지만, **세미/안티 조인은 존재/부재만 판단**하므로 중복 확장이 없습니다. 그래서 `EXISTS`를 일반 조인으로 잘못 바꾸면 결과가 달라집니다(중복 발생).

---

## ✅ 이 절(4.4.2)의 핵심 통찰

1. 서브쿼리는 **① 필터 오퍼레이션(no_unnest)** 또는 **② Unnesting(unnest)** 으로 처리된다.
2. **필터 오퍼레이션**은 메인을 고정 드라이빙으로 두고 건건이 반복 필터 — **서브쿼리 캐싱**(true/false를 PGA에 캐싱)으로 완화한다.
3. **Unnesting('서브쿼리 Flattening')** 은 서브쿼리를 같은 레벨로 풀어 조인화 → 옵티마이저가 순서·방식을 자유롭게 최적화한다.
4. `EXISTS/IN → 세미 조인`, `NOT EXISTS/NOT IN → 안티 조인`으로 변환된다.
5. 세미/안티 조인은 **존재/부재만 판단**해 일반 조인과 달리 중복 확장이 없다.

## 🔗 흐름 속 위치

4.4.1(변환이 필요한 이유)을 구체화한 절입니다. 여기서 본 세미/안티 조인은 DML 서브쿼리(6.1.1)에서 그대로 쓰이고, 다음 4.4.3(뷰와 조인)·4.4.4(스칼라 서브쿼리)는 다른 종류의 서브쿼리 변환을 다룹니다.

📌 한 줄 요약: 서브쿼리는 **필터 오퍼레이션(반복+캐싱)** 또는 **Unnesting(조인화)** 으로 처리되며, EXISTS/NOT EXISTS는 중복 없는 세미/안티 조인으로 풀립니다.
