## 4.3.3 대용량 Build Input 처리

### 핵심 개념
Build Input이 PGA에 다 들어가지 않으면 Oracle은 **분할·정복(Divide & Conquer)** 방식으로 해시 조인을 수행합니다. 양쪽 모두 대용량이어도 마찬가지입니다.

```text
① 파티션 단계
   양쪽 집합의 조인 컬럼에 해시 함수 적용
   → 같은 해시 값끼리 동적 파티셔닝 → 파티션 짝(pair) 생성
   (큰 파티션은 TEMP에 기록)
   ↓
② 조인 단계
   독립적으로 처리 가능한 작은 파티션 짝 단위로
   메모리에 올려 Build/Probe 수행
```

큰 문제 하나를 독립적으로 풀 수 있는 작은 부분 문제들로 쪼갠다는 점에서 전형적인 Divide & Conquer입니다. 6.3.1 파티션 개념과 연결됩니다.

| 상태 | 결과 |
|---|---|
| 메모리 내 처리(Optimal) | 가장 빠름 |
| One-pass | TEMP 사용 있지만 제한적 |
| Multi-pass | TEMP 반복 사용, 매우 느림 |

### CS 대응
메모리에 HashMap이 다 안 들어가면 외부 해시 조인처럼 데이터를 버킷별 파일로 나누고, 버킷 단위로 다시 로드해 처리합니다.

---

## ✅ 이 절(4.3.3)의 핵심 통찰
1. Build Input은 작을수록 좋습니다.
2. PGA 부족은 TEMP I/O로 전환되어 성능을 크게 떨어뜨립니다.
3. 해시 조인도 메모리 조건이 맞아야 강점을 유지합니다.

📌 한 줄 요약: 해시 조인은 Build Input이 메모리에 들어갈 때 가장 강하고, 넘치면 TEMP 비용이 붙습니다.
