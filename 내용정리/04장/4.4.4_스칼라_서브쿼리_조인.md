## 4.4.4 스칼라 서브쿼리 조인

### 핵심 개념
스칼라 서브쿼리는 **메인 쿼리의 각 행마다 단일 값을 반환하는 서브쿼리**입니다.

```sql
SELECT c.고객번호,
       (SELECT MAX(o.주문일자)
        FROM 주문 o
        WHERE o.고객번호 = c.고객번호) AS 최근주문일자
FROM 고객 c;
```

```text
고객 1건
   ↓
주문에서 MAX 조회
   ↓
고객 다음 건
   ↓
주문에서 MAX 조회 반복
```

| 관점 | 설명 |
|---|---|
| 장점 | 표현이 간결하고 단일 값 조회에 자연스러움 |
| 위험 | 메인 행마다 반복 수행될 수 있음 |
| 개선 | 조인/집계 선처리/캐싱/인덱스 구성 |

### CS 대응
Outer loop 안에서 매번 lookup 함수를 호출하는 구조입니다. lookup이 O(1)에 가깝고 캐시가 잘 되면 괜찮지만, 내부에서 범위 스캔과 집계를 반복하면 비용이 커집니다. 실행 형태는 메인 쿼리를 Outer로 둔 NL 조인(`use_nl`)과 같습니다. 단, **결정적 차이는 캐싱**입니다.

### 스칼라 서브쿼리 캐싱 (Query Execution Cache)

스칼라 서브쿼리로 조인하면 오라클은 조인 횟수를 줄이려고 **입력 값과 출력 값을 PGA 내부 캐시(Query Execution Cache)에 저장**합니다. 메인 쿼리 행을 처리할 때마다,

```text
입력 값(메인 쿼리 참조 컬럼)으로 캐시 조회
   ├─ 캐시에 있음 → 저장된 출력 값 즉시 반환 (서브쿼리 실행 안 함)
   └─ 캐시에 없음 → 서브쿼리 실행 → 결과를 캐시에 저장 후 반환
```

- **입력 값** = 서브쿼리가 참조하는 메인 쿼리 컬럼(예: `e.deptno`)
- **출력 값** = 서브쿼리 반환 값(예: `d.dname`)
- 캐싱은 쿼리 단위로, 쿼리 시작 시 PGA에 공간을 할당하고 끝나면 반환합니다.
- 5.2.2의 필터 서브쿼리 캐싱과 같은 메커니즘입니다.

### 핵심 규칙 — 입력 값 NDV가 적을 때만 효과

캐시 공간은 매우 작습니다(9i는 256개 엔트리, 10g+는 `_query_execution_cache_max_size`로 결정).

| 입력 값 종류(NDV) | 결과 |
|---|---|
| **적음** (예: 거래구분코드 20개) | 최초 20번만 조인, 이후 50,000건은 캐시 히트 → **큰 이득** |
| **많음** (예: 고객번호 100만) | 대부분 캐시 미스 → 매번 캐시 확인 비용까지 더해져 **일반 조인보다 느림**, CPU·메모리 낭비 |

> ⚠️ 캐싱은 공짜가 아닙니다. 입력 값이 다양하면 캐시 확인이 헛수고가 되어 오히려 역효과입니다. "스칼라 서브쿼리 = 항상 캐싱 이득"이 아니라 **"입력 값 종류가 소수일 때만 이득"** 입니다.

### 응용 — SELECT-LIST 함수 호출 줄이기

SELECT-LIST의 사용자 정의 함수는 메인 쿼리 결과 건수만큼 호출됩니다. 이를 스칼라 서브쿼리로 감싸면 캐싱이 적용되어 호출 횟수를 줄일 수 있습니다(입력 값 NDV가 적을 때). 3.3.12 함수호출부하 해소와 연결됩니다.

```sql
-- 함수가 행마다 호출됨
select empno, ename, get_dname(e.deptno) dname from emp e;

-- 스칼라 서브쿼리로 감싸 deptno 종류만큼만 호출
select empno, ename, (select get_dname(e.deptno) from dual) dname from emp e;
```

### 튜닝 방향
```text
반복 스칼라 서브쿼리
   ↓
집계 결과를 먼저 생성
   ↓
조인으로 연결
```

## ✅ 이 절(4.4.4)의 핵심 통찰
1. 스칼라 서브쿼리는 메인 쿼리를 Outer로 둔 NL 조인과 같은 실행 구조입니다.
2. 핵심 차이는 캐싱 — 입력 값/출력 값을 PGA의 Query Execution Cache에 저장해 조인 횟수를 줄입니다.
3. 캐싱은 **입력 값 종류(NDV)가 소수일 때만 이득**이고, 다양하면 캐시 확인 비용 때문에 역효과입니다.
4. SELECT-LIST 함수를 스칼라 서브쿼리로 감싸면 캐싱으로 호출 횟수를 줄일 수 있습니다(3.3.12 연결).
5. 내부 스캔이 크고 입력 값이 다양하면 집계 후 조인으로 재구성하는 것이 낫습니다.

📌 한 줄 요약: 스칼라 서브쿼리의 성능은 캐싱에 달려 있고, 캐싱은 입력 값 종류가 적을 때만 이득입니다.
