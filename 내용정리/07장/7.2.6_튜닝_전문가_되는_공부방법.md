## 7.2.6 튜닝 전문가 되는 공부방법

### 데이터베이스 튜닝이란?

'데이터베이스 튜닝'이라 할 때의 튜닝은 일반적으로 **'성능(Performance) 튜닝'** 을 말합니다. 엄밀히는 우리가 튜닝하는 대상이 데이터 파일·리두로그 파일 같은 '파일'이 아니라 **데이터를 읽고 쓰는 소프트웨어 엔진, 즉 DBMS**이므로, '데이터베이스 튜닝'은 정확히는 **'DBMS 성능 튜닝'** 을 의미합니다.

> DBMS는 데이터를 읽고 쓸 때 SQL을 이용합니다. 따라서 **SQL이 병목 없이 빠르고 안정적으로 수행되도록 조치하는 모든 활동**을 데이터베이스 튜닝이라 정의할 수 있습니다.

데이터베이스 튜닝은 크게 세 가지로 나뉩니다(그림 7-2).

```text
              ③ 인스턴스 튜닝
       SQL                인스턴스(SGA: Buffer/Shared Pool/Log Buffer
   ────────►   ① SQL       + Server Process, DBWn/LGWR/SMON ...)
                튜닝              ▲▼
                            데이터베이스(데이터 파일들)
                              ② DB 설계
```

| 구분 | 내용 |
|------|------|
| **① SQL 튜닝** | I/O 효율화, DB Call 최소화, SQL 파싱 최소화 등 |
| **② DB 설계** | 논리적 데이터 구조 설계, 물리적 저장 구조 설계 등 |
| **③ 인스턴스 튜닝** | Lock/Latch 모니터링 및 해소, 메모리 설정, 프로세스 설정 등 |

---

### 비유 — 좋은 소리는 '소스'와 '공간'에서

> **오디오 튜닝 ('천상의 소리를 찾아서')**: 좋은 음악을 들으려면 장비(스피커·앰프) 투자도 필요하지만, 사실 **음원(소스)이 가장 중요**합니다. 그리고 소리 울림을 담당하는 **'공간'** 도 중요합니다.

> **데이터베이스 튜닝 ('전광석화처럼 빠르게')**: 마찬가지로 **좋은 소스 + 좋은 공간**이 중요합니다.
> - **좋은 소스 = 효율적으로 작성한 SQL** — *Garbage In, Garbage Out!* (나쁜 SQL을 넣으면 나쁜 성능이 나온다)
> - **좋은 공간 = 좋은 데이터 구조** (빵빵한 하드웨어가 아닙니다)

좋은 하드웨어가 필요 없다는 뜻은 결코 아닙니다. 필요하지만, **좋은 소스(SQL)와 좋은 데이터 구조가 선행해야** 합니다 — 아무리 좋은 스피커·앰프를 갖춰도 음반·녹음이 나쁘면, 그리고 울림 공간이 나쁘면 좋은 사운드를 재생할 수 없는 것과 같습니다.

---

### 튜닝 전문가가 되려면 무엇을, 어떻게 공부하는가

대부분 SI 프로젝트는 두 팀을 구성합니다 — **DBA팀**과 **SQL 튜닝팀**. 어느 팀에서 일하고 싶은지(역할)에 따라 공부방법이 달라집니다. (각주 1: 소규모 프로젝트는 DBA팀이 두 역할을 겸하기도, 대규모는 성능검증/테스트팀을 따로 두기도 함. DB·SQL뿐 아니라 WAS·OS·네트워크·스토리지 전문가도 참여)

**DBA팀** — 데이터베이스 **자체**에 대한 연구가 우선.
- 설치, 백업/복구, 오브젝트 생성/변경, 보안 등 DB로서 갖추어야 할 기본 기술
- 아키텍처를 완벽히 숙지하고, 운영 중 생기는 여러 장애를 **모니터링·해결하는 기술력**과 자신만의 스크립트 개발

**SQL 튜닝팀** — 데이터베이스 자체보다 **SQL 중심**으로 공부. 필요한 지식·기술력은 다음 7가지입니다.

```text
1. 옵티마이저가 SQL을 파싱하고 통계정보를 활용해 실행계획을 생성하는 원리   (7.1)
2. 옵티마이저 쿼리변환 원리를 바탕으로 실행계획을 분석하는 방법
3. 옵티마이저 힌트를 이용해 실행계획을 제어하는 방법                       (7.2.3)
4. 옵티마이저가 좋은 실행계획을 생성하도록 유도하기 위한 효과적인 SQL 작성법  (7.2.5)
5. 애플리케이션에서 SQL을 실행할 때 사용하는 프로그래밍 인터페이스          (6.1.2)
6. SQL을 빠르게 처리할 수 있는 좋은 데이터 구조와 파티션/인덱스 설계        (2·3·6장)
7. 정확성과 안정성을 확보할 수 있는 통계정보 수집 정책                     (7.1.2)
```

> SQL이 뷰·서브쿼리로 복잡하게 얽히면 실행계획 분석·튜닝도 매우 복잡하고 어렵습니다. 힌트만으로 튜닝에 성공하면 다행이지만, 복잡하게 얽힌 SQL은 **다른 구조로 변환**해야 하며 디버깅을 통해 **결과가 맞는지도 반드시 확인**해야 합니다. ("틀린 결과를 빠르게 조회하도록 돕는 것은 튜닝이 아니다.")

> SQL 튜닝의 이런 어려움 때문인지, 데이터베이스 튜닝 전문가가 되고 싶다면서 정작 **SQL 다루기를 등한시**하는 분이 많습니다. DB 다루는 기술력은 인정받는데 SQL 다루는 기술력에선 좀처럼 확신을 주지 못합니다.

---

### SQL 튜닝이 곧 데이터베이스 튜닝

> 데이터베이스 튜닝은 **SQL이 병목·지연 없이 빠르고 안정적으로 수행되도록 조치하는 모든 활동**입니다. 따라서 **SQL과 친숙해지지 않고 데이터베이스 튜닝 전문가가 되겠다는 생각은 버려야** 합니다. 수많은 SQL과 실행계획을 분석해 보지 않고 SQL 튜닝 전문가가 되는 길은 요원합니다.
>
> 데이터베이스 튜닝 전문가가 되고 싶다면, 데이터베이스를 움직이게 하는 소스인 **SQL과 좋은 데이터 구조를 설계하는 방법**에 관심을 두고 더 많은 시간과 노력을 기울이기 바랍니다.

### CS 대응

튜닝은 DB 전용 암기가 아니라 **시스템 성능 분석의 응용**입니다 — 자료구조 선택, 메모리 locality·캐시, 외부 정렬, 해시 파티셔닝, 비용 기반 탐색이 모두 연결됩니다. 책 전체를 관통한 `reduce I/O`가 "좋은 소스(SQL) + 좋은 공간(데이터 구조)"으로 구체화된 것이 7.2.6의 결론입니다.

---

## ✅ 이 절(7.2.6)의 핵심 통찰

1. 데이터베이스 튜닝 = **DBMS 성능 튜닝**이며, **① SQL 튜닝 ② DB 설계 ③ 인스턴스 튜닝** 세 가지로 나뉜다.
2. 좋은 성능은 **좋은 소스(효율적 SQL) + 좋은 공간(데이터 구조)** 에서 나온다 — *Garbage In, Garbage Out*. 하드웨어는 그다음이다.
3. 진로에 따라 **DBA팀**(DB 자체·운영·장애)과 **SQL 튜닝팀**(SQL 중심)으로 공부가 갈린다.
4. SQL 튜닝팀이 익힐 **7가지**: 실행계획 생성 원리·분석법·힌트 제어·SQL 작성법·프로그래밍 인터페이스·데이터 구조 설계·통계 수집 정책.
5. **SQL 튜닝이 곧 데이터베이스 튜닝** — SQL과 친숙해지지 않고는 전문가가 될 수 없다. 수많은 SQL·실행계획을 직접 분석하는 것이 유일한 길이다.

## 🔗 흐름 속 위치

7장의, 그리고 책 전체의 마지막 절입니다. 1~6장이 SQL 튜닝의 구체 기법(I/O·인덱스·조인·소트·DML)이었고 7장이 그 위의 옵티마이저 이해였다면, 7.2.6은 이 모두를 **"무엇을·어떻게 공부해 전문가가 되는가"** 로 묶어 마무리합니다. 7가지 학습 항목이 곧 이 책 전체의 목차이기도 합니다.

📌 한 줄 요약: 데이터베이스 튜닝은 곧 **SQL 튜닝**이며, 전문가가 되는 길은 좋은 소스(SQL)와 좋은 공간(데이터 구조)에 대해 **수많은 실행계획을 직접 분석**하며 7가지 역량을 쌓는 것입니다.
