## 7장 SQL 옵티마이저 전체 흐름

### 전체 축
7장은 앞 장에서 배운 모든 튜닝 원리를 옵티마이저 관점에서 다시 묶습니다. 핵심은 **옵티마이저가 통계정보를 이용해 비용을 추정하고, 그 결과로 실행계획을 선택한다**는 점입니다.

```text
SQL 문장
   ↓
통계정보 확인
   ↓
선택도/카디널리티 추정
   ↓
액세스 경로와 조인 방식 후보 생성
   ↓
비용 계산
   ↓
실행계획 선택
```

## 7.1 통계정보와 비용 계산

### 핵심 개념
옵티마이저 판단은 행 수 예측에서 시작합니다.

| 절 | 핵심 |
|---|---|
| 7.1.1 | 선택도는 비율, 카디널리티는 예상 행 수 |
| 7.1.2 | 통계정보는 데이터 분포 요약 지도 |
| 7.1.3 | 비용은 예상 I/O와 CPU 기반 상대값 |

```text
조건절
   ↓
선택도
   ↓
카디널리티
   ↓
비용
```

### 앞 장과의 연결
| 앞 장 개념 | 옵티마이저가 보는 방식 |
|---|---|
| 인덱스 손익분기점 | 인덱스 비용 vs Full Scan 비용 |
| 클러스터링 팩터 | ROWID 테이블 액세스 비용 |
| 조인 방식 | NL/Hash/Merge 후보 비용 |
| 소트 연산 | CPU/PGA/TEMP 비용 |
| 파티션 | 읽을 파티션 수와 블록 수 |

## 7.2 옵티마이저 이해와 개발자의 역할

### 핵심 개념
옵티마이저는 제한된 정보로 최선의 실행계획을 고르는 엔진입니다. 개발자는 이 엔진이 좋은 판단을 하도록 SQL과 환경을 정돈해야 합니다.

| 절 | 핵심 |
|---|---|
| 7.2.1 | 현대 튜닝의 중심은 CBO |
| 7.2.2 | `ALL_ROWS`와 `FIRST_ROWS_N`의 목표 차이 |
| 7.2.3 | 통계, 인덱스, 파라미터, 힌트, 바인드가 계획에 영향 |
| 7.2.4 | 옵티마이저는 통계와 가정의 한계를 가짐 |
| 7.2.5 | 개발자는 옵티마이저의 입력과 선택지를 정돈 |
| 7.2.6 | 튜닝 전문가는 실행계획을 I/O 흐름으로 해석 |

```text
옵티마이저가 나쁜 계획 선택
   ↓
통계정보가 틀렸는가?
   ↓
SQL 표현이 변환을 막는가?
   ↓
인덱스 선택지가 부적절한가?
   ↓
바인드/히스토그램/skew 문제가 있는가?
```

## ✅ 7장의 핵심 통찰
1. 옵티마이저는 SQL 실행 전에 가능한 계획들의 비용을 추정합니다.
2. 비용 추정의 출발점은 선택도와 카디널리티입니다.
3. 통계정보가 틀리면 인덱스, 조인, 소트, 파티션 판단이 모두 흔들립니다.
4. `ALL_ROWS`와 `FIRST_ROWS_N`은 전체 처리량과 초기 응답시간 중 무엇을 우선할지의 차이입니다.
5. 옵티마이저의 한계는 제한된 통계와 가정 위에서 미래 비용을 예측해야 한다는 데 있습니다.
6. 개발자의 역할은 옵티마이저가 좋은 계획을 선택할 수 있도록 SQL, 인덱스, 통계정보를 정돈하는 것입니다.
7. 책 전체의 통일 원리인 `reduce I/O`는 옵티마이저 비용 모델 안에서 실행계획 선택 기준으로 드러납니다.

## 🔗 흐름 속 위치
1장은 SQL 성능의 본질을 I/O로 설명했고, 2~3장은 인덱스로 I/O를 줄이는 방법을 다뤘습니다. 4장은 조인, 5장은 소트, 6장은 DML과 파티션까지 확장했습니다. 7장은 이 모든 선택을 옵티마이저가 어떻게 판단하는지 정리하는 마무리 장입니다.

📌 한 줄 요약: 7장은 옵티마이저를 “SQL을 실행계획으로 바꾸는 비용 기반 의사결정 엔진”으로 이해하게 만드는 장입니다.
