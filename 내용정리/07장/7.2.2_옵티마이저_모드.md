## 7.2.2 옵티마이저 모드

### 핵심 개념 — 내비게이션의 검색 모드

내비게이션이 경로를 보여주기 전에 **검색 모드**(고속도로 우선=이동시간 단축 / 일반도로 우선=통행료 절감 / 최단거리=유류비 절감)를 고르듯, 옵티마이저도 **최적화 목표**를 설정하는 모드가 있습니다.

| 모드 | 목표 | 비유 |
|------|------|------|
| **ALL_ROWS** | 전체 결과 처리속도 최적화 | "전체 이동시간 단축" |
| **FIRST_ROWS** | 최초 응답속도 최적화 | (⚠️ deprecated, 사용 금지) |
| **FIRST_ROWS_N** | 최초 N건 응답속도 최적화 | "앞부분 빨리 보여주기" |

```text
ALL_ROWS      → 결과 전체를 읽는다는 전제 → 시스템 리소스(I/O·CPU·메모리) 최소
FIRST_ROWS_N  → 앞쪽 N건만 읽고 멈춘다는 전제 → 초기 응답 최소
```

---

### ALL_ROWS — 전체 처리속도

`ALL_ROWS`로 설정하면 옵티마이저는 **'결과집합 전체를 읽는 것'을 전제로** 시스템 리소스(I/O, CPU, 메모리 등)를 가장 적게 사용하는 실행계획을 선택합니다. 즉 **전체 처리속도**가 목표입니다.

### FIRST_ROWS_N — 최초 N건 응답속도

`FIRST_ROWS_N`은 사용자가 **'앞쪽 N개 로우만 읽고 멈추는 것'을 전제로** 응답 속도가 가장 빠른 실행계획을 선택합니다. `ALL_ROWS`와 비교하면 **Table Full Scan보다 인덱스를, 해시 조인·소트 머지 조인보다 NL 조인을 더 많이** 선택하는 경향이 있습니다(부분범위 처리·소트 생략에 유리한 방향).

```sql
-- 시스템/세션 레벨: N으로 지정 가능한 값은 1, 10, 100, 1000 네 가지
alter session set optimizer_mode = first_rows_1;
alter session set optimizer_mode = first_rows_10;
alter session set optimizer_mode = first_rows_100;
alter session set optimizer_mode = first_rows_1000;

-- 힌트 레벨: 괄호 안에 0보다 큰 정수면 무엇이든 가능
select /*+ first_rows(30) */ col1, col2, col3 from t where ...;
```

### ⚠️ FIRST_ROWS는 왜 쓰면 안 되나

`FIRST_ROWS`는 사용자가 데이터를 **어느 정도 읽다 멈출지를 지정하지 않으므로** 정확한 비용 산정이 어렵습니다. 반면 `FIRST_ROWS_N`은 **읽을 데이터 건수(N)를 명시**하므로 더 정확한 비용 산정이 가능합니다 — 이것이 `FIRST_ROWS_N`이 `FIRST_ROWS`보다 더 완벽한 CBO 모드로 작동하는 이유이며, `FIRST_ROWS`가 deprecated된 이유입니다.

| 구분 | FIRST_ROWS | FIRST_ROWS_N |
|------|------|------|
| 멈출 지점 | 미지정 | N건 명시 |
| 비용 산정 | 부정확 | 정확 |
| 상태 | **deprecated** | 권장 |

### ⚠️ 모드 설정이 중요한 이유

> 예를 들어 100만 건 결과집합 **전체를 파일로 저장**하려는데 모드가 `FIRST_ROWS`로 설정돼 있으면, 옵티마이저는 인덱스로 소트 연산을 생략하는 실행계획을 택합니다. (Order By 컬럼에 인덱스가 있으면 무조건 인덱스를 쓰는 RBO와 비슷) 그런데 사용자가 중간에 멈추지 않고 **결과를 끝까지 다 읽으면**, 테이블 스캔에 비해 전체 수행 속도가 **오히려 더 느려집니다.** 작업 성격(앞부분만? 전체?)에 모드를 맞춰야 합니다.

---

## ✅ 이 절(7.2.2)의 핵심 통찰

1. 옵티마이저 모드는 **전체 처리속도(ALL_ROWS) vs 최초 응답속도(FIRST_ROWS_N)** 의 선택이다.
2. `ALL_ROWS`는 결과 전체를 읽는 전제로 리소스를 최소화하고, `FIRST_ROWS_N`은 앞 N건만 읽는 전제로 인덱스·NL 조인을 선호한다.
3. **`FIRST_ROWS`는 deprecated** — 멈출 지점을 모르니 비용 산정이 부정확하다. **`FIRST_ROWS_N`(N 명시)** 을 써야 비용 산정이 정확하다.
4. `first_rows_N`은 시스템/세션에서 N∈{1,10,100,1000}, 힌트 `first_rows(n)`은 양의 정수 자유 지정.
5. 작업 성격과 모드가 어긋나면(전체 처리인데 FIRST_ROWS) **오히려 느려진다** — 모드 설정 자체가 튜닝이다.

## 🔗 흐름 속 위치

7.2.1이 RBO의 정렬 비용 맹점(Order By 인덱스 무조건)을 지적했다면, 7.2.2의 `ALL_ROWS`/`FIRST_ROWS_N`은 CBO가 그 문제를 **목표 모드로 명시적으로 구분**하는 방식입니다. 3.2(부분범위 처리)와 직결됩니다.

📌 한 줄 요약: 옵티마이저 모드는 "전체를 빨리 끝낼 것인가(ALL_ROWS), 앞 N건을 빨리 보여줄 것인가(FIRST_ROWS_N)"의 선택이며, 멈출 지점을 모르는 FIRST_ROWS는 쓰지 않습니다.
