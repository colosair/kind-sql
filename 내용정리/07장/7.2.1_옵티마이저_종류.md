## 7.2.1 옵티마이저 종류

### 핵심 개념 — 자동차 내비게이션 비유

> SQL 옵티마이저와 자동차 내비게이션은 여러모로 흡사합니다.

옵티마이저는 SQL을 어떤 방식으로 실행할지 결정하는 엔진입니다. 두 종류가 있습니다.

| 구분 | 판단 기준 | 내비게이션 비유 |
|------|------|------|
| **RBO** (Rule-Based) | 정해진 **규칙 우선순위** | "고속도로 → 일반도로" 같은 **고정 규칙**으로 길 안내 |
| **CBO** (Cost-Based) | **통계정보 기반 비용** | **실시간 교통상황·사고·공사** 정보까지 반영해 가장 빠른 길 안내 |

```text
SQL 문장
   ↓
후보 실행계획 도출
   ↓
딕셔너리 통계로 예상 비용 산정 (CBO)
   ↓
가장 낮은 비용의 실행계획 선택
```

> "자동차 내비게이션도 내부적으로 규칙을 사용합니다 — 목적지까지 고속도로로 최대한 가까이 이동한 후, 자동차 전용도로, 시내 주요간선도로, 일반도로, 골목도로 순으로 찾아갑니다. 이런 단순 규칙도 어느 정도 보편타당하지만, **가장 빠른 길로 안내하려면 실시간 교통상황·사고·공사 정보를 활용**해야 합니다."

이 "실시간 정보"가 곧 **통계정보**이고, 그것을 쓰는 것이 CBO입니다.

---

### RBO(규칙기반 옵티마이저)와 그 한계

RBO는 액세스 경로에 대한 **우선순위 규칙**에 따라 실행계획을 만듭니다. **통계정보를 전혀 활용하지 않고** 단순 규칙에만 의존하므로 대용량 데이터 처리에 부적합합니다. 인덱스 구조·연산자·조건절 형태가 순위를 결정합니다.

| 순위 | 액세스 경로 (요약) |
|:---:|------|
| 1 | Single Row by Rowid |
| 4 | Single Row by Unique or Primary Key |
| 8 | Composite Index |
| 9 | Single-Column Indexes |
| 10 | **Bounded Range Search**(닫힌 범위, BETWEEN) on Indexed Columns |
| 11 | **Unbounded Range Search**(열린 범위, 부등호) on Indexed Columns |
| 14 | ORDER BY on Indexed Column |
| 15 | **Full Table Scan** |

[표 7-5] — 총 15순위. 핵심은 **Full Table Scan이 항상 꼴찌(15위)** 라는 점입니다 → RBO는 인덱스가 있으면 거의 무조건 인덱스를 씁니다.

**RBO 실패 예제 ① — 선택도 무시**
```sql
select * from 고객 where 고객유형코드 = 'CC0123';
```
고객유형코드에 인덱스가 있으면 RBO는 **무조건 인덱스**를 씁니다(FTS가 15위라서). 하지만 'CC0123'이 전체의 90%를 차지한다면 인덱스는 결코 좋은 선택이 아닙니다(3.1.3 손익분기점). RBO는 **데이터 분포를 모릅니다.**

**RBO 실패 예제 ② — 정렬 비용 무시**
```sql
select * from 고객 order by 고객명;
```
고객명에 인덱스가 있으면 RBO는 무조건 인덱스를 씁니다(ORDER BY on Indexed=14위 > FTS=15위). **부분범위 처리**가 가능하면 소트 생략으로 유리하지만, 그렇지 못한 상황에서 인덱스로 전체 레코드를 액세스하는 것은 나쁜 선택입니다. → CBO는 `ALL_ROWS`/`FIRST_ROWS` 두 모드로 이를 구분합니다(7.2.2).

**RBO 실패 예제 ③ — 어느 조건이 더 효과적인지 모름**
```sql
select * from 사원 where 연령 >= 60 and 연봉 between 3000 and 6000;
```
RBO 규칙상 BETWEEN(닫힌 범위, 10위)이 부등호(열린 범위, 11위)보다 우선순위가 높아 **연봉 인덱스**를 씁니다. 하지만 실제로는 60세 이상 사원이 연봉 3000~6000 사원보다 훨씬 적습니다 — **연령 인덱스가 더 효과적**입니다. RBO는 이를 모릅니다.

---

### CBO(비용기반 옵티마이저)

CBO는 후보 실행계획을 도출하고, **데이터 딕셔너리에 수집된 통계정보**(데이터량·컬럼 값 종류/분포·인덱스 높이·클러스터링 팩터 등)로 각 후보의 예상 비용을 산정해 **최저 비용**을 선택합니다.

> "RBO의 한계 때문에 CBO가 탄생했다. 따라서 대용량·초대용량을 넘어 빅데이터 시대에 사는 우리에게 CBO 선택은 필수다."

CBO를 이해하려면 앞 절의 **선택도·카디널리티(7.1.1)·통계정보(7.1.2)·비용 계산(7.1.3)** 을 함께 이해해야 합니다. 그리고 개발자/DBA는 **데이터·시스템 특성을 파악해 통계 수집 정책을 수립**함으로써, 옵티마이저가 스스로 최적의 결정을 하도록 돕는 데 역량을 집중해야 합니다.

---

## ✅ 이 절(7.2.1)의 핵심 통찰

1. 옵티마이저는 내비게이션과 같다 — **RBO는 고정 규칙**, **CBO는 실시간 정보(통계) 기반**이다.
2. RBO는 **Full Table Scan을 항상 꼴찌(15위)** 로 둬, 인덱스가 있으면 거의 무조건 인덱스를 쓴다.
3. RBO의 3대 맹점: **선택도(90% 값에도 인덱스), 정렬 비용(부분범위 아닌데 인덱스), 조건 효과(연령 vs 연봉)** 를 전혀 모른다 — 데이터 분포를 안 보기 때문.
4. CBO는 이 맹점을 **통계 기반 비용 비교**로 극복하므로, 대용량 환경에서 필수다.
5. 개발자의 핵심 역할은 **좋은 통계 수집 정책으로 CBO가 옳게 판단하게** 만드는 것이다.

## 🔗 흐름 속 위치

7.1이 "비용을 어떻게 계산하는가"였다면, 7.2.1은 "그 비용으로 판단하는 주체(CBO)는 무엇이며 왜 RBO를 대체했는가"입니다. 다음 7.2.2는 CBO가 가진 두 모드(ALL_ROWS/FIRST_ROWS)로 이어집니다.

📌 한 줄 요약: RBO는 데이터를 안 보고 규칙으로만 길을 정하다 실패하며, CBO는 통계라는 실시간 정보로 여러 실행계획의 예상 비용을 비교해 고릅니다.
