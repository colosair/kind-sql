## 7.2.5 개발자의 역할

### 핵심 개념 — "이 시스템은 참고용으로 사용하십시오"

> *"안내사항이 실제 도로상황과 다를 수 있으니 이 시스템은 참고용으로 사용하십시오."* — 내비게이션을 켤 때 나오는 경고입니다. 내비게이션 스스로 경고하듯, **운전자는 본인의 판단과 선택으로 운전해야** 합니다.

데이터베이스도 마찬가지입니다. **불완전한 옵티마이저(7.2.4)에 의존만 할 것이 아니라, 개발자 스스로 옵티마이저가 되어야** 합니다.

> "능력이 없어 맡기는 것이 아니라, 바빠서 맡긴다고 생각하기 바란다. 바쁘니 일단 옵티마이저에게 맡기지만, 그 결과물이 올바른지 **실행계획을 통해 늘 점검**하고, 개선할 여지는 없는지 찾으려고 노력해야 한다."

### "SQL 성능도 내가 지킨다" — 장인정신

RDBMS 이전, 데이터 추출 프로그램을 일일이 로직으로 개발하던 시절엔 성능이 느리면 **본인의 역할**로 인식했습니다. 그런데 RDBMS 환경 개발자들은 "결과만 올바르게 나오면 내 역할이고, SQL 성능은 DBA·튜닝팀 역할"이라 생각하는 경향이 있습니다 — **실행계획 확인 방법조차 모르는 개발자**도 많습니다(자기 위치를 모른 채 내비에만 의존하는 운전자와 같습니다).

> (일본 DB 전문가 일화) "내 프로그램은 내가 지킨다"는 **장인정신**이 투철해, 외부 전문가의 조언을 싫어할 만큼 스스로 품질 향상에 노력합니다. 한국 개발자도 장인정신이 있지만 SQL에 대해서만큼은 조금 부족합니다 — **"SQL 성능도 내가 지킨다"** 는 장인정신이 필요합니다.

> 고성능·고효율 DB 애플리케이션을 구축하려면 **소수의 DBA·튜너보다 다수 개발자의 역할이 더 중요**합니다. 새 개발 언어를 익히는 노력 이상으로 **SQL 수행원리와 튜닝방법**을 공부해야 하는 이유입니다.

## 개발자가 할 일 세 가지

```text
① 필요한 최소 블록만 읽도록 쿼리 작성   (SQL 자체)
② 최적의 옵티마이징 팩터 제공            (물리적 환경)
③ 필요하다면, 옵티마이저 힌트로 유도      (마지막 수단)
```

### (1) 필요한 최소 블록만 읽도록 쿼리 작성

SQL 작성자는 결과집합을 논리적으로 잘 정의하고, 그 결과집합을 만드는 데 DB 프로세스가 **최소의 일반 하드록 쿼리**를 효율적으로 작성하는 것이 중요합니다 — `reduce I/O`의 SQL 버전입니다.

**웹 게시판 페이징 쿼리** 예제로, 흔한 두 가지 비효율을 봅니다.

```sql
-- ❌ 비효율: 출력 대상이 아닌 게시물에도 함수·스칼라서브쿼리·조인 수행
SELECT *
FROM ( SELECT ROWNUM NO, 등록일자, 번호, 제목, 회원명, 게시판유형명, 질문유형명
            , 아이콘, 댓글개수
       FROM ( SELECT A.등록일자, A.번호, A.제목, B.회원명, C.게시판유형명, D.질문유형명
                   , GET_ICON(A.질문유형코드) 아이콘, (SELECT ... FROM ...) 댓글개수
              FROM 게시판 A, 회원 B, 게시판유형 C, 질문유형 D
              WHERE A.게시판유형 = :TYPE AND B.회원번호 = A.작성자번호
              AND   C.게시판유형 = A.게시판유형 AND D.질문유형 = A.질문유형
              ORDER BY A.등록일자 DESC, A.질문유형, A.번호 )
       WHERE ROWNUM <= (:page * 10) )
WHERE NO >= (:page-1) * 10 + 1;
```
숨은 비효율 두 가지:
1. **화면에 출력할 10건이 아닌 게시물 전체**에 대해 `GET_ICON` 함수와 댓글개수 스칼라 서브쿼리를 수행
2. **회원·게시판유형·질문유형 테이블과의 조인** — 데이터 모델상 출력 대상(10건)을 확정한 **후에 조인해도 되는** 테이블

```sql
-- ✅ 개선: ROWNUM으로 페이지 10건을 먼저 확정한 뒤, 그 10건에만 함수·조인 수행
SELECT /*+ ORDERED USE_NL(B) USE_NL(C) USE_NL(D) */
       A.등록일자, A.번호, A.제목, B.회원명, C.게시판유형명, D.질문유형명
     , GET_ICON(A.질문유형코드) 아이콘, (SELECT ... FROM ...) 댓글개수   -- 최종 10건에만
FROM ( SELECT 등록일자, 번호, 제목, 작성자번호, 게시판유형, 질문유형
       FROM ( SELECT ... , ROWNUM NO
              FROM 게시판
              WHERE 게시판유형 = :TYPE ...
              ORDER BY 등록일자 DESC, 질문유형, 번호 )
       WHERE ROWNUM <= (:page * 10) ) A   -- 먼저 10건 확정
     , 회원 B, 게시판유형 C, 질문유형 D    -- 그 10건에만 NL 조인
WHERE NO >= (:page-1)*10 + 1
AND   B.회원번호 = A.작성자번호 ...;
```

> **통계정보에 담기 힘든 업무적 특성**(예: "사용자는 한 페이지 10건만 본다")까지 고려해 SQL을 효과적으로 작성하는 일은 **누가 뭐래도 개발자 역할**입니다.

### (2) 최적의 옵티마이징 팩터 제공

Optimizing(최적화)은 옵티마이저가 수행하지만, **잘할 수 있도록 적절한 수단(팩터)을 제공하는 것은 사용자 몫**입니다. (1)에서 최소 블록 쿼리를 작성해 공을 옵티마이저에 넘겼더라도, 옵티마이저는 인덱스·파티션·클러스터를 **실시간으로 만들어 내며** 최적화할 수는 없습니다 — 물리적 DB 구성은 사람의 몫입니다.

| 옵티마이징 팩터 | 핵심 | 군대 비유 |
|------|------|------|
| **전략적인 인덱스 구성** | 가장 기본. 인덱스는 **SQL 조건절 기준으로** 설계. 어떤 테이블을 어떤 조건으로 자주 액세스하는지는 **DBA보다 개발자가 잘 안다** | "전장 나가는 병사에게 무기를 안 주고 싸우라?" |
| **DBMS 기능 활용** | 파티션·클러스터·IOT·MV·Result Cache 적극 활용 | "탱크·박격포 미는 적에게 소총 하나?" (1억 건 중 100만 건 집계를 인덱스 하나로 몇 초에?) |
| **옵티마이저 모드 설정** | 전략·목표를 명확히 (7.2.2) | "직진" 공격 명령처럼 목적이 분명해야 |
| **정확·안정적 통계정보** | 옵티마이저 판단의 정보력 (7.1.2) | 정보 부족하면 10만 대군에 20~30명이 무모하게 돌진 |

> 정리하면: ① 옵티마이저 모드 포함 각종 **파라미터를 적절히 설정 + 통계를 잘 수집**하는 것이 기본 → 이 위에 ② **전략적 인덱스 구성**이 필수로 뒷받침 → 그리고 ③ **DBMS 기능을 적극 활용**해 옵티마이저에 다양한 선택지를 제공.

### (3) 필요하다면, 옵티마이저 힌트로 최적 액세스 경로 유도

옵티마이징 팩터를 다 제공했으면 옵티마이저에 맡기고 싶지만, **옵티마이저에는 한계가 있습니다(7.2.4)**. 옵티마이저가 최적 실행계획을 못 세울 때 개발자가 **힌트로 직접 액세스 경로를 선택**합니다. 이미 최적으로 실행 중이어도, **절대 다른 방식으로 바뀌지 않게 실행계획을 고정**해야 하는 시스템도 있습니다.

> DB 애플리케이션 개발자라면 **옵티마이저가 미처 생각하지 못한 최적의 액세스 경로를 찾아내고, 실행계획을 그 방식으로 유도할 수 있는 능력을 반드시 갖춰야** 합니다.

---

## ✅ 이 절(7.2.5)의 핵심 통찰

1. 옵티마이저는 내비처럼 **"참고용"** — 개발자가 스스로 옵티마이저가 되어 실행계획을 늘 점검해야 한다. **"SQL 성능도 내가 지킨다"** 는 장인정신이 필요하다.
2. 고성능 DB 앱은 소수 DBA보다 **다수 개발자의 역할**에 달려 있고, 개발자는 SQL 수행원리·튜닝을 새 언어 배우듯 공부해야 한다.
3. 개발자의 일 ①: **필요한 최소 블록만 읽는 쿼리** — 페이징은 10건을 먼저 확정한 뒤 함수·스칼라서브쿼리·조인을 수행한다(업무 특성은 통계가 모른다).
4. 개발자의 일 ②: **최적의 옵티마이징 팩터 제공** — 전략적 인덱스(개발자가 조건을 가장 잘 안다) + DBMS 기능(파티션/MV/Result Cache) + 모드 + 통계.
5. 개발자의 일 ③: **힌트로 최적 경로 유도·고정** — 옵티마이저의 한계를 보완하는 마지막 수단.

## 🔗 흐름 속 위치

7.2.4에서 본 옵티마이저의 한계에 대한 답이 7.2.5입니다. 여기서 말하는 세 가지 일은 곧 **2~6장 전체의 요약**입니다 — 최소 블록 쿼리(1~5장), 인덱스·파티션·MV(2·3·6장), 힌트(전 장). 다음 7.2.6은 이 역량을 어떻게 **공부**해 전문가가 되는지로 마무리합니다.

📌 한 줄 요약: 개발자의 역할은 옵티마이저를 억누르는 것이 아니라, **최소 블록 쿼리 + 옵티마이징 팩터 + (필요시) 힌트**로 옵티마이저가 좋은 선택을 하도록 판을 정리하는 것이며, 그 바탕은 "SQL 성능도 내가 지킨다"는 장인정신입니다.
