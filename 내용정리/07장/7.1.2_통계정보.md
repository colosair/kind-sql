## 7.1.2 통계정보

### 핵심 개념

통계정보는 옵티마이저가 비용을 계산하기 위해 사용하는 **데이터 분포 요약 정보**입니다. 7.1.1에서 본 선택도·카디널리티의 입력값(NDV 등)이 모두 여기서 나옵니다. 크게 **오브젝트 통계**와 **시스템 통계**로 나뉩니다.

```text
통계정보
├─ 오브젝트 통계
│   ├─ 테이블 통계  : NUM_ROWS, BLOCKS, AVG_ROW_LEN ...
│   ├─ 인덱스 통계  : BLEVEL, LEAF_BLOCKS, CLUSTERING_FACTOR ...
│   └─ 컬럼 통계    : NUM_DISTINCT(NDV), DENSITY, 히스토그램 ...
└─ 시스템 통계      : CPU 속도, Single/Multiblock I/O 속도 ...
```

핵심 인식: 옵티마이저는 **실제 데이터를 매번 다 읽어보지 않습니다.** 미리 수집해 둔 통계라는 "지도"를 보고 비용을 추정합니다. 따라서 통계가 실제와 다르면(노후화·skew) 비용 계산이 통째로 틀어져 비효율적인 실행계획이 나옵니다.

---

### (1) 테이블 통계

```sql
-- 수집
begin
  dbms_stats.gather_table_stats('scott', 'emp');
end;
/
-- 조회 (all_tab_statistics 뷰에서도 동일)
select num_rows, blocks, avg_row_len, sample_size, last_analyzed
from   all_tables
where  owner = 'SCOTT' and table_name = 'EMP';
```

| 통계항목 | 의미 |
|------|------|
| **NUM_ROWS** | 테이블에 저장된 총 레코드 개수 |
| **BLOCKS** | '사용된' 익스텐트(데이터가 한 건이라도 입력된 적 있는 모든 익스텐트)에 속한 총 블록 수 (※ '할당된' 총 블록 수는 `dba_segments`/`user_segments` 뷰에서 확인) |
| AVG_ROW_LEN | 레코드당 평균 길이(Bytes) |
| SAMPLE_SIZE | 샘플링한 레코드 수 |
| LAST_ANALYZED | 통계정보 수집일시 |

[표 7-1] — `BLOCKS`(사용된 블록)는 Full Scan 비용 추정의 직접 입력값입니다.

---

### (2) 인덱스 통계

```sql
-- 인덱스 통계만 수집
begin dbms_stats.gather_index_stats(ownname=>'scott', indname=>'emp_x01'); end;
/
-- 테이블 통계를 수집하면서 인덱스 통계도 같이 (cascade)
begin dbms_stats.gather_table_stats('scott', 'emp', cascade=>true); end;
/
-- 조회 (all_ind_statistics 뷰에서도 동일)
select blevel, leaf_blocks, num_rows, distinct_keys
     , avg_leaf_blocks_per_key, avg_data_blocks_per_key, clustering_factor
     , sample_size, last_analyzed
from   all_indexes
where  owner = 'SCOTT' and table_name = 'EMP' and index_name = 'EMP_X01';
```

| 통계항목 | 의미 | 용도 |
|------|------|------|
| **BLEVEL** | 브랜치 레벨. 루트에서 리프 직전까지 읽는 블록 수 | **수직적 탐색** 비용 (2.1.1 / 3.3) |
| **LEAF_BLOCKS** | 인덱스 리프 블록 총 개수 | **수평적 탐색** 비용 (2.1.1 / 3.3) |
| NUM_ROWS | 인덱스에 저장된 레코드 개수 | |
| **DISTINCT_KEYS** | 인덱스 키값 조합으로 만들어지는 값의 종류 개수. `=` 조회 시 선택도 계산에 사용 | 수평적 탐색 비용 |
| AVG_LEAF_BLOCKS_PER_KEY | 키값을 모두 `=`로 조회할 때 읽을 리프 블록 개수 | |
| AVG_DATA_BLOCKS_PER_KEY | 키값을 모두 `=`로 조회할 때 읽을 테이블 블록 개수 | 테이블 액세스 비용 |
| **CLUSTERING_FACTOR** | 인덱스 순서로 스캔하며 테이블 레코드를 찾을 때 읽게 될 테이블 블록 개수의 추정치 | **테이블 액세스** 비용 (3.1.2) |

[표 7-2] — 인덱스 통계는 **수직 탐색(BLEVEL) → 수평 탐색(LEAF_BLOCKS) → 테이블 액세스(CLUSTERING_FACTOR)** 라는 인덱스 비용 3단계(7.1.3)에 그대로 매핑됩니다.

### (3) 컬럼 통계

테이블 통계를 수집할 때 함께 수집됩니다.

```sql
-- 조회 (all_tab_col_statistics 뷰에서도 동일)
select num_distinct, density, avg_col_len, low_value, high_value, num_nulls
     , last_analyzed, sample_size
from   all_tab_columns
where  owner = 'SCOTT' and table_name = 'EMP' and column_name = 'DEPTNO';
```

| 통계항목 | 의미 |
|------|------|
| **NUM_DISTINCT** | 컬럼 값의 종류 개수 (NDV). 예: 성별 컬럼이면 2 |
| **DENSITY** | `=` 조건 선택도를 미리 구해 둔 값. **히스토그램이 없고 100% 균일 분포면 `1/NUM_DISTINCT`와 일치** |
| AVG_COL_LEN | 컬럼 평균 길이(Bytes) |
| LOW_VALUE / HIGH_VALUE | 최소값 / 최대값 |
| NUM_NULLS | 값이 NULL인 레코드 수 |

[표 7-3] — `=` 선택도는 `1/NUM_DISTINCT`로 구하거나 미리 구해 둔 `DENSITY`를 씁니다. 둘이 일치하지 않는 경우가 바로 데이터가 균일하지 않을 때이며, 그때 필요한 것이 히스토그램입니다.

---

### 컬럼 히스토그램

`=` 선택도를 `1/NUM_DISTINCT`나 `DENSITY`로 구하는 공식은 **분포가 균일한 컬럼**에는 잘 맞지만, **분포가 균일하지 않은(skew) 컬럼**에는 맞지 않습니다. 선택도를 잘못 구하면 비용 산정이 틀어지므로, 옵티마이저는 일반 컬럼 통계 외에 **히스토그램**을 추가로 활용합니다.

[그림 7-1] — 히스토그램은 (아파트매물 테이블의 '시도구분'처럼) **값별 데이터 비중/빈도를 미리 계산해 둔** 통계입니다. 실제 데이터를 읽어 계산한 값이므로 분포가 크게 변하지 않는 한 정확합니다.

| 히스토그램 유형 | 설명 |
|------|------|
| **도수분포 (FREQUENCY)** | 값별로 빈도수 저장 |
| **높이균형 (HEIGHT-BALANCED)** | 각 버킷의 높이가 동일하도록 분포 관리 |
| 상위도수분포 (TOP-FREQUENCY) | 상위 n개 값에 대한 빈도수 저장 (12c 이상) |
| 하이브리드 (HYBRID) | 도수분포 + 높이균형의 결합 (12c 이상) |

[표 7-4] — 오라클 12c 기준 4가지 유형.

```sql
-- 히스토그램 수집: method_opt 파라미터 지정
begin
  dbms_stats.gather_table_stats('scott', 'emp'
    , cascade=>false, method_opt=>'for columns ename size 10, deptno size 4');
end;
/
-- for all columns size 75      → 모든 컬럼에 버킷 75개
-- for all columns size auto    → 오라클이 컬럼별로 자동 판단

-- 조회 (all_tab_histograms 뷰)
select endpoint_value, endpoint_number
from   all_histograms
where  owner = 'SCOTT' and table_name = 'EMP' and column_name = 'DEPTNO'
order by endpoint_value;
--  ENDPOINT_VALUE  ENDPOINT_NUMBER
--  --------------  ---------------
--             10                3
--             20                8
--             30               14
```
(결과값 해석은 히스토그램 유형마다 다릅니다.)

### (4) 시스템 통계

시스템 통계는 **애플리케이션·하드웨어 성능 특성**을 측정한 것으로, 다음을 포함합니다.

- CPU 속도
- 평균 Single Block I/O 속도
- 평균 Multiblock I/O 속도 / 평균 Multiblock I/O 개수
- I/O 서브시스템 최대 처리량(Throughput)
- 병렬 Slave의 평균 처리량(Throughput)

> 과거에는 옵티마이저가 이들을 **고정 상수값**으로 처리해, 실제 하드웨어가 그 가정보다 좋거나 나쁘면 최적이 아닌 실행계획을 세울 수 있었습니다. 9i부터 오라클은 **실제 시스템을 측정한 시스템 통계**를 도입해, 설치된 하드웨어와 애플리케이션 특성(OLTP/DW)을 반영하도록 했습니다.

```sql
-- 조회 (sys.aux_stats$ 뷰)
SQL> select sname, pname, pval1, pval2 from sys.aux_stats$;
--  SYSSTATS_MAIN  CPUSPEEDNW   1189.15  ...   ← CPU 속도
--  SYSSTATS_MAIN  IOSEEKTIM      10     ...   ← I/O Seek 시간
--  SYSSTATS_MAIN  SREADTIM      5.253   ...   ← Single block 읽기 시간
--  SYSSTATS_MAIN  MREADTIM      3.122   ...   ← Multiblock 읽기 시간
--  SYSSTATS_MAIN  MBRC           15     ...   ← Multiblock 읽기 개수
```

### CS 대응

통계정보는 쿼리 실행 전에 만든 **메타데이터 기반 휴리스틱**입니다. 데이터 전체를 매번 직접 세지 않고, 샘플·요약 정보로 비용을 추정하는 것 — DB 통계 수집은 일종의 **사전 인덱싱된 분포 요약**이고, 히스토그램은 그 요약이 실패하는 skew 구간을 보정하는 **값-빈도 매핑(value→frequency)** 입니다.

---

## ✅ 이 절(7.1.2)의 핵심 통찰

1. 통계는 **오브젝트 통계(테이블·인덱스·컬럼)** 와 **시스템 통계**로 나뉘며, 옵티마이저는 실제 데이터가 아니라 이 통계로 비용을 추정한다.
2. 인덱스 통계 **BLEVEL/LEAF_BLOCKS/CLUSTERING_FACTOR**는 인덱스 비용 3단계(수직·수평 탐색·테이블 액세스)에 1:1로 대응한다.
3. `=` 선택도는 `1/NUM_DISTINCT` 또는 `DENSITY`로 구하며, **균일 분포면 둘이 일치**한다.
4. skew 컬럼은 이 공식이 빗나가므로 **히스토그램**(FREQUENCY/HEIGHT-BALANCED/TOP-FREQUENCY/HYBRID)을 `method_opt`으로 수집해 보정한다.
5. **시스템 통계(9i~)** 는 고정 상수 대신 실제 하드웨어/워크로드 특성을 반영해 비용 계산을 더 현실화한다.

## 🔗 흐름 속 위치

7.1.1의 선택도·카디널리티가 "무엇을 추정하는가"였다면, 7.1.2는 "그 추정의 입력값(통계)이 어디서 오는가"입니다. 다음 7.1.3은 이 통계 항목들을 실제 **비용 공식**에 대입해 옵티마이저가 비용을 계산하는 원리를 보여줍니다.

📌 한 줄 요약: 통계정보는 옵티마이저가 실행 전에 데이터 세계를 추정하는 **지도**이며, skew는 히스토그램으로, 하드웨어 현실은 시스템 통계로 보정합니다.
