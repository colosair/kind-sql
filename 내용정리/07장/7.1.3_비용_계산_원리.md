## 7.1.3 비용 계산 원리

### 핵심 개념
옵티마이저 비용은 실행시간 그 자체가 아니라, **예상 I/O와 CPU 사용량을 바탕으로 계산한 상대적 비용 값**입니다.

```text
통계정보
   ↓
카디널리티 추정
   ↓
액세스 경로별 I/O/CPU 비용 계산
   ↓
가장 낮은 비용의 실행계획 선택
```

| 비용 요소 | 의미 |
|---|---|
| Single Block I/O | 인덱스 ROWID 기반 랜덤 액세스 |
| Multi Block I/O | Full Scan 등 대량 블록 읽기 |
| CPU Cost | 조건 평가, 해시, 정렬, 함수 계산 |
| Memory/TEMP | 정렬·해시 작업공간 부족 시 영향 |

### 인덱스 액세스 비용 공식

단일 테이블을 인덱스로 액세스할 때의 비용은 인덱스 통계만으로 계산됩니다. 세 항은 각각 **수직 탐색 + 수평 탐색 + 테이블 랜덤 액세스**에 대응합니다(2장·3장 구조 그대로).

**인덱스 키값을 모두 `=` 조건으로 검색할 때:**

```text
비용 = BLEVEL                      ← 인덱스 수직적 탐색 (루트→리프)
     + AVG_LEAF_BLOCKS_PER_KEY     ← 인덱스 수평적 탐색
     + AVG_DATA_BLOCKS_PER_KEY     ← 테이블 랜덤 액세스
```

**키값이 모두 `=`가 아닐 때(범위 조건 포함):**

```text
비용 = BLEVEL                              ← 수직적 탐색
     + LEAF_BLOCKS      × 유효 인덱스 선택도  ← 수평적 탐색 (스캔할 리프 범위)
     + CLUSTERING_FACTOR × 유효 테이블 선택도  ← 테이블 랜덤 액세스
```

| 항 | 의미 | 연결 |
|---|---|---|
| `BLEVEL` | 브랜치 깊이(수직 탐색) | [[2.1.3_인덱스_수직적_탐색]] |
| `LEAF_BLOCKS × 유효 인덱스 선택도` | ①번 **인덱스 액세스 조건**이 결정하는 리프 스캔량 | [[3.3.3_액세스_조건과_필터_조건]] |
| `CLUSTERING_FACTOR × 유효 테이블 선택도` | ①+② 조건이 결정하는 테이블 방문량 | [[3.1.2_인덱스_클러스터링_팩터]] |

핵심 구분은 **유효 인덱스 선택도**(인덱스 액세스 조건만)와 **유효 테이블 선택도**(액세스+필터 조건 모두)입니다. 필터 조건은 인덱스 스캔량은 줄이지 못하지만 테이블 액세스량은 줄입니다 — 3.3.3의 액세스/필터 구분이 그대로 비용 공식의 두 선택도로 나타납니다.

### 💡 비용(Cost)의 정확한 의미 — I/O 비용 모델 vs CPU 비용 모델

위 비용 계산식은 **'I/O 비용 모델'** 기준입니다. 이 모델에서 실행계획에 나타나는 `Cost`는 **'예상 I/O Call 횟수'** 를 의미합니다.

반면 최신 **'CPU 비용 모델'** 에서 `Cost`는 **Single Block I/O를 기준으로 한 상대적 시간**을 표현합니다.

```text
Cost = 100  →  "우리 시스템에서 Single Block I/O를 100번 하는 정도의 시간"
            (절대 시간이 아니라 상대적 시간 개념)
```

| 모델 | Cost의 의미 |
|------|------|
| I/O 비용 모델 | 예상 **I/O Call 횟수** |
| CPU 비용 모델 (최신) | Single Block I/O 기준 **상대적 시간** |

**CPU 비용 모델을 개발한 이유** — 같은 실행계획으로 같은 양의 데이터를 읽어도 애플리케이션·하드웨어 성능 특성에 따라 **절대 소요시간이 다를 수 있기** 때문입니다. 똑같이 I/O Call을 100번 해도 그것이 Single Block I/O냐 Multiblock I/O냐에 따라 속도가 다르고, 시스템마다 그 속도도 다릅니다. 이 차이를 반영하는 것이 7.1.2의 **시스템 통계**(CPU 속도, Single/Multiblock I/O 속도)입니다.

### 핵심 연결
1장부터 반복한 `reduce I/O` 관점이 여기서 비용 모델로 구현됩니다. 옵티마이저는 각 실행계획 후보가 얼마나 많은 블록을 읽고, 얼마나 많은 CPU 작업을 할지 추정합니다. 위 공식에서 `CLUSTERING_FACTOR`가 크면(나쁜 CF) 테이블 랜덤 액세스 비용이 커져 인덱스가 불리해지는데, 이것이 3.1.3 손익분기점의 비용 모델 표현입니다.

---

## ✅ 이 절(7.1.3)의 핵심 통찰
1. 비용은 실제 시간이 아니라 실행계획 간 비교를 위한 추정값입니다.
2. 인덱스 비용 = `BLEVEL`(수직) + `LEAF_BLOCKS×유효 인덱스 선택도`(수평) + `CLUSTERING_FACTOR×유효 테이블 선택도`(테이블 액세스).
3. 유효 인덱스 선택도(액세스 조건)와 유효 테이블 선택도(액세스+필터)의 구분이 3.3.3 그대로입니다.
4. CF가 크면 테이블 랜덤 액세스 비용이 커져 인덱스가 불리해집니다(3.1.3 손익분기점).
5. 비용 계산이 틀리는 주된 원인은 잘못된 카디널리티 추정입니다.

📌 한 줄 요약: 옵티마이저 비용은 “이 계획이 얼마나 많은 I/O와 CPU를 쓸 것인가”에 대한 상대적 추정치입니다.
