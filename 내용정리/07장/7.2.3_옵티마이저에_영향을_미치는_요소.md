## 7.2.3 옵티마이저에 영향을 미치는 요소

### 핵심 개념

옵티마이저는 SQL 문장만 보고 판단하지 않습니다. 실행계획은 **SQL + 주변 메타데이터의 합작품**입니다. 책은 영향 요소를 6가지로 정리합니다.

```text
SQL 텍스트
   + 연산자 형태 + 옵티마이징 팩터 + 제약 + 통계 + 힌트 + 파라미터
   ↓
비용 계산 → 실행계획 선택
```

| # | 요소 | 영향 |
|---|------|------|
| (1) | SQL과 연산자 형태 | 같은 결과라도 `=`/`IN`/`LIKE`/`BETWEEN`/부등호 등에 따라 다른 선택 |
| (2) | 옵티마이징 팩터 | 인덱스·IOT·클러스터·파티션·MV 구성에 따라 실행계획·성능 |
| (3) | 제약 설정 | PK·FK·Check·Not Null = 무결성 + 옵티마이저 최적화용 메타정보 |
| (4) | 통계정보 | **가장 강력** — 부정확하면 갑작스러운 성능 저하·장애 |
| (5) | 옵티마이저 힌트 | **가장 절대적** — directives로 인식해 그대로 따름 |
| (6) | 옵티마이저 관련 파라미터 | 버전·파라미터 변경으로 행동 변화 |

---

### (1)~(3) SQL 형태 · 옵티마이징 팩터 · 제약

- **(1) SQL과 연산자 형태**: 결과가 같아도 어떤 연산자(`=`, `IN`, `LIKE`, `BETWEEN`, 부등호)로 작성했는지에 따라 옵티마이저가 다른 선택을 하고, 궁극적으로 성능에 영향을 줍니다.
- **(2) 옵티마이징 팩터**: 인덱스·IOT·클러스터·파티션·MV(Materialized View)를 **어떻게 구성했는지**에 따라 가능한 액세스 경로와 성능이 크게 달라집니다.
- **(3) 제약 설정**: PK·FK·Check·Not Null 제약은 데이터 무결성을 보장할 뿐 아니라, **옵티마이저가 쿼리 성능을 최적화하는 데 매우 중요한 메타 정보**로 활용됩니다.

---

### (4) 통계정보 — 갑작스러운 장애의 주범

> 1장에서 봤듯 통계정보는 옵티마이저에 가장 강력한 영향을 줍니다. 내비게이션이 쓰는 주소·도로 정보가 부정확하면 운전자가 곤란하고 위험해지듯, **잘 작동하던 프로그램이 어느 날 갑자기 느려진다면 십중팔구 통계정보가 원인**입니다.

통계정보 문제로 시스템 장애가 발생하는 전형적 시나리오:

| 시나리오 | 비고 |
|------|------|
| 특정 테이블 통계정보를 갑자기 삭제 | |
| 대량 데이터를 지웠다 재입력하기 **직전, 빈 상태에서** 자동 수집됨 | 스케줄링된 통계수집 패키지가 동작 |
| 3년간 갱신 안 하던 테이블 통계를 어느 날 갑자기 재수집 | 실행계획 급변 |
| 통계 없이 관리하던 테이블에 **인덱스를 재생성** | ※ 10g부터 인덱스 생성 시 인덱스 통계 자동 수집 |
| 테이블/인덱스를 재생성하며 **파티션 단위로만** 통계 수집 | ※ 바인드 변수 쓰면 파티션 통계가 아닌 Global 통계 사용 |

핵심: 통계는 "수집했는가"뿐 아니라 **"언제·어떤 상태에서 수집했는가"** 가 중요합니다. 빈 테이블에서 수집된 통계나, 오래 묵었다 급변한 통계가 곧 장애로 이어집니다.

---

### (5) 옵티마이저 힌트 — 가장 절대적, 그러나 안 들을 때

옵티마이저에게 **가장 절대적인** 영향을 주는 요소가 힌트입니다. 옵티마이저는 힌트를 **명령어(directives)로 인식하고 그대로 따릅니다.** 그런데 힌트가 잘 작동하지 않는다면 다음 5가지 중 하나일 가능성이 높습니다.

```text
힌트가 안 듣는 경우
 1. 문법적으로 맞지 않게 힌트를 기술
 2. 잘못된 참조 사용 (없는 테이블/별칭 지정 등)
 3. 의미적으로 맞지 않게 힌트를 기술
 4. 논리적으로 불가능한 액세스 경로 (예: 없는 인덱스 강제)
 5. 버그
```

핵심: 힌트가 안 먹힐 때는 옵티마이저를 탓하기 전에 **힌트 자체의 문법·참조·논리**를 먼저 의심해야 합니다.

---

### (6) 옵티마이저 관련 파라미터

> SQL·데이터·통계정보·하드웨어 등 **모든 환경이 같은데도** 오라클 버전을 업그레이드하면 옵티마이저가 다르게 작동하는 경험을 흔히 합니다. 이런 행동 변화는 대개 **파라미터 추가 또는 기본값 변경**에 기인합니다.

따라서 버전 업그레이드 시에는 옵티마이저 관련 파라미터의 변화를 점검해야 합니다.

---

### 정밀화 — 옵티마이저를 탓하기 전에

옵티마이저가 이상한 계획을 골랐다고 해서 항상 옵티마이저 자체가 문제인 것은 아닙니다. 대부분은 **입력 정보(통계)가 부정확**하거나, **SQL 표현이 쿼리 변환을 막거나**, **인덱스·제약 선택지가 부적절**한 경우입니다. 힌트는 **마지막 수단**에 가깝고, 먼저 (1)~(4)의 입력 정보와 구조를 점검하는 것이 순서입니다.

---

## ✅ 이 절(7.2.3)의 핵심 통찰

1. 실행계획은 SQL 문장과 **6가지 요소(연산자 형태·옵티마이징 팩터·제약·통계·힌트·파라미터)** 의 합작품이다.
2. **통계정보가 가장 강력** — "어느 날 갑자기 느려짐"의 십중팔구는 통계 문제(삭제·빈 상태 수집·급격한 재수집·인덱스 재생성·파티션 단위 수집).
3. **힌트가 가장 절대적**이지만, 안 들으면 문법·참조·의미·논리 오류·버그 5가지를 의심한다.
4. 환경이 같아도 **버전 업그레이드**로 옵티마이저가 달라지는 건 대개 파라미터 변화 탓이다.
5. 옵티마이저를 탓하기 전에 **입력 정보(통계)·SQL 표현·구조**를 먼저 점검한다. 힌트는 마지막 수단.

## 🔗 흐름 속 위치

7.2.1~7.2.2가 옵티마이저의 종류와 모드였다면, 7.2.3은 "무엇이 그 판단을 흔드는가"입니다. 다음 7.2.4는 이 요소들이 완벽해도 남는 **옵티마이저 자체의 한계**로 이어집니다.

📌 한 줄 요약: 옵티마이저는 SQL만이 아니라 통계·인덱스·제약·파라미터·힌트까지 보고 계획을 고르며, 그중 **통계가 가장 강력하고 힌트가 가장 절대적**입니다.
