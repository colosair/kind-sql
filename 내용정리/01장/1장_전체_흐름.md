## 1장 SQL 처리 과정과 I/O 전체 흐름

### 핵심 개념

1장은 책 전체의 기초입니다. SQL이 선언형 언어이기 때문에 옵티마이저가 실행계획을 만들고, 그 실행계획의 성능은 결국 얼마나 많은 I/O를 발생시키는지에 의해 좌우됩니다.

```text
SQL은 선언형 언어
   ↓
DBMS가 실행 절차 필요
   ↓
옵티마이저가 실행계획 선택
   ↓
실행계획은 데이터 액세스 방식 결정
   ↓
데이터 액세스는 블록 I/O 발생
   ↓
SQL 성능은 I/O 양에 의해 좌우
```

| 절 | 핵심 질문 | 핵심 답 |
|---|---|---|
| 1.1.1 | SQL은 어떤 언어인가 | 구조적, 집합적, 선언적 언어입니다. |
| 1.1.2 | SQL은 어떻게 실행 가능한 절차가 되는가 | 파싱, 최적화, 로우 소스 생성 과정을 거칩니다. |
| 1.1.3 | 옵티마이저는 무엇을 하는가 | 후보 실행계획의 비용을 비교해 하나를 선택합니다. |
| 1.1.4 | 실행계획과 Cost는 무엇인가 | 실행 절차와 예상 비용입니다. |
| 1.1.5 | 힌트는 왜 필요한가 | 옵티마이저 선택을 보정/강제하기 위해 사용합니다. |
| 1.2.1 | 소프트/하드 파싱 차이는 무엇인가 | 실행 프로시저 재사용 여부입니다. |
| 1.2.2 | 바인드 변수는 왜 중요한가 | SQL 공유를 가능하게 해 하드 파싱을 줄입니다. |
| 1.3.1 | SQL이 왜 느린가 | 대부분 I/O, 특히 디스크 I/O 때문입니다. |
| 1.3.2 | 데이터는 어떻게 저장되는가 | 블록, 익스텐트, 세그먼트, 테이블스페이스 구조입니다. |
| 1.3.3 | I/O 단위는 무엇인가 | 로우가 아니라 블록입니다. |
| 1.3.4 | 액세스 패턴은 어떻게 나뉘는가 | 시퀀셜 액세스와 랜덤 액세스입니다. |
| 1.3.5 | 논리/물리 I/O 차이는 무엇인가 | 캐시 접근과 디스크 접근입니다. |
| 1.3.6 | Single/Multiblock I/O 차이는 무엇인가 | 한 번에 읽는 블록 수가 다릅니다. |
| 1.3.7 | Full Scan과 Index Scan은 언제 유리한가 | 대량은 Full Scan, 소량은 Index Scan이 유리합니다. |
| 1.3.8 | 캐시는 어떻게 탐색하는가 | 해시 구조로 블록을 찾습니다. |

### 1장의 통합 원리

```text
SQL 튜닝
   ↓
좋은 실행계획 유도
   ↓
블록 접근량 감소
   ↓
논리적 I/O 감소
   ↓
물리적 I/O 가능성 감소
   ↓
응답시간 개선
```

## ✅ 1장의 핵심 통찰

1. SQL은 선언형 언어이고, 옵티마이저가 절차를 만듭니다.
2. 실행계획은 SQL의 실제 처리 경로입니다.
3. 하드 파싱은 비싸며, 바인드 변수는 SQL 공유를 가능하게 합니다.
4. SQL 성능의 근본 병목은 I/O입니다.
5. DBMS는 로우가 아니라 블록 단위로 데이터를 읽습니다.
6. 논리적 I/O 감소가 물리적 I/O 감소보다 더 근본적인 튜닝 레버입니다.

📌 한 줄 요약: 1장은 SQL 튜닝을 "옵티마이저가 선택한 실행계획이 얼마나 적은 블록을 읽게 만드는가"라는 문제로 정식화합니다.
