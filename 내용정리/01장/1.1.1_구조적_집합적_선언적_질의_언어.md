## 1.1.1 구조적, 집합적, 선언적 질의 언어

### 핵심 개념

두 테이블(EMP, DEPT)을 부서번호(DEPTNO)로 조인해 사원명(ENAME) 순으로 정렬하는 로직(그림 1-1)을, 예전에는 절차적 언어로 어렵게 프로그래밍해야 했지만(20~30년 전), 요즘은 SQL로 간단히 처리합니다.

```sql
SELECT E.EMPNO, E.ENAME, E.JOB, D.DNAME, D.LOC
FROM   EMP E, DEPT D
WHERE  E.DEPTNO = D.DEPTNO
ORDER BY E.ENAME;
```

**SQL = Structured Query Language(구조적 질의 언어)** 의 줄임말입니다. 위키피디아의 정의가 SQL의 본질을 잘 보여줍니다.

> - SQL is designed for a specific purpose: to query data contained in a relational database.
> - SQL is a **set-based, declarative** query language, **not an imperative** language such as C or BASIC.

즉 SQL은 사용자가 원하는 결과집합을 **구조적(structured)·집합적(set-based)·선언적(declarative)** 으로 표현하는 언어입니다. 사용자는 "어떻게 처리할지"가 아니라 "무엇을 얻고 싶은지"를 기술합니다. (오라클 PL/SQL, SQL Server T-SQL처럼 절차적 기능을 구현하는 확장 언어도 제공되지만, SQL 자체는 선언형입니다.)

원하는 결과집합을 구조적·집합적으로 선언하지만, 그 결과집합을 **만드는 과정은 절차적일 수밖에** 없습니다. 즉 프로시저가 필요한데, 그런 프로시저를 만들어 내는 DBMS 내부 엔진이 바로 **SQL 옵티마이저**입니다. 옵티마이저가 프로그래밍을 대신해 주는 셈입니다(그림 1-2).

```text
사용자  ──SQL──►  옵티마이저  ──►  실행계획(프로시저)
선언            절차 생성        실제 실행 코드
```

### SQL과 절차형 언어의 차이

| 구분 | SQL | 절차형 언어 |
|---|---|---|
| 사고 방식 | 결과집합 선언 | 처리 절차 직접 작성 |
| 처리 대상 | 집합 | 개별 레코드/반복문 |
| 실행 절차 | 옵티마이저가 결정 | 개발자가 결정 |
| 핵심 질문 | 무엇을 원하는가 | 어떻게 처리할 것인가 |

CS 관점에서 보면 SQL은 "알고리즘을 직접 구현하는 코드"가 아니라 "문제의 제약조건과 목표를 선언하는 DSL"에 가깝습니다. 실제 탐색 전략은 옵티마이저가 선택합니다.

```text
개발자: 결과 조건 선언
옵티마이저: 가능한 실행 절차 탐색
로우 소스: 실제 실행 가능한 절차 생성
```

---

## ✅ 이 절(1.1.1)의 핵심 통찰

1. SQL은 구조적, 집합적, 선언적 언어입니다.
2. 사용자는 결과집합을 선언하고, 처리 절차는 DBMS가 만듭니다.
3. SQL 튜닝은 선언을 바꾸거나 힌트를 주어 옵티마이저가 더 좋은 절차를 선택하게 만드는 일입니다.

## 🔗 흐름 속 위치

이 절은 SQL 튜닝의 출발점입니다. SQL이 선언형 언어이기 때문에 내부에서 반드시 실행계획을 만드는 옵티마이저가 필요합니다.

📌 한 줄 요약: SQL은 절차를 쓰는 언어가 아니라 결과집합을 선언하고, 절차는 옵티마이저가 만드는 언어입니다.
