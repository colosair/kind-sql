## 1.3.7 Table Full Scan vs. Index Range Scan

### 핵심 개념

`Table Full Scan`은 테이블 전체 블록을 순차적으로 읽는 방식이고, `Index Range Scan`은 인덱스에서 조건에 맞는 범위를 찾은 뒤 ROWID로 테이블을 액세스하는 방식입니다.

```text
Table Full Scan
   ↓
테이블 블록을 순차적으로 읽음
   ↓
조건 필터링

Index Range Scan
   ↓
인덱스 수직 탐색
   ↓
리프 블록 범위 스캔
   ↓
ROWID로 테이블 랜덤 액세스
```

### 비교

| 구분 | Table Full Scan | Index Range Scan |
|---|---|---|
| 접근 방식 | 시퀀셜 액세스 | 인덱스 + 랜덤 액세스 |
| I/O 방식 | Multiblock I/O | Single Block I/O 중심 |
| 유리한 상황 | 대량 조회 | 소량 선택 조회 |
| 주요 비용 | 전체 블록 읽기 | 테이블 랜덤 액세스 반복 |

### 선택 기준

인덱스는 조건에 맞는 데이터가 적을 때 유리합니다. 하지만 조건에 맞는 데이터가 많으면 인덱스 경유 랜덤 액세스 비용이 Full Scan보다 커질 수 있습니다.

```text
선택도 낮음
   ↓
적은 ROWID
   ↓
Index Range Scan 유리

선택도 높음
   ↓
많은 ROWID
   ↓
랜덤 액세스 폭증
   ↓
Table Full Scan 유리 가능
```

### 통일 원리

어느 방식이든 핵심은 읽는 블록 수입니다.

```text
Index Range Scan 비용
   =
인덱스 블록
+ 테이블 랜덤 액세스 블록

Table Full Scan 비용
   =
테이블 전체 블록 / Multiblock 효율
```

---

## ✅ 이 절(1.3.7)의 핵심 통찰

1. Full Scan은 대량 순차 처리에 강합니다.
2. Index Range Scan은 소량 조건 검색에 강합니다.
3. 인덱스 경유 테이블 랜덤 액세스가 많아지면 Full Scan보다 느릴 수 있습니다.
4. 선택도와 클러스터링이 실행계획 선택에 큰 영향을 줍니다.

## 🔗 흐름 속 위치

이 절은 1.3.4~1.3.6의 액세스 패턴과 I/O 단위를 실제 실행계획 선택 문제로 연결합니다.

📌 한 줄 요약: 인덱스와 Full Scan의 승부는 결국 어느 쪽이 더 적은 비용으로 필요한 블록을 읽느냐입니다.
