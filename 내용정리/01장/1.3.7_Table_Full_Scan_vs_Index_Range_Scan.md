## 1.3.7 Table Full Scan vs. Index Range Scan

### 핵심 개념

테이블 데이터를 읽는 방식은 두 가지입니다.

- **Table Full Scan**: 테이블에 속한 블록 '전체'를 읽어 사용자가 원하는 데이터를 찾음 — **시퀀셜 액세스 + Multiblock I/O**
- **Index Range Scan**: 인덱스에서 '일정량'을 스캔해 얻은 ROWID로 테이블 레코드를 찾음 — **랜덤 액세스 + Single Block I/O**

```text
Table Full Scan          Index Range Scan
테이블 블록 순차 읽기      인덱스 수직 탐색 → 리프 범위 스캔
   ↓ (Multiblock)            ↓
조건 필터링                ROWID로 테이블 랜덤 액세스 (Single Block, 반복)
```

| 구분 | Table Full Scan | Index Range Scan |
|------|------|------|
| 접근 방식 | 시퀀셜 액세스 | 인덱스 + **랜덤 액세스** |
| I/O 방식 | **Multiblock I/O** | **Single Block I/O** 중심 |
| 유리한 상황 | 대량 조회 | 소량 선택 조회 |
| 주요 비용 | 전체 블록 읽기 | 테이블 랜덤 액세스 반복 |

---

### Index Range Scan이 대량에서 불리한 두 가지 이유

**① 물리적 I/O 측면** — 랜덤 + Single Block I/O라, 캐시 미스 시 **레코드 하나를 읽으려 매번 잠(I/O Call)** 을 잡습니다(1.3.1). 대량이면 이 잠이 폭증해 Full Scan보다 느립니다.

**② 논리적 I/O 측면** — 더 결정적입니다. 한 블록에 평균 500개 레코드가 있다면:

```text
인덱스로 '전체 레코드'를 액세스 → 같은 블록을 최대 500번 반복해서 읽음
   (모든 블록을 평균 500번씩 → 논리적 I/O 폭증)

Table Full Scan → 각 블록을 단 한 번만 읽음
```

즉 같은 데이터를 읽어도 **인덱스는 같은 블록을 수백 번 중복 방문**할 수 있습니다(그림 1-26). 그래서 읽을 데이터가 일정량을 넘으면 인덱스보다 Full Scan이 유리합니다(→ 3.1.3 손익분기점).

---

### ⚠️ 인덱스 맹신은 금물 — 이 절의 핵심 메시지

> "데이터베이스를 효과적으로 이용하는 데 인덱스의 중요성은 아무리 강조해도 지나치지 않다. **하지만 인덱스에 대한 맹신은 금물이다.** 인덱스가 항상 옳은 것은 아니며, 바꿔 말해 **Table Full Scan이 항상 나쁜 것도 아니다.** 인덱스는 큰 테이블에서 아주 적은 일부 데이터를 빨리 찾는 도구일 뿐이므로, **모든 성능 문제를 인덱스로 해결하려 해선 안 된다.**"

대량 처리(집계용 SQL·배치)에는 오히려 Full Scan이 정확하고 빠릅니다 — Full Scan 위주 SQL은 OLTP SQL보다 튜닝하기가 비교적 쉽고, 상당수는 **Full Scan으로 유도해야 성능이 빨라집니다.**

#### 💬 필드 스토리

> SQL 튜닝을 처음 공부하는 독자라면 다소 당황스러울 수 있습니다 — 보통 성능을 논할 때 인덱스를 빼고 얘기할 수 없는데, 이 책은 Full Scan을 두둔하는 입장에서 쓰였기 때문입니다. 이는 **'경험에서 우러나온 한탄(?)'** 입니다. 튜닝팀에 의뢰되는 SQL을 분석해 보면, **인덱스를 안 타서 느린 경우보다 불필요하게 인덱스를 타서 느린 경우가 더 많습니다.** 실제 튜닝 프로젝트의 개선 포인트 대부분이 **'힌트를 이용한 Table Full Scan 유도'** 였습니다. 쿼리 툴을 만든다면, Full Scan뿐 아니라 **예상 카디널리티가 일정량을 넘는데도 인덱스로 테이블을 액세스하는 부분에 빨간색을 표시**하고 싶을 정도입니다.

> 그래서 "실행계획에서 Table Full Scan을 찾아내는 식"의 분석은 SQL 성능 향상에 큰 도움이 되지 않습니다. **무엇을, 얼마나 읽는가**가 핵심입니다.

---

### 선택 기준 — 결국 읽는 블록 수

```text
선택도 낮음(소량) → 적은 ROWID → Index Range Scan 유리
선택도 높음(대량) → 많은 ROWID → 랜덤·중복 액세스 폭증 → Table Full Scan 유리

Index Range Scan 비용 = 인덱스 블록 + 테이블 랜덤 액세스 블록(중복 포함)
Table Full Scan 비용  = 테이블 전체 블록 / Multiblock 효율
```

---

## ✅ 이 절(1.3.7)의 핵심 통찰

1. Full Scan은 **시퀀셜+Multiblock**(대량 강함), Index Range Scan은 **랜덤+Single Block**(소량 강함)이다.
2. 인덱스가 대량에서 불리한 이유는 ① 물리 I/O(매번 잠) + ② **논리 I/O(같은 블록을 최대 500번 중복 방문)** 다. Full Scan은 각 블록을 한 번만 읽는다.
3. **인덱스 맹신은 금물 — Table Full Scan이 항상 나쁜 게 아니다.** 인덱스는 큰 테이블에서 소량을 찾는 도구일 뿐이다.
4. 실무에서는 **인덱스를 안 타서 느린 것보다 불필요하게 타서 느린 경우가 더 많다** — 개선의 다수가 'Full Scan 유도'다.
5. 승부는 결국 **어느 쪽이 더 적은 블록을 읽느냐**(선택도·손익분기점, 3.1.3)로 갈린다.
