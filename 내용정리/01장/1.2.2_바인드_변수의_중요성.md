## 1.2.2 바인드 변수의 중요성

### 핵심 개념

Oracle에서 SQL은 텍스트 자체가 이름 역할을 합니다. 값만 달라도 SQL 텍스트가 달라지면 서로 다른 SQL로 인식되어 하드 파싱이 반복됩니다. 바인드 변수는 SQL 텍스트를 고정하고 값만 파라미터로 전달하게 해 SQL 공유를 가능하게 합니다.

```text
Literal SQL
   ↓
값이 바뀔 때마다 SQL 텍스트 변경
   ↓
라이브러리 캐시 공유 실패
   ↓
하드 파싱 반복

Bind SQL
   ↓
SQL 텍스트 고정
   ↓
값만 바인딩
   ↓
실행계획 공유
```

### 이름 없는 SQL 문제

프로시저나 함수는 이름이 있지만 SQL은 별도 이름이 없습니다. SQL 전체 텍스트가 식별자 역할을 합니다.

```sql
select * from emp where empno = 7900;
select * from emp where empno = 7901;
```

의미상 같은 패턴이어도 텍스트가 다르므로 다른 SQL입니다.

### 바인드 변수

```sql
select *
from customer
where login_id = :1;
```

```text
고객 100만 명 로그인
   ↓
Literal 방식: SQL 100만 개
   ↓
하드 파싱 폭증

Bind 방식: SQL 1개
   ↓
값만 다르게 실행
   ↓
라이브러리 캐시 공유
```

CS 관점으로 보면 Literal SQL은 값마다 함수를 새로 컴파일하는 것과 같습니다. 바인드 변수는 파라미터를 받는 하나의 함수를 재사용하는 구조입니다.

---

## ✅ 이 절(1.2.2)의 핵심 통찰

1. SQL은 텍스트 자체가 식별자입니다.
2. Literal 값이 달라지면 SQL 공유가 깨집니다.
3. 바인드 변수는 SQL 텍스트를 고정해 실행계획 재사용을 가능하게 합니다.
4. 동시성이 높은 OLTP에서는 바인드 변수가 필수입니다.
