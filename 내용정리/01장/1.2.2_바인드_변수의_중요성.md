## 1.2.2 바인드 변수의 중요성

### 이름 없는 SQL 문제

사용자 정의 함수·프로시저·트리거·패키지는 생성될 때부터 **이름**을 갖습니다. 컴파일된 상태로 딕셔너리에 영구 저장되어 재사용됩니다. 반면 **SQL은 이름이 없습니다** — **전체 SQL 텍스트가 곧 이름** 역할을 합니다.

> 오라클 10g에 등장한 **SQL ID**도 SQL 텍스트가 변하면 따라서 변합니다(텍스트와 1:1). 실제로 IBM DB2 같은 DBMS는 SQL을 영구 저장(이름 부여)하지만, **오라클·SQL Server는 그렇게 하지 않습니다** — ad hoc·1회성·무효화될 SQL이 많아 영구 저장하면 공간과 속도 비용이 크기 때문입니다.

```sql
-- 의미상 같은 패턴이어도 텍스트가 다르면 모두 '다른 SQL'
select * from emp where empno = 7900;
select * from emp where empno = 7901;
```

---

### 공유 가능 SQL — 텍스트가 완전히 같아야 한다

라이브러리 캐시에서 SQL을 찾는 키가 **'SQL 문 텍스트 자체'** 이므로, 아래는 **모두 다른 SQL**로 취급됩니다(각자 하드파싱·별도 캐시 공간).

```sql
SELECT * FROM emp WHERE empno = 7900;          -- 기준
select * from emp where empno = 7900;          -- 대소문자 다름
select * from  emp where empno = 7900;         -- 공백 다름
select * from emp where empno = 7900 ;         -- 세미콜론 앞 공백
select * from scott.emp where empno = 7900;    -- 스키마명 붙음
select /* comment */ * from emp where empno = 7900;   -- 주석
select /*+ first_rows */ * from emp where empno = 7900; -- 힌트
```

### Literal SQL의 재앙 — 500만 고객 쇼핑몰

```java
// ❌ Literal 방식: login_id 값을 SQL 텍스트에 직접 결합
public void login(String login_id) throws Exception {
  String SQLStmt = "SELECT * FROM CUSTOMER WHERE LOGIN_ID = '" + login_id + "'";
  Statement st = con.createStatement();
  ResultSet rs = st.executeQuery(SQLStmt);
  ...
}
```

12시 정각 30분간 대대적 할인 이벤트 → 100만 고객이 12시 전후 동시 접속.

```text
고객마다 LOGIN_ID 값이 다름 → SQL 텍스트가 100만 개
   ↓
라이브러리 캐시(V$SQL)가 아래로 가득 참
   SELECT * FROM CUSTOMER WHERE LOGIN_ID = 'oraking'
   SELECT * FROM CUSTOMER WHERE LOGIN_ID = 'javaking'
   SELECT * FROM CUSTOMER WHERE LOGIN_ID = 'tommy'  ...
   ↓
하드파싱 폭증 → CPU 급증 + 라이브러리 캐시 경합 → 로그인 마비
```

**DBMS 부하는 대개 과도한 I/O가 원인이지만, 이 경우는 I/O가 아닌데도 CPU 사용률이 급증**합니다 — 하드파싱 자체가 CPU 집약적이기 때문입니다(1.2.1).

> **비유**: 이는 고객마다 `LOGIN_ORAKING()`, `LOGIN_JAVAKING()`, `LOGIN_TOMMY()` 프로시저를 **따로 만드는** 셈입니다. 내부 로직이 똑같으니, **파라미터를 받는 `LOGIN(login_id)` 하나를 공유**해야 마땅합니다.

---

### 해법 — 바인드 변수 (파라미터 Driven)

```java
// ✅ Bind 방식: SQL 텍스트를 고정하고 값은 파라미터로
public void login(String login_id) throws Exception {
  String SQLStmt = "SELECT * FROM CUSTOMER WHERE LOGIN_ID = ?";
  PreparedStatement st = con.prepareStatement(SQLStmt);
  st.setString(1, login_id);
  ResultSet rs = st.executeQuery();
  ...
}
```

```text
라이브러리 캐시에 로그인 SQL이 단 하나만 존재
   SELECT * FROM CUSTOMER WHERE LOGIN_ID = :1
   ↓
하드파싱 최초 1번 → 캐싱된 SQL을 100만 고객이 공유·재사용
```

CS 관점: Literal SQL은 **값마다 함수를 새로 컴파일**하는 것이고, 바인드 변수는 **파라미터를 받는 하나의 함수를 재사용**하는 구조입니다.

> ⚠️ **단, 양면성이 있습니다.** 바인드 변수는 파싱 시점에 값을 모르므로 **컬럼 히스토그램을 활용하지 못합니다.** 분포가 균일하면 문제없지만, skew 컬럼에서는 선택도 추정이 빗나갈 수 있습니다(7.2.4 — Bind Variable Peeking으로 보완). "하드파싱을 줄인다"와 "선택도 추정을 흐린다"는 동전의 양면입니다.

---

## ✅ 이 절(1.2.2)의 핵심 통찰

1. SQL은 이름이 없어 **전체 텍스트가 식별자**다 — 대소문자·공백·주석·힌트·스키마명 하나만 달라도 다른 SQL로 하드파싱된다.
2. **Literal SQL**은 값마다 SQL이 양산돼(100만 값 → 100만 SQL) 하드파싱 폭증·**CPU 급증**·캐시 경합을 부른다.
3. **바인드 변수**는 SQL 텍스트를 고정하고 값만 전달해(`LOGIN_ID = :1`) 하드파싱을 최초 1번으로 줄이고 실행계획을 공유한다.
4. 비유하면 고객별 프로시저(LOGIN_ORAKING…) 대신 **파라미터 함수 LOGIN(login_id) 하나**를 재사용하는 것이다.
5. 동시성 높은 **OLTP에서는 바인드 변수가 필수** — 단, 히스토그램 미활용이라는 트레이드오프가 있다(7.2.4).

## 🔗 흐름 속 위치

1.2.1에서 하드파싱의 비용을 봤고, 이 절은 그것을 줄이는 가장 중요한 코딩 원칙(바인드 변수)을 다룹니다. 1.2(SQL 공유 및 재사용)의 결론이며, 바인드 변수의 부작용(히스토그램 무력화)은 7.2.4 옵티마이저의 한계로 이어집니다.

📌 한 줄 요약: 바인드 변수는 값만 바꾸고 SQL 텍스트를 공유하게 만들어 하드파싱(CPU 부하)을 격감시키지만, 그 대가로 히스토그램을 활용하지 못합니다.
