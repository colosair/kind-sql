## 1.2.1 소프트 파싱 vs. 하드 파싱

### 핵심 개념

SQL 파싱, 최적화, 로우 소스 생성 과정을 거쳐 만든 내부 프로시저는 라이브러리 캐시에 저장되어 재사용됩니다. 이미 캐시에 있으면 소프트 파싱, 없으면 하드 파싱입니다.

```text
SQL 입력
   ↓
라이브러리 캐시 탐색
   ↓
찾음 → 소프트 파싱 → 실행
못 찾음 → 하드 파싱 → 최적화 → 로우 소스 생성 → 실행
```

> "SQL을 캐시에서 찾아 곧바로 실행단계로 넘어가는 것을 소프트 파싱이라 하고, 찾는 데 실패해 최적화 및 로우 소스 생성 단계까지 모두 거치는 것을 하드 파싱이라고 한다."

### 라이브러리 캐시

라이브러리 캐시는 SGA의 Shared Pool 안에 있는 영역으로, SQL 실행을 위한 내부 프로시저를 저장합니다.

```text
SGA
   └─ Shared Pool
        ├─ Library Cache
        └─ Data Dictionary Cache
```

### 하드 파싱이 비싼 이유 — 탐색 공간 폭발 (내비 비유)

내비게이션이 서울→부산 경로를 위해 모든 도로를 충망라하면 시간이 어마어마하게 걸려, 주요 도로 중심으로 후보군을 좁혀 가장 빠른 길을 찾듯, 옵티마이저도 거대한 탐색 공간을 다룹니다.

```text
5개 테이블 조인 → 조인 순서만 120(=5!)가지
  × 조인 방식 (NL / 소트 머지 / 해시)
  × 액세스 방식 (Index Range/Unique/Full/Fast Full/Skip Scan, Full Scan)
  × 테이블당 여러 인덱스 후보
   ↓
대충 계산해도 수십만 가지 → 순식간에 엄청난 연산 → CPU 폭증
```

데이터베이스 작업 대부분은 **I/O 중심**이지만, 하드 파싱은 **CPU를 많이 쓰는** 대표 작업입니다. 옵티마이저가 하드 파싱 시 사용하는 정보는 다음과 같습니다.

| 정보 | 내용 |
|------|------|
| 구조 정보 | 테이블·컬럼·인덱스 구조에 관한 기본 정보 |
| 오브젝트 통계 | 테이블·인덱스 통계, 히스토그램 포함 컬럼 통계 (7.1.2) |
| 시스템 통계 | CPU 속도, Single/Multiblock I/O 속도 등 (7.1.2) |
| 파라미터 | 옵티마이저 관련 초기화 파라미터 (7.2.3) |

---

## ✅ 이 절(1.2.1)의 핵심 통찰

1. 소프트 파싱은 기존 실행 프로시저를 재사용합니다.
2. 하드 파싱은 최적화와 로우 소스 생성까지 다시 수행합니다.
3. 하드 파싱은 CPU를 많이 소비합니다.
4. 라이브러리 캐시는 하드 파싱 비용을 줄이기 위한 핵심 구조입니다.
