## 1.2.1 소프트 파싱 vs. 하드 파싱

### 핵심 개념

SQL 파싱, 최적화, 로우 소스 생성 과정을 거쳐 만든 내부 프로시저는 라이브러리 캐시에 저장되어 재사용됩니다. 이미 캐시에 있으면 소프트 파싱, 없으면 하드 파싱입니다.

```text
SQL 입력
   ↓
라이브러리 캐시 탐색
   ↓
찾음 → 소프트 파싱 → 실행
못 찾음 → 하드 파싱 → 최적화 → 로우 소스 생성 → 실행
```

> "SQL을 캐시에서 찾아 곧바로 실행단계로 넘어가는 것을 소프트 파싱이라 하고, 찾는 데 실패해 최적화 및 로우 소스 생성 단계까지 모두 거치는 것을 하드 파싱이라고 한다."

### 라이브러리 캐시

라이브러리 캐시는 SGA의 Shared Pool 안에 있는 영역으로, SQL 실행을 위한 내부 프로시저를 저장합니다.

```text
SGA
   └─ Shared Pool
        ├─ Library Cache
        └─ Data Dictionary Cache
```

### 하드 파싱이 비싼 이유

하드 파싱은 단순 문법 분석이 아닙니다. 옵티마이저가 수많은 실행계획 후보를 만들고 통계정보를 읽어 비용을 비교해야 합니다.

```text
조인 순서 후보
× 조인 방식 후보
× 인덱스 후보
× 스캔 방식 후보
   ↓
거대한 탐색 공간
   ↓
CPU 사용량 증가
```

데이터베이스 작업 대부분은 I/O 중심이지만, 하드 파싱은 CPU를 많이 쓰는 대표 작업입니다.

---

## ✅ 이 절(1.2.1)의 핵심 통찰

1. 소프트 파싱은 기존 실행 프로시저를 재사용합니다.
2. 하드 파싱은 최적화와 로우 소스 생성까지 다시 수행합니다.
3. 하드 파싱은 CPU를 많이 소비합니다.
4. 라이브러리 캐시는 하드 파싱 비용을 줄이기 위한 핵심 구조입니다.

## 🔗 흐름 속 위치
