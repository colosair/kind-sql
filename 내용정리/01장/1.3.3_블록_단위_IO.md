## 1.3.3 블록 단위 I/O

### 핵심 개념

Oracle은 데이터를 한 행씩 읽지 않고 블록 단위로 읽습니다. 한 행만 필요해도 그 행이 들어 있는 블록 전체를 읽습니다.

```text
특정 로우 필요
   ↓
해당 로우가 포함된 블록 탐색
   ↓
블록 전체 읽기
   ↓
블록 안에서 로우 찾기
```

### 왜 블록 단위인가 — 파일 단위 비유

클라우드의 문서는 파일 단위로 저장·읽습니다(그림 1-13). 데이터베이스에서 읽고 쓰는 단위는 무엇일까요(그림 1-14)?

- **파일 단위**(예: 2GB)로 매번 읽고 쓰는 건 상상하기 어렵습니다.
- **세그먼트 단위**(예: 100MB)도 마찬가지입니다.
- **익스텐트**는 공간을 확장하는 단위일 뿐입니다.
- → **블록이 바로 DBMS가 데이터를 읽고 쓰는 단위**입니다.

| 관점 | 단위 |
|---|---|
| SQL 결과 | 로우 |
| **DBMS I/O** | **블록** |
| 저장장치 | 물리 섹터/페이지 |

### 블록 크기 — 1Byte를 읽어도 8KB를 읽는다

> 데이터 I/O 단위가 블록이므로 **특정 레코드 하나를 읽고 싶어도 해당 블록을 통째로** 읽습니다. **심지어 1Byte짜리 컬럼 하나만 읽고 싶어도 블록을 통째로** 읽습니다.

오라클은 기본적으로 **8KB 블록**을 사용하므로, 1Byte를 읽기 위해 8KB를 읽는 셈입니다. (각주: 2KB·4KB·16KB도 가능, OS에 따라 32KB도 지원. MS-SQL Server는 8KB 페이지)

```sql
SQL> show parameter block_size
--  NAME            TYPE     VALUE
--  db_block_size   integer  8192      ← 8KB
```

### 튜닝 관점

블록 단위 I/O 때문에 다음 원리가 중요해집니다.

```text
필요한 로우들이 같은 블록에 모여 있음
   ↓
적은 블록 읽기
   ↓
빠름

필요한 로우들이 여러 블록에 흩어져 있음
   ↓
많은 블록 읽기
   ↓
느림
```

따라서 인덱스로 로우를 몇 건 찾느냐보다, 그 로우들이 몇 개 테이블 블록에 흩어져 있는지가 중요합니다.

---

## ✅ 이 절(1.3.3)의 핵심 통찰

1. Oracle I/O는 블록 단위로 발생합니다.
2. 로우 1건 조회도 블록 전체를 읽습니다.
3. 필요한 로우가 같은 블록에 모여 있을수록 효율적입니다.
4. 블록 단위 I/O는 클러스터링 팩터와 랜덤 액세스 비용의 기반입니다.

## 🔗 흐름 속 위치
