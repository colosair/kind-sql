## 1.3.3 블록 단위 I/O

### 핵심 개념

Oracle은 데이터를 한 행씩 읽지 않고 블록 단위로 읽습니다. 한 행만 필요해도 그 행이 들어 있는 블록 전체를 읽습니다.

```text
특정 로우 필요
   ↓
해당 로우가 포함된 블록 탐색
   ↓
블록 전체 읽기
   ↓
블록 안에서 로우 찾기
```

### 왜 블록 단위인가

디스크와 메모리는 바이트 단위로 데이터가 존재하지만, DBMS는 효율을 위해 고정 크기 블록 단위로 관리합니다.

| 관점 | 단위 |
|---|---|
| SQL 결과 | 로우 |
| DBMS I/O | 블록 |
| 저장장치 | 물리 섹터/페이지 |

### 튜닝 관점

블록 단위 I/O 때문에 다음 원리가 중요해집니다.

```text
필요한 로우들이 같은 블록에 모여 있음
   ↓
적은 블록 읽기
   ↓
빠름

필요한 로우들이 여러 블록에 흩어져 있음
   ↓
많은 블록 읽기
   ↓
느림
```

따라서 인덱스로 로우를 몇 건 찾느냐보다, 그 로우들이 몇 개 테이블 블록에 흩어져 있는지가 중요합니다.

---

## ✅ 이 절(1.3.3)의 핵심 통찰

1. Oracle I/O는 블록 단위로 발생합니다.
2. 로우 1건 조회도 블록 전체를 읽습니다.
3. 필요한 로우가 같은 블록에 모여 있을수록 효율적입니다.
4. 블록 단위 I/O는 클러스터링 팩터와 랜덤 액세스 비용의 기반입니다.

## 🔗 흐름 속 위치

1.3.2에서 저장 구조를 봤습니다. 이 절은 그중 블록이 왜 SQL 성능의 핵심 단위인지 설명합니다.

📌 한 줄 요약: DBMS는 행을 읽는 것이 아니라 행이 들어 있는 블록을 읽습니다.
