## 1.3.2 데이터베이스 저장 구조

### 핵심 개념

Oracle 데이터베이스는 데이터를 블록, 익스텐트, 세그먼트, 테이블스페이스, 데이터파일 계층으로 저장합니다. SQL 튜닝에서 중요한 최소 I/O 단위는 블록입니다.

```text
테이블 / 인덱스
   ↓
세그먼트
   ↓
익스텐트
   ↓
블록
   ↓
데이터파일
   ↓
디스크
```

### 저장 구조 계층 (그림 1-10)

| 단위 | 설명 |
|---|---|
| **블록(Block)** | 데이터를 읽고 쓰는 **최소 단위** = 실제 레코드 저장 공간. (DB2·SQL Server는 '페이지(page)'라 부름) |
| **익스텐트(Extent)** | **공간 확장 단위** = 연속된 블록의 집합. 공간 부족 시 테이블스페이스에서 추가 할당 |
| **세그먼트(Segment)** | 테이블·인덱스처럼 저장공간이 필요한 오브젝트. 파티션이면 **각 파티션이 하나의 세그먼트**, LOB도 별도 세그먼트 |
| **테이블스페이스(Tablespace)** | 세그먼트를 담는 컨테이너 |
| **데이터파일(Datafile)** | 테이블스페이스를 구성하는 디스크 상의 물리적 OS 파일 |

**중요한 두 가지 독점 규칙**:
- **블록은 하나의 테이블이 독점** — 한 블록에 저장된 레코드는 모두 같은 테이블 레코드입니다. (예외: 다중 테이블 클러스터링 — 3.1.7)
- **익스텐트도 하나의 테이블이 독점** — 한 익스텐트의 블록은 모두 같은 테이블 블록입니다. (MS-SQL Server는 한 익스텐트를 여러 오브젝트가 공유 가능)

**익스텐트끼리는 연속 공간이 아닙니다**(그림 1-11). 한 세그먼트의 익스텐트들이 같은 데이터파일에 없을 수 있고, 오히려 **서로 다른 데이터파일에 흩어질 가능성이 더 높습니다** — 파일 경합을 줄이려 DBMS가 일부러 여러 데이터파일로 분산하기 때문입니다.

```sql
-- 세그먼트에 할당된 익스텐트 목록 조회 (dba_extents 뷰)
SQL> select segment_type, tablespace_name, extent_id, file_id, block_id, blocks
  2  from dba_extents
  3  where owner = USER and segment_name = 'MY_SEGMENT'
  4  order by extent_id;
```

### 튜닝 관점

SQL 성능을 이해하려면 "몇 건을 읽었는가"보다 "몇 블록을 읽었는가"가 더 중요합니다.

```text
로우 1건 조회
   ↓
해당 로우가 있는 블록 읽기

로우 여러 건 조회
   ↓
같은 블록에 모여 있으면 I/O 적음
   ↓
흩어져 있으면 I/O 많음
```

이는 이후 클러스터링 팩터, 인덱스 손익분기점, 랜덤 액세스 비용의 기반이 됩니다.

---

## ✅ 이 절(1.3.2)의 핵심 통찰

1. Oracle의 기본 I/O 단위는 블록입니다.
2. 테이블과 인덱스는 세그먼트로 저장됩니다.
3. SQL 성능은 로우 수보다 블록 수에 더 직접적으로 좌우됩니다.
4. 데이터가 블록에 어떻게 모여 있는지가 인덱스 효율을 결정합니다.

## 🔗 흐름 속 위치

1.3.1에서 I/O가 느림의 원인임을 봤습니다. 이 절은 그 I/O가 어떤 저장 단위를 대상으로 발생하는지 설명합니다.

📌 한 줄 요약: SQL 튜닝은 결국 로우가 아니라 블록을 얼마나 적게 읽느냐의 문제입니다.
