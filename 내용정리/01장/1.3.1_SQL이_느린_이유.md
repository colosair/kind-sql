## 1.3.1 SQL이 느린 이유

### 핵심 개념

SQL이 느린 이유는 대부분 I/O 때문입니다. 특히 디스크 I/O는 프로세스를 대기 상태로 만들기 때문에 성능에 치명적입니다.

> "SQL이 느린 이유는 십중팔구 I/O 때문이다."

```text
프로세스 실행
   ↓
디스크 I/O 필요
   ↓
OS I/O Call
   ↓
CPU 반환 후 Waiting
   ↓
I/O 완료 대기
   ↓
다시 Runnable
```

책에서는 I/O를 `SLEEP`으로 설명합니다.

```text
I/O = SLEEP
```

### 프로세스 생명주기와 I/O (그림 1-8)

프로세스는 '실행 중인 프로그램'이며, 생성(new) 이후 종료(terminated) 전까지 준비(ready)·실행(running)·대기(waiting)를 반복합니다.

```text
 new ─admitted─► ready ─dispatched─► running ─exit─► terminated
                   ▲                    │
                   │ ◄──interrupted─────┤   (CPU 선점, 다시 ready로)
                   │                    │ I/O 요청 또는 이벤트 대기
                   └──I/O 완료/이벤트완료── waiting ◄┘
```

핵심: 여러 프로세스가 하나의 CPU를 공유하므로, **디스크에서 데이터를 읽어야 할 땐 CPU를 OS에 반환(I/O Call)하고 `waiting`(Wait Queue)에서 잠**을 잡니다(그림 1-9). 열심히 일해야 할 프로세스가 한가하게 잠을 자니 I/O가 많으면 느릴 수밖에 없습니다.

### I/O Call 속도 — 여전히 느리다

| 스토리지 | Single Block I/O 1회 | 초당 읽는 블록 |
|------|------|------|
| 일반 디스크 | 평균 **10ms** | 약 100 블록 |
| 큰 캐시 SAN 스토리지 | 4~8ms | 125~250 블록 |
| 최신 SSD | 1~2ms | 500~1,000 블록 |

> 스토리지가 빨라지고 있지만 여전히 우리 기대엔 못 미칩니다. 어떤 SQL이 Single Block I/O로 100만 블록을 읽는다면, 가장 최신 SSD에서도 **10초 이상** 기다려야 합니다 — 동시다발 I/O로 디스크 경합이 심하면 **20초**가 될 수도 있습니다. **"디스크 I/O가 SQL 성능을 좌우한다"** 는 말이 과언이 아닌 이유입니다.

### 튜닝 관점

```text
I/O 많음
   ↓
프로세스 대기 증가
   ↓
응답시간 증가
   ↓
SQL 느림
```

따라서 SQL 튜닝의 핵심은 결국 읽어야 할 블록 수를 줄이는 것입니다.

---

## ✅ 이 절(1.3.1)의 핵심 통찰

1. SQL 성능 문제의 핵심은 대부분 I/O입니다.
2. 디스크 I/O 중 프로세스는 일을 하지 못하고 대기합니다.
3. I/O가 많으면 CPU가 빨라도 SQL은 느릴 수밖에 없습니다.
4. SQL 튜닝의 통일 원리는 읽는 블록 수를 줄이는 것입니다.

## 🔗 흐름 속 위치

1.1~1.2가 SQL 실행 절차와 공유 문제였다면, 1.3부터는 SQL 성능의 물리적 기반인 I/O 메커니즘을 다룹니다.

📌 한 줄 요약: SQL이 느린 근본 이유는 프로세스를 잠들게 만드는 I/O 때문입니다.
