## 1.1.5 옵티마이저 힌트

### 핵심 개념

자동차 내비게이션이 보편적으로 좋은 선택을 하지만 항상 최선은 아니듯, SQL 옵티마이저도 완벽하진 않습니다. 운전자가 아는 정보·경험으로 더 빨리 도착하듯, **통계정보에 담을 수 없는 데이터/업무 특성을 아는 개발자가 직접 액세스 경로를 지정**할 수 있습니다 — 그 수단이 **옵티마이저 힌트**입니다.

```sql
-- 주석 기호에 '+'를 붙이면 힌트
SELECT /*+ INDEX(A 고객_PK) */
       고객명, 연락처, 주소, 가입일시
FROM   고객 A
WHERE  고객ID = '000000008';
```

---

### 힌트 표기 — `/*+ ... */` 권장

`--+` 방식도 있지만 **가급적 쓰지 말아야** 합니다.

```sql
-- ⚠️ --+ 방식: 문자열로 SQL을 조립할 때 줄바꿈이 사라지면 아래쪽 라인이 전부 주석 처리됨
SQLStmt = "SELECT --+ INDEX(A 고객_PK) "
        + "       고객명, 연락처, 주소, 가입일시 "   ← 모두 주석으로 처리되는 사고
        + " FROM 고객 A WHERE 고객ID = '000000008' ";
```

### 힌트 사용 주의사항

| 주의사항 | 예시 |
|------|------|
| 힌트 **사이에 콤마 금지** (인자 나열엔 콤마 OK) | `/*+ INDEX(A A_X01) INDEX(B B_X03) */` → 모두 유효<br>`/*+ INDEX(C), FULL(D) */` → **첫 번째만 유효** |
| 테이블에 **스키마명 금지** | `/*+ FULL(SCOTT.EMP) */` → **무효** |
| FROM에 Alias 썼으면 **힌트도 Alias** | `FROM EMP E`인데 `/*+ FULL(EMP) */` → **무효** (`FULL(E)`로 써야 함) |

핵심: 힌트 문법이 틀리면 **에러가 아니라 조용히 무시**됩니다(7.2.3 힌트 무시 5경우).

---

### 자율이냐 강제냐 — 부분 유도 vs 전체 고정

내비게이션에 음성으로 *"중부내륙고속도로로 가다가 대구에서 경부고속도로로 진입! 나머지 경로는 알아서 결정"* 이라고 지시하듯, 일부 경로만 정하고 나머지는 옵티마이저에 맡길 수 있습니다.

```sql
-- 부분 유도: 주문 테이블 액세스 인덱스만 지정, 조인 순서·방식·고객 액세스는 옵티마이저 판단
SELECT /*+ INDEX(A (주문일자)) */
       A.주문번호, A.주문금액, B.고객명, B.연락처, B.주소
FROM   주문 A, 고객 B
WHERE  A.주문일자 = :ORD_DT AND A.고객ID = B.고객ID;
```
```sql
-- 전체 고정: 옵티마이저가 절대 다른 방식을 못 고르도록 빈틈없이 지정
SELECT /*+ LEADING(A) USE_NL(B) INDEX(A (주문일자)) INDEX(B 고객_PK) */
       A.주문번호, A.주문금액, B.고객명, B.연락처, B.주소
FROM   주문 A, 고객 B
WHERE  A.주문일자 = :ORD_DT AND A.고객ID = B.고객ID;
```

> 옵티마이저의 작은 실수가 기업에 큰 손실을 끼치는 **중대한 시스템**이라면, 자율 판단에 맡기지 말고 힌트로 **빈틈없이 기술**해야 할 때가 있습니다.

---

### 자주 사용하는 힌트 목록 [표 1-1]

| 분류 | 힌트 | 설명 |
|------|------|------|
| **최적화 목표** | `ALL_ROWS` / `FIRST_ROWS(N)` | 전체 처리속도 / 최초 N건 응답속도 최적화 |
| **액세스 방식** | `FULL` | Table Full Scan 유도 |
| | `INDEX` / `INDEX_DESC` | Index Scan / 역순 Index Scan |
| | `INDEX_FFS` / `INDEX_SS` | Index Fast Full Scan / Index Skip Scan |
| **조인 순서** | `ORDERED` | FROM 절에 나열된 순서대로 조인 |
| | `LEADING` | 괄호에 기술한 순서대로 조인 (예: `LEADING(T1 T2)`) |
| | `SWAP_JOIN_INPUTS` | 해시 조인 시 Build Input을 명시적으로 선택 |
| **조인 방식** | `USE_NL` / `USE_MERGE` / `USE_HASH` | NL / 소트 머지 / 해시 조인 유도 |
| | `NL_SJ` / `MERGE_SJ` / `HASH_SJ` | NL / 소트 머지 / 해시 **세미조인** 유도 |
| **서브쿼리 팩토링** | `MATERIALIZE` / `INLINE` | WITH 절 집합을 물리적으로 생성 / 생성 안 하고 인라인 |
| **쿼리 변환** | `MERGE` / `NO_MERGE` | 뷰 머징 유도 / 방지 |
| | `UNNEST` / `NO_UNNEST` | 서브쿼리 Unnesting 유도 / 방지 |
| | `PUSH_PRED` / `NO_PUSH_PRED` | 조인조건 Pushdown 유도 / 방지 |
| | `USE_CONCAT` / `NO_EXPAND` | OR·IN-List를 OR-Expansion 유도 / 방지 |
| **병렬 처리** | `PARALLEL` / `PARALLEL_INDEX` | 테이블·DML / 인덱스 스캔 병렬 처리 (예: `PARALLEL(T1 2)`) |
| | `PQ_DISTRIBUTE` | 병렬 수행 시 데이터 분배 방식 결정 |
| **기타** | `APPEND` | Direct-Path Insert 유도 (6.2.2) |
| | `DRIVING_SITE` | DB Link Remote 쿼리 최적화·실행 주체 지정 |
| | `PUSH_SUBQ` / `NO_PUSH_SUBQ` | 서브쿼리를 가급적 빨리 / 늦게 필터링 |

(몇 가지를 제외하곤 이후 장에서 자세히 설명되므로, 지금은 "이런 제어 수단이 있다" 정도만 파악하면 됩니다.)

## ✅ 이 절(1.1.5)의 핵심 통찰

1. 힌트는 통계로 표현 못 하는 업무·데이터 특성을 아는 개발자가 **실행계획 선택에 개입**하는 수단이다.
2. 표기는 **`/*+ ... */`** 를 쓴다 — `--+`는 문자열 조립 시 줄바꿈 사고로 뒷줄이 주석 처리될 수 있다.
3. 문법이 틀리면 **에러 없이 조용히 무시**된다 — 힌트 사이 콤마·스키마명·Alias 누락이 대표 실수다.
4. **부분 유도**(일부 경로만)와 **전체 고정**(LEADING+USE_NL+INDEX로 빈틈없이) 둘 다 가능하다.
5. 표 1-1의 힌트는 최적화 목표·액세스·조인순서/방식·서브쿼리·쿼리변환·병렬·기타로 분류되며, 이 책 전체의 튜닝 도구 지도다.

## 🔗 흐름 속 위치

1.1.4에서 본 실행계획이 기대와 다를 때, 개발자가 개입하는 방법이 힌트입니다. 1.1(SQL 파싱과 최적화)의 마지막 절로, 이후 모든 장의 튜닝 기법은 결국 "옵티마이저가 좋은 계획을 고르게 하거나, 힌트로 직접 유도"하는 것으로 귀결됩니다(7.2.5 개발자의 역할).

📌 한 줄 요약: 힌트는 옵티마이저의 자율 판단을 보정·제한하는 실행계획 제어 장치이며, `/*+ ... */`로 정확히 기술하지 않으면 조용히 무시됩니다.
