## 1.1.5 옵티마이저 힌트

### 핵심 개념

옵티마이저 힌트는 개발자가 옵티마이저에게 특정 실행경로를 선택하도록 지시하는 수단입니다. 옵티마이저가 항상 최선의 선택을 하는 것은 아니므로, 업무 특성이나 데이터 특성을 아는 개발자가 개입할 수 있습니다.

```sql
SELECT /*+ INDEX(A 고객_PK) */
       고객명, 연락처, 주소, 가입일시
FROM 고객 A
WHERE 고객ID = '000000008';
```

```text
옵티마이저 기본 판단
   ↓
통계정보 기반 선택
   ↓
개발자가 더 나은 경로를 알고 있음
   ↓
힌트로 실행계획 유도
```

### 힌트 사용 주의사항

| 주의사항 | 설명 |
|---|---|
| `/*+ ... */` 사용 | `--+` 방식은 줄바꿈 오류에 취약합니다. |
| 힌트 사이 콤마 금지 | 힌트와 힌트 사이에 콤마를 넣으면 일부만 유효할 수 있습니다. |
| 스키마명 금지 | 힌트의 테이블 지정에 스키마명을 쓰면 무효입니다. |
| Alias 사용 | FROM 절에 Alias를 썼으면 힌트에도 Alias를 써야 합니다. |

### 자율과 강제

힌트는 일부만 줄 수도 있고, 실행계획 전체를 강하게 고정할 수도 있습니다.

```sql
SELECT /*+ INDEX(A (주문일자)) */
...
```

```text
주문 테이블 액세스 인덱스만 지정
   ↓
조인 순서/조인 방식은 옵티마이저 판단
```

```sql
SELECT /*+ LEADING(A) USE_NL(B) INDEX(A (주문일자)) INDEX(B 고객_PK) */
...
```

```text
조인 순서
+ 조인 방식
+ 각 테이블 액세스 방식
   ↓
실행계획 강하게 고정
```

### 대표 힌트

| 범주 | 힌트 |
|---|---|
| 최적화 목표 | `ALL_ROWS`, `FIRST_ROWS(N)` |
| 액세스 방식 | `FULL`, `INDEX`, `INDEX_DESC`, `INDEX_FFS`, `INDEX_SS` |
| 조인 순서 | `ORDERED`, `LEADING` |
| 조인 방식 | `USE_NL`, `USE_MERGE`, `USE_HASH` |
| 서브쿼리 | `UNNEST`, `NO_UNNEST`, `PUSH_SUBQ` |
| 병렬/DML | `PARALLEL`, `APPEND` |

---

## ✅ 이 절(1.1.5)의 핵심 통찰

1. 힌트는 옵티마이저의 실행계획 선택에 개입하는 수단입니다.
2. 힌트 문법이 틀리면 조용히 무시될 수 있습니다.
3. 일부 경로만 유도할 수도 있고, 전체 실행계획을 강하게 고정할 수도 있습니다.
4. 중요한 시스템에서는 힌트를 쓸 때 빈틈없이 기술해야 할 때가 있습니다.

## 🔗 흐름 속 위치
