## 1.1.4 실행계획과 비용

### 핵심 개념

내비게이션에 '경로 미리보기' 기능이 있듯, DBMS에도 **'SQL 실행경로 미리보기'** 기능이 있습니다 — 그것이 **실행계획(Execution Plan)** 입니다. 옵티마이저가 생성한 처리절차를 사용자가 트리 구조로 확인할 수 있게 표현한 것입니다.

```text
SQL → 옵티마이저 → 실행계획 생성 → Cost 비교 → 최종 실행계획 선택
```

### 실행계획 읽기

실행계획을 보면 SQL이 테이블을 Full Scan하는지 인덱스를 스캔하는지, 어떤 조인 방식을 선택했는지 확인할 수 있고, 예상과 다르면 실행경로를 변경할 수 있습니다.

```text
Execution Plan
---------------------------------------------------------------------
0   SELECT STATEMENT Optimizer=ALL_ROWS (Cost=209 Card=5 Bytes=175)
1 0   TABLE ACCESS (BY INDEX ROWID) OF 'EMP'  (Cost=2 Card=5 Bytes=85)
2 1    NESTED LOOPS  (Cost=209 Card=5 Bytes=175)
3 2     TABLE ACCESS (BY INDEX ROWID) OF 'DEPT' (Cost=207 Card=1 Bytes=18)
4 3      INDEX (RANGE SCAN) OF 'DEPT_LOC_IDX' (NON-UNIQUE) (Cost=7 Card=1)
5 3     INDEX (RANGE SCAN) OF 'EMP_DEPTNO_IDX' (NON-UNIQUE) (Cost=1 Card=5)
```
각 단계에 **`Cost`(예상 비용)·`Card`(예상 카디널리티)·`Bytes`(예상 데이터량)** 가 함께 표시됩니다.

---

### 실험 — 같은 SQL, 세 가지 비용

옵티마이저가 실행계획을 선택하는 근거가 무엇인지 직접 확인합니다.

```sql
-- 테스트 테이블 (scott.emp 14건 × 1000 = 14,000건)
SQL> create table t as
  2  select d.no, e.* from scott.emp e
  3       , (select rownum no from dual connect by level <= 1000) d;

SQL> create index t_x01 on t(deptno, no);
SQL> create index t_x02 on t(deptno, job, no);
SQL> exec dbms_stats.gather_table_stats(user, 't');   -- 통계 수집
```

같은 조건(`deptno=10 and no=1`)을 세 가지 경로로 강제해 비용을 비교합니다.

```sql
SQL> set autotrace traceonly exp;

-- ① 옵티마이저 선택 (T_X01)
SQL> select * from t where deptno = 10 and no = 1;
--  TABLE ACCESS BY INDEX ROWID BATCHED  T      Cost=2
--   INDEX RANGE SCAN                    T_X01  Cost=1

-- ② T_X02 강제
SQL> select /*+ index(t t_x02) */ * from t where deptno = 10 and no = 1;
--  TABLE ACCESS BY INDEX ROWID BATCHED  T      Cost=19

-- ③ Full Scan 강제
SQL> select /*+ full(t) */ * from t where deptno = 10 and no = 1;
--  TABLE ACCESS FULL                    T      Cost=29
```

| 실행경로 | Cost | 결과 |
|------|---:|------|
| **T_X01 인덱스** | **2** | 옵티마이저가 자동 선택 ★ |
| T_X02 인덱스 | 19 | 더 많은 I/O 예상 |
| Full Scan | 29 | 이 예에서 가장 비쌈 |

옵티마이저가 T_X01을 선택한 근거가 바로 **비용(Cost)** 입니다. (T_X01은 `deptno + no`로 조건 두 개를 모두 인덱스로 거르지만, T_X02는 `deptno + job + no`라 `no` 앞의 `job`이 조건에 없어 비효율 — 3.3장 액세스 조건 참고)

---

### Cost는 '예상치'다 — 내비게이션 비유

> 비용(Cost)은 쿼리를 수행하는 동안 발생할 것으로 **예상하는** I/O 횟수 또는 예상 소요시간을 표현한 값입니다.

내비게이션이 보여주는 도착 예상시간이 토요일 오후엔 30분 늦거나 새벽엔 빨리 도착하듯, **SQL Cost도 어디까지나 예상치**입니다. 통계정보가 부정확하거나 데이터 분포가 왜곡되면 실제 성능과 다를 수 있습니다.

```text
통계정보 → 예상 카디널리티 → 예상 I/O → Cost
                                   (실측 아님!)
```

---

## ✅ 이 절(1.1.4)의 핵심 통찰

1. 실행계획은 옵티마이저가 선택한 처리절차의 **미리보기(트리 구조)** 이며, 단계마다 Cost/Card/Bytes가 붙는다.
2. 같은 SQL·같은 조건이라도 경로에 따라 Cost가 다르다 — 실험에서 **T_X01=2, T_X02=19, Full Scan=29**.
3. 옵티마이저는 **가장 낮은 Cost(2)** 의 경로를 고른다 — 이것이 실행계획 선택의 근거다.
4. **Cost는 실측이 아니라 통계 기반 예상치** — 내비 도착시간처럼 실제와 어긋날 수 있다.
5. 실행계획을 읽을 줄 알아야 "왜 이 경로를 골랐나"를 진단하고 힌트로 교정할 수 있다.

## 🔗 흐름 속 위치

1.1.3에서 옵티마이저가 비용 기반으로 실행계획을 고른다고 봤고, 이 절은 그 결과물(실행계획)과 근거(Cost)를 실험으로 확인했습니다. 다음 1.1.5는 그 선택을 사용자가 **힌트**로 바꾸는 방법입니다.

📌 한 줄 요약: 실행계획은 옵티마이저의 선택 결과이고 Cost는 그 선택의 **예상** 근거이며, 같은 SQL도 경로에 따라 Cost가 크게 갈립니다(2 vs 29).
