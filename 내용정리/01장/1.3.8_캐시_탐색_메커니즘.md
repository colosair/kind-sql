## 1.3.8 캐시 탐색 메커니즘

### 핵심 개념

Direct Path I/O를 제외한 **모든 블록 I/O는 메모리 버퍼캐시를 경유**합니다. 구체적으로 다음 오퍼레이션이 모두 버퍼캐시 탐색을 거칩니다.

```text
1. 인덱스 루트 블록을 읽을 때                       (루트 주소는 파싱 시 SQL 커서에 담김)
2. 인덱스 루트에서 얻은 주소로 브랜치 블록을 읽을 때
3. 인덱스 브랜치에서 얻은 주소로 리프 블록을 읽을 때
4. 인덱스 리프에서 얻은 주소로 테이블 블록을 읽을 때
5. 테이블 블록을 Full Scan 할 때                    (읽을 블록은 익스텐트 맵에서, 1.3.4)
```

---

### 해시 구조로 탐색 (그림 1-27)

버퍼캐시는 **해시(Hash) 구조**로 관리됩니다. 블록의 고유 주소인 **DBA(Data Block Address)** 를 해시 함수에 넣어 해시 체인을 찾고, 그 체인 안에서 원하는 버퍼를 탐색합니다.

> **DBA(Data Block Address)**: 모든 데이터블록이 "몇 번 데이터파일의 몇 번째 블록"인지 나타내는 디스크상의 고유 주소.

```text
해시 함수 f(x) = mod(x, 5)          해시 체인(버킷)
DBA를 입력 ──► mod(20,5)=0 ──► [0] → 5 → 20 → 15
              mod(27,5)=2 ──► [2] → 7            ← 27 없음
                                                  → 디스크에서 읽어 체인 2에 연결
```

**탐색 예**:
- 20번 블록 요청 → `mod(20,5)=0` → 첫 번째 체인 탐색 → **캐싱돼 있으면 거기서 발견(논리 I/O)**
- 27번 블록 요청 → `mod(27,5)=2` → 세 번째 체인 탐색 → **없으면 디스크에서 읽어(물리 I/O)** 체인에 연결

**해시 구조의 특징** (HashMap과 동일):

| 특징 | 의미 |
|------|------|
| 같은 입력값 → 같은 체인 | 동일 DBA는 항상 같은 버킷 |
| 다른 입력값도 같은 체인 가능 | 예: 4와 9가 같은 버킷 (해시 충돌 = Separate Chaining) |
| 체인 내 정렬 보장 안 됨 | 충돌 엔트리는 연결 리스트 |

```text
key    = DBA (데이터파일 번호 + 블록 번호)
hash   = mod(DBA, 버킷수)
bucket = 해시 체인
chain  = 충돌 엔트리 연결 리스트
```

### 메모리 공유자원 액세스 직렬화 — 래치(Latch)

버퍼캐시는 SGA 구성요소이므로 **모든 프로세스가 공유**합니다. 두 프로세스가 같은 버퍼블록(또는 해시 체인)에 **동시에 접근하면 정합성 문제**가 생깁니다. 그래서 한 프로세스씩 **순차적으로(직렬화, serialization)** 접근하도록 '줄 세우기'를 합니다.

> **공유경제(Sharing Economy) 비유**: 카셰어링은 차를 같이 쓰는 것 같지만, **한순간에는 한 사람만** 운전합니다 — 다른 사람은 줄 서서 기다려야 합니다. 숙박 공유도 방 하나를 동시에 쓰는 게 아니라 시간을 달리해 씁니다. 버퍼캐시도 특정 순간엔 한 프로세스만 접근하고, 이런 **줄 세우기를 가능하게 하는 메커니즘이 래치(Latch)** 입니다.

#### 캐시버퍼 체인 래치 (Cache Buffers Chains Latch)

대량 데이터를 읽으면 모든 블록에 대해 해시 체인을 탐색합니다. 체인을 스캔하는 동안 다른 프로세스가 체인 구조를 변경하는 것을 막기 위해 **해시 체인 래치**가 존재합니다(그림 1-27에서 0~4 다섯 체인 앞의 자물쇠). **자물쇠를 열 키(Key)를 획득한 프로세스만 체인에 진입**할 수 있습니다.

```text
논리적 I/O 증가
   ↓
해시 체인 탐색 증가
   ↓
캐시버퍼 체인 래치 경합 ('줄 세우기' 길어짐)
   ↓
CPU 사용 증가 + 대기
```

→ 따라서 **"캐시에 있으니 공짜"가 아닙니다.** 캐시에 있더라도 논리적 I/O(읽는 블록 수)가 많으면 해시 탐색·래치 경합 비용이 쌓입니다. 1.3.5의 "논리적 I/O를 줄여라"가 이 메커니즘 때문에 정당화됩니다.

---

## ✅ 이 항목의 핵심 통찰

1. Direct Path I/O를 뺀 **모든 블록 I/O는 버퍼캐시를 경유**한다(인덱스 루트/브랜치/리프/테이블 액세스/Full Scan).
2. 버퍼캐시는 **DBA를 키로 한 해시 구조**(HashMap + Separate Chaining)로 탐색한다 — 같은 DBA는 같은 체인, 충돌 가능, 체인 내 정렬 없음.
3. 버퍼캐시는 공유자원이라 **래치(Latch)로 접근을 직렬화**한다 — 공유경제(카셰어링)처럼 한순간엔 한 프로세스만.
4. **캐시버퍼 체인 래치**는 해시 체인 진입을 통제하므로, 논리적 I/O가 많으면 래치 경합·CPU가 증가한다.
5. 그래서 **캐시에 있어도 많이 읽으면 느리다** — "논리적 I/O를 줄여라"의 근본 이유다.

## 🔗 흐름 속 위치

1.3.5에서 "논리적 I/O가 더 근본적 레버"라고 했고, 1.3.8은 그 논리적 I/O가 내부적으로 **해시 탐색 + 래치 경합 비용**을 가짐을 보여 그 주장을 완성합니다. 여기서 본 래치·직렬화 개념은 6.4(Lock과 동시성 제어)로 이어지며, 1장 전체의 통일 원리 `reduce I/O`를 메커니즘 수준에서 마무리합니다.

📌 한 줄 요약: 버퍼캐시는 DBA를 키로 한 HashMap처럼 빠르게 찾지만, 해시 탐색과 래치(줄 세우기) 비용이 있으므로 **논리적 I/O 자체를 줄여야** 합니다.
