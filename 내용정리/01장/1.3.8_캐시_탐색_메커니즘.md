## 1.3.8 캐시 탐색 메커니즘

### 핵심 개념

Oracle은 블록을 읽을 때 먼저 버퍼캐시에서 해당 블록이 있는지 찾습니다. 버퍼캐시는 해시 구조로 관리되며, 블록 주소를 해시 함수에 넣어 해시 체인을 찾고 그 안에서 원하는 버퍼를 탐색합니다.

```text
블록 요청
   ↓
해시 함수 적용
   ↓
해시 버킷 선택
   ↓
해시 체인 탐색
   ↓
버퍼 발견 → 논리적 I/O
버퍼 없음 → 물리적 I/O
```

### HashMap 비유

CS 관점에서는 버퍼캐시 탐색이 `HashMap`과 유사합니다.

```text
key   = 데이터파일 번호 + 블록 번호
hash  = 해시 함수(key)
bucket = hash bucket
chain = 충돌 엔트리 연결 리스트
```

다만 일반 HashMap과 달리 DBMS는 여러 프로세스가 동시에 접근하므로 래치/Lock 같은 동시성 제어가 필요합니다.

### 캐시 탐색 비용

캐시에 있는 블록을 읽는 논리적 I/O도 공짜가 아닙니다. 해시 체인 탐색, 래치 획득, 버퍼 상태 확인 비용이 발생합니다.

```text
논리적 I/O 증가
   ↓
해시 체인 탐색 증가
   ↓
래치/버퍼 경합 가능
   ↓
CPU 사용 증가
```

따라서 "캐시에 있으니 괜찮다"가 아니라, 캐시에 있더라도 읽는 블록 수를 줄여야 합니다.

---

## ✅ 이 절(1.3.8)의 핵심 통찰

1. 버퍼캐시는 해시 구조로 탐색합니다.
2. 블록 주소가 캐시 탐색의 키 역할을 합니다.
3. 논리적 I/O도 해시 탐색과 동시성 제어 비용을 가집니다.
4. 캐시에 있어도 많이 읽으면 느립니다.

## 🔗 흐름 속 위치

1.3.5에서 논리적 I/O가 중요하다고 봤습니다. 이 절은 논리적 I/O가 내부적으로 어떤 탐색 비용을 갖는지 설명합니다.

📌 한 줄 요약: 버퍼캐시는 HashMap처럼 빠르게 찾지만, 논리적 I/O도 공짜가 아니므로 블록 수를 줄여야 합니다.
