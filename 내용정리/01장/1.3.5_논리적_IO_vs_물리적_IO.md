## 1.3.5 논리적 I/O vs. 물리적 I/O

### 핵심 개념

논리적 I/O는 버퍼캐시에서 블록을 읽는 작업이고, 물리적 I/O는 디스크에서 블록을 읽어오는 작업입니다. SQL 튜닝에서 더 근본적인 목표는 논리적 I/O를 줄이는 것입니다.

```text
블록 요청
   ↓
버퍼캐시 탐색
   ↓
캐시에 있음 → 논리적 I/O
캐시에 없음 → 물리적 I/O 후 캐시에 적재
```

### 비교 (그림 1-20)

| 구분 | 논리적 I/O | 물리적 I/O |
|---|---|---|
| 위치 | 메모리 버퍼캐시 | 디스크/스토리지 |
| 작동 | **전기적 신호** (블록 읽기 요청) | **액세스 암(Arm)** 의 물리적 작용 |
| 속도 | 빠름 | 메모리보다 **약 10,000배 느림** (경합 심하면 더) |
| 발생 관계 | SQL 처리 중 읽은 **총** 블록 수 | 캐시 미스로 디스크까지 간 블록 (논리 I/O의 일부) |
| 튜닝 방향 | 읽는 블록 수 자체를 줄임 | 캐시 효율/스토리지 성능 개선 |

> 메모리 버퍼캐시 크기는 `show sga`의 `Database Buffers` 항목(또는 `V$SGA` 뷰)으로 확인합니다. 서버 프로세스는 데이터파일과의 사이에 버퍼캐시가 있어, 운 좋게 캐시에서 블록을 찾으면(논리 I/O) I/O Call로 잠을 자지 않아도 됩니다. 버퍼캐시는 공유메모리라 같은 블록을 읽는 다른 프로세스도 덕을 봅니다.

### 자전거 비유 — 내생변수 vs 외생변수

책은 이 관계를 강변 자전거 도로 비유로 설명합니다. 집에서 여의도까지 거리가 15km라면, 페달을 밟는 **논리적 일량**(바퀴 회전 수)은 이사하지 않는 한 항상 같습니다 — 26인치(둘레 약 2m) 바퀴라면 **15km = 약 7,500번 회전**으로 고정입니다. 그러나 같은 15km라도 실제로 힘들게 밟아야 하는 **물리적 일량**은 바람·도로 경사·노면 상태에 따라 매번 다릅니다.

| 비유 | DB 대응 | 성질 |
|---|---|---|
| 거리 15km (항상 동일) | 논리적 I/O | **내생변수** — SQL 구조로 통제 가능 |
| 바람·경사·노면 | BCHR(캐시 상태) | 외생변수 — 시스템 상황에 좌우 |
| 실제 밟는 페달 양 | 물리적 I/O | 외생변수에 의해 결정됨 |

여의도까지 빨리 가려면 통제 불가능한 바람을 탓할 게 아니라, **통제 가능한 내생변수(거리=논리적 I/O)** 를 줄여야 합니다. 이것이 5장까지 이어지는 `reduce I/O`의 정확한 의미입니다.

### 왜 논리적 I/O가 근본인가 — BCHR 공식

버퍼캐시 효율을 재는 전통 지표가 버퍼캐시 히트율(Buffer Cache Hit Ratio, BCHR)입니다.

```text
BCHR = (캐시에서 곧바로 찾은 블록 수 / 총 읽은 블록 수) × 100
     = (논리적 I/O − 물리적 I/O) / 논리적 I/O × 100
```

이 공식을 물리적 I/O에 대해 풀면 핵심 관계가 드러납니다.

```text
물리적 I/O = 논리적 I/O × (100% − BCHR)
```

논리적 I/O는 일정하므로 **물리적 I/O는 BCHR이라는 외생변수가 결정**합니다. BCHR은 메모리를 늘리는 것 외에 직접 통제할 수 없습니다. 따라서 SQL 튜닝이 손댈 수 있는 유일한 레버는 **논리적 I/O를 줄이는 것**뿐입니다.

정량적으로 보면 효과가 분명합니다.

```text
시스템 BCHR 70%, 논리적 I/O 10,000
   ↓
물리적 I/O = 10,000 × 30% = 3,000

논리적 I/O를 1,000으로 줄이면
   ↓
물리적 I/O = 1,000 × 30% = 300   (성능 약 10배 향상)
```

> ⚠️ BCHR이 높다고 SQL이 좋은 것은 아닙니다. 같은 블록을 캐시에서 반복해 읽으면 BCHR은 99%로 올라가지만 논리적 I/O 자체가 많아 느릴 수 있습니다. **BCHR은 결과 지표일 뿐, 튜닝 목표가 아닙니다.**

### 트레이스에서 읽는 법

SQL 트레이스의 Call 통계에서 논리적/물리적 I/O를 직접 읽을 수 있습니다.

| 트레이스 항목 | 의미 |
|---|---|
| `Query`(Consistent 모드) + `Current` 모드 | 논리적 I/O (버퍼캐시에서 읽은 총 블록 수) |
| `Disk` | 물리적 I/O (디스크에서 읽은 블록 수) |

예: `Query 1,351,677 + Current 12,367 = 논리적 I/O 1,364,044`, `Disk = 물리적 I/O 601,458`.

### 튜닝 관점

```text
SQL 튜닝
   ↓
좋은 인덱스/조인/소트 전략
   ↓
읽는 블록 수 감소
   ↓
논리적 I/O 감소
   ↓
물리적 I/O 감소 가능
```

---

## ✅ 이 절(1.3.5)의 핵심 통찰

1. 논리적 I/O는 버퍼캐시 블록 접근(총 읽은 블록 수)이고, 물리적 I/O는 그중 캐시 미스로 디스크까지 간 블록입니다.
2. `물리적 I/O = 논리적 I/O × (100% − BCHR)` — 물리적 I/O는 통제 불가능한 외생변수입니다.
3. 자전거 비유처럼 논리적 I/O(거리)만이 통제 가능한 내생변수이므로, 여기를 줄이는 것이 SQL 튜닝입니다.
4. BCHR이 높다고 좋은 SQL이 아닙니다. BCHR은 결과 지표일 뿐 튜닝 목표가 아닙니다.
5. 트레이스에서 논리적 I/O = `Query + Current`, 물리적 I/O = `Disk`로 확인합니다.

## 🔗 흐름 속 위치

1.3.1에서 I/O가 느림의 원인임을 봤습니다. 이 절은 I/O를 논리/물리 계층으로 나누어 튜닝 우선순위를 세웁니다.

📌 한 줄 요약: 물리적 I/O를 줄이려면 먼저 논리적 I/O, 즉 읽는 블록 수 자체를 줄여야 합니다.
