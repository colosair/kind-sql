## 6.1.4 인덱스 및 제약 해제를 통한 대량 DML 튜닝

### 핵심 개념

6.1.1에서 봤듯 인덱스와 무결성 제약은 DML 성능에 큰 영향을 끼칩니다. **OLTP(동시 트랜잭션 처리) 시스템에서는 이들을 해제할 수 없지만**, 동시 트랜잭션 없이 대량 데이터를 적재하는 **배치(Batch) 프로그램**에서는 이들을 일시 해제함으로써 큰 성능개선 효과를 얻을 수 있습니다.

```text
[유지한 채 적재]  매 행 → 인덱스 수직탐색·리프변경 + 제약확인 + Redo/Undo  → 느림
[해제 후 적재]    제약·인덱스 OFF → 대량 적재 → 인덱스 일괄 rebuild → 제약 재활성화
```

핵심 원리는 6.1.2~6.1.3과 동일한 결입니다 — **"반복되는 랜덤 작업을 일괄(batch) 작업으로 바꾼다."** 적재 중 매 행마다 정렬 위치를 찾아 인덱스를 갱신하는 대신, 데이터를 다 넣고 **전체를 한 번에 정렬해 인덱스를 빌드**하면 훨씬 빠릅니다.

---

### 기준 테스트 — 인덱스·제약을 유지한 채 1,000만 건 적재

이번엔 데이터를 10배 늘려 SOURCE에 1,000만 건을 준비하고, TARGET에는 PK 제약(→ Unique 인덱스 자동 생성)과 일반 인덱스 1개, 총 인덱스 2개가 있는 상태로 적재합니다.

```sql
SQL> alter table target add constraint target_pk primary key(no, empno);  -- Unique 인덱스 자동 생성
SQL> create index target_x1 on target(ename);

SQL> insert /*+ append */ into target select * from source;
10000000 개의 행이 만들어졌습니다.
-- 경과: 00:01:19.10   ← 1분 19초
```

> `/*+ append */` 힌트는 6.2.2 Direct Path Insert를 유발합니다. 여기서도 이미 사용 중이지만, 인덱스·제약 유지 비용이 그 위에 얹혀 1분 19초가 걸립니다.

---

### 해제 방식 1 — PK에 Unique 인덱스를 사용하는 경우

PK·일반 인덱스를 모두 해제한 뒤 적재합니다.

```sql
SQL> truncate table target;

-- PK 제약 비활성화 + Unique 인덱스 Drop
SQL> alter table target modify constraint target_pk disable drop index;

-- 일반 인덱스는 Unusable 상태로 (Drop 대신)
SQL> alter index target_x1 unusable;

-- Unusable 인덱스가 있어도 INSERT가 막히지 않도록
SQL> alter session set skip_unusable_indexes = true;   -- 기본값 true (확인용)

SQL> insert /*+ append */ into target select * from source;
10000000 개의 행이 만들어졌습니다.
-- 경과: 00:00:05.84   ← 5.84초!
```

적재가 끝나면 인덱스를 재생성하고 PK 제약을 다시 활성화합니다.

```sql
SQL> alter table target modify constraint target_pk enable NOVALIDATE;  -- PK 인덱스 자동 재생성
-- 경과: 00:00:06.77
SQL> alter index target_x1 rebuild;
-- 경과: 00:00:08.26
```

| 단계 | 시간 |
|------|---:|
| 적재(INSERT) | 5.84초 |
| PK enable (인덱스 재생성 포함) | 6.77초 |
| 일반 인덱스 rebuild | 8.26초 |
| **합계** | **약 21초** (vs 유지 79초) |

**`NOVALIDATE`의 역할** — 제약을 활성화할 때 **기존에 들어있던 데이터의 무결성 체크를 생략**하는 옵션입니다. 이것이 시간 단축의 또 다른 열쇠입니다(체크하면 전체 스캔 비용 발생). 단, 무결성에 확신이 없으면 활성화 전에 직접 확인해야 합니다.

```sql
-- NOVALIDATE 쓰기 전, 중복 PK가 없는지 미리 확인
SQL> select no, empno, count(*) from source
  2  group by no, empno having count(*) > 1;
```

---

### 해제 방식 2 — PK에 Non-Unique 인덱스를 사용하는 경우 ★

방식 1에는 함정이 있습니다. **PK가 Unique 인덱스에 기반하면, 그 인덱스를 Unusable로 두는 순간 INSERT 자체가 막힙니다.**

```sql
SQL> alter index target_pk unusable;   -- Unique 기반 PK 인덱스를 unusable로
SQL> insert /*+ append */ into target select * from source;
1행에 오류:
ORA-01502: index 'TARGET_PK' or partition of such index is in unusable state
-- 또는
ORA-26026: unique index TARGET_PK initially in unusable state
```

이유: **Unusable 상태에서는 데이터를 입력할 수 없는데, Unique 인덱스는 입력할 때마다 유일성을 검증해야** 하므로 Unusable인 채로는 INSERT를 허용할 수 없습니다. (그래서 방식 1은 PK 인덱스를 Unusable이 아니라 **Drop**했던 것입니다.)

PK 인덱스를 Drop하지 않고 **Unusable 상태로 두면서도** 입력하고 싶다면, **PK 제약을 Non-Unique 인덱스에 기반**하도록 구성합니다.

```sql
SQL> alter table target drop primary key drop index;

-- ★ Non-Unique 인덱스를 먼저 생성
SQL> create index target_pk on target(no, empno);

-- ★ 그 Non-Unique 인덱스를 PK 제약이 사용하도록 지정
SQL> alter table target add constraint target_pk primary key(no, empno)
  2  using index target_pk;

-- PK 제약 비활성화하되 인덱스는 남김(keep), 인덱스는 Unusable로
SQL> alter table target modify constraint target_pk disable keep index;
SQL> alter index target_pk unusable;
SQL> alter index target_x1 unusable;

-- 이제 Unusable 상태에서도 대량 INSERT 가능
SQL> insert /*+ append */ into target select * from source;
10000000 개의 행이 만들어졌습니다.
-- 경과: 00:00:05.53
SQL> commit;
```
```sql
-- 재생성·재활성화
SQL> alter index target_x1 rebuild;                                  -- 7.24초
SQL> alter index target_pk rebuild;                                  -- 5.27초
SQL> alter table target modify constraint target_pk enable novalidate; -- 0.06초
-- 합계 약 18초 (vs 유지 79초)
```

**왜 Non-Unique면 Unusable이어도 입력되나?** Non-Unique 인덱스는 입력 시점에 유일성을 검증할 의무가 없으므로, Unusable인 채로 두고 나중에 rebuild할 때 한꺼번에 정렬·구축하면 됩니다. 유일성 보장은 PK 제약을 `enable`하는 시점(rebuild + 검증)에 이뤄집니다.

| 구분 | 방식 1 (Unique 인덱스 PK) | 방식 2 (Non-Unique 인덱스 PK) |
|------|------|------|
| 적재 전 PK 인덱스 처리 | **Drop 해야 함** (Unusable 불가) | Unusable로 두면 됨 (keep index) |
| Unusable 상태 INSERT | 불가 (ORA-01502/26026) | **가능** |
| 적재 후 | 제약 enable 시 인덱스 자동 재생성 | 인덱스 rebuild + 제약 enable |
| 결과(1,000만 건) | 약 21초 | 약 18초 |

### 적용 판단

```text
대량 DML(배치/적재/마이그레이션)인가?
   ↓ Yes
동시 트랜잭션이 없어(또는 중단 가능해) 무결성·조회 영향이 허용되는가?
   ↓ Yes
인덱스/제약 유지 비용 > 해제+재생성 비용 인가?
   ↓ Yes
해제 후 적재 + 일괄 rebuild + NOVALIDATE 재활성화
```

| 체크 항목 | 내용 |
|------|------|
| OLTP 부적합 | 실시간 동시 트랜잭션 시스템에서는 인덱스/제약을 해제할 수 없음 |
| 무결성 위험 | 해제 중 중복·오류 데이터가 들어올 수 있음 → NOVALIDATE 전 사전 검증 |
| 재생성 비용 | 인덱스 rebuild·제약 검증도 비용이 큼. **유지 비용과 비교** 후 결정 |
| 적합 영역 | 대량 배치, 초기 적재, 마이그레이션 |

---

## ✅ 이 절(6.1.4)의 핵심 통찰

1. 대량 적재에서 인덱스·제약은 최대 병목 — 1,000만 건이 유지 시 **79초**, 해제 시 **18~21초**로 줄었다.
2. 원리는 **"매 행 랜덤 갱신 → 적재 후 일괄 빌드"** — 6.1.2/6.1.3과 같은 "반복을 배치로" 전략의 인덱스 버전이다.
3. **Unique 인덱스 기반 PK는 Unusable이면 INSERT 불가**(ORA-01502/26026)라 Drop해야 한다. **Non-Unique 인덱스 기반 PK는 Unusable 상태로도 입력 가능**해 `keep index`로 남길 수 있다.
4. `NOVALIDATE`는 제약 활성화 시 기존 데이터 검증을 생략해 시간을 단축한다 — 단, 무결성 확신이 없으면 사전에 중복을 직접 확인해야 한다.
5. 이 기법은 **OLTP에는 부적합**하고 동시성이 없는 배치·적재·마이그레이션에 적합하다 — 재생성 비용까지 합산해 유지 비용과 비교 후 적용한다.

## 🔗 흐름 속 위치

6.1.1에서 식별한 "인덱스·제약의 DML 부가비용"을 대량 작업에서 직접 제거하는 절입니다. INSERT 가속에 쓰인 `/*+ append */`(Direct Path Insert)는 6.2.2에서, 다음 6.1.5(수정가능 조인 뷰)·6.1.6(MERGE)은 절차적 UPDATE를 One SQL로 바꾸는 6.1.2의 연장선입니다.

📌 한 줄 요약: 대량 배치에서는 인덱스를 유지하며 한 건씩 갱신하기보다, **잠시 내려놓고(Non-Unique PK면 Unusable, Unique PK면 Drop) 적재 후 한 번에 다시 빌드**하는 편이 훨씬 빠릅니다.
