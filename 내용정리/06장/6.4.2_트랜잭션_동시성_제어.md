## 6.4.2 트랜잭션 동시성 제어

### 핵심 개념 — 비관적 vs 낙관적

동시성 제어는 같은 데이터를 여러 사용자가 동시에 다룰 때 정합성을 보장하는 것입니다. 두 가지 철학으로 나뉩니다.

| 구분 | 가정 | 읽는 시점 Lock | 핵심 |
|------|------|:---:|------|
| **비관적(Pessimistic)** | 사용자들이 같은 데이터를 **동시에 수정할 것** | **건다** | 첫 트랜잭션 완료 전까지 다른 사용자 수정 불가 |
| **낙관적(Optimistic)** | 사용자들이 **동시에 수정 안 할 것** | 안 건다 | 수정 시점에 "그새 바뀌었나" **반드시 검사** |

> "낙관적 입장에 섰다고 해서 동시 트랜잭션에 의한 잘못된 데이터 갱신을 신경 쓰지 않아도 된다는 것은 절대 아니다. 읽는 시점에 Lock을 사용하지 않았지만, **데이터를 수정하고자 하는 시점에 앞서 읽은 데이터가 다른 사용자에 의해 변경되었는지 반드시 검사해야 한다.**"

⚠️ 비관적 제어는 잘못 쓰면 동시성을 심각하게 떨어뜨립니다 — "잘못 사용하면"에 주목해야 합니다. 잘 쓰면 약, 못 쓰면 독입니다.

---

### 비관적 동시성 제어 — FOR UPDATE

읽는 시점에 Lock을 걸어 다른 트랜잭션의 변경을 원천 차단합니다.

```sql
-- ❌ FOR UPDATE 없으면: SELECT 이후 UPDATE 전에 다른 트랜잭션이 끼어들 수 있음
select 적립포인트, 방문횟수, 최근방문일시, 구매실적 from 고객
where 고객번호 = :cust_num;
-- (새로운 적립포인트 계산 ... 이 사이에 다른 트랜잭션이 변경하면 문제)
update 고객 set 적립포인트 = :적립포인트 where 고객번호 = :cust_num;

-- ✅ FOR UPDATE: 읽는 순간 로우 Lock → 다른 트랜잭션이 못 건드림
select 적립포인트, 방문횟수, 최근방문일시, 구매실적 from 고객
where 고객번호 = :cust_num for update;
```

문제는 동시성 저하입니다. 그래서 **무한정 기다리지 않도록 WAIT/NOWAIT** 옵션을 함께 씁니다.

```sql
for update nowait   → 대기 없이 즉시 Exception (ORA-00054)
for update wait 3   → 3초 대기 후 Exception (ORA-30006)
```

> WAIT/NOWAIT로 Lock 충돌 시 Exception을 받으면 **"다른 사용자에 의해 변경 중이니 다시 시도하십시오"** 메시지를 출력하고 트랜잭션을 종료할 수 있습니다. 따라서 오히려 동시성을 증가시킵니다.

#### 💡 큐(Queue) 테이블 동시성 제어 — SKIP LOCKED

여러 데몬이 큐 테이블의 입금 정보를 나눠 처리하는 상황입니다. `FOR UPDATE WAIT`는 Lock이 풀릴 때까지 기다리느라 느려집니다.

```sql
-- WAIT 3: 다른 데몬의 Lock이 풀릴 때까지 최대 3초 대기 (느릴 수 있음)
select cust_id, rcpt_amt from cust_rcpt_Q
where yn_upd = 'Y' and rownum <= 100 FOR UPDATE WAIT 3;
```

**`SKIP LOCKED`** 를 쓰면 Lock 걸린 레코드는 **건너뛰고** 다음 레코드를 계속 읽어, 데몬들이 서로 기다리지 않고 일을 나눠 갖습니다.

```sql
select cust_id, rcpt_amt from cust_rcpt_Q
where yn_upd = 'Y' FOR UPDATE SKIP LOCKED;
```
> SQL에서 `rownum` 조건절을 제거하고, 클라이언트 단에서 100개씩 범위로 나눠 Array Fetch하는 것이 핵심입니다(한 건씩 Fetch하면 성능이 나쁨). 이것이 여러 워커가 작업 큐를 경합 없이 분배하는 전형적 패턴입니다.

### 낙관적 동시성 제어 — 변경 여부 검사

읽을 때는 Lock을 안 걸고, 저장 시점에 "그새 바뀌었나"를 조건절로 검사합니다.

**방법 ①: 읽은 모든 컬럼을 UPDATE 조건에 포함**
```sql
select 적립포인트, 방문횟수, 최근방문일시, 구매실적 into :a, :b, :c, :d
from 고객 where 고객번호 = :cust_num;
-- 새로운 적립포인트 계산
update 고객 set 적립포인트 = :적립포인트
where 고객번호 = :cust_num
and 적립포인트 = :a and 방문횟수 = :b and 최근방문일시 = :c and 구매실적 = :d;

if sql%rowcount = 0 then
  alert('다른 사용자에 의해 변경되었습니다.');   -- 0건 = 그새 누가 바꿈
end if;
```
단점: 컬럼이 많으면 조건절을 일일이 기술하기 번거롭습니다.

**방법 ②: 최종변경일시(변경일시) 컬럼 하나로** ★
```sql
select 적립포인트, 방문횟수, 최근방문일시, 구매실적, 변경일시
into :a, :b, :c, :d, :mod_dt
from 고객 where 고객번호 = :cust_num;
-- 계산
update 고객 set 적립포인트 = :적립포인트, 변경일시 = SYSDATE
where 고객번호 = :cust_num
and  변경일시 = :mod_dt;          -- 최종 변경일시가 앞서 읽은 값과 같은지만 비교

if sql%rowcount = 0 then
  alert('다른 사용자에 의해 변경되었습니다.');
end if;
```
테이블에 `변경일시` 같은 컬럼이 있으면, **이 한 컬럼만** 비교해 간단히 갱신 여부를 판단합니다. 이것이 흔히 말하는 **버전 컬럼(version column) 기반 낙관적 락**입니다.

**방법 ③: 낙관적 + FOR UPDATE NOWAIT 결합** — UPDATE 전에 한 번 더 SELECT로 Lock을 확인해, Lock 대기 없이 예외처리합니다.
```sql
select 고객번호 from 고객
where 고객번호 = :cust_num and 변경일시 = :mod_dt
for update nowait;
```

### ⚠️ "동시성 제어 없는 낙관적 프로그래밍"의 함정

낙관적 제어는 Lock 시간이 짧아 동시성에 유리하지만, **변경 여부 검사 자체를 빼먹는** 경우가 자주 보입니다. 검사를 안 하면 그건 낙관적 제어가 아니라 그냥 버그입니다.

```text
        t1                t2                t3        → time
TX1 ── 상품조회:1,000원 ─────────────────── 결제:1,000원? 1,200원?
TX2 ──────────────── 가격변경:1,000→1,200 ──────────
```
[그림 6-27] — TX1이 t1에 1,000원으로 조회했는데, 결제(t3) 직전 TX2가 1,200원으로 바꿨습니다. **조회 시점 가격 기준으로 주문을 처리하는 것이 업무 규칙이라면** 그대로 처리해도 되지만, 아니라면 변경 여부를 체크해 취소시키거나 사용자에게 알려야 합니다.

```sql
insert into 주문
select :상품코드, :고객ID, :주문일시, :상점번호, ...
from   상품
where  상품코드 = :상품코드
and    가격 = :가격;         -- 주문을 시작한 시점의 가격

if sql%rowcount = 0 then
  alert('상품가격이 변경되었습니다.');   -- 그새 가격이 바뀌면 0건
end if;
```

> "그런데 그런 로직을 찾기 힘들다. 주문을 진행하는 동안 상품 공급업체가 가격을 변경하지 않을 것이라고 낙관적으로 생각하기 때문이다." → 좌우 두 화면에서 회원정보를 따로 수정·저장하면, **수정한 필드만 UPDATE하지 않고 화면에 뿌려진 전체 필드를 UPDATE**하기 때문에 한쪽 변경이 되돌아가는 **Lost Update**가 발생합니다.

### 로우 Lock 대상 테이블 지정 — FOR UPDATE OF

조인 쿼리에 `FOR UPDATE`를 걸면 **조인된 모든 테이블**에 로우 Lock이 걸립니다. 특정 테이블만 걸려면 `OF`로 지정합니다(그림 6-28: 계좌마스터 1 : 주문 M).

```sql
-- 계좌마스터·주문 양쪽 모두 로우 Lock
select b.주문수량 from 계좌마스터 a, 주문 b
where a.고객번호 = :cust_no and b.계좌번호 = a.계좌번호 and b.주문일자 = :ord_dt
for update;

-- 주문 테이블에만 로우 Lock (OF 뒤 컬럼이 속한 테이블만)
select b.주문수량 from 계좌마스터 a, 주문 b
where a.고객번호 = :cust_no and b.계좌번호 = a.계좌번호 and b.주문일자 = :ord_dt
for update of b.주문수량;
```
필요 없는 테이블까지 Lock을 거는 것은 동시성 낭비이므로, **꼭 변경할 테이블만 `OF`로 지정**합니다.

### 데이터 품질과 동시성 향상을 위한 제언

> "성능도 중요하지만, 성능보다 데이터 품질이 더 중요하다. ... **FOR UPDATE 사용을 두려워하지 말자.** 다중 트랜잭션이 존재하는 환경에서 공유 자원에 대한 액세스 직렬화는 필수다."

- FOR UPDATE가 필요하면 **정확히 사용**하되, WAIT/NOWAIT 예외처리로 동시성을 지킨다.
- **낙관적·비관적을 함께** 쓸 수도 있다 — 낙관적으로 시도하다 변경이 감지되면, 롤백 후 다시 시도할 때 비관적(FOR UPDATE)으로 전환.
- **동시성 향상의 기본은 결국 SQL 튜닝** — 대상을 빨리 찾으면 Lock을 빨리 풀 수 있다. One SQL·Array Processing으로 처리 시간을 줄이면 Lock 점유 시간도 준다(6.1.2). "Lock에 대한 고민은 트랜잭션 내 모든 SQL을 완벽히 튜닝하고 나서 해도 늦지 않다."

---

## ✅ 이 절(6.4.2)의 핵심 통찰

1. **비관적**(읽을 때 Lock, FOR UPDATE)과 **낙관적**(저장 시 변경 검사) 두 철학이 있다. 낙관적이라고 검사를 생략하면 안 된다.
2. 비관적은 **`FOR UPDATE WAIT n`/`NOWAIT`**(ORA-30006/ORA-00054)로 무한 대기를 피한다. 작업 큐 분배에는 **`SKIP LOCKED`**(Lock 걸린 건 건너뜀)가 핵심이다.
3. 낙관적은 읽은 컬럼 전부를 조건에 넣거나, **`변경일시`(버전) 컬럼 하나**로 갱신 여부를 판단(`sql%rowcount = 0`)한다.
4. **Lost Update**는 수정한 필드만이 아니라 화면 전체 필드를 UPDATE해서 생긴다 — "동시성 제어 없는 낙관적 프로그래밍"의 전형(그림 6-27).
5. 조인 쿼리는 `FOR UPDATE OF 컬럼`으로 **변경할 테이블만** Lock한다. 그리고 동시성 향상의 토대는 결국 **SQL 튜닝**이다.

## 🔗 흐름 속 위치

6.4.1이 Lock 메커니즘이었다면, 6.4.2는 **업무 트랜잭션에서 정합성을 지키는 실전 패턴**(FOR UPDATE/SKIP LOCKED/버전 컬럼)입니다. 다음 6.4.3은 동시 INSERT의 대표 경합 지점인 **채번(numbering)** 으로, Lock 논의를 구체 사례로 마무리합니다.

📌 한 줄 요약: 동시성 제어는 DB Lock에만 맡길 일이 아니라 **업무 규칙(변경 여부 검사·버전 컬럼·FOR UPDATE OF)을 SQL에 반영**하는 일이며, 그 토대는 Lock을 빨리 푸는 SQL 튜닝입니다.
