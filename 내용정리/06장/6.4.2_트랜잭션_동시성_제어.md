## 6.4.2 트랜잭션 동시성 제어

### 핵심 개념

트랜잭션 동시성 제어의 핵심은 같은 데이터를 여러 사용자가 동시에 다룰 때 정합성을 보장하는 것입니다. Lock만으로 모든 업무 정합성이 자동 보장되는 것은 아니며, 업무 규칙에 맞는 검증 로직이 필요합니다.

```text
사용자 A 조회
   ↓
사용자 B 변경 후 Commit
   ↓
사용자 A가 예전 값 기준으로 변경
   ↓
업무 정합성 문제 가능
```

### Lost Update 문제

대표적인 문제는 Lost Update입니다. 두 화면에서 같은 회원 정보를 열고 각각 다른 필드를 수정했는데, 두 번째 저장이 첫 번째 저장 내용을 덮어쓰는 상황입니다.

```text
화면 1: 전화번호 변경 후 저장
화면 2: 생년월일 변경 후 저장
   ↓
화면 2가 전체 컬럼 UPDATE
   ↓
전화번호가 예전 값으로 되돌아감
```

원인은 사용자가 수정한 컬럼만 UPDATE하지 않고, 화면에 조회된 전체 컬럼을 그대로 UPDATE하기 때문입니다.

### 낙관적 동시성 제어

업무상 조회 시점의 값이 바뀌었는지 확인해야 한다면 UPDATE 조건에 기존 값을 포함해야 합니다.

```sql
update 상품
set 가격 = :new_price
where 상품코드 = :prod_cd
and 가격 = :old_price;
```

```text
조회 당시 가격 = 1000
   ↓
저장 시 where 가격 = 1000
   ↓
다른 트랜잭션이 1200으로 바꿨다면 update 0건
   ↓
사용자에게 변경 사실 알림
```

### 동시성 제어 판단

| 방식 | 설명 | 적합 |
---|---|---|
| 비관적 제어 | 먼저 Lock을 잡고 작업 | 충돌 가능성이 높고 반드시 순차 처리해야 할 때 |
| 낙관적 제어 | 저장 시 변경 여부 확인 | 일반 화면 업무, 충돌 가능성이 낮을 때 |
| 컬럼 단위 UPDATE | 수정한 필드만 변경 | Lost Update 위험 감소 |

### 튜닝 포인트

동시성 제어는 성능과 정합성의 균형입니다.

```text
충돌 가능성 낮음
   ↓
낙관적 제어

충돌 가능성 높음 / 순서 중요
   ↓
명시적 Lock 또는 직렬화
```

---

## ✅ 이 절(6.4.2)의 핵심 통찰

1. Lock만으로 모든 업무 동시성 문제가 해결되지는 않습니다.
2. Lost Update는 화면 기반 업무에서 흔히 발생합니다.
3. 저장 시 기존 값을 조건에 포함하면 변경 충돌을 감지할 수 있습니다.
4. 수정한 컬럼만 UPDATE하는 것도 중요한 동시성 제어입니다.
5. 업무 규칙에 따라 비관적/낙관적 제어를 선택해야 합니다.

## 🔗 흐름 속 위치

6.4.1이 Lock의 기본 메커니즘이었다면, 6.4.2는 실제 업무 트랜잭션에서 정합성을 어떻게 지킬지 다룹니다.

📌 한 줄 요약: 동시성 제어는 DB Lock에만 맡길 일이 아니라 업무 규칙을 SQL 조건에 반영하는 일입니다.
