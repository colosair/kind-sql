## 6.3.5 파티션을 활용한 대량 INSERT 튜닝

### 핵심 개념

대량 INSERT 튜닝의 출발점은 6.1.4와 같습니다 — **인덱스를 Unusable시켰다가 적재 후 재생성**하면 더 빠를 수 있습니다. 6.3.5의 핵심 기여는 **파티션 테이블 + 로컬 인덱스**일 때, 이 인덱스 비활성화를 **파티션 단위로 국소화**할 수 있다는 점입니다. 전체 인덱스를 내릴 필요 없이, **작업 대상 파티션의 인덱스 조각만** 내렸다 올리면 됩니다.

---

### 비파티션 테이블일 때 (6.1.4 복습)

손익분기점을 넘는 대량 INSERT라면, 인덱스를 Unusable시켰다 재생성하는 방식이 더 빠를 수 있습니다(그림 6-25).

```sql
-- ① (가능하면) 테이블을 nologging 모드로
alter table target_t nologging;

-- ② 인덱스를 Unusable 상태로
alter index target_t_x01 unusable;

-- ③ (Direct Path Insert로) 대량 입력
insert /*+ append */ into target_t select * from source_t;

-- ④ (nologging으로) 인덱스 재생성
alter index target_t_x01 rebuild nologging;

-- ⑤ (nologging이었다면) logging 모드로 전환
alter table target_t logging;
alter index target_t_x01 logging;
```

**문제점:** 초대용량 테이블이면 인덱스가 **전체** Unusable되는 동안 그 인덱스를 쓰는 다른 조회가 모두 막힙니다. 그래서 실무에서는 시간이 더 걸려도 **인덱스를 그대로 둔 채** INSERT하는 경우가 많습니다 — 인덱스 전체 재생성 부담이 그만큼 크기 때문입니다.

---

### 파티션 테이블일 때 — 파티션 단위 인덱스 재생성 ★

테이블이 파티셔닝되어 있고 인덱스도 다 로컬 파티션이라면 고민이 해결됩니다. **파티션 단위로 인덱스를 재생성**할 수 있기 때문입니다(그림 6-26). 작업 대상 파티션(예: `p_201712`)만 건드리고, 나머지 파티션과 그 인덱스는 **온라인 상태 그대로** 둡니다.

```sql
-- ① (가능하면) 작업 대상 테이블 파티션만 nologging
alter table target_t modify partition p_201712 nologging;

-- ② 작업 대상 파티션과 매칭되는 인덱스 파티션만 Unusable
alter index target_t_x01 modify partition p_201712 unusable;

-- ③ (Direct Path Insert로) 해당 파티션 범위 대량 입력
insert /*+ append */ into target_t
select * from source_t where dt between '20171201' and '20171231';

-- ④ (nologging으로) 인덱스 파티션만 재생성
alter index target_t_x01 rebuild partition p_201712 nologging;

-- ⑤ (nologging이었다면) 작업 파티션만 logging으로 전환
alter table target_t modify partition p_201712 logging;
alter index target_t_x01 modify partition p_201712 logging;
```

```text
[비파티션]  인덱스 전체 Unusable → 적재 → 전체 Rebuild   (그 동안 인덱스 전체 사용 불가)

[파티션]    p_201712 인덱스만 Unusable → 적재 → p_201712만 Rebuild
            └ p_201710, p_201711 ... 나머지는 정상 서비스 유지
```

**핵심 차이:** 비파티션 방식이 "인덱스 전체를 내렸다 올린다"면, 파티션 방식은 **"한 조각만 내렸다 올린다"** 입니다. 영향 범위(blast radius)가 파티션 하나로 국한되므로, **서비스 중단 없이** 대량 적재가 가능합니다. 6.1.4의 기법을 파티션 입도(granularity)로 정밀화한 것입니다.

| | 비파티션 테이블 | 파티션 테이블(로컬 인덱스) |
|------|------|------|
| Unusable 범위 | 인덱스 **전체** | **작업 파티션 인덱스만** |
| Rebuild 범위 | 인덱스 **전체** | **작업 파티션만** |
| 다른 데이터 조회 | 해당 인덱스 전부 영향 | 영향 없음 (나머지 파티션 정상) |
| 적재 방식 | Direct Path Insert | Direct Path Insert |

---

### 튜닝 포인트

| 포인트 | 설명 |
|------|------|
| Direct Path Insert | `/*+ append */`로 HWM 바깥에 순차 적재(6.2.2) |
| NOLOGGING | 복구 정책이 허용하면 Redo 최소화 — 테이블/파티션 속성으로 지정 |
| 로컬 파티션 인덱스 | 파티션 단위 Unusable/Rebuild의 **전제 조건** |
| 파티션 단위 국소화 | 작업 파티션만 내려 나머지 서비스 유지 |

---

## ✅ 이 절(6.3.5)의 핵심 통찰

1. 대량 INSERT 튜닝의 기본은 6.1.4와 같다 — **인덱스 Unusable → Direct Path 적재 → Rebuild**.
2. 비파티션 테이블은 인덱스 **전체**를 내려야 해서 그동안 조회가 막히고, 재생성 부담도 커 실무에선 그냥 두고 INSERT하는 경우가 많다.
3. **파티션 테이블 + 로컬 인덱스**면 **작업 대상 파티션의 인덱스 조각만** Unusable/Rebuild하면 된다 — 6.1.4의 파티션 입도 버전이다.
4. 영향 범위가 파티션 하나로 국한되어, **나머지 파티션은 정상 서비스를 유지**한 채 대량 적재가 가능하다.
5. 이 모든 것의 전제는 역시 **로컬 파티션 인덱스**(= PK에 파티션 키 포함)이다.

## 🔗 흐름 속 위치

6.2의 Direct Path Insert와 6.3의 로컬 파티션 인덱스가 만나는 절입니다. 6.3.3(UPDATE-Exchange)·6.3.4(DELETE-Drop/Truncate)·6.3.5(INSERT-파티션 단위 Rebuild)는 모두 **로컬 파티션 인덱스를 전제로 대량 DML을 세그먼트/파티션 단위 작업으로 바꾸는** 3종 세트를 이루며, 6.3을 마무리합니다. 다음 6.4는 동시성(Lock)으로 넘어갑니다.

📌 한 줄 요약: 대량 INSERT는 인덱스 전체를 내리지 말고, **로컬 파티션 인덱스의 작업 대상 조각만** 내렸다 올려(Direct Path 적재 + 파티션 단위 Rebuild) 나머지 서비스를 멈추지 않는 것이 핵심입니다.
