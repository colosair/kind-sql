## 6.1.5 수정가능 조인 뷰

### 핵심 개념

다른 테이블과 조인해 그 값으로 UPDATE해야 할 때, 전통적 UPDATE 문은 **참조 테이블을 여러 번 반복 조회**하는 비효율을 안고 있습니다. **수정가능 조인 뷰(Updatable Join View)** 는 조인을 한 번만 수행한 결과 집합을 대상으로 UPDATE/DELETE/INSERT하는 기법으로, 이 반복 조회를 제거합니다. 6.1.2의 "One SQL" 원리를 UPDATE에 적용하는 대표 도구입니다.

---

### 전통적인 방식의 UPDATE와 그 비효율

고객별로 최근 한 달 거래 통계를 갱신하는 흔한 UPDATE입니다.

```sql
update 고객 c
set    최종거래일시 = (select max(거래일시) from 거래
                       where 고객번호 = c.고객번호
                       and   거래일시 >= trunc(add_months(sysdate, -1)))
     , 최근거래횟수 = (select count(*)    from 거래
                       where 고객번호 = c.고객번호
                       and   거래일시 >= trunc(add_months(sysdate, -1)))
     , 최근거래금액 = (select sum(거래금액) from 거래
                       where 고객번호 = c.고객번호
                       and   거래일시 >= trunc(add_months(sysdate, -1)))
where exists (select 'x' from 거래
              where 고객번호 = c.고객번호
              and   거래일시 >= trunc(add_months(sysdate, -1)));
```

스칼라 서브쿼리 3개가 **같은 거래 데이터를 세 번** 읽습니다. SET절을 묶으면 한 번으로 줄일 수 있습니다.

```sql
update 고객 c
set    (최종거래일시, 최근거래횟수, 최근거래금액) =
       (select max(거래일시), count(*), sum(거래금액)
        from   거래
        where  고객번호 = c.고객번호
        and    거래일시 >= trunc(add_months(sysdate, -1)))
where exists (select 'x' from 거래
              where 고객번호 = c.고객번호
              and   거래일시 >= trunc(add_months(sysdate, -1)));
```

하지만 이래도 **거래 데이터를 두 번**(SET절 1번 + `exists` 1번) 조회합니다. 비효율은 전부 사라지지 않았습니다. 추가로 두 가지 튜닝 포인트가 있습니다.

**① 총 고객 수가 많으면 `exists`를 세미 조인으로 유도** — 모든 고객을 한 건씩 확인하는 대신 해시 세미조인으로 대상을 한 번에 추립니다.
```sql
where exists (select /*+ unnest hash_sj */ 'x' from 거래
              where 고객번호 = c.고객번호
              and   거래일시 >= trunc(add_months(sysdate, -1)));
```

**② 같은 값으로 갱신되는 비중이 높으면 Redo 절감** — 변경이 없어도 UPDATE는 Redo를 발생시킵니다. `decode`/`nvl`로 **실제로 바뀌는 행만** 갱신 대상에 남깁니다.
```sql
set (최종거래일시, 최근거래횟수, 최근거래금액) =
    (select nvl(max(거래일시), c.최종거래일시)
          , decode(count(*), 0, c.최근거래횟수, count(*))
          , nvl(sum(거래금액), c.최근거래금액)
     from 거래 where 고객번호 = c.고객번호
     and  거래일시 >= trunc(add_months(sysdate, -1)))
```

> "이처럼 다른 테이블과 조인이 필요할 때 전통적인 UPDATE 문을 사용하면 비효율을 완전히 해소할 수 없다." — 거래를 최소 두 번 읽는 구조가 한계입니다.

### 수정가능 조인 뷰로 비효율 제거

조인을 **한 번만** 수행하고 그 결과를 UPDATE합니다.

```sql
update
( select /*+ ordered use_hash(c) no_merge(t) */
         c.최종거래일시, c.최근거래횟수, c.최근거래금액
       , t.거래일시, t.거래횟수, t.거래금액
  from   (select 고객번호
               , max(거래일시) 거래일시, count(*) 거래횟수, sum(거래금액) 거래금액
          from   거래
          where  거래일시 >= trunc(add_months(sysdate, -1))
          group by 고객번호) t          -- 거래를 단 한 번 집계
       , 고객 c
  where  c.고객번호 = t.고객번호
)
set 최종거래일시 = 거래일시
  , 최근거래횟수 = 거래횟수
  , 최근거래금액 = 거래금액;
```

거래 테이블을 **단 한 번만** 읽어 비효율이 사라졌습니다. (이 쿼리는 12c 이상에서만 정상 실행되며, 10g에서는 UPDATE 옆에 `bypass_ujvc` 힌트가 필요합니다. 11g에서는 실행되지 않는데, 이유는 아래에서 설명합니다.)

---

### 조인 뷰와 수정가능 조인 뷰의 정의

> **조인 뷰(Join View)**: FROM 절에 두 개 이상 테이블을 가진 뷰.
> **수정가능 조인 뷰(Updatable/Modifiable Join View)**: 입력·수정·삭제가 허용되는 조인 뷰. 단, **1쪽 집합과 조인하는 M쪽 집합에만** 입력·수정·삭제가 허용된다.

`scott.emp`(M쪽)와 `scott.dept`(1쪽)를 조인한 뷰로 동작을 살펴봅니다.

```sql
SQL> create table emp as select * from scott.emp;
SQL> create table dept as select * from scott.dept;
SQL> create or replace view EMP_DEPT_VIEW as
  2  select e.rowid emp_rid, e.*, d.rowid dept_rid, d.dname, d.loc
  3  from   emp e, dept d
  4  where  e.deptno = d.deptno;
```

**위험한 예 — M쪽이 아닌 1쪽 컬럼을 수정**
```sql
SQL> update EMP_DEPT_VIEW set loc = 'SEOUL' where job = 'CLERK';
```
CLERK 사원이 10·20·30 부서에 흩어져 있는데, 이 UPDATE는 세 부서의 loc를 **모두** 'SEOUL'로 바꿔버립니다 — 다른 job 사원의 부서 소재지까지 바뀌는 의도치 않은 결과입니다. (1쪽 DEPT는 조인 결과에서 여러 번 중복되므로 갱신이 모호함)

**M쪽 컬럼 수정도 처음엔 막힘**
```sql
SQL> update EMP_DEPT_VIEW set comm = nvl(comm,0) + (sal * 0.1) where sal <= 1500;
ORA-01779: cannot modify a column which maps to a non key-preserved table

SQL> delete from EMP_DEPT_VIEW where job = 'CLERK';
ORA-01752: cannot delete from view without exactly one key-preserved table
```
옵티마이저가 **어느 테이블이 1쪽인지 알 수 없어서** 발생하는 에러입니다.

**해결 — 1쪽 집합에 PK 제약 또는 Unique 인덱스 부여**
```sql
SQL> alter table dept add constraint dept_pk primary key(deptno);
SQL> update EMP_DEPT_VIEW set comm = nvl(comm,0) + (sal * 0.1) where sal <= 1500;
4 rows updated.       -- ★ 이제 성공
```

이렇게 하면 EMP는 **키 보존 테이블(Key-Preserved Table)**, DEPT는 **비 키 보존 테이블(Non Key-Preserved Table)** 로 남습니다.

---

### 키 보존 테이블(Key-Preserved Table)이란

> "키 보존 테이블이란, 조인된 결과집합을 통해서도 **중복 값 없이 Unique하게 식별 가능한** 테이블을 말한다. **Unique한 1쪽 집합과 조인되는 테이블**이어야 식별이 가능하다."

뷰에 rowid를 함께 출력하면 직관적으로 이해됩니다.

```sql
SQL> select ROWID, emp_rid, dept_rid, empno, deptno from EMP_DEPT_VIEW;

ROWID            EMP_RID          DEPT_RID         EMPNO DEPTNO
---------------  ---------------  ---------------  ----- ------
AAAMt4AAGAA...A  AAAMt4AAGAA...A  AAAM5AAGAA...AB   7369     20
AAAMt4AAGAA...B  AAAMt4AAGAA...B  AAAM5AAGAA...AB   7499     30
...
```
- `emp_rid`(EMP): **중복 없음** → 뷰의 `ROWID`와 일치 → **키 보존 테이블**
- `dept_rid`(DEPT): **중복 값 나타남** → 비 키 보존 테이블

> 단적으로 말해 **"키 보존 테이블이란 뷰에 ROWID를 제공하는 테이블"** 입니다. DML은 결국 ROWID로 대상 로우를 특정하므로, 뷰에 자기 ROWID를 제공하는 M쪽 테이블만 수정 가능한 것입니다.

DEPT의 Unique 인덱스(PK)를 제거하면 뷰에 키 보존 테이블이 없어져 rowid조차 못 뽑습니다.
```sql
SQL> alter table dept drop primary key;
SQL> select rowid, ... from EMP_DEPT_VIEW;
ORA-01445: cannot select ROWID from, or sample, a join view without a key-preserved table
```

---

### ORA-01779 회피의 역사 — bypass_ujvc → MERGE → 12c

집계(GROUP BY)한 결과로 1쪽 테이블을 갱신할 때 문제가 불거집니다. DEPT에 부서별 평균급여 컬럼을 두고 EMP를 deptno로 집계해 채우는 경우입니다.

```sql
SQL> alter table dept add avg_sal number(7,2);
SQL> update
  2  ( select d.deptno, d.avg_sal as d_avg_sal, e.avg_sal as e_avg_sal
  3    from   (select deptno, round(avg(sal),2) avg_sal from emp group by deptno) e
  4         , dept d
  5    where  d.deptno = e.deptno )
  6  set d_avg_sal = e_avg_sal;
ORA-01779: cannot modify a column which maps to a non key-preserved table
```
EMP를 deptno로 GROUP BY했으므로 deptno는 사실상 Unique인데도, **11g 이하 옵티마이저는 이를 인지 못 해** 불필요한 제약을 겁니다.

| 버전 | ORA-01779 회피 방법 | 비고 |
|------|------|------|
| **10g** | `/*+ bypass_ujvc */` 힌트 | Updatable Join View Check를 생략하라고 지시 |
| **11g** | (힌트 사용 불가) | 위 UPDATE를 실행할 방법이 없음 → **MERGE로 전환**(6.1.6) |
| **12c~** | 힌트 없이 정상 동작 | GROUP BY한 집합과 조인한 테이블은 키 보존된다고 옵티마이저가 인정 |

> "오해하지 말 것은, `bypass_ujvc` 힌트 사용이 중단됐을 뿐 수정가능 조인 뷰 사용이 중단된 게 아니다. 11g에도 **1쪽 집합에 Unique 인덱스가 있으면** 수정가능 조인 뷰를 이용한 UPDATE가 가능하다."

**12c의 개선** — GROUP BY 결과는 키가 보존된다는 사실을 옵티마이저가 인정하기 시작해, 1쪽에 Unique 인덱스가 **없는** 임시 테이블에도 적용 가능해졌습니다.
```sql
-- 12c: 고객_t에 Unique 인덱스가 없어도 GROUP BY로 키 보존 인정 → 동작
update (
  select o.주문금액, o.할인금액
  from   주문_t o
       , (select 고객번호 from 고객_t where 고객등급 = 'A' group by 고객번호) c
  where  o.고객번호 = c.고객번호
  and    o.주문금액 >= 1000000 )
set 할인금액 = 주문금액 * 0.2, 주문금액 = 주문금액 * 0.8;
```

> "배치 프로그램이나 데이터 이행 프로그램에서 사용하는 중간 임시 테이블에는 일일이 PK 제약이나 인덱스를 생성하지 않으므로 이 패턴이 유용할 수 있다."

---

## ✅ 이 절(6.1.5)의 핵심 통찰

1. 다른 테이블 값으로 갱신하는 전통적 UPDATE는 **참조 테이블을 2번 이상 반복 조회**한다(SET절 + WHERE exists). 수정가능 조인 뷰는 조인을 **1번**으로 줄인다.
2. 보조 튜닝: `exists`는 `unnest hash_sj`로 세미조인 유도, 같은 값 갱신이 많으면 `decode`/`nvl`로 변경 행만 남겨 **Redo를 절감**한다.
3. 수정가능 조인 뷰는 **1쪽과 조인하는 M쪽 집합에만** DML이 허용된다. 1쪽 컬럼 수정은 의도치 않은 다중 갱신을 부른다.
4. **키 보존 테이블 = 뷰에 자기 ROWID를 제공하는, 조인 후에도 Unique한 M쪽 테이블.** 1쪽에 PK/Unique 인덱스가 있어야 성립한다(없으면 ORA-01779/01752/01445).
5. ORA-01779 회피: 10g는 `bypass_ujvc`, 11g는 불가(→MERGE), **12c는 GROUP BY 키 보존을 인정해 힌트 없이 동작** — 인덱스 없는 임시 테이블 갱신에 유용하다.

## 🔗 흐름 속 위치

6.1.2의 One SQL 원리를 UPDATE에 적용한 절입니다. 11g에서 수정가능 조인 뷰가 막히는 상황의 대안이자 더 범용적인 도구가 바로 다음 절 **MERGE 문(6.1.6)** 입니다.

📌 한 줄 요약: 수정가능 조인 뷰는 **참조 테이블을 한 번만 조인**해 루프/반복 조회 UPDATE를 집합 UPDATE로 바꾸는 기법이며, 그 성립 조건이 **"1쪽이 Unique → M쪽이 키 보존"** 입니다.
