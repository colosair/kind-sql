## 6.3.3 파티션을 활용한 대량 UPDATE 튜닝

### 전제 — 왜 "로컬 파티션 인덱스 + PK 설계"가 먼저인가

6.3.2의 Unique 인덱스 제약(파티션 키가 인덱스 구성 컬럼이어야 함) 때문에, **PK 인덱스를 로컬 파티셔닝하지 못하면** 파티션 Drop/Truncate/**Exchange**/Split/Merge 같은 구조 변경 작업도 쉽지 않습니다. 글로벌·비파티션 인덱스는 이런 작업 순간 Unusable이 되어 Rebuild·서비스 중단이 따르기 때문입니다.

> "서비스 중단 없이 파티션 구조를 빠르게 변경하려면, **PK를 포함한 모든 인덱스가 로컬 파티션 인덱스여야 한다.**"

6.3.3~6.3.5는 모두 파티션 **구조 변경 작업**(특히 Exchange)을 수반합니다. 따라서 **ILM(Information Lifecycle Management)** 체계를 효과적으로 운영하려면, 테이블 설계 단계에서 **대량으로 추가/변경/삭제할 기준 컬럼을 PK에 포함**시켜 인덱스를 로컬 파티션으로 구성해 둬야 합니다. (각주 14: ILM = 데이터 생성부터 소멸까지 생명주기를 효율적으로 관리하는 체계)

---

### 인덱스를 둔 채 대량 UPDATE하면 — 손익분기점 5%

6.1.4에서 봤듯 인덱스는 DML 성능에 큰 영향을 줍니다. 그래서 대량 작업 시 인덱스를 Drop/Unusable 후 작업하는 방법을 많이 씁니다. **손익분기점은 5% 정도**로 봅니다 — 입력/수정/삭제 대상이 전체의 5%를 넘으면 인덱스를 둔 채 작업하기보다 인덱스 없이 작업 후 재생성하는 게 더 빠릅니다. (각주 15: 실제로는 5% 이하에서 결정되는 때가 많고, 테이블이 클수록 손익분기점은 더 낮아짐)

**사례: 거래 테이블 10억 건** (그림 6-21, `거래_X2 = 상태코드 + 거래일자`)
```sql
-- 2015년 1월 1일 전 거래 중 상태코드 <> 'ZZZ'인 5,000만 건(5%)을 'ZZZ'로 변경
update 거래 set 상태코드 = 'ZZZ'
where 상태코드 <> 'ZZZ' and 거래일자 < '20150101';
```

딜레마가 정확히 손익분기점에 걸립니다.
- **인덱스를 둔 채** 5,000만 건 UPDATE + 거래_X2 인덱스 실시간 관리 → 엄청난 시간.
- **인덱스를 Drop** 후 작업 → 5,000만 건 위해 **10억 건짜리 인덱스를 재생성**하는 비용도 만만치 않음.

> "어느 쪽이 더 빠를지는 상황에 따라 다르므로 테스트해 봐야 알 수 있지만, ... 실무적으로 이 정도 대용량 테이블이라면 인덱스를 그대로 둔 상태에서 작업하는 경우가 많다. 인덱스 전체를 재생성해야 하는 부담이 그만큼 크기 때문이다."

→ 두 선택지 모두 비싸다면, **제3의 길이 파티션 Exchange**입니다.

---

### 파티션 Exchange를 이용한 대량 데이터 변경 ★

테이블이 파티셔닝되어 있고 인덱스도 다 로컬 파티션이라면, 좋은 해법이 있습니다 — **수정된 값을 갖는 새 세그먼트를 만들어 원본 파티션과 통째로 바꿔치기**하는 방식입니다(그림 6-22). (가정: 2015년 1월 1일 전 거래는 모두 첫 번째 파티션 `P201412`에 저장)

```sql
-- 1. 임시 테이블 생성 (가능하면 nologging)
create table 거래_t nologging as select * from 거래 where 1 = 2;

-- 2. 원본을 읽어 임시 테이블에 입력하면서 상태코드 수정 (Direct Path)
insert /*+ append */ into 거래_t
select 고객번호, 거래일자, 거래순번, ...
     , (case when 상태코드 <> 'ZZZ' then 'ZZZ' else 상태코드 end) 상태코드
from   거래
where  거래일자 < '20150101';

-- 3. 임시 테이블에 원본과 같은 구조의 인덱스 생성 (nologging)
create unique index 거래_t_pk on 거래_t(고객번호, 거래일자, 거래순번) nologging;
create index 거래_t_x1 on 거래_t(거래일자, 고객번호) nologging;
create index 거래_t_x2 on 거래_t(상태코드, 거래일자) nologging;

-- 4. 원본 파티션과 임시 테이블을 Exchange (그림 6-23)
alter table 거래
exchange partition p201412 with table 거래_t
including indexes without validation;

-- 5. 임시 테이블 Drop
drop table 거래_t;

-- 6. (nologging이었다면) 파티션·인덱스를 logging 모드로 전환
alter table 거래       modify partition p201412 logging;
alter index 거래_pk    modify partition p201412 logging;
alter index 거래_x1    modify partition p201412 logging;
alter index 거래_x2    modify partition p201412 logging;
```

[그림 6-23] — `EXCHANGE PARTITION`의 정체:
```text
변경 전:  거래.P201412 (수정 안 된 상태코드)  ↔  거래_T (수정된 상태코드)
            ──────────────── Exchange ────────────────
변경 후:  거래.P201412 (수정된 상태코드)      ↔  거래_T (수정 안 된 옛 데이터, Drop 대상)
```

**핵심 메커니즘 — Exchange는 데이터를 옮기지 않는다.** `EXCHANGE PARTITION`은 파티션 세그먼트와 임시 테이블 세그먼트의 **딕셔너리 포인터만 맞바꾸는** 순수 메타데이터 작업입니다. 5,000만 건이든 5억 건이든 **교환 자체는 거의 순식간**입니다. `including indexes`는 로컬 인덱스까지 함께 교환하고, `without validation`은 데이터 검증을 생략해 더 빠릅니다.

```text
일반 대량 UPDATE        : 5,000만 행 × (행 변경 + 인덱스 갱신 + Undo/Redo)  → 매우 느림
파티션 Exchange 방식    : ① Direct Path로 새로 적재(빠름) + ② 포인터 교환(즉시)  → 비교 불가로 빠름
```

비용이 ②(느린 랜덤 변경 반복)에서 ①(빠른 순차 적재 한 번)으로 이동한 것입니다 — 6.1.2~6.2.2를 관통하는 "반복을 일괄로" 원리의 파티션 버전입니다.

### 판단 기준

```text
UPDATE 대상이 전체의 5%를 넘는 대량인가?
   ↓ Yes
대상이 특정 파티션에 한정되는가? (시계열 컬럼 등)
   ↓ Yes
모든 인덱스가 로컬 파티션인가? (PK에 파티션 키 포함)
   ↓ Yes
행 단위 UPDATE 대신 → 임시 테이블에 수정본 생성 + EXCHANGE PARTITION
```

## ✅ 이 절(6.3.3)의 핵심 통찰

1. 대량 UPDATE(5% 초과)는 "인덱스 둔 채 작업"도 "인덱스 Drop 후 재생성"도 모두 비싸다 — 10억 건 사례에서 둘 다 한계.
2. **파티션 Exchange**가 제3의 길: 수정본을 임시 테이블에 만들고 원본 파티션과 **세그먼트 포인터만 교환**한다.
3. `EXCHANGE PARTITION`은 **데이터를 옮기지 않는 메타데이터 작업**이라 건수와 무관하게 거의 순식간이다. `including indexes`로 로컬 인덱스까지 교환된다.
4. 이 기법이 가능하려면 **PK 포함 모든 인덱스가 로컬 파티션**이어야 한다 → 설계 단계에서 대량 변경 기준 컬럼을 PK에 포함해야 한다(ILM).
5. 비용이 "느린 반복 변경"에서 "빠른 Direct Path 적재 + 즉시 교환"으로 이동한 것 — 6장 전체의 "반복을 일괄로" 원리의 파티션 적용이다.

## 🔗 흐름 속 위치

6.3.1~6.3.2의 파티션·로컬 인덱스 구조를 **대량 UPDATE**에 적용한 절입니다. 이어지는 6.3.4(DELETE)·6.3.5(INSERT)도 같은 EXCHANGE/파티션 구조 변경 원리를 각 DML에 맞춰 변주합니다.

📌 한 줄 요약: 대량 UPDATE는 한 행씩 고치지 말고, **수정본을 새로 만들어 파티션과 통째로 바꿔치기(EXCHANGE PARTITION)** — 데이터 이동 없는 포인터 교환이 핵심입니다.
