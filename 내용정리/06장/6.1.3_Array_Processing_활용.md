## 6.1.3 Array Processing 활용

### 핵심 개념

실무에서 절차적 프로그램을 One SQL로 구현하는 일은 **절대 쉽지 않습니다.** 복잡한 업무 로직을 포함하는 경우가 많기 때문입니다. 그럴 때 **Array Processing**을 활용하면 One SQL로 구현하지 않고도 **Call 부하를 획기적으로 줄일 수 있습니다.**

> "Array Processing 기능을 활용하면 One SQL로 구현하지 않고도 Call 부하를 획기적으로 줄일 수 있다."

핵심 통찰은 6.1.2의 "Call을 0으로(One SQL)" 만들지 못하더라도, **"Call을 1/N로"** 만들면 거의 같은 효과를 얻는다는 것입니다.

| 구분 | 건건이 처리 | Array Processing |
|------|------|------|
| 묶음 단위 | 1건당 1 Call | N건을 배열로 묶음 |
| Call 횟수 | 100만 Call (폭증) | N건당 1 Call (예: 100만 → 100) |

> 절차적으로 구현한 프로그램을 One SQL로 바꾸는 데 유용한 구문이 **Insert Into Select**(6.1.2), **수정가능 조인 뷰**(6.1.5), **Merge 문**(6.1.6)입니다. 이들이 어려울 때의 차선이 Array Processing입니다.

---

### PL/SQL Array Processing — BULK COLLECT + FORALL

PL/SQL에서는 **`BULK COLLECT`**(여러 행을 배열로 한 번에 Fetch)와 **`FORALL`**(배열을 한 번에 DML)을 씁니다.

```sql
SQL> declare
  2    cursor c is select * from source;
  3    type typ_source is table of c%rowtype;
  4    l_source typ_source;
  5
  6    l_array_size number default 10000;       -- 배열 크기 = 한 번에 묶을 건수
  7
  8    procedure insert_target(p_source in typ_source) is
  9    begin
 10      forall i in p_source.first .. p_source.last     -- ★ 배열을 한 번에 INSERT
 11        insert into target values p_source(i);
 12    end insert_target;
 13
 14  begin
 15    open c;
 16    loop
 17      fetch c bulk collect into l_source limit l_array_size;   -- ★ 1만 건씩 Fetch
 18
 19      insert_target(l_source);
 20
 21      exit when c%notfound;
 22    end loop;
 23    close c;
 24
 25    commit;
 26  end;
 27  /
-- 경과: 00:00:03.99
```

```text
BULK COLLECT ... LIMIT 10000
   ↓
1만 행을 배열로 한 번에 Fetch       ← Fetch Call 1/10000
   ↓
FORALL ... INSERT
   ↓
1만 행을 한 번에 INSERT             ← Execute Call 1/10000
```

**결과: 절차적 루프 29.31초 → Array Processing 3.99초** (약 7배 단축). 처리 건수는 똑같이 100만 건인데, **Call 횟수만 1/10000로 줄여** 얻은 효과입니다.

### Java Array Processing — setFetchSize + addBatch/executeBatch

Java에서는 읽기 쪽 **`setFetchSize`**, 쓰기 쪽 **`addBatch`/`executeBatch`** 가 핵심입니다.

```java
public class JavaArrayProcessing {
  public void execute() throws Exception {
    int arraySize = 10000;
    long[] no = new long[arraySize];   // ... 컬럼별 배열 준비

    String SQLStmt = "select no, empno, ... from source";
    PreparedStatement st = con.prepareStatement(SQLStmt);
    st.setFetchSize(arraySize);        // ★ 한 번에 1만 행씩 Fetch
    ResultSet rs = st.executeQuery();

    int i = 0;
    while (rs.next()) {
      no[i] = rs.getLong(1); ...       // 배열에 적재
      if (++i == arraySize) {          // 1만 건 모이면
        insertTarget(i, no, empno, ...);  // 묶음 INSERT
        i = 0;
      }
    }
    if (i > 0) insertTarget(i, no, ...);  // 남은 자투리 처리
  }

  public void insertTarget(int length, long[] p_no, ...) throws Exception {
    PreparedStatement st = con.prepareStatement(SQLStmt);
    for (int i = 0; i < length; i++) {
      st.setLong(1, p_no[i]); ...
      st.addBatch();           // ★ INSERT할 값들을 배열에 저장(아직 실행 X)
    }
    st.executeBatch();         // ★ 배열에 저장된 값을 한 번에 INSERT
    st.close();
  }
}
```

**결과: 절차적 Java 루프 218초 → Array Processing 11.813초** (약 **18배** 단축).

핵심은 두 가지가 함께 줄었다는 점입니다.
- `setFetchSize`: SELECT의 **Fetch Call(User Call)** 감소
- `addBatch`/`executeBatch`: INSERT의 **Execute Call(User Call)** 감소 — 100만 번 발생하던 네트워크 왕복을 100번으로

```text
[before] while: getXxx → execute()  →  User Call 100만 회 (218초)
[after]  while: getXxx → addBatch    →  executeBatch  →  User Call 100회 (11.8초)
```

---

### One SQL vs Array Processing vs 건건이 처리

| 방식 | Call | 100만 건 시간(PL/SQL · Java) | 특징 |
|------|------|------|------|
| **One SQL** (Insert Into Select) | 1회 | 1.46초 | 가장 빠름. 복잡 로직 구현은 어려울 수 있음 |
| **Array Processing** | 1/N회 | 3.99초 · 11.81초 | 절차 로직 유지하며 Call 격감. One SQL에 준하는 효과 |
| **건건이 처리** | N회 | 29초(PL/SQL) · 218초(Java) | 구현은 단순, 성능 최악 |

> "Call을 단 하나로 줄이지 못하더라도 Array Processing을 활용해 10~100번 수준으로 줄일 수 있다면 One SQL에 준하는 성능효과를 얻을 수 있다."

여기서 **수확 체감(diminishing returns)** 패턴이 중요합니다. 한 번에 1건 vs 만 건의 차이는 극적이지만, 만 건 vs 백만 건의 차이는 크지 않습니다. 즉 배열 크기를 무한정 키울 필요 없이, 수천~만 단위면 충분히 One SQL급 효과에 수렴합니다(메모리 사용량과의 균형도 고려).

```text
배열 크기 →  시간 단축
1     ────────────────  매우 느림
10    ──────            급격히 빨라짐
100   ──                거의 수렴
10000 ─                 One SQL에 근접  ← 이 이상은 효과 미미, 메모리만 증가
```

---

## ✅ 이 절(6.1.3)의 핵심 통찰

1. One SQL이 어려우면 **Array Processing**이 차선 — "Call을 0으로"가 아니라 **"Call을 1/N로"** 만드는 전략이다.
2. PL/SQL은 **`BULK COLLECT`(읽기) + `FORALL`(쓰기)** → 29초가 3.99초로.
3. Java는 **`setFetchSize`(읽기) + `addBatch`/`executeBatch`(쓰기)** → 218초가 11.8초로(약 18배).
4. 처리 **건수**가 아니라 **Call 횟수**를 줄이는 것이 본질이다. 건수는 그대로 100만 건이다.
5. 효과는 수확 체감 곡선을 그린다 — 배열 크기를 수천~만 단위로 잡으면 One SQL에 준하는 성능에 수렴하고, 그 이상은 메모리만 더 쓴다.

## 🔗 흐름 속 위치

6.1.2가 "One SQL로 Call을 0으로"를 강조했다면, 6.1.3은 **One SQL이 불가능한 현실에서 Call을 1/N로 줄이는** 실전 절충안입니다. 다음 6.1.4는 대량 DML 시 인덱스·제약 부하 자체를 일시 제거하는 또 다른 축의 튜닝으로 이어집니다.

📌 한 줄 요약: One SQL이 불가능하면 건건이 처리하지 말고 **배열 단위로 묶어** Call 횟수를 1/N로 줄이면, One SQL에 준하는 성능을 얻습니다.
