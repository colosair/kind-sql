## 6.4.3 채번 방식에 따른 INSERT 성능 비교

### 핵심 개념

INSERT/UPDATE/DELETE/MERGE 중 가장 중요하고 튜닝 요소가 많은 것이 **INSERT**입니다 — 수행빈도가 가장 높고, 무엇보다 **채번(numbering) 방식에 따른 성능 차이가 매우 크기** 때문입니다. 신규 데이터를 입력하려면 PK 중복을 방지할 번호 생성이 선행되어야 합니다. 가장 많이 쓰는 세 방식을 비교합니다.

| 방식 | 한 줄 요약 |
|------|------|
| **채번 테이블** | 별도 테이블에서 최종 번호를 관리·갱신 |
| **시퀀스 오브젝트** | 오라클 시퀀스 사용 |
| **MAX + 1 조회** | 대상 테이블의 현재 최대값 + 1 |

> **구분 속성** 정의: PK가 「상담원ID + 상담일자 + 상담순번」처럼 복합컬럼일 때, 순번 이외의 컬럼(상담원ID, 상담일자)을 '구분 속성'이라 부릅니다. 이 개념이 절 전체의 핵심 변수입니다.

---

### ① 채번 테이블

별도 테이블에 구분 속성별 최종 번호를 저장하고, 입력 시 그 레코드를 읽어 +1 갱신해 새 번호를 얻습니다. **채번 레코드를 변경하는 과정에서 자연스럽게 액세스가 직렬화(트랜잭션 줄 세우기)** 되므로 중복을 **원천 방지**합니다.

**장점**
- 범용성이 좋다.
- INSERT 중복 예외처리에 크게 신경 쓸 필요 없다 (↔ MAX+1).
- 결번을 방지할 수 있다 (↔ 시퀀스).
- PK가 복합컬럼일 때도 사용할 수 있다 (↔ 시퀀스).

**단점 — 성능.** 채번 레코드를 변경하기 위한 **로우 Lock 경합** 때문입니다(그림 6-29). 로우 Lock은 (자율 트랜잭션을 안 쓰면) 대상 테이블 INSERT를 마치고 **커밋할 때까지 지속**됩니다.

```text
프로세스1 ─┐
프로세스2 ─┼→ 채번레코드 D(최종번호 123456) → 123457 갱신 시도 → 🔒 블로킹
프로세스3 ─┘
```
[그림 6-29]

동시 INSERT가 아주 많으면 채번 **레코드**뿐 아니라 채번 테이블 **블록** 자체에도 경합이 생깁니다 — 서로 다른 레코드를 변경하는 프로세스끼리도 경합합니다. (각주 18: 버퍼 Lock과 ITL 경합, 1.3.8 참조) PK가 복합컬럼이면 구분 속성별로 채번이 분산돼 Lock 경합이 줄지만, **구분 속성 값 종류가 소수일 때만** 채번 테이블을 쓰므로(각주 19) 경합 가능성은 여전히 높습니다.

#### 💡 자율 트랜잭션으로 Lock 빨리 풀기

`pragma autonomous_transaction`으로 채번 함수를 선언하면, 그 안의 커밋이 **메인 트랜잭션과 독립**되어 채번 테이블 로우 Lock을 메인보다 빨리 해제합니다.

```sql
create or replace function seq_nextval(l_gubun number) return number
as
  pragma autonomous_transaction;          -- ★ 자율 트랜잭션 선언
  l_new_seq seq_tab.seq%type;
begin
  update seq_tab set seq = seq + 1 where gubun = l_gubun;
  select seq into l_new_seq from seq_tab where gubun = l_gubun;
  commit;                                 -- 메인과 무관하게 즉시 커밋 → Lock 해제
  return l_new_seq;
end;
```
```sql
insert into target_tab values (seq_nextval(123), :x, :y, :z);
```
메인 트랜잭션이 최종 커밋 전 다른 작업을 오래 해도, 채번 테이블 로우 Lock은 **이미 해제**되어 다른 트랜잭션을 블로킹하지 않습니다.

### ② 시퀀스 오브젝트

**가장 큰 장점은 성능**입니다. 중복 예외처리도 필요 없고, 사용이 편리합니다. 하지만 성능 이슈가 **없는 건 아닙니다** — 시퀀스도 결국 **오라클 내부가 관리하는 채번 테이블**(`SYS.SEQ$`, `DBA_SEQUENCES` 뷰)이기 때문입니다(그림 6-30). 값을 읽고 변경하는 과정에 Lock이 작동합니다.

**시퀀스 Lock 3가지**

| Lock | 정체 | 대기 이벤트 |
|------|------|------|
| ① 로우 캐시 Lock | 딕셔너리(`SEQ$`)를 로우 캐시(SGA)에서 변경 — nextval마다 `last_number` 변경 | `row cache lock` (각주 21) |
| ② 시퀀스 캐시 Lock | 시퀀스 캐시(공유 캐시)에서 값 얻을 때 직렬화 = **SQ Lock** | `enq: SQ - contention` (각주 22) |
| ③ SV Lock | RAC에서 `ORDER` 옵션 시 캐시를 노드 간 공유하며 직렬화 | `DFS lock handle` (각주 23) |

**①의 해결책이 `CACHE`** 입니다 — 번호를 미리 한 묶음 확보해 nextval 대부분을 메모리에서 처리하고 딕셔너리 갱신 횟수를 줄입니다(기본값 20).

```sql
SQL> create sequence MYSEQ cache 1000;
-- last_number = 1
SQL> select MYSEQ.NEXTVAL from dual;   -- 1
-- last_number = 1001  ← 1000개를 미리 확보, 소진 후 다시 2001로 점프
```

**③ SV Lock과 RAC 순서 문제** — 시퀀스 캐시는 인스턴스마다 따로입니다. RAC에서 1번 인스턴스가 1~1000, 2번이 1001~2000을 받으면, 교대 호출 시 값이 `1→1001→2→1002...`로 입력됩니다.
- 식별자의 기본 조건은 **유일(Unique) + Not Null**뿐 — **순서대로 입력할 의무는 없습니다.** 위처럼 입력해도 식별자 조건은 전혀 위배하지 않습니다.
- 하지만 **일련번호(Serial Number)** 처럼 순서를 보장하려면 `ORDER` 옵션이 필요하고, 그러면 캐시를 모든 RAC 노드가 공유하며 **SV Lock으로 직렬화** → 1·2번이 교대로 빠르게 nextval하면 **INSERT 성능이 급락**합니다.
- > "RAC 환경에서 ORDER 옵션을 사용했더니 성능이 급격히 나빠졌다. 해결방법은?" — **답은 단 하나, ORDER 옵션을 제거하는 것**입니다.

**RAC에서 다른 방식을 써도** 인스턴스 간 블록을 주고받는 부작용은 똑같습니다(시퀀스 캐시 / 채번 레코드 블록 / MAX 인덱스 블록). 그래도 ORDER를 주고 안 주고에 따라 차이가 크며, **결론적으로 시퀀스가 가장 빠릅니다.** "동시 INSERT가 많은" 테이블에 단일컬럼 일련번호 식별자를 뒀다면 시퀀스가 최선입니다.

**시퀀스의 단점 (1) — PK 복합컬럼 부적합.** 기본적으로 PK가 단일컬럼일 때만 적합합니다. 복합컬럼에도 쓸 수 있지만, "PK는 유일성을 보장하는 최소 컬럼으로 구성"이라는 **최소성(Minimality)** 요건을 위배합니다.
> 💡 PK가 복합컬럼인데 동시성이 높아 시퀀스가 꼭 필요하면 **순환(cycle) 옵션** 시퀀스를 고려 — 하루 도달 불가한 최대값을 두고 1부터 다시 시작시켜 순번 컬럼 길이를 최소화합니다.

**시퀀스의 단점 (2) — 결번.** 원인은 (ㄱ) 채번 후 트랜잭션 롤백, (ㄴ) CACHE 시퀀스가 캐시에서 밀려남(또는 인스턴스 재기동). NOCACHE면 방지되나 성능이 떨어집니다. Shared Pool에 KEEP하는 방법도 있습니다.
```sql
EXEC SYS.DBMS_SHARED_POOL.KEEP('SCOTT.MY_SEQ', 'Q');
```
> 그런데 결번을 막을 필요가 있을까? 대개 없습니다 — 채번 테이블·MAX+1도 데이터를 삭제하면 결번을 막을 수 없습니다.

### ③ MAX + 1 조회

대상 테이블의 최종 일련번호를 조회해 +1 해서 INSERT합니다.

```sql
insert into 상품거래(거래일련번호, 계좌번호, 거래일시, 상품코드, 거래가격, 거래수량)
values ((select max(거래일련번호) + 1 from 상품거래)
       , :acnt_no, sysdate, :prod_cd, :trd_price, :trd_qty);
```

**장점**
- 시퀀스·별도 채번 테이블 **관리 부담 없음**.
- 동시 충돌이 많지 않으면 **매우 빠름**.
- PK 복합컬럼(구분 속성별 순번)에도 사용 가능 — 채번 테이블은 구분 속성 NDV가 적을 때만 쓰지만, **MAX+1은 값 종류가 많을수록 오히려 성능이 좋아집니다**(입력 값 중복에 의한 로우 Lock 경합·재실행이 분산되어 줄기 때문, 그림 6-31).

**단점**
- 레코드 중복에 대비한 **세밀한 예외처리 필수**.
- 동시 채번이 심하면 시퀀스보다 훨씬 느려질 수 있다 — 로우 Lock이 선행 트랜잭션 커밋/롤백까지 지속되고, 선행이 롤백하지 않는 한 후행 INSERT는 결국 실패해 **채번+INSERT를 재실행**해야 합니다(그림 6-31).
- 로우 Lock 외에, **MAX 값 조회에 최적화된 인덱스가 없으면** 성능 이슈 발생 (5.3.3 '최소값/최대값 구하기' 참조).

### 채번 방식별 Lock 경합 정리 [표 6-3]

| 방식 | 식별자 구조 | 주요 경합 | 부수적 경합 | 비고 |
|------|------|------|------|------|
| **채번 테이블** | 일련번호 | (값 변경 위한) 로우 Lock 경합 | 동시성↑이면 테이블 블록 경합 | 테이블 관리 부담 |
| | 구분+순번 | 단일 일련번호일 때보다 Lock 감소 | | |
| **시퀀스** | 일련번호 | 시퀀스 경합 | 시퀀스 해소 시 인덱스 블록 경합 | 관리 부담 + 결번 가능성 |
| **MAX+1** | 일련번호 | (입력 값 중복 시) 로우 Lock + 재실행 | 동시성↑이면 인덱스 블록 경합 | 별도 오브젝트 없음 / 중복 예외처리 필수 / PK에 인덱스 구성 따라 성능차 |
| | 구분+순번 | 단일 일련번호일 때보다 Lock 감소 (구분 NDV 많으면 현저히 감소) | | |

### 채번 방식 선택 기준 [그림 6-32]

```text
다중 트랜잭션 동시 채번이 많은가?
   │
   ├─ No ───────────────────────────→ MAX + 1  (관리 부담 적어 선호)
   │
   └─ Yes → PK가 단일컬럼?
              ├─ Yes ────────────────→ 시퀀스 오브젝트
              └─ No → 구분 속성 값 종류가 많은가?
                        ├─ Yes ──────→ MAX + 1  (구조적으로 좋음, Minimality)
                        └─ No ───────→ (순환옵션) 시퀀스 오브젝트
```

1. **동시 채번 적음** → 셋 다 무방, 관리 부담 적은 **MAX+1** 선호.
2. **동시 채번 많고 PK 단일컬럼** → **시퀀스**가 최선.
3. **동시 채번 많고 PK 구분 속성 값 종류 많음** → 중복 경합·재실행이 낮아 **MAX+1**이 구조적(Minimality)으로 유리.
4. **동시 채번 많고 PK 구분 속성 값 종류 적음** → MAX+1 성능 문제 → **순환(cycle) 옵션 시퀀스**.

### 🏆 시퀀스보다 좋은 솔루션 — 입력일시를 PK에

순번 대신 **입력일시(주문일시·계약일시·청구일시 등) 컬럼을 PK에** 두면, 채번·INSERT 과정의 Lock 이슈를 **거의 해소**할 수 있습니다. 채번 과정을 생략하고 `SYSDATE`/`SYSTIMESTAMP`만 호출하면 되기 때문에 빠르고 간편합니다.

| 구분 속성 값 종류 | 권장 타입 |
|------|------|
| 많음 | `DATE` |
| 적음 | `TIMESTAMP` (소수 초까지 → 중복 거의 없음) |

> `TIMESTAMP`는 OS에 따라 정밀도가 다릅니다 — 유닉스/리눅스는 microsecond(6자리), MS 윈도우는 millisecond(3자리). 이 정도면 데이터 중복이 거의 없습니다. 그래도 중복 예외처리는 필요합니다.

이는 6.3 ILM과도 연결됩니다 — 입력 기준 컬럼을 PK에 포함하면 6.3.4의 "서비스 중단 없는 파티션 DELETE" 3조건(파티션 키 = 삭제 기준 = PK 구성 컬럼)도 자연히 충족됩니다.

### ⚠️ INSERT가 너무 빨라도 문제 — 인덱스 블록 경합

채번 경합을 없애고 나면 숨어 있던 복병, **인덱스 블록 경합**이 드러납니다. 특히 **Right Growing 인덱스**에서 흔합니다.

인덱스는 키순으로 정렬되므로, **일련번호·입력일시·변경일시처럼 순차 증가하는 단일컬럼 인덱스는 항상 맨 우측 리프 블록에만** 데이터가 입력됩니다(그림 6-33). 값이 달라도 같은 블록을 갱신하려 해 **버퍼 Lock 경합**이 생기고, 동시 INSERT가 많을 때(특히 RAC — Current 블록을 노드 간 주고받음) 성능을 떨어뜨립니다.

```text
        [B*Tree]
            │
   ┌────────┼────────┐
  ...      ...    ┌─────────┐
                  │ ...91   │  ← 모든 INSERT가
                  │ ...92   │     맨 우측 블록으로 몰림
                  │ ...93   │  ⬆⬆⬆ 버퍼 Lock 경합
                  └─────────┘
```
[그림 6-33] Right Growing 인덱스 (구분 속성을 앞에 두면 완화되지만, 동시성이 매우 높고 구분 NDV가 적으면 여전히 경합 — 그림 6-34)

**해결책**

| 방법 | 원리 |
|------|------|
| **인덱스 해시 파티셔닝** | 값이 순차 증가해도 해시 함수가 서로 다른 파티션으로 분산 |
| **Reverse Key 인덱스** | 키를 뒤집어 저장해 입력 위치를 흩뜨림 |
| **12c 글로벌+세션 시퀀스 조합** | `g_seq.currval ‖ s_seq.nextval` → 세션마다 다른 리프 블록 |
| **18c Scalable 시퀀스** | 인스턴스번호+세션ID+시퀀스번호 조합으로 분산 |

```sql
-- 18c Scalable 시퀀스
SQL> create sequence my_seq maxvalue 9999 SCALE EXTEND;
SQL> select my_seq.nextval from dual;   -- 예: 1011410001 (앞쪽이 인스턴스/세션 식별)

-- 하위 버전에서 같은 효과 (수동 조합)
select sys_context('userenv','instance') as 인스턴스번호
     , sys_context('userenv','sid')      as 세션ID
     , my_seq.nextval                    as 시퀀스번호
from dual;
```

### 💡 12c 시퀀스 신기능

```sql
-- (1) 컬럼 기본값으로 시퀀스 지정
create sequence my_seq;
create table t (c1 number default my_seq.nextval not null, c2 varchar2(5));

-- (2) IDENTITY 컬럼
create table t (c1 number generated always as identity, c2 varchar2(5));
-- generated always는 직접 입력 시 ORA-32792 → 사용자 입력 허용하려면:
create table t (c1 number generated by default as identity, c2 varchar2(5));

-- (3) 세션 시퀀스 (세션 종료 시 초기화, 세션 내에서만 유효, Lock 불필요→빠름)
create sequence g_seq GLOBAL;    -- 글로벌(기본)
create sequence s_seq SESSION;   -- 세션
```
세션 시퀀스는 **스테이징 테이블 적재**에 유용합니다 — 여러 세션이 공유하지 않고, Truncate 후 재적재하므로 이전 값을 이어 채번할 필요가 없어 더 빠릅니다.

## ✅ 이 절(6.4.3)의 핵심 통찰

1. 채번은 INSERT 동시성·정합성을 결정하는 핵심 설계다. 세 방식의 본질은 모두 **"누가 다음 번호를 잡느냐"의 직렬화 비용**이다.
2. **채번 테이블**은 결번 방지·복합 PK에 강하나 로우 Lock 경합이 크다 — `pragma autonomous_transaction`으로 Lock을 빨리 풀 수 있다.
3. **시퀀스**가 가장 빠르지만 로우 캐시 Lock·SQ Lock·(RAC) SV Lock이 있다. `CACHE`로 완화하고, **RAC에서 `ORDER`는 성능 급락의 주범이니 꼭 필요할 때만** 쓴다. 식별자는 유일하면 되지 순서 보장 의무는 없다.
4. **MAX+1**은 관리 부담이 없고 구분 속성 값 종류가 많을수록 빨라지지만, 중복 시 재실행·예외처리가 필요하다. 선택 트리(그림 6-32)로 정리된다.
5. **궁극의 해법은 입력일시(SYSDATE/SYSTIMESTAMP)를 PK에** 둬 채번 자체를 없애는 것. 단 채번을 없애면 **Right Growing 인덱스 블록 경합**이 드러나며, 이는 **해시 파티셔닝·Reverse 키·Scalable 시퀀스**로 분산한다.

## 🔗 흐름 속 위치

6.4.1~6.4.2의 Lock·동시성 제어를 **INSERT의 가장 흔한 병목인 채번**에 적용한, 6장의 마지막이자 클라이맥스 절입니다. 채번 테이블의 로우 Lock(6.4.1), 자율 트랜잭션·예외처리(6.4.2), 입력일시 PK와 ILM(6.3), 인덱스 블록 경합(2·3장)까지 6장 전체를 한데 묶습니다.

📌 한 줄 요약: 채번은 단순 번호 생성이 아니라 **INSERT 직렬화 비용을 어디에 둘지의 설계**이며, 동시성이 높을수록 시퀀스(단일 PK) 또는 입력일시 PK로 채번을 없애되 **그때 드러나는 Right Growing 인덱스 경합까지** 분산해야 완성됩니다.
