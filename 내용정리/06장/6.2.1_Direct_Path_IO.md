## 6.2.1 Direct Path I/O

### 핵심 개념 (6.2 도입 포함)

온라인 트랜잭션(OLTP)은 기준성 데이터·특정 고객·최근 거래 등을 **반복적으로** 읽기 때문에 버퍼캐시가 성능에 도움을 줍니다. 반면 **정보계(DW/OLAP)나 배치 프로그램**은 주로 대량 데이터를 처리하므로 버퍼캐시를 경유하는 I/O 메커니즘이 **오히려 성능을 떨어뜨릴 수 있습니다.** 그래서 오라클은 버퍼캐시를 경유하지 않고 곧바로 데이터 블록을 읽고 쓰는 **Direct Path I/O** 기능을 제공합니다.

```text
[일반 블록 I/O]   데이터파일 ↔ DB Buffer Cache(SGA) ↔ Server Process
                  찾고 → 없으면 디스크 읽기 → 변경은 DBWR가 나중에 반영

[Direct Path I/O] 데이터파일 ↔ Server Process (PGA)
                  버퍼캐시 우회 → 탐색·적재 부담 없음
```

---

### 왜 대량 작업에서 버퍼캐시가 독이 되는가

일반 I/O는 **버퍼캐시 경유**가 전제입니다. 읽을 블록을 캐시에서 찾고, 없으면 디스크에서 캐시로 적재한 뒤 읽습니다. 변경 시에도 캐시의 버퍼블록을 고치고 DBWR가 나중에 Dirty 블록을 데이터파일에 반영합니다. 이 구조가 대량 작업에서는 두 가지 비용을 만듭니다.

```text
대량 Full Scan / 적재
   ↓
① 디스크 → 버퍼캐시 적재 부담  (재사용성 낮은데도 캐시에 올림)
② 핫 블록(OLTP 자주 쓰는 블록) 밀려남  → 캐시 오염
   ↓
대량 작업도 느리고, OLTP 성능도 악화
```

> "대량 블록을 건건이 디스크로부터 버퍼캐시에 적재하고서 읽어야 하는 부담도 크다. ... Full Scan 위주로 수행되는 대용량 처리 프로그램이 읽어 들인 데이터는 대개 재사용성이 낮다. 그런 데이터 블록들이 버퍼캐시를 점유하면 다른 프로그램 성능에도 나쁜 영향을 미친다."

핵심은 **재사용성**입니다. 반복해서 읽힐 데이터(OLTP)는 캐시에 두는 게 이득이지만, 한 번 훑고 버릴 데이터(대량 배치)는 캐시를 거치는 것 자체가 낭비이자 오염원입니다.

---

### Direct Path I/O가 작동하는 6가지 경우

| # | 경우 | 비고 |
|---|------|------|
| 1 | **병렬 쿼리로 Full Scan**을 수행할 때 | Direct Path Read |
| 2 | **병렬 DML**을 수행할 때 | Direct Path Read + Direct Path Insert (6.2.3) |
| 3 | **Direct Path Insert**를 수행할 때 | `/*+ append */` 등 (6.2.2) |
| 4 | Temp 세그먼트 블록을 읽고 쓸 때 | 정렬/해시 작업의 디스크 영역 |
| 5 | `direct` 옵션을 지정하고 export할 때 | |
| 6 | `nocache` 옵션을 지정한 LOB 컬럼을 읽을 때 | |

이 중 **1~3번이 활용도가 가장 높습니다.** 2·3번(병렬 DML, Direct Path Insert)은 6.2.2~6.2.3에서 다루므로, 여기서는 본서에서 따로 설명하지 않는 **1번 병렬 쿼리**를 짧게 정리합니다.

---

### 병렬 쿼리 (Direct Path Read)

쿼리문에 `parallel` 또는 `parallel_index` 힌트를 주면, 지정한 **병렬도(Parallel Degree)** 만큼 병렬 프로세스가 동시에 작업합니다.

```sql
-- 테이블을 4개 병렬 프로세스로 Full Scan
SQL> select /*+ full(t) parallel(t 4) */ * from big_table t;

-- 인덱스를 4개 병렬 프로세스로 Fast Full Scan
SQL> select /*+ index_ffs(t big_table_x1) parallel_index(t big_table_x1 4) */ count(*)
  2  from big_table t;
```

> "놀랍게도, 위처럼 병렬도를 4로 지정하면, 성능이 네 배 빨라지는 게 아니라 **수십 배 빨라진다.** 바로 Direct Path I/O 때문에 나타나는 효과다. 버퍼캐시를 탐색하지 않고, 디스크로부터 버퍼캐시에 적재하는 부담도 없으니 빠른 것이다."

여기서 중요한 통찰: **병렬도 4 → 4배가 아니라 수십 배.** 단순히 일꾼이 4명으로 늘어난 효과(×4)에 더해, **각 일꾼이 버퍼캐시를 우회(Direct Path)** 하는 효과가 곱해지기 때문입니다. 병렬화 자체보다 Direct Path I/O 전환이 더 큰 레버입니다.

> 참고로 Order By, Group By, 해시 조인, 소트 머지 조인 등을 처리할 때는 힌트로 지정한 병렬도보다 **두 배 많은 프로세스**가 사용됩니다(생산자-소비자 구조).

---

### 튜닝 관점 — 항상 좋은 것은 아니다

| 상황 | 적합성 | 이유 |
|------|------|------|
| 대량 배치 적재 / Direct Path Insert | 적합 | 재사용성 낮은 대량 쓰기 |
| 대량 병렬 Full Scan | 적합 | 버퍼캐시 우회로 수십 배 가속 |
| 소량 OLTP 변경/조회 | **부적합** | 캐시 히트가 더 빠름 |
| 반복 재사용되는 데이터 | 부적합 | 버퍼캐시 활용이 유리 |

판단 기준은 일관되게 **재사용성**입니다 — 한 번 훑고 버릴 대량 데이터면 Direct Path, 반복해서 읽힐 소량 데이터면 버퍼캐시.

---

## ✅ 이 절(6.2.1)의 핵심 통찰

1. Direct Path I/O는 **버퍼캐시를 우회**해 데이터파일과 직접 I/O한다. 일반 I/O는 반드시 버퍼캐시를 경유한다.
2. 대량·재사용성 낮은 작업에서 버퍼캐시는 **적재 부담 + 캐시 오염(핫 블록 밀어냄)** 이라는 이중 비용을 만든다. Direct Path가 이를 제거한다.
3. 작동하는 6가지 경우 중 **병렬 쿼리·병렬 DML·Direct Path Insert(1~3)** 가 핵심이다.
4. **병렬도 N → N배가 아니라 수십 배** 빨라지는데, 이는 병렬화에 Direct Path I/O 효과가 곱해진 결과다.
5. 판단 기준은 **재사용성** — 반복 조회 소량 데이터(OLTP)는 버퍼캐시, 한 번 훑는 대량 데이터(배치/정보계)는 Direct Path.

## 🔗 흐름 속 위치

6.1이 DML의 기본 부가비용(인덱스·제약·Redo·Undo·Lock·커밋·Call)을 다뤘다면, 6.2는 한 단계 더 나아가 **I/O 경로 자체를 버퍼캐시 바깥으로 바꾸는** 대량 처리 전략입니다. 다음 6.2.2는 그 쓰기 버전인 Direct Path Insert를, 6.2.3은 이를 병렬화한 병렬 DML을 다룹니다.

📌 한 줄 요약: Direct Path I/O는 **재사용성 낮은 대량 작업을 버퍼캐시 바깥으로 빼서** 적재 부담과 캐시 오염을 동시에 없애는 전략입니다.
