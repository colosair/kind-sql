## 6.2.3 병렬 DML

### 핵심 개념

병렬 DML은 여러 병렬 프로세스가 동시에 INSERT/UPDATE/DELETE/MERGE를 수행해 대량 작업의 경과시간을 줄이는 방식입니다. 핵심 함정은 **활성화 여부**에 있습니다.

| 기능 | 기본 상태 |
|------|------|
| 병렬 쿼리(SELECT) | 활성화 — 힌트만 주면 바로 병렬 |
| 병렬 DDL | 활성화 |
| **병렬 DML** | **비활성화** — 세션에서 명시적으로 켜야 함 |

```sql
SQL> alter session enable parallel dml;   -- ★ 먼저 활성화해야 함
```

그 후 힌트로 병렬도를 지정합니다. **대상 레코드를 찾는 작업**(INSERT는 SELECT 쿼리, UPDATE/DELETE는 조건 검색)은 물론 **데이터 추가/변경/삭제**도 병렬로 진행됩니다.

```sql
insert /*+ parallel(c 4) */ into 고객 c
select /*+ full(o) parallel(o 4) */ * from 외부가입고객 o;

update /*+ full(c) parallel(c 4) */ 고객 c set 고객상태코드 = 'WD'
where 최종거래일시 < '20100101';

delete /*+ full(c) parallel(c 4) */ from 고객 c
where 탈퇴일시 < '20100101';
```

**활성화를 깜빡하면?** 대상 레코드를 찾는 작업은 병렬로 진행되지만, **추가/변경/삭제는 QC(Query Coordinator)가 혼자 담당**하므로 거기서 병목이 생깁니다.

> (각주 12) **QC(Query Coordinator)**: SQL을 병렬 실행하면 병렬도만큼 별도 병렬 프로세스를 피워 동시 작업하는데, 최초 접속해 SQL을 수행하는 프로세스가 QC 역할을 맡습니다. 병렬 처리가 불가능하거나 병렬도를 지정하지 않은 작업은 QC가 직접 처리합니다.

---

### 💡 두 단계 전략 (MVCC 연결)

6.1.1에서 본 **Undo의 MVCC 모델**이 여기서 다시 쓰입니다. "병렬 DML을 활성화해야 대상 찾기와 추가/변경/삭제가 **모두** 병렬로 처리된다"는 사실의 배경입니다.

```text
오라클 DML의 두 단계 전략
   ① Consistent 모드로 대상 레코드를 찾고   ← 읽기(SELECT)와 같은 모드
   ② Current 모드로 추가/변경/삭제한다       ← 실제 변경
```

병렬 DML 비활성 시에는 ①만 병렬이고 ②가 직렬(QC)이라 반쪽짜리가 됩니다. 활성화해야 ②까지 병렬 프로세스가 나눠 맡습니다.

---

### 병렬 INSERT와 Direct Path Insert

**병렬 INSERT는 `append` 힌트를 주지 않아도 Direct Path Insert 방식을 사용합니다.** 다만 병렬 DML이 작동하지 않을 경우를 대비해 `append`를 **함께** 쓰는 것이 좋습니다 — 병렬이 안 되더라도 QC가 Direct Path Insert는 사용해 어느 정도 성능을 냅니다.

```sql
insert /*+ append parallel(c 4) */ into 고객 c
select /*+ full(o) parallel(o 4) */ * from 외부가입고객 o;
```

**12c부터는 `enable_parallel_dml` 힌트**로 `alter session` 없이 문장 단위로 병렬 DML을 켤 수 있습니다.

```sql
insert /*+ enable_parallel_dml parallel(c 4) */ into 고객 c
select /*+ full(o) parallel(o 4) */ * from 외부가입고객 o;

update /*+ enable_parallel_dml full(c) parallel(c 4) */ 고객 c
set 고객상태코드 = 'WD' where 최종거래일시 < '20100101';

delete /*+ enable_parallel_dml full(c) parallel(c 4) */ from 고객 c
where 탈퇴일시 < '20100101';
```

---

### 주의점 — Exclusive TM Lock

병렬 DML을 사용하면 **테이블에 Exclusive 모드 TM Lock**이 걸립니다(Direct Path Insert와 동일). 트랜잭션이 빈번한 주간에 이 옵션을 쓰는 것은 절대 금물입니다.

```text
병렬도 증가
   ↓
처리 분담으로 경과시간 단축
   ↓ (그러나)
CPU / I/O / Undo / Redo 경합 증가 + 테이블 Exclusive Lock
   ↓
OLTP 업무와 충돌 → 특정 지점 이후 오히려 악화
```

---

### 병렬 DML이 잘 작동하는지 확인하는 방법 ★

실행계획에서 **UPDATE/DELETE/INSERT가 `PX COORDINATOR` 기준 어디에 있는가**로 판별합니다.

**✅ 작동함 — DML이 `PX COORDINATOR` 아래쪽** (각 병렬 프로세스가 DML 처리)
```text
| Id | Operation               | Name | TQ    |IN-OUT| PQ Distrib |
-----------------------------------------------------------------
|  0 | UPDATE STATEMENT        |      |       |      |            |
|  1 |  PX COORDINATOR         |      |       |      |            |
|  2 |   PX SEND QC (RANDOM)   |:TQ10000| Q1,00| P->S | QC (RAND) |
|  3 |    UPDATE               | 고객 | Q1,00 | PCWP |            |   ← 아래쪽
|  4 |     PX BLOCK ITERATOR   |      | Q1,00 | PCWC |            |
|  5 |      TABLE ACCESS FULL  | 고객 | Q1,00 | PCWP |            |
```

**❌ 작동 안 함 — DML이 `PX COORDINATOR` 위쪽** (QC가 혼자 DML 처리)
```text
| Id | Operation               | Name | TQ    |IN-OUT| PQ Distrib |
-----------------------------------------------------------------
|  0 | UPDATE STATEMENT        |      |       |      |            |
|  1 |  UPDATE                 | 고객 |       |      |            |   ← 위쪽
|  2 |   PX COORDINATOR        |      |       |      |            |
|  3 |    PX SEND QC (RANDOM)  |:TQ10000| Q1,00| P->S | QC (RAND) |
|  4 |     PX BLOCK ITERATOR   |      | Q1,00 | PCWC |            |
|  5 |      TABLE ACCESS FULL  | 고객 | Q1,00 | PCWP |            |
```

판별 원리: `PX COORDINATOR`는 병렬 프로세스의 결과를 QC가 취합하는 지점입니다. DML이 그 **아래**(병렬 프로세스 영역)에 있으면 각 프로세스가 직접 변경한 것이고, **위**(QC 영역)에 있으면 병렬 프로세스는 대상 찾기만 하고 변경은 QC가 직렬로 처리한 것입니다. 후자라면 `alter session enable parallel dml`을 깜빡한 것입니다.

---

## ✅ 이 절(6.2.3)의 핵심 통찰

1. 병렬 쿼리·DDL은 기본 활성화지만 **병렬 DML은 기본 비활성화** — `alter session enable parallel dml`(또는 12c `enable_parallel_dml` 힌트)로 켜야 한다.
2. 활성화를 깜빡하면 **대상 찾기만 병렬, 변경은 QC가 직렬**로 처리해 병목이 생긴다. 이는 DML의 **두 단계 전략(Consistent로 찾고 Current로 변경)** 과 직결된다.
3. **병렬 INSERT는 `append` 없이도 Direct Path Insert**를 쓴다. 안전하게 `append`를 함께 지정하면 병렬이 안 돼도 Direct Path 효과는 남는다.
4. 병렬 DML은 **Exclusive TM Lock**을 걸므로 OLTP 시간대에는 금물이고, 병렬도가 높을수록 리소스 경합이 커진다.
5. 작동 여부는 실행계획에서 **DML이 `PX COORDINATOR` 아래에 있으면 병렬(정상), 위에 있으면 QC 직렬(미작동)** 로 확인한다.

## 🔗 흐름 속 위치

6.2.1(Direct Path I/O 개념)·6.2.2(Direct Path Insert)가 I/O 경로를 바꾼 것이라면, 6.2.3은 그 위에 **처리 주체를 병렬화**해 경과시간을 줄이는 마지막 단계입니다. 세 절 모두 "대량·재사용성 낮은 작업"이라는 같은 전제에서 출발하며, 6장 6.2의 결론을 이룹니다.

📌 한 줄 요약: 병렬 DML은 대량 작업을 여러 프로세스로 나눠 빠르게 끝내지만, **반드시 세션에서 활성화해야 변경까지 병렬화**되고, Exclusive Lock과 리소스 경합이라는 비용을 동반합니다.
