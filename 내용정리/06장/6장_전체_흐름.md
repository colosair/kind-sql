## 6장 DML 튜닝 전체 흐름

### 핵심 개념

6장의 통일 원리는 **쓰기 작업의 부가 비용을 줄이는 것**입니다. SELECT 튜닝이 읽을 블록을 줄이는 데 집중했다면, DML 튜닝은 변경 대상 탐색뿐 아니라 인덱스 유지, 제약 확인, Redo/Undo, Lock, Commit, Call까지 함께 줄여야 합니다.

```text
DML 수행
   ↓
대상 로우 탐색
   ↓
테이블 변경
   ↓
인덱스/제약/Redo/Undo/Lock/Commit 비용
   ↓
대량 처리라면 Call/I/O/파티션/병렬화까지 고려
```

| 절 | 핵심 질문 | 튜닝 방향 |
---|---|---|
| 6.1.1 | DML 비용은 어디서 생기는가 | 인덱스, 제약, Redo/Undo, Lock, Commit을 함께 봅니다. |
| 6.1.2 | Call이 너무 많은가 | One SQL로 바꿉니다. |
| 6.1.3 | One SQL이 어려운가 | Array Processing으로 Call을 줄입니다. |
| 6.1.4 | 대량 DML에서 인덱스/제약 유지가 병목인가 | 해제 후 일괄 작업/재생성을 검토합니다. |
| 6.1.5 | 루프 UPDATE인가 | 수정가능 조인 뷰로 집합 UPDATE합니다. |
| 6.1.6 | UPDATE/INSERT 분기 루프인가 | MERGE로 One SQL화합니다. |
| 6.2.1 | 대량 I/O가 버퍼캐시를 오염시키는가 | Direct Path I/O를 검토합니다. |
| 6.2.2 | 대량 INSERT인가 | Direct Path Insert/APPEND를 검토합니다. |
| 6.2.3 | 작업 시간이 긴 대량 DML인가 | 병렬 DML을 검토합니다. |
| 6.3.1 | 작업 범위를 파티션으로 줄일 수 있는가 | 테이블 파티션을 활용합니다. |
| 6.3.2 | 파티션 관리 시 인덱스가 병목인가 | 로컬/글로벌 인덱스를 선택합니다. |
| 6.3.3 | 대량 UPDATE인가 | 파티션 재구성/교환을 검토합니다. |
| 6.3.4 | 대량 DELETE인가 | DELETE보다 파티션 제거를 검토합니다. |
| 6.3.5 | 대량 INSERT인가 | 스테이징 후 파티션 교환을 검토합니다. |
| 6.4.1 | Lock 경합이 있는가 | Lock 범위를 줄이고 트랜잭션을 짧게 유지합니다. |
| 6.4.2 | 업무 동시성 문제가 있는가 | 낙관적/비관적 제어를 업무 규칙에 맞게 적용합니다. |
| 6.4.3 | INSERT 채번이 병목인가 | 채번 테이블/시퀀스/MAX+1을 비교합니다. |

### 통합 튜닝 흐름

```text
DML 성능 문제 발생
   ↓
1. 대상 로우 탐색이 느린가?
      → 인덱스/조인 튜닝
   ↓
2. 인덱스/제약 유지 비용이 큰가?
      → 인덱스 정비, 제약 해제, 재생성 검토
   ↓
3. Call이 많은가?
      → One SQL 또는 Array Processing
   ↓
4. 대량 작업인가?
      → Direct Path, 병렬, 파티션 활용
   ↓
5. Lock/Commit/채번 경합인가?
      → 트랜잭션 설계와 채번 방식 점검
```

## ✅ 6장의 핵심 통찰

1. DML은 읽기보다 더 많은 부가 비용을 동반합니다.
2. 인덱스는 SELECT에는 약이지만 DML에는 비용입니다.
3. 대량 처리에서는 One SQL이 가장 강력하고, 어려우면 Array Processing을 사용합니다.
4. Direct Path, 병렬 DML, 파티션은 대량 작업의 핵심 도구입니다.
5. Lock과 Commit은 성능 문제이면서 동시에 정합성 문제입니다.
6. INSERT 성능은 채번 방식에 크게 좌우됩니다.

📌 한 줄 요약: 6장 DML 튜닝의 본질은 변경 자체보다 변경을 둘러싼 인덱스, 로그, Lock, Call, 채번 비용을 줄이는 것입니다.
