## 6.4.1 오라클 Lock

### 핵심 개념

Lock은 데이터 정합성을 지키기 위한 동시성 제어 장치입니다. DML은 변경 대상 로우에 Lock을 걸어 다른 트랜잭션이 동시에 같은 데이터를 모순되게 변경하지 못하게 합니다.

```text
트랜잭션 A가 로우 변경
   ↓
해당 로우 Lock 획득
   ↓
트랜잭션 B가 같은 로우 변경 시도
   ↓
대기
   ↓
A Commit/Rollback 후 진행
```

### Lock의 성격

MVCC 모델을 사용하는 Oracle은 **`for update`가 없는 SELECT 문에는 로우 Lock을 아예 사용하지 않습니다.** 다른 트랜잭션이 변경 중인(아직 커밋하지 않은) 로우를 읽을 때도 Lock이 풀릴 때까지 기다리지 않고, Undo로 복사본을 만들어 쿼리 시작 시점으로 되돌려 읽습니다(읽기 일관성). 따라서 읽기에는 Lock이 필요 없습니다.

이로부터 오라클 동시성의 핵심 명제가 나옵니다.

```text
DML  ↔ SELECT : 서로 방해하지 않음 (읽기는 Undo 복사본으로 처리)
SELECT ↔ SELECT : 서로 방해하지 않음
DML  ↔ DML   : 같은 로우면 서로 방해함 (Row Lock 대기) ← 유일한 충돌
```

> ⚠️ 참고: MVCC를 쓰지 **않는** DBMS는 SELECT에 공유 Lock을 사용해 DML과 SELECT가 서로 막을 수 있습니다. "읽기가 쓰기를 막지 않는다"는 오라클의 특성이지 모든 DBMS의 특성이 아닙니다.

| 충돌 | 처리 |
---|---|
| 읽기 vs 쓰기 | Lock 없음 — Undo 기반 읽기 일관성으로 처리 |
| 쓰기 vs 쓰기 | Row Lock 대기 |
| DDL vs DML | 테이블(TM) Lock 경합 가능 |

### DML Lock

오라클은 로우 Lock을 걸기 전에 테이블 Lock(TM Lock)을 먼저 설정합니다. 다른 트랜잭션이 테이블 구조를 바꾸지 못하게 막기 위해서입니다. 로우 Lock에는 항상 배타적 모드를 사용합니다.

```text
UPDATE / DELETE
   ↓
수정·삭제 대상 로우에 배타적 Row Lock
   (다른 트랜잭션은 같은 로우 UPDATE/DELETE 불가)

INSERT
   ↓
대개 Lock 경합 없음
   단, Unique 인덱스가 있고 두 트랜잭션이 같은 값을 입력하면
   → 먼저 입력한 쪽이 커밋할 때까지 뒤 트랜잭션이 대기
```

Lock은 Commit 또는 Rollback까지 유지됩니다. 따라서 트랜잭션을 오래 열어두면 다른 세션의 대기 시간이 늘어납니다.

### 튜닝 포인트

Lock 튜닝은 Lock을 없애는 것이 아니라, Lock을 짧게 잡고 충돌 가능성을 낮추는 것입니다.

```text
트랜잭션 범위 명확화
   ↓
대상 로우를 빠르게 찾는 인덱스
   ↓
불필요한 대량 Lock 방지
   ↓
적절한 Commit
```

---

## ✅ 이 절(6.4.1)의 핵심 통찰

1. Lock은 정합성을 위한 필수 장치입니다.
2. Oracle은 `for update` 없는 SELECT에 로우 Lock을 사용하지 않습니다(Undo 읽기 일관성).
3. 핵심 명제: DML↔SELECT 비방해, SELECT↔SELECT 비방해, **DML↔DML만 충돌**합니다.
4. INSERT는 보통 경합이 없으나, Unique 인덱스에 같은 값을 동시 입력하면 대기가 생깁니다.
5. Lock은 Commit/Rollback까지 유지되므로 트랜잭션을 짧고 명확하게 설계해야 합니다.

## 🔗 흐름 속 위치

6.1에서 Lock이 DML 성능 요소임을 봤고, 6.4는 이를 동시성 제어 관점에서 본격적으로 설명합니다.

📌 한 줄 요약: Lock은 성능의 적이 아니라 정합성의 비용이며, 튜닝 목표는 짧고 정확하게 잡는 것입니다.
