## 6.4.1 오라클 Lock

### 핵심 개념 (6.4 도입 포함)

> "Lock은 데이터베이스의 특징을 결정짓는 가장 핵심적인 메커니즘이다. 자신이 사용하는 데이터베이스의 고유한 Lock 메커니즘을 이해하지 못한다면, 고품질·고성능 애플리케이션을 구축하기 어렵다."

오라클은 공유 리소스와 사용자 데이터를 보호하기 위해 여러 종류의 Lock을 사용합니다.

| Lock 종류 | 보호 대상 |
|------|------|
| **DML Lock** | 사용자 데이터 (다중 트랜잭션 동시 액세스로부터) |
| **DDL Lock** | 딕셔너리(객체 정의) — Dictionary Lock이라고도 함 (각주 16) |
| 래치(Latch) | SGA 공유 자료구조 (1.3.8 참조) |
| 버퍼 Lock | 버퍼 블록 액세스 직렬화 |
| 라이브러리 캐시 Lock/Pin | 공유 SQL 커서, PL/SQL 프로그램 |

> (각주 16) 가장 중요하고 구현이 복잡한 것이 **TM Lock(=DML 테이블 Lock)**, 라이브러리 캐시 Lock/Pin입니다. DDL Lock, TM Lock, 라이브러리 캐시 Lock/Pin의 작용으로 나타나는 현상을 또 다른 Lock으로 분류하기도 합니다.

애플리케이션 측면에서 가장 중요한 것은 **DML Lock**이며, DML Lock에는 **테이블 Lock(TM)** 과 **로우 Lock(TX)** 이 있습니다.

---

### DML 로우 Lock — 오라클 동시성의 핵심 명제

DML 로우 Lock은 두 동시 트랜잭션이 **같은 로우**를 변경하는 것을 방지합니다. 어떤 DBMS든 로우 Lock에는 **배타적 모드**를 사용하므로, UPDATE/DELETE 진행 중(미커밋)인 로우를 다른 트랜잭션이 UPDATE/DELETE할 수 없습니다.

**INSERT의 로우 Lock 경합은 Unique 인덱스가 있을 때만** 발생합니다 — 두 트랜잭션이 같은 값을 입력하려 할 때 블로킹이 생기고, 먼저 입력한 쪽이 커밋하면 뒤 트랜잭션은 INSERT에 실패(롤백하면 성공)합니다. 서로 다른 값이거나 Unique 인덱스가 없으면 INSERT 경합은 없습니다.

**MVCC 모델 덕분에, 오라클은 `for update`가 없는 SELECT에 로우 Lock을 사용하지 않습니다.** 다른 트랜잭션이 변경 중인(미커밋) 로우를 읽을 때도 Lock 해제를 기다리지 않고, Undo로 복사본을 만들어 쿼리 시작 시점으로 되돌려 읽습니다(6.1.1 읽기 일관성). 여기서 오라클 동시성의 핵심 명제가 나옵니다.

```text
SELECT ↔ SELECT : 서로 방해하지 않음
DML    ↔ SELECT : 서로 방해하지 않음  (읽기는 Undo 복사본으로)
DML    ↔ DML    : 같은 로우면 충돌 (Row Lock 대기)  ← 유일한 충돌
```

> "다른 트랜잭션이 읽고 있는 로우를 변경하려면 다음 레코드로 이동할 때까지 기다려야 하고, 다른 트랜잭션이 변경 중인 로우를 읽으려면 커밋할 때까지 기다려야 한다." — **MVCC를 쓰지 않는 DBMS**의 이야기입니다. 그런 DBMS는 SELECT에 **공유 Lock**을 써서 DML과 SELECT가 서로 막습니다. "읽기가 쓰기를 막지 않는다"는 **오라클의 특성**이지 모든 DBMS의 특성이 아닙니다.

로우 Lock 성능 저하를 막으려면, 온라인 트랜잭션 처리 주간에 Lock을 필요 이상 오래 유지하지 않도록 커밋 시점을 조절해야 합니다. → **1~5장의 SQL 튜닝(대상을 빨리 찾기)이 곧 Lock 튜닝**입니다.

### DML 테이블 Lock (TM Lock) — 6가지 모드

오라클은 로우 Lock을 걸기 전에 **테이블 Lock(TM)을 먼저** 설정합니다 — 현재 트랜잭션이 갱신 중인 테이블 **구조를 다른 트랜잭션이 못 바꾸게** 막기 위해서입니다.

로우 Lock은 항상 배타적 모드지만, **테이블 Lock에는 6가지 모드**가 있고 모드 간 호환성(Compatibility)이 정해져 있습니다.

| | Null | RS | RX | S | SRX | X |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **Null** | O | O | O | O | O | O |
| **RS** | O | O | O | O | O | |
| **RX** | O | O | O | | | |
| **S** | O | O | | O | | |
| **SRX** | O | O | | | | |
| **X** | O | | | | | |

[표 6-2] (O = 두 모드 간 호환 가능)

| 약어 | 의미 |
|------|------|
| **RS** | row share (= SS: sub share) |
| **RX** | row exclusive (= SX: sub exclusive) |
| **S** | share |
| **SRX** | share row exclusive (= SSX: share/sub exclusive) |
| **X** | exclusive |

**어떤 작업이 어떤 모드를 거는가:**
- `INSERT`, `UPDATE`, `DELETE`, `MERGE` → 로우 Lock 전에 **RX(=SX)** 모드 테이블 Lock
- `SELECT FOR UPDATE` → **10gR1 이하 RS(=SS)**, **10gR2 이상 RX(=SX)**

RS·RX는 어떤 조합으로도 호환되므로(표에서 둘 다 O), **SELECT FOR UPDATE나 DML문 수행 시 테이블 Lock에 의한 경합은 절대 발생하지 않습니다.** 충돌은 오직 **같은 로우를 갱신할 때 로우 Lock**에서만 일어납니다. (S·SRX·X는 DDL 등이 사용)

**테이블 Lock = 플래그(Flag)** 개념이 핵심입니다.
> "오라클이 말하는 테이블 Lock은, 자신(테이블 Lock을 설정한 트랜잭션)이 해당 테이블에 현재 어떤 작업을 수행 중인지를 알리는 일종의 **푯말(Flag)** 이다."

테이블 전체에 Lock이 걸려 다른 트랜잭션이 레코드를 못 건드린다고 오해하기 쉽지만, DML은 항상 테이블 Lock을 먼저 설정하므로 그렇게 이해하면 안 됩니다(그러면 동시성이 불가능). 푯말에 기록된 Lock 모드와 후행 트랜잭션이 하려는 작업의 호환성에 따라 **진행할지/기다릴지/포기할지**가 결정됩니다. (각주 17: SQL Server는 이런 테이블 Lock을 'Intent Lock'이라 부름)

### 대상 리소스가 사용 중일 때 — 진로 선택 3가지

Lock을 얻으려는 리소스가 사용 중이면, 프로세스는 세 가지 중 하나를 택합니다. `SELECT FOR UPDATE`가 셋을 모두 노출합니다.

```sql
1. Lock이 해제될 때까지 기다린다     -- select * from t for update
2. 일정 시간만 기다리다 포기한다       -- select * from t for update wait 3
3. 기다리지 않고 즉시 포기한다         -- select * from t for update nowait
```

| 작업 | 기본 동작 | 옵션 |
|------|------|------|
| 일반 DML | 묵시적 테이블 Lock, **기다림**(1번) | — |
| `LOCK TABLE` | 기본 기다림 | `NOWAIT`로 즉시 포기 |
| DDL | 내부적 테이블 Lock, **`NOWAIT` 자동** | 11g~ `ddl_lock_timeout`(초)으로 대기 가능 |

```sql
lock table emp in exclusive mode NOWAIT;
```

### 블로킹 vs 교착상태 — 커밋이 푸는 열쇠

가끔 **블로킹**과 **교착상태**를 구분 못 하는 경우가 있습니다.

| 구분 | 정의 | 해소 방법 |
|------|------|------|
| **블로킹(Blocking)** | 선행 트랜잭션의 Lock 때문에 후행이 멈춰 있는 상태 | **커밋(또는 롤백)** 뿐 |
| **교착상태(Deadlock)** | 두 트랜잭션이 서로가 잡은 리소스에 Lock을 걸려고 마주 대기 (좁은 골목 두 차량) | 오라클이 자동 감지 후 한쪽 **문장 수준 롤백** |

**교착상태가 발생하면** 오라클이 먼저 인지한 트랜잭션에 **문장 수준 롤백**을 진행한 뒤 아래 에러를 던집니다.
```text
ORA-00060: deadlock detected while waiting for resource
```
⚠️ 주의: 이 메시지를 받은 트랜잭션은 교착상태에선 벗어났지만 **여전히 블로킹 상태**입니다. 따라서 **커밋 또는 롤백을 직접 결정**해야 합니다. 프로그램에서 이 에러에 대한 예외처리(커밋/롤백)를 하지 않으면 대기 상태가 지속되므로 주의해야 합니다.

**커밋이 Lock을 푸는 유일한 열쇠**이므로, 트랜잭션 길이의 균형이 중요합니다.

```text
트랜잭션 너무 김  → 롤백 시 시간 폭증 + Undo 세그먼트 부족 위험 + 블로킹 장기화
트랜잭션 너무 짧음 → 잦은 커밋 → LGWR 동기(sync) I/O 반복 → 성능 급락
        ↓
같은 데이터 동시 갱신 안 되게 설계 + 적절한 시점에 커밋
```

> 오라클은 읽기에 Lock을 안 써서 다른 DBMS보다 Lock 경합이 적어 트랜잭션을 충분히 길게 가져갈 수 있지만, 그래도 Undo 부족과 롤백 비용 때문에 너무 길어선 안 됩니다.

#### 비동기 커밋 — COMMIT WRITE 옵션 (10gR2~)

잦은 커밋이 느린 이유는 커밋이 LGWR의 **동기식(Sync) 로그 기록**을 기다리기 때문입니다(6.1.1). 10gR2부터는 이 동작을 옵션으로 조절할 수 있습니다.

| 옵션 | 의미 |
|------|------|
| **WAIT** (기본) | LGWR가 로그버퍼를 파일에 기록 완료할 때까지 기다림 (**동기식 커밋**) |
| **NOWAIT** | LGWR 완료 메시지를 안 기다리고 바로 다음 트랜잭션 진행 (**비동기식 커밋**) |
| **IMMEDIATE** (기본) | 커밋 명령마다 LGWR가 로그버퍼를 파일에 기록 |
| **BATCH** | 세션 내부에 트랜잭션 데이터를 일정량 버퍼링했다 일괄 기록 |

```sql
SQL> COMMIT WRITE IMMEDIATE WAIT;    -- ① (기본 동작)
SQL> COMMIT WRITE IMMEDIATE NOWAIT;  -- ②
SQL> COMMIT WRITE BATCH WAIT;        -- ③
SQL> COMMIT WRITE BATCH NOWAIT;      -- ④
```

NOWAIT/BATCH는 빠르지만, 커밋 후 인스턴스 Crash가 나면 **마지막 트랜잭션이 유실될 수 있는** 영속성(Durability) 트레이드오프가 있으므로 신중히 써야 합니다.

## ✅ 이 절(6.4.1)의 핵심 통찰

1. 오라클 Lock은 DML/DDL Lock, 래치, 버퍼·라이브러리 캐시 Lock 등 다양하며, 애플리케이션에서 가장 중요한 것은 **DML Lock**(테이블 TM + 로우 TX)이다.
2. 핵심 명제: **DML↔SELECT 비방해, SELECT↔SELECT 비방해, DML↔DML(같은 로우)만 충돌.** MVCC가 없는 DBMS는 SELECT에 공유 Lock을 써서 읽기가 쓰기를 막는다.
3. INSERT 로우 경합은 **Unique 인덱스에 같은 값을 동시 입력할 때만** 발생한다.
4. **테이블 Lock은 "내가 뭘 하는지 알리는 푯말"** — DML은 RX 모드라 서로 호환되어 **테이블 Lock 경합은 없고, 로우 Lock에서만** 충돌한다(표 6-2).
5. **블로킹은 커밋으로만 풀리고, 교착상태(ORA-00060)는 오라클이 한쪽을 문장 롤백**하되 여전히 블로킹이므로 예외처리(커밋/롤백)가 필요하다. Lock 튜닝 = 짧고 정확한 트랜잭션 + 빠른 대상 탐색.

## 🔗 흐름 속 위치

6.1.1에서 Lock이 DML 부가비용임을 짚었고, 6.4는 이를 동시성 제어 관점에서 본격적으로 다룹니다. 다음 6.4.2(트랜잭션 동시성 제어)는 비관적/낙관적 제어 등 실전 구현으로, 6.4.3은 채번(INSERT 경합)으로 이어집니다.

📌 한 줄 요약: 오라클은 **읽기에 Lock을 쓰지 않아 DML↔DML(같은 로우)만 충돌**하며, 테이블 Lock은 푯말일 뿐이다 — Lock 튜닝의 목표는 Lock을 없애는 게 아니라 **짧고 정확하게 잡고 제때 커밋**하는 것입니다.
