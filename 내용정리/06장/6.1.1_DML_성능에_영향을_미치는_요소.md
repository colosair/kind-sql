## 6.1.1 DML 성능에 영향을 미치는 요소

### 핵심 개념

지금까지 배운 인덱스·조인 튜닝은 DML 문에도 **그대로** 적용됩니다. 하지만 SELECT와 달리 DML(`INSERT`/`UPDATE`/`DELETE`)은 데이터를 **바꾸는** 작업이므로, "대상 로우를 잘 찾는" 비용 위에 "바꾸는 동안 발생하는 부가 비용"이 얹힙니다. 이 부가 비용이 6장 전체의 주제입니다.

> "DML 성능에 영향을 미치는 요소는 인덱스, 무결성 제약, 조건절, 서브쿼리, Redo 로깅, Undo 로깅, Lock, 커밋이다."

```text
DML 한 건 수행
   ↓
① 대상 로우 탐색          ← 조건절·서브쿼리 (SELECT/조인 튜닝 원리 그대로)
   ↓
② 테이블 블록 변경
   ↓
③ 모든 인덱스 변경        ← 인덱스 개수만큼 B*Tree 탐색
   ↓
④ 무결성 제약 확인        ← PK/FK는 실제 데이터 조회
   ↓
⑤ Undo 생성 / Redo 생성  ← 읽기 일관성·복구용
   ↓
⑥ Lock 유지              ← 동시성 제어 비용
   ↓
⑦ Commit                 ← LGWR 동기식(Sync) 로그 기록
```

| 요소 | 성능에 미치는 영향 | 적용되는 튜닝 원리 |
|------|------|------|
| 인덱스 | 입력·삭제·변경 시 모든 인덱스를 함께 관리 (수직 탐색 + 리프 변경) | 인덱스 개수 최소화 |
| 무결성 제약 | PK/FK는 실제 데이터 조회 필요 → Check/Not Null보다 비용 큼 | 제약 종류별 비용 차이 인식 |
| 조건절 | 대상 로우 탐색 = SELECT와 동일 | 2·3장 인덱스 튜닝 |
| 서브쿼리 | 세미/안티/일반 조인으로 변환 | 4장 조인 튜닝 |
| Redo 로깅 | 모든 변경을 복구용 로그에 기록 | 6.2 Direct Path로 생략 |
| Undo 로깅 | 롤백·읽기 일관성을 위해 이전 이미지 저장 (생략 불가) | MVCC 이해 |
| Lock | 동시성 제어 비용, 블로킹 유발 | 4절 동시성 제어 |
| 커밋 | LGWR 동기식 디스크 I/O | 적절한 커밋 주기 |

핵심은 **"왜 SELECT보다 비싼가"** 입니다. 읽기는 블록을 찾아 보기만 하면 끝나지만, 쓰기는 (1) 정렬 자료구조인 인덱스를 갱신하고, (2) 무결성을 검증하고, (3) 되돌릴 수 있도록 과거/미래 이미지를 모두 로깅하고, (4) 충돌을 막기 위해 락을 잡고, (5) 영속성을 위해 디스크 동기화를 거칩니다. 이 다섯 가지가 통일 원리 `reduce I/O`의 "쓰기 쪽 비용"을 구성합니다.

### ① 인덱스와 DML 성능

테이블에 레코드 하나를 입력하면 **모든 인덱스에도** 입력해야 합니다. 여기서 테이블과 인덱스의 자료구조 차이가 비용 차이를 만듭니다.

- **테이블**: 빈 공간이 있는 블록(`Freelist`로 관리)에 그냥 넣으면 됩니다. 정렬 의무가 없으므로 "아무 빈 곳"이면 됩니다.
- **인덱스**: 정렬된 B*Tree이므로, 키 값이 들어갈 **정확한 리프 블록**을 수직 탐색으로 찾아내야 합니다.

> "테이블은 Freelist를 통해 입력할 블록을 할당받지만, 인덱스는 정렬된 자료구조이므로 수직적 탐색을 통해 입력할 블록을 찾아야 한다. 인덱스에 입력하는 과정이 더 복잡하므로 DML 성능에 미치는 영향도 더 크다."

> **Freelist**: 테이블마다 데이터 입력이 가능한(여유 공간이 있는) 블록 목록을 관리하는데, 이를 'Freelist'라고 합니다. (각주 1)

```text
INSERT 한 건
   ↓
테이블: Freelist에서 빈 블록 받아 즉시 입력
   ↓
인덱스마다 반복:
   루트 → 브랜치 → 리프  (수직 탐색)
   ↓
   해당 리프 블록에 인덱스 엔트리 삽입 (정렬 위치 유지)
```

**연산별 인덱스 처리** — 세 DML이 인덱스에 가하는 작업량이 다릅니다.

| DML | 테이블 작업 | 인덱스 작업 | 비고 |
|------|------|------|------|
| `INSERT` | 빈 블록에 입력 | 모든 인덱스에 엔트리 삽입 | 인덱스마다 수직 탐색 |
| `DELETE` | 로우 삭제 | 모든 인덱스에서 해당 엔트리 삭제 | 엔트리를 모두 찾아 삭제 |
| `UPDATE` | 변경 컬럼만 수정 | **변경 컬럼을 참조하는 인덱스만** 처리 | 키당 삭제+삽입 2회 |

`UPDATE`가 특히 미묘합니다. 인덱스는 정렬 자료구조이므로, 키 값을 'A'에서 'K'로 바꾸면 **저장 위치 자체가 달라집니다.** 따라서 제자리 수정이 불가능하고, **기존 위치에서 삭제 → 새 위치에서 삽입**이라는 두 번의 오퍼레이션으로 처리됩니다.

```text
UPDATE 인덱스 컬럼 (A → K)
   ↓
기존 리프 위치에서 'A' 엔트리 삭제
   ↓
'K'가 들어갈 위치를 수직 탐색
   ↓
새 리프 위치에 'K' 엔트리 삽입   ← 한 번의 UPDATE가 인덱스엔 삭제+삽입
```

단, 변경하지 않는 컬럼만 가진 인덱스는 건드리지 않습니다. 그래서 "어떤 컬럼을 인덱스로 두느냐"가 UPDATE 부하를 좌우합니다.

**테스트: 인덱스 개수 → 100만 건 INSERT 시간** (p395~397)

```sql
-- SOURCE: 100만 건, TARGET: 비어 있음
SQL> create table source as
  2  select b.no, a.* from (select * from emp where rownum <= 10) a
  3       , (select rownum as no from dual connect by level <= 100000) b;

SQL> create table target as select * from source where 1 = 2;
SQL> alter table target add constraint target_pk primary key(no, empno);

-- (1) PK 인덱스만 있는 상태
SQL> set timing on
SQL> insert into target select * from source;   -- 경과: 00:00:04.95

-- (2) 일반 인덱스 2개 추가 후
SQL> truncate table target;
SQL> create index target_x1 on target(ename);
SQL> create index target_x2 on target(deptno, mgr);
SQL> insert into target select * from source;   -- 경과: 00:00:38.98

-- (3) 인덱스·PK 모두 제거 후
SQL> drop index target_x1;
SQL> drop index target_x2;
SQL> alter table target drop primary key;
SQL> truncate table target;
SQL> insert into target select * from source;   -- 경과: 00:00:01.32
```

| PK 제약/인덱스 | 일반 인덱스(2개) | 소요시간 |
|:---:|:---:|---:|
| O | O | **38.98초** |
| O | X | 4.95초 |
| X | X | **1.32초** |

[표 6-1] — 같은 100만 건 INSERT인데, 인덱스 2개 추가만으로 4.95초 → 38.98초로 **약 8배** 느려졌습니다.

> "인덱스 개수가 DML 성능에 미치는 영향이 매우 큰 만큼, 인덱스 설계에 심혈을 기울여야 한다. 핵심 트랜잭션 테이블에서 인덱스 하나라도 줄이면 TPS(Transaction Per Second)는 그만큼 향상된다."

여기서 **SELECT와 DML의 근본 트레이드오프**가 드러납니다. 인덱스는 조회 성능을 높이지만, 그 대가로 모든 입력·삭제·변경에 부담을 줍니다. OLTP 핵심 테이블일수록 "조회에 정말 필요한 최소 인덱스"만 남기는 설계가 TPS에 직결됩니다.

### ② 무결성 제약과 DML 성능

데이터베이스에 **논리적으로 의미 있는 자료만** 저장되게 하는 무결성 규칙은 네 가지입니다.

| 무결성 규칙 | 의미 | DBMS 구현 제약 |
|------|------|------|
| 개체 무결성 (Entity Integrity) | 각 로우를 유일하게 식별 | PK |
| 참조 무결성 (Referential Integrity) | 참조하는 값이 부모에 실재 | FK |
| 도메인 무결성 (Domain Integrity) | 컬럼 값이 허용 범위 내 | Check, Not Null |
| 사용자 정의 무결성 | 업무 제약 조건 | (애플리케이션 또는 제약) |

이 규칙들을 애플리케이션 코드로 구현할 수도 있지만, DBMS에서 PK·FK·Check·Not Null 같은 **제약(Constraint)** 으로 설정하면 더 완벽하게 무결성을 지켜낼 수 있습니다.

핵심은 **제약 종류에 따라 비용이 다르다**는 점입니다.

> "PK, FK 제약은 Check, Not Null 제약보다 성능에 더 큰 영향을 미친다. Check, Not Null은 정의한 제약 조건을 준수하는지만 확인하면 되지만, PK, FK 제약은 실제 데이터를 조회해 봐야 하기 때문이다."

```text
Check / Not Null  →  입력 값 자체만 검사 (값 < 100 인가? NULL 아닌가?)
                     → 다른 데이터를 볼 필요 없음 → 저렴

PK                →  "이미 같은 키가 있는가?" → 유니크 인덱스 조회 필요
FK                →  "부모 테이블에 이 값이 있는가?" → 부모 조회 필요
                     → 실제 데이터 액세스 발생 → 비쌈
```

즉 PK/FK 검증은 사실상 **추가적인 인덱스/테이블 조회**를 동반하므로, 앞의 인덱스 비용과 같은 맥락에서 DML을 무겁게 만듭니다. (p396 테스트에서 인덱스·PK를 모두 제거했을 때 1.32초가 나온 것도 PK 유니크 검증까지 사라졌기 때문입니다.)

### ③ 조건절과 DML 성능

DML이 대상 로우를 찾는 과정은 SELECT와 **완전히 동일**합니다. 따라서 실행계획도 다르지 않고, 2·3장의 인덱스 튜닝 원리를 그대로 적용할 수 있습니다.

```sql
SQL> update emp set sal = sal * 1.1 where deptno = 40;
```
```text
-------------------------------------------------------------
| Id | Operation            | Name    | Rows | Cost (%CPU) |
-------------------------------------------------------------
|  0 | UPDATE STATEMENT     |         |    1 |   2  (0)    |
|  1 |  UPDATE              | EMP     |      |             |
|  2 |   INDEX RANGE SCAN   | EMP_X01 |    1 |   1  (0)    |
-------------------------------------------------------------
```
```sql
SQL> delete from emp where deptno = 40;
```
```text
-------------------------------------------------------------
| Id | Operation            | Name    | Rows | Cost (%CPU) |
-------------------------------------------------------------
|  0 | DELETE STATEMENT     |         |    1 |   2  (0)    |
|  1 |  DELETE              | EMP     |      |             |
|  2 |   INDEX RANGE SCAN   | EMP_X01 |    1 |   1  (0)    |
-------------------------------------------------------------
```

> "SELECT 문과 실행계획이 다르지 않으므로 이들 DML 문에는 2장과 3장에서 학습한 인덱스 튜닝 원리를 그대로 적용할 수 있다."

핵심: `WHERE deptno = 40`을 처리하는 `INDEX RANGE SCAN`은 SELECT의 그것과 똑같습니다. UPDATE/DELETE 튜닝의 절반은 "대상을 어떻게 빨리 찾는가" — 즉 익숙한 SELECT 튜닝입니다.

### ④ 서브쿼리와 DML 성능

서브쿼리를 포함한 DML은 **4장의 조인 튜닝 원리**가 그대로 적용됩니다. 특히 4.4 '서브쿼리 조인'과 밀접합니다 — 옵티마이저가 서브쿼리를 세미조인/안티조인/일반조인으로 변환하기 때문입니다.

```sql
SQL> update emp e set sal = sal * 1.1
  2  where exists (select 'x' from dept where deptno = e.deptno and loc = 'CHICAGO');
```
```text
----------------------------------------------------------------------
| Id | Operation                       | Name     | Rows | Cost(%CPU)|
----------------------------------------------------------------------
|  0 | UPDATE STATEMENT                |          |   5  |  5 (20)  |
|  1 |  UPDATE                         | EMP      |      |          |
|  2 |   NESTED LOOPS                  |          |   5  |  5 (20)  |
|  3 |    SORT UNIQUE                  |          |   1  |  1  (0)  |
|  4 |     TABLE ACCESS BY INDEX ROWID | DEPT     |   1  |  2  (0)  |
|  5 |      INDEX RANGE SCAN           | DEPT_X01 |   1  |  1  (0)  |
|  6 |    INDEX RANGE SCAN             | EMP_X01  |   5  |  1  (0)  |
----------------------------------------------------------------------
```
```sql
SQL> delete from emp e
  2  where exists (select 'x' from dept where deptno = e.deptno and loc = 'CHICAGO');
```
```text
----------------------------------------------------------------------
| Id | Operation                       | Name     | Rows | Cost(%CPU)|
----------------------------------------------------------------------
|  0 | DELETE STATEMENT                |          |   5  |  4 (25)  |
|  1 |  DELETE                         | EMP      |      |          |
|  2 |   HASH JOIN SEMI                |          |   5  |  4 (25)  |
|  3 |    INDEX FULL SCAN              | EMP_X01  |  14  |  1  (0)  |
|  4 |    TABLE ACCESS BY INDEX ROWID  | DEPT     |   1  |  2  (0)  |
|  5 |     INDEX RANGE SCAN            | DEPT_X01 |   1  |  1  (0)  |
----------------------------------------------------------------------
```
```sql
SQL> insert into emp select e.* from emp_t e
  2  where exists (select 'x' from dept where deptno = e.deptno and loc = 'CHICAGO');
```
```text
----------------------------------------------------------------------
| Id | Operation                       | Name     | Rows | Cost(%CPU)|
----------------------------------------------------------------------
|  0 | INSERT STATEMENT                |          |   5  |  6 (17)  |
|  1 |  LOAD TABLE CONVENTIONAL        | EMP      |      |          |
|  2 |   HASH JOIN SEMI                |          |   5  |  6 (17)  |
|  3 |    TABLE ACCESS FULL            | EMP_T    |  14  |  3  (0)  |
|  4 |    TABLE ACCESS BY INDEX ROWID  | DEPT     |   1  |  2  (0)  |
|  5 |     INDEX RANGE SCAN            | DEPT_X01 |   1  |  1  (0)  |
----------------------------------------------------------------------
```

> "SELECT 문과 실행계획이 다르지 않으므로 이들 DML 문에는 4장에서 학습한 조인 튜닝 원리를 그대로 적용할 수 있다. 특히, 4장 4절(4.4) '서브쿼리 조인'과 밀접한 관련이 있다."

주목할 점: 같은 `exists` 서브쿼리라도 옵티마이저는 상황에 따라 `NESTED LOOPS`(UPDATE), `HASH JOIN SEMI`(DELETE/INSERT) 등 다른 조인 방식을 선택합니다. DML 서브쿼리 튜닝 = 세미조인 튜닝입니다.

### ⑤ Redo 로깅과 DML 성능

오라클은 데이터파일과 컨트롤 파일에 가해지는 **모든 변경사항**을 Redo 로그에 기록합니다. (데이터파일 변경은 캐시된 버퍼블록을 통해 이루어집니다 — 각주 2)

> "Redo 로그는 트랜잭션 데이터가 어떤 이유에서건 유실됐을 때, 트랜잭션을 재현함으로써 유실 이전 상태로 복구하는 데 사용된다."

DML을 수행할 때마다 Redo를 생성해야 하므로 Redo 로깅은 DML 성능에 영향을 미칩니다. INSERT 작업에 대해 Redo 로깅 **생략** 기능(Direct Path, 6.2.2)을 제공하는 이유가 여기에 있습니다.

**Redo 로그의 세 가지 용도** (p400)

| 용도 | 설명 | 시점 |
|------|------|------|
| ① Database Recovery | 디스크(데이터파일)가 깨졌을 때(Media Fail) Archived Redo로 복구 = 'Media Recovery' | 매체 장애 시 |
| ② Cache Recovery | 인스턴스 비정상 종료(Crash) 시, 캐시에만 있고 디스크에 못 쓴 변경을 roll forward = 'Instance Recovery' | 인스턴스 재기동 시 |
| ③ Fast Commit | 변경을 로그에 **먼저** 기록해 커밋을 빠르게 처리 | 커밋 시 |

세 번째가 성능과 직결됩니다. 변경된 메모리 버퍼블록을 디스크 데이터블록에 반영하는 작업은 **랜덤 액세스**라 느립니다. 반면 로그는 **Append(순차) 방식**이라 빠릅니다.

```text
변경 발생
   ↓
[느림] Dirty 버퍼블록 → 데이터파일 (랜덤 액세스)   ← 나중에 배치(Batch)로
[빠름] 변경분 → Redo 로그 (Append 순차 기록)        ← 즉시, 커밋 시 동기화
```

> "변경된 메모리 버퍼블록과 데이터파일 블록 간 동기화는 적절한 수단(DBWR, Checkpoint)을 이용해 나중에 배치(Batch) 방식으로 일괄 수행한다. 사용자의 갱신내용이 메모리상의 버퍼블록에 기록된 채 아직 디스크에 기록되지 않았지만 Redo 로그를 믿고 빠르게 커밋을 완료한다는 의미에서 이를 'Fast Commit'이라고 부른다."

즉 "느린 랜덤 쓰기는 미루고, 빠른 순차 로그만 안전하게 남겨 커밋을 끝낸다" — 이것이 모든 DBMS 커밋의 공통 설계이며 ⑧ 커밋 메커니즘의 토대입니다.

### ⑥ Undo 로깅과 DML 성능

과거에는 롤백(Rollback)이라 불렀지만 9i부터 오라클은 **Undo**라는 용어를 씁니다. Redo와 Undo는 방향이 정반대입니다.

```text
      ┌──────────── Redo ───────────→┐
   과거                              현재
      └←─────────── Undo ────────────┘

Redo : 트랜잭션을 재현 → 과거를 현재로 (복구)
Undo : 트랜잭션을 롤백 → 현재를 과거로 (되돌림)
```

[그림 6-3]

> "Redo는 트랜잭션을 재현하는 데 필요한 정보를 로깅하고, Undo에는 변경된 블록을 이전 상태로 되돌리는 데 필요한 정보를 로깅한다."

DML을 수행할 때마다 Undo를 생성해야 하므로 Undo 로깅도 DML 성능에 영향을 미칩니다. 다만 **Undo는 생략할 수 없습니다.** (Redo는 Direct Path로 생략 가능하지만, Undo는 롤백·읽기 일관성의 근간이라 오라클이 생략 수단을 제공하지 않습니다.)

**Undo의 세 가지 용도** (p402)

| 용도 | 설명 |
|------|------|
| ① Transaction Rollback | 변경을 커밋하지 않고 되돌릴 때 |
| ② Transaction Recovery | Instance Crash 후 roll forward로 복구된 미커밋 트랜잭션을 롤백 (rollback 단계) |
| ③ **Read Consistency** | **읽기 일관성** — SQL 튜닝 관점에서 가장 중요 |

> "읽기 일관성을 위해 Consistent 모드로 데이터를 읽는 오라클에선 동시 트랜잭션이 많을수록 블록 I/O가 증가하면서 성능 저하로 이어지기 때문이다."

#### MVCC (Multi-Version Concurrency Control) 모델

Undo의 읽기 일관성 용도를 이해하려면 MVCC를 알아야 합니다. 오라클은 데이터를 **두 가지 모드**로 읽습니다.

| 모드 | 읽는 방식 | 사용 주체 |
|------|------|------|
| **Current 모드** | 디스크에서 캐시로 적재된 **원본(Current) 블록**을 현재 상태 그대로 읽음 | DML의 변경 대상 찾기 |
| **Consistent 모드** | 다른 트랜잭션이 변경한 블록을 만나면 원본으로부터 **복사본(CR Copy)** 을 만들고, Undo를 적용해 **쿼리 시작 시점**으로 되돌려 읽음 | SELECT (읽기 일관성) |

```text
        ┌─────────────┐         ┌──────────────────────┐
        │ Current 블록 │  ───→   │  CR Copy (복사본들)    │
        │  15시 버전   │         │  14시 50분 버전        │
        │ (최종 버전)  │         │  14시 40분 버전        │
        └─────────────┘         │  14시 20분 버전        │
                                └──────────────────────┘
```
[그림 6-4] — 원본 블록 하나에 여러 복사본이 캐시에 공존할 수 있습니다.

**SCN(System Commit Number)으로 시점 관리** — Consistent 모드를 정확히 이해하는 열쇠입니다.

```text
SCN : 시스템에서 마지막 커밋이 발생한 시점정보 (Global 변수, 트랜잭션 커밋마다 +1)
   ↓
블록 SCN : 각 블록 헤더에 마지막 변경 시점을 기록
쿼리 SCN : 쿼리가 시작될 때 Global SCN을 먼저 확인해 기록
   ↓
Consistent 모드 동작:
   블록 SCN  >  쿼리 SCN  →  "내가 시작한 뒤 바뀐 블록"
      ↓
   CR Copy 생성 후 Undo 적용 → 쿼리 시작 시점으로 되돌려 읽음
```

이때 Undo 데이터가 다른 트랜잭션에 의해 **재사용되어** 쿼리 시작 시점으로 되돌릴 수 없으면 그 유명한 **Snapshot too old (ORA-01555)** 에러가 발생합니다.

> "SELECT 문은 (몇몇 예외 케이스를 제외하면) 항상 Consistent 모드로 데이터를 읽는다. 반면, DML 문은 Consistent 모드로 대상 레코드를 찾고, Current 모드로 추가/변경/삭제한다. 즉, Consistent 모드로 DML 문이 '시작된 시점'에 존재했던 데이터 블록을 찾고, 다시 Current 모드로 원본 블록을 찾아서 갱신한다." (각주 3: Consistent 모드로 읽은 레코드의 ROWID를 이용)

정리하면 **DML은 두 모드를 모두** 씁니다 — "무엇을 바꿀지"는 시작 시점 기준(Consistent)으로 정하되, 실제 변경은 반드시 원본(Current)에 가합니다. 동시 트랜잭션이 많을수록 CR Copy 생성과 Undo 적용 때문에 블록 I/O가 늘어 읽기마저 느려진다는 것이 SQL 튜닝 관점의 핵심입니다.

> Undo 세그먼트에 기록한 데이터가 차지하는 공간은 해당 트랜잭션이 커밋하는 순간 다른 트랜잭션이 재사용할 수 있는 상태로 바뀝니다. (가장 오래 전에 커밋한 Undo 공간부터 재사용 → 그래서 ORA-01555가 발생)

### ⑦ Lock과 DML 성능

Lock은 DML 성능에 **매우 크고 직접적인** 영향을 미칩니다. 그리고 양방향으로 위험합니다.

```text
Lock을 너무 많이/길게  →  동시성↓ (대기·블로킹 폭증) → DML 느려짐
Lock을 너무 적게/짧게  →  데이터 품질↓ (무결성 깨짐)
```

> "성능과 데이터 품질이 모두 중요한데, 이 둘은 트레이드 오프(Trade-off) 관계여서 어렵다. 두 마리 토끼를 다 잡으려면 매우 세심한 동시성 제어가 필요하다."

동시성 제어(Concurrency Control)란 동시에 실행되는 트랜잭션 수를 **최대화(고성능)** 하면서도 입력·수정·삭제·검색 시 데이터 무결성을 **유지(고품질)** 하기 위한 노력입니다. Lock과 트랜잭션 동시성 제어는 6.4절에서 자세히 다룹니다.

> **트랜잭션 격리성 수준 4단계** (각주 4): Read Uncommitted, Read Committed, Repeatable Read, Serializable. 오라클은 Read Committed가 기본이며 Serializable도 지원합니다.
>
> **Lock 레벨** (각주 5): 로우 레벨, 블록 레벨, 테이블 레벨 등. 낮은 레벨로 잠금 대상이 많아지면 겹쳐서 상위 레벨로 올리는데 이를 'Lock 에스컬레이션(Escalation)'이라 합니다. **오라클은 Lock 에스컬레이션을 사용하지 않습니다.**

### ⑧ 커밋과 DML 성능

커밋은 DML과 별개로 실행되지만, DML을 끝내려면 커밋까지 완료해야 하므로 서로 밀접합니다. 특히 DML이 Lock에 의해 **블로킹(Blocking)** 된 경우, 락을 푸는 열쇠가 바로 커밋이므로 커밋은 DML 성능과 직결됩니다.

> "커밋은 결코 가벼운 작업이 아니다."

모든 DBMS가 Fast Commit을 구현합니다(구현 방식은 서로 다름). Fast Commit의 도움으로 커밋을 순간적으로 처리하지만, 그 내부에는 다음 구성요소가 맞물려 있습니다.

**(1) DB 버퍼캐시** — 서버 프로세스가 사용자를 대신해 모든 일을 처리하며, 버퍼캐시를 통해 데이터를 읽고 씁니다. 변경된 블록(Dirty 블록)을 모아 주기적으로 데이터파일에 일괄 기록하는 일은 **DBWR(Database Writer)** 가 맡습니다(건건이 아닌 배치 처리).

**(2) Redo 로그버퍼** — 버퍼캐시는 휘발성이라 DBWR가 Dirty 블록을 반영할 때까지 불안하지만, 변경사항을 Redo 로그에도 기록하므로 안심해도 됩니다. Redo 로그도 파일이지만 Append라도 디스크 I/O는 느리므로, 먼저 **로그버퍼**에 기록하고 나중에 **LGWR(Log Writer)** 가 Redo 로그파일에 일괄(Batch) 기록합니다.

**(3) 트랜잭션 데이터 저장 과정** (그림 6-5)

```text
        ┌───────────── SGA ─────────────┐        Server Process
        │ Shared Pool                   │        ② 레코드 변경
        │  ├ Library Cache              │        ③ Redo 로깅
        │  └ Data Dictionary Cache      │             ↓
        │                               │        ┌────────┐  ① commit
        │  DB Buffer Cache ───┐         │  ←──    │ Redo   │ ───→ LGWR
        │                     │         │         │ Log    │       ↓
        │                     │         │         │ Buffer │  Online Redo Log
        └─────────────────────┼─────────┘         └────────┘   (로그파일 저장)
                              │ DBWR
                              ↓ DB 파일에 저장
                    Datafile  Datafile  Datafile
```
[그림 6-5]

```text
① DML 문을 실행하면 Redo 로그버퍼에 변경사항을 기록한다.
② 버퍼블록에서 데이터를 변경(레코드 추가/수정/삭제)한다.
   (버퍼캐시에서 블록을 못 찾으면 데이터파일에서 읽는 작업부터)
③ 커밋한다.
④ LGWR 프로세스가 Redo 로그버퍼 내용을 로그파일에 일괄 저장한다.
⑤ DBWR 프로세스가 변경된 버퍼블록들을 데이터파일에 일괄 저장한다.
```

**WAL (Write Ahead Logging)** — 핵심 순서 규칙입니다.

> "오라클은 데이터를 변경하기 전에 **항상 로그부터** 기록한다. 서버 프로세스가 버퍼블록에서 데이터를 변경(①)하기 전에 Redo 로그버퍼에 로그를 먼저 기록(②)하고, DBWR 프로세스가 Dirty 블록을 디스크에 기록(⑧)하기 전에 LGWR 프로세스가 Redo 로그파일에 로그를 먼저 기록(④)한다. 이를 'Write Ahead Logging'이라고 부른다."

**Log Force at Commit** — WAL에서 한 가지 의문이 생깁니다. 로그버퍼도 휘발성인데 Redo를 거기 남기면 같이 유실되지 않을까? 답은:

> "잠자던 DBWR와 LGWR 프로세스는 '주기적으로' 깨어나 각각 Dirty 블록과 Redo 로그버퍼를 파일에 기록하지만, LGWR 프로세스는 서버 프로세스가 **커밋을 발행했다고 신호를 보낼 때도** 깨어나 활동을 시작한다. '적어도 커밋시점에는' Redo 로그버퍼 내용을 로그파일에 기록한다는 뜻이다. 이를 'Log Force at Commit'이라고 부른다."

즉 변경된 버퍼블록을 디스크에 못 썼더라도, **커밋 시점에 Redo만 디스크에 안전하게 기록되면 그 순간부터 트랜잭션의 영속성이 보장**됩니다. (인스턴스 Crash가 나도 Redo로 roll forward 가능)

**(4) '커밋 = 저장 버튼'** — 워드프로세서 비유로 정리합니다.

```text
워드프로세서          ↔   데이터베이스
─────────────────────────────────────────
입력 → 메모리 기록    ↔   DML → 버퍼캐시 변경
저장 버튼 → 디스크    ↔   커밋 → Redo 로그파일 동기화(Sync)
저장 끝날 때까지 대기 ↔   LGWR 기록 완료까지 서버 프로세스 대기
```

저장(커밋)은 **Sync 방식**이라, LGWR가 Redo 로그를 디스크에 기록(디스크 I/O)할 때까지 서버 프로세스가 다음 작업을 진행할 수 없습니다. 그래서 **"커밋은 생각보다 느립니다."**

문서 작업의 나쁜 저장 습관처럼, 커밋에도 양 극단의 함정이 있습니다.

| 나쁜 습관 (저장 버튼) | 대응 문제 (커밋) | 결과 |
|------|------|------|
| 작업 끝날 때까지 한 번도 안 누름 | 너무 오래 커밋 안 함 | Undo 공간 부족 → 시스템 장애 위험 |
| 너무 자주·수시로 누름 | 루프 돌며 건건이 커밋 | 동기식 로그 I/O 반복 → 성능 급락 |
| 습관적으로 두 번씩 누름 | (불필요한 중복 커밋) | 낭비 |

> "트랜잭션은 필요 이상으로 길게 정의해서도 안 되고 너무 자주 커밋해서도 안 된다. ... 트랜잭션을 논리적으로 잘 정의함으로써 불필요한 커밋이 발생하지 않도록 구현해야 한다."

이것이 6.1 이후 절들(Array Processing, Direct Path 등)이 결국 **"커밋·로깅·Call 횟수를 줄여 쓰기 부가비용을 낮추는"** 방향으로 흐르는 이유입니다.

## ✅ 이 절(6.1.1)의 핵심 통찰

1. **DML 비용 = 탐색 비용 + 변경 부가비용.** SELECT는 "잘 찾기"가 전부지만, DML은 인덱스·제약·Redo·Undo·Lock·커밋이라는 쓰기 전용 비용이 추가된다.
2. **인덱스는 SELECT엔 약, DML엔 독.** 정렬 자료구조라 INSERT/DELETE는 수직 탐색+엔트리 갱신, UPDATE는 키당 삭제+삽입 2회가 발생한다. 100만 건 테스트에서 인덱스 2개만으로 8배(4.95→38.98초) 느려졌다 — 핵심 트랜잭션 테이블의 인덱스 최소화가 TPS에 직결된다.
3. **제약도 비용이 다르다.** Check/Not Null은 값만 검사하지만 PK/FK는 실제 데이터를 조회하므로 더 비싸다.
4. **조건절은 2·3장(인덱스), 서브쿼리는 4장(조인) 튜닝이 그대로 적용**된다. DML 튜닝의 절반은 이미 배운 SELECT 튜닝이다.
5. **Redo는 복구·Fast Commit용(생략 가능), Undo는 롤백·읽기 일관성용(생략 불가).** Undo의 읽기 일관성이 MVCC의 토대다.
6. **MVCC: SELECT는 Consistent 모드, DML은 둘 다 사용.** SCN으로 쿼리 시작 시점을 판정하고, CR Copy + Undo로 과거 버전을 복원하며, Undo가 재사용되면 ORA-01555(Snapshot too old)가 난다.
7. **커밋은 가볍지 않다.** WAL(로그 먼저)과 Log Force at Commit(커밋 시 Redo 강제 기록)으로 영속성을 보장하되, 커밋은 LGWR의 동기식(Sync) 디스크 I/O라 느리다. 너무 길어도(Undo 부족), 너무 잦아도(동기 I/O 반복) 안 된다.

## 🔗 흐름 속 위치

1~5장의 인덱스·조인·소트 튜닝은 DML의 **대상 로우 탐색** 과정(조건절·서브쿼리)에 그대로 흡수됩니다. 6장 6.1.1은 그 위에 **쓰기 작업만의 부가 비용**(인덱스 갱신·제약 검증·Redo/Undo·Lock·커밋)을 처음으로 펼쳐 보여주는 절입니다. 이후 6.1.2(Call과 성능), 6.1.3(Array Processing), 6.2(Direct Path I/O)는 모두 여기서 식별한 부가 비용 — 특히 **Call·로깅·커밋 횟수** — 를 줄이는 구체 기법으로 이어집니다.

📌 한 줄 요약: DML 튜닝은 "잘 찾기(SELECT 튜닝)"에 더해, **바꾸는 동안 발생하는 인덱스·제약·Redo·Undo·Lock·커밋의 부가 비용을 줄이는 것** 입니다.
