## 6.3.1 테이블 파티션

### 핵심 개념 (6.3 도입 포함)

파티션을 이용하면 대량 추가/변경/삭제 작업을 빠르게 처리할 수 있습니다. **파티셔닝(Partitioning)** 이란 테이블 또는 인덱스 데이터를 **특정 컬럼(파티션 키) 값에 따라 별도 세그먼트에 나눠 저장**하는 것을 말합니다.

```text
큰 테이블  →  파티션 키(계절)로 분할  →  각 칸이 별도 세그먼트

   [ 봄 ]   [ 여름 ]   [ 가을 ]   [ 겨울 ]     (계절별 옷장 서랍)
```
[그림 6-10] — 계절별로 옷을 나눠 관리하면 외출할 때 필요한 옷을 쉽고 빠르게 찾습니다. 데이터도 월별·분기별·반기별·연별로 나눠 저장하면 빠르게 조회하고 관리하기 쉽습니다.

**파티션이 필요한 이유**는 관리적·성능적 두 측면입니다.

| 측면 | 효과 |
|------|------|
| 관리적 측면 | 파티션 단위 백업·추가·삭제·변경 → **가용성 향상** |
| 성능적 측면 | 파티션 단위 조회 및 DML → **경합 또는 부하 분산** |

파티션에는 **Range, 해시(Hash), 리스트(List)** 세 종류가 있습니다.

---

### Range 파티션 — 날짜 기준 (오라클 8~)

가장 기초적인 방식으로, 주로 **날짜 컬럼**을 기준으로 파티셔닝합니다.

```sql
create table 주문 (주문번호 number, 주문일자 varchar2(8), 고객ID varchar2(5)
                 , 배송일자 varchar2(8), 주문금액 number, ...)
partition by range(주문일자) (
  partition P2017_Q1 values less than ('20170401')   -- < 20170401
, partition P2017_Q2 values less than ('20170701')
, partition P2017_Q3 values less than ('20171001')
, partition P2017_Q4 values less than ('20180101')
, partition P2018_Q1 values less than ('20180401')
, partition P9999_MX values less than (MAXVALUE)      -- >= 20180401 전부
);
```
[그림 6-11] — 각 레코드는 파티션 키(주문일자) 값에 따라 해당 분기 파티션에 저장됩니다. `MAXVALUE` 파티션은 어떤 경계에도 안 걸리는 값을 모두 받습니다.

---

### Partition Pruning — 파티션 성능의 핵심 원리

> "파티션 테이블에 대한 SQL 성능 향상 원리는 **파티션 Pruning(=Elimination)** 에 있다. 'prune'은 '쓸데없는 가지를 치다', '불필요한 부분을 제거한다'는 뜻이다. 파티션 Pruning은 SQL 하드파싱이나 실행 시점에 **조건절을 분석해 읽지 않아도 되는 파티션 세그먼트를 액세스 대상에서 제외**하는 기능이다."

```sql
select * from 주문
where 주문일자 >= '20120401' and 주문일자 <= '20120630';
```

이 조건은 전체 1,200만 건 중 25%(300만 건)에 해당합니다(그림 6-12).

```text
[비파티션]  1,200만 건이 한 덩어리
            → 25%(300만) 조회를 위해 1,200만 건 전체를 Full Scan

[파티션]    100만 건 × 12개 파티션
            P1 : 100만  (건너뜀, Pruning)
            P2 : 100만  (건너뜀, Pruning)
            P3 : 100만  ◄┐
            P4 : 100만  ◄┼ 조건에 걸리는 3개(300만)만 Full Scan
            P5 : 100만  ◄┘
            ...        (나머지 9개는 모두 건너뜀)
```
[그림 6-12]

**왜 파티션이 답인가** — 300만 건(25%)이라는 양이 딜레마를 만듭니다.
- **인덱스로 랜덤 액세스**하면? 25%는 너무 많아 테이블 Full Scan보다 **오히려 느립니다**(3장 손익분기점).
- **전체를 Full Scan**하면? 1,200만 건(예: 100GB) 전체를 훑어야 해 너무 큽니다.

테이블을 100만 건 단위 파티션으로 나누면, Full Scan을 하더라도 **전체가 아닌 해당 파티션 3개**(300만 건)만 읽고 멈춥니다. 인덱스도 Full Scan도 아닌 제3의 길입니다. 여기에 **병렬 처리가 만나면 효과는 배가**됩니다.

> 참고로 파티션 테이블도 인덱스로 액세스할 수 있지만, 파티션 Pruning을 이용한 테이블 스캔보다 훨씬 느립니다.

파티션은 **클러스터·IOT와 마찬가지로 관련 데이터를 물리적으로 인접하게 저장하는 군집(클러스터링) 기술**에 속합니다 — 클러스터(3.1.7)는 블록 단위, IOT(3.1.6)는 정렬 순서로 모은다는 점만 다릅니다.

### 해시 파티션 — 분포 균등 컬럼 기준 (오라클 8i~)

파티션 키 값을 **해시 함수**에 입력해 반환받은 값이 같은 데이터를 같은 세그먼트에 저장합니다. 파티션 **개수만 사용자가 정하고** 분산 알고리즘은 오라클 내부 해시함수가 결정합니다.

```sql
create table 고객 (고객ID varchar2(5), 고객명 varchar2(10), ...)
partition by hash(고객ID) partitions 4;
```
[그림 6-13] — `func(고객id)` 값(1~4)에 따라 SYS_P001~SYS_P004에 분산됩니다.

| 특징 | 내용 |
|------|------|
| 적합 컬럼 | 고객ID처럼 **변별력 좋고 분포가 고른** 컬럼 |
| Pruning 조건 | **등치(=) 또는 IN-List 조건일 때만** 작동 |

핵심 제약: 해시는 값을 흩뜨리므로, 검색 시 조건절 비교 값(상수/변수)에 **똑같은 해시 함수**를 적용해 읽을 파티션을 정합니다. 따라서 **범위(BETWEEN, 부등호) 조건으로는 Pruning이 불가능**합니다 — 해시값은 원래 값의 순서를 보존하지 않기 때문입니다.

---

### 리스트 파티션 — 값 목록 기준 (오라클 9i~)

**사용자가 정의한 그룹핑 기준**에 따라 데이터를 분할 저장합니다.

```sql
create table 인터넷매물 (물건코드 varchar2(5), 지역분류 varchar2(4), ...)
partition by list(지역분류) (
  partition P_지역1 values ('서울')
, partition P_지역2 values ('경기', '인천')
, partition P_지역3 values ('부산', '대구', '대전', '광주')
, partition P_기타  values (DEFAULT)    -- 나머지 전부
);
```
[그림 6-14] — Range가 "경계 미만"으로 자동 분할한다면, List는 **어떤 값이 어느 파티션에 갈지 직접 열거**합니다. `DEFAULT`는 열거되지 않은 값을 모두 받습니다.

---

### 세 파티션 비교

| 종류 | 도입 | 기준 | Pruning 작동 조건 | 대표 용도 |
|------|------|------|------|------|
| **Range** | 8 | 값의 범위(경계 미만) | =, IN, **범위 모두** | 날짜/시계열 데이터 |
| **Hash** | 8i | 해시 함수 결과 | **= 또는 IN-List만** | 분포 균등 컬럼 부하 분산 |
| **List** | 9i | 값 목록 직접 열거 | =, IN | 코드성 분류(지역·등급) |

---

## ✅ 이 절(6.3.1)의 핵심 통찰

1. 파티셔닝은 파티션 키로 데이터를 **별도 세그먼트에 나눠 저장**하는 것 — 관리적(가용성)·성능적(부하 분산) 효과가 있다.
2. 성능 향상의 핵심은 **Partition Pruning** — 조건절을 분석해 읽지 않아도 되는 파티션을 액세스 대상에서 제외한다.
3. 파티션은 "인덱스 랜덤 액세스(25%엔 비싸다) vs 전체 Full Scan(너무 크다)"의 **제3의 길** — 해당 파티션만 Full Scan한다. 클러스터·IOT와 같은 군집 기술이다.
4. **Range**(범위, 날짜)는 모든 연산자에서 Pruning되지만, **Hash**(분포 균등 컬럼)는 해시가 순서를 안 지켜 **= / IN-List에서만** Pruning된다.
5. **List**는 어떤 값이 어느 파티션에 갈지 직접 열거하며, Range/List는 `MAXVALUE`/`DEFAULT`로 나머지를 받는다.

## 🔗 흐름 속 위치

6.2가 대량 DML의 I/O 경로(Direct Path)와 병렬화를 다뤘다면, 6.3은 **작업 범위를 파티션 단위로 줄이는** 축입니다. 이어지는 6.3.2(인덱스 파티션)는 인덱스에 같은 원리를 적용하고, 6.3.3~6.3.5는 파티션을 활용한 대량 UPDATE/DELETE/INSERT 튜닝의 구체 기법으로 이어집니다.

📌 한 줄 요약: 파티션은 대량 DML과 조회를 **전체 테이블 작업에서 특정 조각(파티션) 작업으로** 바꾸는 장치이며, 그 성능의 정체는 불필요한 조각을 건너뛰는 **Partition Pruning** 입니다.
