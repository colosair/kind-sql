## 6.1.6 MERGE 문 활용

### 핵심 개념

`MERGE`는 소스 집합을 타깃과 조인해 **매칭되면 UPDATE, 매칭 안 되면 INSERT**를 한 문장으로 처리하는 구문입니다(UPSERT = UPDATE + INSERT). 오라클 9i에서 도입되었으며, 원래 동기가 **DW(데이터웨어하우스)** 작업입니다.

> "DW에서 가장 흔히 발생하는 오퍼레이션은 기간계 시스템에서 가져온 신규 트랜잭션 데이터를 반영함으로써 두 시스템 간 데이터를 동기화하는 작업이다. ... 이 중 3번 데이터 적재 작업을 효과적으로 지원하기 위해 오라클 9i에서 MERGE 문이 도입됐다."

```text
1. Extraction   : 기간계에서 변경분 추출 → customer_delta 생성
2. Transportation: delta 테이블을 DW로 전송
3. Loading      : MERGE로 DW에 반영 ← MERGE가 도입된 이유
```

```sql
-- 1. 변경분 추출
SQL> create table customer_delta as
  2  select * from customer
  3  where mod_dt >= trunc(sysdate)-1 and mod_dt < trunc(sysdate);

-- 3. MERGE로 적재 (UPSERT)
SQL> merge into customer t using customer_delta s on (t.cust_id = s.cust_id)
  2  when matched then
  3    update set t.cust_nm = s.cust_nm, t.email = s.email, ...
  4  when not matched then
  5    insert (cust_id, cust_nm, email, tel_no, region, addr, reg_dt)
  6    values (s.cust_id, s.cust_nm, s.email, s.tel_no, s.region, s.addr, s.reg_dt);
```

MERGE는 Source를 기준으로 Target과 **Left Outer Join** 방식으로 조인하여, 조인에 **성공하면 UPDATE, 실패하면 INSERT** 합니다(그림 6-7).

```text
            INSERT  ┌─────────┐
  신규 데이터 ─────→ │  Target │
  기존 데이터 ─────→ │ (기존)  │
            UPDATE  └─────────┘
  Source = Customer_Delta  ──Left Outer Join──  Target = Customer
```
[그림 6-7]

### Optional Clauses — MATCHED / NOT MATCHED 단독 사용

`when matched`와 `when not matched`는 **각각 단독으로** 쓸 수 있습니다.

```sql
-- UPDATE만 (순수 갱신)
merge into customer t using customer_delta s on (t.cust_id = s.cust_id)
when matched then update set t.cust_nm = s.cust_nm, ...;

-- INSERT만 (순수 입력)
merge into customer t using customer_delta s on (t.cust_id = s.cust_id)
when not matched then insert (...) values (...);
```

이 덕분에 MERGE는 **6.1.5의 수정가능 조인 뷰를 대체**할 수 있습니다. 11g에서 수정가능 조인 뷰가 막히던 부서별 평균급여 갱신을 MERGE로 깔끔하게 처리합니다.

```sql
-- 〈수정가능 조인 뷰〉 (11g에서 ORA-01779)
update (select d.deptno, d.avg_sal d_avg_sal, e.avg_sal e_avg_sal
        from (select deptno, round(avg(sal),2) avg_sal from emp group by deptno) e, dept d
        where d.deptno = e.deptno)
set d_avg_sal = e_avg_sal;

-- 〈MERGE 문〉 (버전 무관하게 동작)
merge into dept d
using (select deptno, round(avg(sal),2) avg_sal from emp group by deptno) e
on (d.deptno = e.deptno)
when matched then update set d.avg_sal = e.avg_sal;
```

### Conditional Operations — 추가 조건절

`ON` 절의 조인 조건 외에, MATCHED/NOT MATCHED 각각에 `where`로 추가 조건을 걸 수 있습니다.

```sql
merge into customer t using customer_delta s on (t.cust_id = s.cust_id)
when matched then
  update set t.cust_nm = s.cust_nm, t.email = s.email, ...
  where reg_dt >= to_date('20080101','yyyymmdd')    -- 이 조건 맞는 행만 UPDATE
when not matched then
  insert (...) values (...)
  where reg_dt < trunc(sysdate);                     -- 이 조건 맞는 행만 INSERT
```

### DELETE Clause — 두 가지 함정

MERGE는 조건에 따라 삭제하는 기능도 제공합니다.

```sql
merge into customer t using customer_delta s on (t.cust_id = s.cust_id)
when matched then
  update set t.cust_nm = s.cust_nm, t.email = s.email, ...
  delete where t.withdraw_dt is not null    -- 탈퇴일시가 null 아닌 레코드 삭제
when not matched then
  insert (...) values (...);
```

**함정 ① — DELETE는 UPDATE된 결과로 판단한다.** MERGE의 DELETE 절은 `when matched`의 UPDATE가 **이뤄진 결과**를 기준으로 삭제 여부를 정합니다. 즉 탈퇴일시가 null이 아니었어도, UPDATE 결과가 null이면 삭제되지 않습니다.

> "결국 DELETE 절은, 조인에 성공한(Matched) 데이터를 모두 UPDATE 하고서 그 결과 값이 DELETE WHERE 조건절을 만족하면 삭제하는 기능이다."

**함정 ② — DELETE는 조인 성공(Matched) 데이터만 삭제한다.** Source에서 삭제된 데이터를 Target에서도 지우고 싶을 때 MERGE의 DELETE는 쓸 수 없습니다. Source에서 삭제된 데이터는 **조인에 실패**(Not Matched)하기 때문입니다. 조인 실패 데이터는 UPDATE도 DELETE도 할 수 없습니다(그림 6-8).

```text
            INSERT  ┌─────────┐
  신규 데이터 ─────→ │  Target │
  기존 데이터 ─────→ │ (기존)  │
            UPDATE  └─────────┘
  삭제 데이터  ──✗── (조인 실패 → 손댈 수 없음)
```
[그림 6-8]

### Merge Into 활용 예 — UPSERT를 한 문장으로

저장하려는 레코드가 있으면 UPDATE, 없으면 INSERT하는 전형적 UPSERT를 단계적으로 개선합니다.

**① 항상 SQL 두 번** (SELECT 1번 + INSERT/UPDATE 1번)
```sql
select count(*) into :cnt from dept where deptno = :val1;
if :cnt = 0 then
  insert into dept(deptno, dname, loc) values(:val1, :val2, :val3);
else
  update dept set dname = :val2, loc = :val3 where deptno = :val1;
end if;
```

**② 최대 SQL 두 번** (UPDATE 먼저, 영향 행 없으면 INSERT)
```sql
update dept set dname = :val2, loc = :val3 where deptno = :val1;
if sql%rowcount = 0 then
  insert into dept(deptno, dname, loc) values(:val1, :val2, :val3);
end if;
```

**③ SQL 한 번** (MERGE + dual)
```sql
merge into dept a
using (select :val1 deptno, :val2 dname, :val3 loc from dual) b
on (b.deptno = a.deptno)
when matched then     update set a.dname = b.dname, a.loc = b.loc
when not matched then insert (a.deptno, a.dname, a.loc) values (b.deptno, b.dname, b.loc);
```

| 방식 | SQL 수행 횟수 | 비고 |
|------|------|------|
| ① count 후 분기 | 항상 2번 | 가장 비효율 |
| ② update 후 rowcount 분기 | 최대 2번 | 매칭되면 1번 |
| ③ MERGE | **1번** | One SQL UPSERT |

### 💡 수정가능 조인 뷰 vs MERGE 문 — 안티패턴 주의

UPDATE 문이 위기를 맞고 있습니다 — INSERT 없는 **단순 UPDATE에도** MERGE를 쓰는 개발자가 늘고 있기 때문입니다. 실행계획만 같다면 무엇을 쓰든 상관없지만, 다음과 같은 **성능이 나쁜 패턴**이 자주 보입니다.

```sql
-- ⚠️ 안티패턴: SELECT로 ROWID 검증 후, 바깥에서 ROWID로 MERGE
MERGE INTO emp T2
USING (SELECT T.ROWID AS RID, S.ENAME
       FROM   EMP T, EMP_SRC S
       WHERE  T.EMPNO = S.EMPNO
       AND    T.ENAME <> S.ENAME) S
ON (T2.ROWID = S.RID)
WHEN MATCHED THEN UPDATE SET T2.ENAME = S.ENAME;
```

이 패턴이 나쁜 이유는 자명합니다 — **UPDATE 대상 테이블 EMP를 두 번 액세스**하기 때문입니다(USING 절에서 한 번, 바깥 MERGE에서 ROWID로 또 한 번). `ON`에 ROWID를 썼으니 괜찮다고 생각할 수 있지만, ROWID는 포인터일 뿐 액세스가 사라지는 것은 아닙니다.

차라리 **수정가능 조인 뷰(6.1.5)** 가 더 낫습니다.

```sql
-- ✅ 같은 작업, EMP 한 번만 액세스
UPDATE (
  SELECT S.ENAME AS S_ENAME, T.ENAME AS T_ENAME
  FROM   EMP T, EMP_SRC S
  WHERE  T.EMPNO = S.EMPNO
  AND    T.ENAME <> S.ENAME
)
SET T_ENAME = S_ENAME;
```

> 물론 `EMP_SRC`의 EMPNO에 Unique 인덱스가 있어야 하지만 대개는 있습니다. 없으면 10g는 `bypass_ujvc`, 12c는 GROUP BY로 ORA-01779를 회피합니다(11g는 불가). (각주 7: MERGE는 Using절에 Unique 인덱스를 요구하진 않지만, 정확한 갱신을 위해 소스가 복잡하면 GROUP BY 처리가 필요하긴 합니다.)

**결론: 조인 UPDATE에 MERGE를 쓸지 수정가능 조인 뷰를 쓸지는 각자의 몫이지만, "INSERT 없는 단순 UPDATE를 ROWID MERGE로 감싸 타깃을 두 번 읽는 패턴"은 피해야 합니다.**

### 튜닝 포인트

`MERGE`는 편리하지만 **내부적으로는 조인**이므로 4장 조인 튜닝이 그대로 적용됩니다.

| 포인트 | 설명 |
|------|------|
| `ON` 조건 | 타깃 매칭을 결정 → 타깃 조인 키에 인덱스가 중요 |
| 소스(Using) 집합 | 크기에 따라 NL/Hash 조인 방식 선택이 갈림 |
| UPDATE 컬럼 | 인덱스 컬럼을 바꾸면 인덱스 삭제+삽입 비용 발생(6.1.1) |
| INSERT 경로 | Direct Path Insert(6.2.2) 적용 가능 여부 고려 |
| 타깃 중복 액세스 | 단순 UPDATE를 MERGE로 감싸 타깃을 두 번 읽지 않도록 주의 |

---

## ✅ 이 절(6.1.6)의 핵심 통찰

1. `MERGE`는 매칭 시 UPDATE·미매칭 시 INSERT를 한 문장으로 처리하는 **UPSERT** 도구로, 9i에서 DW 동기화(기간계→DW 적재)를 위해 도입됐다. 내부는 **Source 기준 Left Outer Join**이다.
2. `when matched`/`when not matched`를 **단독 사용**할 수 있어, 11g에서 막히는 **수정가능 조인 뷰를 대체**한다.
3. 추가 `where`(Conditional)와 `delete where`(DELETE 절)를 지원한다. DELETE는 **UPDATE된 결과로 판단**하고 **조인 성공 데이터만** 삭제하므로, Source에서 삭제된 행을 Target에서 지우는 용도로는 못 쓴다.
4. UPSERT는 "항상 2번 → 최대 2번 → MERGE로 1번"으로 SQL 수행 횟수를 줄인다.
5. **안티패턴 경고**: 단순 UPDATE를 ROWID 기반 MERGE로 감싸면 타깃을 두 번 액세스한다. 이 경우 수정가능 조인 뷰가 더 효율적이다.

## 🔗 흐름 속 위치

6.1.5(수정가능 조인 뷰)가 UPDATE를 One SQL로 만드는 도구였다면, 6.1.6은 **UPDATE/INSERT/DELETE 분기까지 포함한** 더 범용적인 One SQL 패턴입니다. 6.1.2~6.1.6은 모두 "절차적 반복을 집합 처리로" 라는 한 줄기이며, 다음 6.2는 I/O 경로 자체를 바꾸는 Direct Path로 넘어갑니다.

📌 한 줄 요약: MERGE는 존재 확인 후 UPDATE/INSERT를 반복하는 절차 로직을 **하나의 집합 DML(UPSERT)** 로 바꾸지만, 단순 UPDATE까지 ROWID MERGE로 감싸 타깃을 중복 액세스하지는 말아야 합니다.
