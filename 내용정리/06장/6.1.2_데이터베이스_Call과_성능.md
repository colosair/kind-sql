## 6.1.2 데이터베이스 Call과 성능

### 핵심 개념

**데이터베이스 Call이 많으면 성능은 느릴 수밖에 없습니다.** 특히 네트워크를 경유하는 User Call의 영향이 매우 큽니다. 같은 100만 건 처리라도 "건건이 Call"하면 수십~수백 초가 걸리고, "한 번의 SQL(One SQL)"로 처리하면 1초대로 끝납니다. 6.1.2의 결론은 단순합니다 — **루프 돌며 부르지 말고, 집합으로 한 번에 처리하라.**

SQL 트레이스 리포트의 Call 통계가 보여주듯, SQL은 세 단계 Call로 나뉘어 실행됩니다.

```text
select cust_nm, birthday from customer where cust_id = :cust_id

call     count   cpu   elapsed   disk   query  current   rows
------- ------ ----- --------- ------ ------- -------- ------
Parse        1  0.00      0.00      0       0        0      0
Execute   5000  0.18      0.14      0       0        0      0
Fetch     5000  0.21      0.25      0   20000        0  50000
------- ------ ----- --------- ------ ------- -------- ------
total    10001  0.39      0.40      0   20000        0  50000
```

| Call 단계 | 의미 | 발생 SQL | 비고 |
|------|------|------|------|
| **Parse Call** | SQL 파싱·최적화 | 모든 SQL | 라이브러리 캐시에서 찾으면 최적화 단계 생략(소프트 파싱) |
| **Execute Call** | SQL 실행 | SELECT·DML | DML은 이 단계에서 모든 과정이 끝남 |
| **Fetch Call** | 결과집합을 사용자에게 전송 | SELECT만 | 전송 데이터가 많으면 여러 번 발생 |

위 리포트에서 `Execute`와 `Fetch`가 각각 5000번 — 즉 한 건씩 5000번 호출했다는 뜻입니다. Call 횟수 자체가 성능 지표입니다.

```text
Call 횟수 많음
   ↓
Parse / Execute / Fetch 반복
   ↓
(User Call이면) 네트워크 왕복 + 서버 프로세스 기동 반복
   ↓
성능 저하
```

### User Call과 Recursive Call

Call은 **어디서 발생하느냐**에 따라 두 종류로 나뉘며, 비용 차이가 결정적입니다.

```text
  Client          WAS              DBMS
 ┌──────┐  Service  ┌──────┐  User  ┌──────┐  Recursive
 │ 화면 │ ───Call──→│ 로직 │ ──Call→│ SQL  │ ──Call──┐
 └──────┘          └──────┘        └──────┘ ←───────┘
 Presentation     Business         Data Layer
   Layer            Layer       (Dictionary / 내부 SQL)
```
[그림 6-6]

| 구분 | 발생 위치 | 예시 | 비용 |
|------|------|------|------|
| **User Call** | DBMS **외부**에서 네트워크를 경유해 인입 | WAS/AP 서버가 보내는 SQL 요청 | **큼** (네트워크 왕복) |
| **Recursive Call** | DBMS **내부**에서 발생 | 파싱 중 데이터 딕셔너리 조회, PL/SQL 함수·프로시저·트리거 내장 SQL | 상대적으로 작음 (네트워크 없음) |

핵심 인식: **3-Tier 아키텍처에서 DBMS 입장의 '사용자'는 최종 브라우저가 아니라 WAS/AP 서버**입니다. 그림 6-6의 맨 왼쪽 Client는 DBMS와 직접 통신하지 않습니다. 따라서 User Call은 WAS↔DBMS 구간에서 발생하며, 이 구간의 왕복 횟수를 줄이는 것이 곧 튜닝입니다. User Call이든 Recursive Call이든 SQL을 실행할 때마다 Parse·Execute·Fetch 단계를 거칩니다.

### 절차적 루프 처리의 문제 (테스트)

같은 100만 건 INSERT를 처리 방식만 바꿔가며 측정합니다.

**① PL/SQL 루프 — Recursive Call 반복** (p410)
```sql
SQL> begin
  2    for s in (select * from source)
  3    loop
  4      insert into target values (s.no, s.empno, s.ename, s.job, s.mgr
  5                               , s.hiredate, s.sal, s.comm, s.deptno);
  6    end loop;
  7    commit;
  8  end;
  9  /
-- 경과: 00:00:29.31
```
루프를 돌며 건건이 Call이 발생하지만, **네트워크를 경유하지 않는 Recursive Call**이므로 그나마 29초.

**② PL/SQL 루프 + 건건이 Commit** (p411)
```sql
SQL> begin
  2    for s in (select * from source)
  3    loop
  4      insert into target values (...);
  5      commit;          -- ← 루프 안으로 커밋 이동
  6    end loop;
  7  end;
  8  /
-- 경과: 00:01:00.50
```
29초 → 60초로 **약 2배** 느려졌습니다. 6.1.1에서 본 대로 커밋은 LGWR 동기식 I/O라, 자주 발행하면 비싸집니다. 게다가 루프 중간 커밋은 트랜잭션 **원자성(Atomicity)** 도 깨뜨립니다.

> 반대로 매우 오래 걸리는 트랜잭션을 한 번도 커밋하지 않으면 Undo 공간 부족으로 부작용이 생깁니다. 원자성을 위해 한 번에 처리해야 한다면 Undo 공간을 늘리되, 그렇지 않다면 **적당한 주기**로 커밋하는 절충안을 씁니다.

```sql
if mod(i, 100000) = 0 then   -- 10만 건마다 한 번씩 커밋
  commit;
end if;
```
이렇게 하면 맨 마지막에 한 번 커밋하는 것과 성능 차이가 크지 않습니다.

**③ Java 루프 — User Call 반복** (p412~413)
```java
public class JavaLoopQuery {
  public void execute() throws Exception {
    String SQLStmt = "select no, empno, ... from source";
    PreparedStatement stmt = con.prepareStatement(SQLStmt);
    ResultSet rs = stmt.executeQuery();
    while (rs.next()) {            // 한 건 읽어서
      ...
      insertTarget(con, no, empno, ...);   // 한 건 INSERT → User Call!
    }
  }
}
```
```text
c:\Test>java JavaLoopQuery
elapsed time : 218.392초
```
같은 루프인데 **218초** — PL/SQL(29초)의 7배 이상. 차이는 단 하나, INSERT가 **네트워크를 경유하는 User Call**이라는 점입니다.

**④ One SQL — Insert Into Select** (p413)
```sql
SQL> insert into target select * from source;
1000000 개의 행이 만들어졌습니다.
-- 경과: 00:00:01.46
```

| 방식 | Call 성격 | 소요시간 | 배수(④ 기준) |
|------|------|---:|---:|
| ④ Insert Into Select (One SQL) | Execute Call 1회 | **1.46초** | ×1 |
| ① PL/SQL 루프 | Recursive Call ×100만 | 29.31초 | ×20 |
| ② PL/SQL 루프 + 건건 Commit | Recursive Call + Commit ×100만 | 60.50초 | ×41 |
| ③ Java 루프 | **User Call** ×100만 | 218.39초 | **×150** |

### One SQL의 중요성과 레이어 이동

가장 빠른 방식은 작업을 **하나의 SQL**로 처리하는 것입니다.

```text
절차적 루프                One SQL
─────────────             ─────────────
100만 번 Execute Call  →   1번 Execute Call
   ↓                          ↓
앱이 한 건씩 지휘          DB 엔진이 집합으로 처리
```

**CS 관점 — 레이어 이동 트레이드오프.** 루프를 애플리케이션 레이어에서 돌리는 대신, **연산을 데이터가 있는 레이어(DB 엔진)로 밀어 넣는** 것입니다. 비싼 레이어(WAS↔DB 네트워크 왕복)의 일을 싼 레이어(DB 내부 집합 연산)로 내립니다.

```text
[비싼 레이어]  WAS ↔ DB 네트워크 왕복 100만 회   ← User Call
[싼 레이어]    DB 엔진 내부 집합 처리 1회         ← One SQL
```

이는 6장 전체가 따르는 통일 원리 `reduce I/O`의 "Call/왕복 축소" 버전입니다. 단 One SQL이 항상 가능한 것은 아니므로, 불가피하게 루프를 돌아야 할 때 Call을 줄이는 절충 기법이 다음 절 **Array Processing(6.1.3)** 입니다.

---

## ✅ 이 절(6.1.2)의 핵심 통찰

1. SQL은 **Parse·Execute·Fetch** 세 Call 단계를 거치며, Call 횟수 자체가 성능을 좌우한다.
2. Call은 **User Call(외부·네트워크 경유, 비쌈)** 과 **Recursive Call(내부, 저렴)** 로 나뉜다. 3-Tier에서 DBMS의 사용자는 WAS/AP 서버다.
3. 같은 루프라도 PL/SQL(Recursive, 29초) vs Java(User Call, 218초)로 **7배 이상** 차이가 난다 — 네트워크 왕복 비용이 본질이다.
4. 루프 안에서 건건이 커밋하면 LGWR 동기 I/O 반복으로 더 느려지고(29→60초) 원자성도 깨진다. 불가피하면 **N건마다 한 번** 커밋한다.
5. 최선은 절차적 루프를 **One SQL(Insert Into Select)** 로 바꾸는 것 — 1.46초. 연산을 데이터가 있는 레이어로 미는 레이어 이동이다.

## 🔗 흐름 속 위치

6.1.1이 DML 부가비용 전체를 개괄했다면, 6.1.2는 그중 **애플리케이션↔DBMS 사이의 Call 비용**을 집중 조명합니다. One SQL이 불가능할 때의 차선책인 6.1.3(Array Processing)으로 직접 이어집니다.

📌 한 줄 요약: 대량 DML의 제1원칙은 **"건건이 부르지 말고 한 번에 처리하라"** — 네트워크 User Call과 커밋 횟수를 줄이는 것이 핵심입니다.
