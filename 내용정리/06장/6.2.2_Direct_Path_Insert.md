## 6.2.2 Direct Path Insert

### 핵심 개념

Direct Path Insert는 버퍼캐시를 거치지 않고 새 블록을 데이터 파일에 직접 적재하는 INSERT 방식입니다. 일반 INSERT가 기존 블록의 빈 공간을 찾아 입력한다면, Direct Path Insert는 세그먼트 끝에 새 공간을 할당해 대량으로 적재합니다.

```text
일반 INSERT
   ↓
빈 블록 탐색
   ↓
버퍼캐시에 블록 적재
   ↓
로우 입력

Direct Path Insert
   ↓
세그먼트 끝에 새 블록 할당
   ↓
PGA에서 데이터 구성
   ↓
데이터 파일에 직접 기록
```

### APPEND 힌트

Oracle에서는 보통 `APPEND` 힌트로 Direct Path Insert를 유도합니다.

```sql
insert /*+ append */ into target
select *
from source;
```

대량 입력에서 이 방식은 일반 INSERT보다 빠를 수 있습니다.

### 장점과 주의점

| 장점 | 설명 |
---|---|
| 빠른 대량 적재 | 버퍼캐시 경유와 빈 공간 탐색 비용을 줄입니다. |
| Redo 감소 가능 | 조건에 따라 NOLOGGING과 함께 Redo를 줄일 수 있습니다. |
| 병렬 INSERT와 궁합 | 대량 병렬 적재에 적합합니다. |

| 주의점 | 설명 |
---|---|
| HWM 증가 | 기존 빈 공간을 재사용하지 않고 세그먼트 끝에 적재할 수 있습니다. |
| Lock 영향 | 테이블 수준 Lock 등 동시성 영향이 있을 수 있습니다. |
| 복구 전략 | NOLOGGING 사용 시 백업/복구 전략이 필요합니다. |
| 소량 INSERT | OLTP 건별 입력에는 부적합합니다. |

### 튜닝 포인트

```text
대량 INSERT인가?
   ↓
동시 OLTP 영향이 허용되는가?
   ↓
복구 정책상 NOLOGGING 가능 여부 확인
   ↓
APPEND / 병렬 DML 검토
```

---

## ✅ 이 절(6.2.2)의 핵심 통찰

1. Direct Path Insert는 버퍼캐시를 우회해 대량 입력을 빠르게 처리합니다.
2. `APPEND` 힌트로 유도할 수 있습니다.
3. 세그먼트 끝에 적재하므로 HWM과 공간 사용을 고려해야 합니다.
4. NOLOGGING과 함께 쓰면 Redo를 줄일 수 있지만 복구 위험이 생깁니다.
5. OLTP 소량 INSERT가 아니라 배치 대량 적재에 적합합니다.

## 🔗 흐름 속 위치

6.2.1이 Direct Path I/O 개념이었다면, 6.2.2는 이를 INSERT에 적용한 절입니다.

📌 한 줄 요약: Direct Path Insert는 대량 입력을 기존 블록 관리 경로가 아니라 세그먼트 끝 직접 적재 경로로 처리합니다.
