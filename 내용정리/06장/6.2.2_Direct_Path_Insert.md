## 6.2.2 Direct Path Insert

### 핵심 개념 — 일반 INSERT는 왜 느린가

Direct Path Insert를 이해하려면 먼저 **일반 INSERT가 거치는 5단계**를 봐야 합니다. 이 단계들을 하나씩 건너뛰는 것이 곧 Direct Path Insert입니다.

```text
일반 INSERT 한 건
   ① Freelist에서 입력 가능 블록 찾기   (HWM 아래 여유 공간 있는 블록 목록)
   ② 할당받은 블록을 버퍼캐시에서 찾기
   ③ 버퍼캐시에 없으면 데이터파일에서 읽어 적재
   ④ INSERT 내용을 Undo 세그먼트에 기록
   ⑤ INSERT 내용을 Redo 로그에 기록
```

> **HWM(High-Water Mark)**: 테이블 세그먼트에서 데이터가 한 번이라도 채워졌던 최고 수위선. Freelist는 HWM 아래쪽의 여유 공간 블록 목록입니다.

---

### Direct Path Insert를 유도하는 4가지 방법

```sql
-- 1. INSERT ... SELECT 문에 append 힌트
insert /*+ append */ into target select * from source;

-- 2. parallel 힌트로 병렬 모드 INSERT (6.2.3)
insert /*+ parallel(t 4) */ into target select * from source;

-- 3. direct 옵션을 지정하고 SQL*Loader(sqlldr)로 적재
-- sqlldr ... direct=true

-- 4. CTAS (create table ... as select)
create table target as select * from source;
```

**Array Processing(6.1.3)도** `append_values` 힌트로 Direct Path Insert가 됩니다.
```sql
procedure insert_target(p_source in typ_source) is
begin
  forall i in p_source.first .. p_source.last
    insert /*+ append_values */ into target values p_source(i);
end;
```

---

### Direct Path Insert가 빠른 이유 (5단계를 건너뛴다)

| 일반 INSERT 단계 | Direct Path Insert |
|------|------|
| ① Freelist 탐색 | **참조 안 함** — HWM **바깥** 영역에 순차 입력 |
| ② 버퍼캐시에서 블록 탐색 | **탐색 안 함** |
| ③ 버퍼캐시에 적재 | **적재 안 함** — 데이터파일에 직접 기록 |
| ④ Undo 로깅 | **안 함** |
| ⑤ Redo 로깅 | **안 하게 할 수 있음**(NOLOGGING) |

**④ Undo 로깅을 안 하는 이유** (각주 8) — Undo의 3대 용도(롤백·복구·읽기 일관성)가 모두 불필요해집니다. HWM **위쪽**에 쌓는 신규 데이터는 (ㄱ) 커밋 전 다른 세션이 읽지 않으니 Read Consistency용 Undo가 필요 없고, (ㄴ) Direct Path Insert는 커밋 전까지 HWM을 이동시키지 않으므로 **롤백은 HWM만 제자리에 두면 끝**입니다(데이터를 일일이 지울 필요 없음). Transaction Recovery용 Undo도 같은 이유로 불필요합니다.

**⑤ Redo 로깅을 줄이는 법** (각주 9 — 정확히는 "최소화", 딕셔너리 변경분만 로깅) — 테이블을 NOLOGGING 모드로 바꾼 뒤 Direct Path Insert합니다.
```sql
alter table t NOLOGGING;
```
> 참고: **Direct Path Insert가 아닌 일반 INSERT를 로깅 안 하게 하는 방법은 없습니다.** (각주 10: Redo는 데이터베이스 영속성(Durability)을 위한 필수 기능)

#### ⚠️ 작동하지 않는 nologging Insert (말풍선)
```sql
-- nologging이 T 테이블의 Alias일 뿐, NOLOGGING 기능과 무관!
insert into t NOLOGGING select * from test;

-- 오라클은 nologging 힌트를 제공하지 않음 (무시됨)
insert /*+ APPEND NOLOGGING */ into t select * from test;
```
NOLOGGING은 **테이블 속성(`alter table`)** 으로만 지정합니다. INSERT 문 안에서 키워드나 힌트로 주는 것은 모두 무효입니다.

---

### 주의점 두 가지

**① Exclusive 모드 TM Lock이 걸린다.** 성능은 비교할 수 없이 빠르지만, **커밋 전까지 다른 트랜잭션이 해당 테이블에 DML을 수행할 수 없습니다.** 트랜잭션이 빈번한 주간(OLTP 시간대)에 이 옵션을 쓰는 것은 절대 금물입니다. (TM Lock은 6.4.1 'DML 테이블 Lock' 참조)

**② HWM 바깥에만 쌓아 여유 공간을 재활용하지 않는다.** Freelist를 조회하지 않으므로, 테이블에 빈 공간이 있어도 쓰지 않고 항상 HWM 위쪽 새 영역에 적재합니다(그림 6-9).

```text
일반 Insert        : ▓░▓░▓░░▓░  → 중간 빈 공간(░) 재활용
Direct Path Insert : ▓▓▓▓▓▓▓▓▓ |░░░░| ↓↓↓  → HWM 바깥에만 추가
```
[그림 6-9]

이 때문에 **과거 데이터를 주기적으로 DELETE해서 공간을 비워도, Direct Path Insert만 계속하면 테이블 사이즈는 줄지 않고 계속 늘어납니다.**

| 테이블 형태 | 공간 회수 방법 |
|------|------|
| **Range 파티션** 테이블 | 과거 데이터를 DELETE가 아니라 **파티션 DROP**(또는 Truncate)으로 제거해야 공간 반환됨 (각주 11) |
| 비파티션 테이블 | 주기적으로 **Reorg(재구성)** 작업 필요 |

> (각주 11) 그래서 테이블을 시계열 컬럼 기준으로 Range 파티셔닝하면, Range 파티션이 INSERT 작업이 신규 파티션에만 일어나 Direct Path Insert와 궁합이 좋습니다.

---

## ✅ 이 절(6.2.2)의 핵심 통찰

1. 일반 INSERT의 5단계(Freelist 탐색→버퍼캐시 탐색→적재→Undo→Redo)를 **모두 건너뛰는** 것이 Direct Path Insert다.
2. 유도 방법은 **`/*+ append */`, 병렬 INSERT, sqlldr direct, CTAS**, 그리고 Array Processing의 `append_values`.
3. **Undo를 안 쓰는 이유**가 핵심 — HWM 위쪽 신규 데이터는 롤백이 "HWM만 제자리"면 끝이라 Undo가 불필요하다. Redo는 NOLOGGING 테이블에서 최소화된다.
4. **NOLOGGING은 테이블 속성(`alter table`)으로만** 켜진다. INSERT 문 안의 키워드·힌트는 무효다.
5. 두 함정: **Exclusive TM Lock**(OLTP 시간대 금물)과 **HWM 바깥 적재로 공간 미재활용**(파티션 DROP 또는 Reorg로 회수).

## 🔗 흐름 속 위치

6.2.1의 Direct Path I/O 개념을 INSERT에 적용한 절입니다. 6.1.4의 대량 적재 테스트에서 이미 `/*+ append */`를 썼던 그 메커니즘의 정체이며, 다음 6.2.3(병렬 DML)은 이 Direct Path Insert를 여러 프로세스로 병렬화합니다.

📌 한 줄 요약: Direct Path Insert는 일반 INSERT의 **버퍼캐시·Freelist·Undo·Redo 경로를 모두 우회**해 HWM 바깥에 직접 순차 적재하는 방식으로, 대량 배치에 최적이지만 TM Lock과 공간 미재활용을 감수해야 합니다.
