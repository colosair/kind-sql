## 6.3.4 파티션을 활용한 대량 DELETE 튜닝

### 핵심 개념

대량 DELETE는 행마다 Undo/Redo와 인덱스 삭제 비용을 만듭니다. 삭제 대상이 파티션 단위와 일치하면 `DELETE`보다 `TRUNCATE PARTITION` 또는 `DROP PARTITION`이 훨씬 효율적입니다.

```text
DELETE
   ↓
각 행 삭제
   ↓
Undo/Redo 생성
   ↓
인덱스 엔트리 삭제

TRUNCATE/DROP PARTITION
   ↓
파티션 세그먼트 단위 제거
   ↓
행 단위 삭제 비용 회피
```

### 파티션 단위 삭제

예를 들어 월별 Range Partition 테이블에서 2024년 1월 데이터를 삭제해야 한다면 다음이 가능합니다.

```sql
alter table 주문 truncate partition p202401;
```

또는 보관이 필요 없으면:

```sql
alter table 주문 drop partition p202401;
```

### 장점과 주의점

| 장점 | 설명 |
---|---|
| 빠름 | 행 단위 삭제가 아니라 세그먼트 단위 처리입니다. |
| Undo 감소 | 행별 Undo 부담이 크게 줄어듭니다. |
| 관리 단순 | 오래된 데이터 정리에 적합합니다. |

| 주의점 | 설명 |
---|---|
| 조건 일치 필요 | 삭제 대상이 파티션 경계와 맞아야 합니다. |
| 글로벌 인덱스 | 인덱스 유지 또는 Unusable 문제가 생길 수 있습니다. |
| 복구 정책 | Drop/Truncate는 일반 DELETE와 복구 성격이 다릅니다. |

---

## ✅ 이 절(6.3.4)의 핵심 통찰

1. 대량 DELETE는 행 단위로 수행하면 매우 비쌉니다.
2. 삭제 범위가 파티션과 일치하면 파티션 단위 제거가 가장 강력합니다.
3. `TRUNCATE PARTITION`은 Undo/Redo와 인덱스 삭제 비용을 크게 줄입니다.
4. 글로벌 인덱스 유지 비용을 반드시 고려해야 합니다.
5. 파티션 설계가 잘 되어 있으면 데이터 보관/삭제 정책이 SQL 튜닝이 됩니다.

## 🔗 흐름 속 위치

6.3.3이 대량 UPDATE를 파티션 재구성으로 바꾸는 전략이었다면, 6.3.4는 대량 DELETE를 파티션 제거로 바꾸는 전략입니다.

📌 한 줄 요약: 대량 삭제는 `DELETE`가 아니라 파티션을 버리는 문제로 바꿀 때 가장 빨라집니다.
