## 6.3.4 파티션을 활용한 대량 DELETE 튜닝

### DELETE가 느린 이유 — 7단계 부수 작업

대량 DELETE는 행마다 다음 부수 작업을 수반하므로 느립니다.

```text
DELETE 한 건
   1. 테이블 레코드 삭제
   2. 테이블 레코드 삭제에 대한 Undo Logging
   3. 테이블 레코드 삭제에 대한 Redo Logging
   4. 인덱스 레코드 삭제              ← 인덱스마다 반복
   5. 인덱스 레코드 삭제에 대한 Undo Logging
   6. 인덱스 레코드 삭제에 대한 Redo Logging
   7. Undo(2번·5번)에 대한 Redo Logging   ← Undo 자체도 Redo로 보호됨
```

특히 **각 인덱스 레코드를 찾아 삭제하는 작업**의 부담이 큽니다 — 건건이 수직적 탐색을 거쳐 대상 레코드를 찾아야 하기 때문입니다(그림 6-24). 인덱스 개수가 DML 성능에 미치는 영향이 다시 한번 확인됩니다.

**DELETE는 UPDATE보다 인덱스 측면에서 더 불리합니다.**

| | 재생성 대상 인덱스 |
|------|------|
| 대량 UPDATE | **변경 대상 컬럼을 포함하는 인덱스만** |
| 대량 DELETE | **모든 인덱스** (로우 자체가 사라지므로) |

> "그렇다고 초대용량 테이블 인덱스를 모두 Drop 했다가 다시 생성하기도 만만치 않다. UPDATE는 변경 대상 컬럼을 포함하는 인덱스만 재생성하면 되지만, DELETE는 모든 인덱스를 재생성해야 한다."

```sql
-- 인덱스를 실시간 관리하며 수천만 건 삭제하면 엄청난 시간 소요
delete from 거래 where 거래일자 < '20150101';
```

→ 그래서 삭제도 파티션 구조를 활용합니다. 두 가지 길이 있습니다.

---

### 길 ①: 파티션 Drop — 삭제 조건이 파티션과 일치할 때

삭제 조건(`거래일자 < '20150101'`)이 파티션 키와 정확히 일치하고 인덱스도 다 로컬 파티션이라면, **문장 하나로 순식간에** 삭제합니다.

```sql
alter table 거래 drop partition p201412;

-- 오라클 11g부터: 파티션 이름 대신 값으로 지정 가능
alter table 거래 drop partition for('20141201');
```

`DROP PARTITION`은 파티션 세그먼트를 통째로 제거하는 메타데이터 작업이라, 행 단위 삭제의 7단계 비용(Undo/Redo/인덱스 삭제)을 **전부 회피**합니다. 데이터를 영구히 버려도 될 때 가장 강력합니다.

---

### 길 ②: 파티션 Truncate + 재적재 — 추가 삭제 조건이 있을 때

삭제 대상이 파티션 경계와 딱 맞지 않고 **또 다른 조건**이 붙는 경우입니다.

```sql
-- 거래일자뿐 아니라 상태코드 조건도 있음
delete from 거래
where 거래일자 < '20150101'
and  (상태코드 <> 'ZZZ' or 상태코드 is null);
```

이때는 **"무엇이 다수인가"** 로 전략이 갈립니다.

- **삭제 대상이 소수**(= 남길 데이터가 대다수)이면 → 위 `DELETE`를 그대로 사용.
- **삭제 대상이 대다수**(= 남길 데이터가 소수)이면 → 지우지 말고 **남길 데이터만 백업했다가 재입력**:

```sql
-- 1. 임시 테이블에 남길 데이터만 복제 (상태코드 = 'ZZZ'인 소수)
create table 거래_t as
select * from 거래
where 거래일자 < '20150101' and 상태코드 = 'ZZZ';   -- 남길 데이터만

-- 2. 삭제 대상 파티션을 Truncate (파티션 전체 비움)
alter table 거래 truncate partition p201412;
-- 11g: alter table 거래 truncate partition for('20141201');

-- 3. 백업해 둔 남길 데이터를 원본에 재입력
insert into 거래 select * from 거래_t;

-- 4. 임시 테이블 Drop
drop table 거래_t;
```

핵심 발상: **"많은 것을 지우는 것보다 적은 것을 남기는 게 싸다."** 9억 건을 DELETE하느니, 살아남을 1억 건만 빼냈다가 파티션을 통째로 비우고(Truncate, 즉시) 다시 넣는 편이 훨씬 빠릅니다.

---

### 서비스 중단 없이 Drop/Truncate하기 위한 3가지 조건

파티션 Drop/Truncate를 서비스 중단 없이 수행하려면 다음을 모두 만족해야 합니다.

| # | 조건 | 예시 |
|---|------|------|
| 1 | 파티션 키와 **커팅(삭제) 기준 컬럼이 일치** | 둘 다 '신청일자' |
| 2 | 파티션 단위와 **커팅 주기가 일치** | 월 단위 파티션을 월 주기로 커팅 |
| 3 | **모든 인덱스가 로컬 파티션 인덱스** | 파티션 키 '신청일자', PK '신청일자 + 신청순번' |

3번이 다시 핵심입니다 — **PK 인덱스는 삭제 기준(파티션 키) 컬럼이 인덱스 구성 컬럼이어야 로컬 파티셔닝이 가능**합니다(6.3.2의 Unique 인덱스 제약). 이 세 조건이 곧 "ILM을 위한 테이블 설계"의 체크리스트입니다.

---

### Drop vs Truncate vs Exchange 정리

| 기법 | 절 | 쓰는 상황 | 결과 |
|------|----|------|------|
| **DROP PARTITION** | 6.3.4 | 삭제 조건 = 파티션 경계, 데이터 영구 폐기 | 파티션 자체 제거 |
| **TRUNCATE + 재적재** | 6.3.4 | 추가 삭제 조건, 남길 데이터가 소수 | 파티션 비우고 일부 복원 |
| **EXCHANGE PARTITION** | 6.3.3 | 대량 변경(값 수정) | 수정본과 포인터 교환 |

세 기법 모두 **"행 단위 반복 → 세그먼트 단위 일괄"** 이라는 공통 원리이며, 모두 **인덱스가 로컬 파티션**이어야 깔끔하게 동작합니다(글로벌·비파티션 인덱스는 Unusable → Rebuild 필요).

---

## ✅ 이 절(6.3.4)의 핵심 통찰

1. DELETE는 테이블·인덱스 각각의 삭제 + Undo + Redo + Undo의 Redo까지 **7단계** 부수 작업을 수반해 느리다.
2. **DELETE는 UPDATE보다 불리** — UPDATE는 변경 컬럼 인덱스만, DELETE는 **모든 인덱스**를 손대야 한다(로우가 사라지므로).
3. 삭제 조건이 파티션과 일치하면 **`DROP PARTITION`** 한 문장으로 7단계 비용을 전부 회피한다(11g `for('값')`).
4. 추가 조건이 있고 삭제 대상이 다수면 **`TRUNCATE PARTITION` + 남길 데이터만 재적재** — "지우기보다 남기기"가 싸다.
5. 모든 파티션 기법(Drop/Truncate/Exchange)은 **로컬 파티션 인덱스**를 전제로 한다.

## 🔗 흐름 속 위치

6.3.3이 대량 UPDATE를 EXCHANGE로 바꿨다면, 6.3.4는 대량 DELETE를 **파티션 제거(Drop)/비움(Truncate)** 으로 바꿉니다. 다음 6.3.5는 거꾸로 대량 INSERT를 EXCHANGE로 반영하는 전략으로, 6.3 파티션 DML 3종 세트를 완성합니다.

📌 한 줄 요약: 대량 삭제는 `DELETE`(행 단위 7단계)가 아니라 **파티션을 버리는(DROP) 또는 비우는(TRUNCATE) 문제**로 바꿀 때 가장 빨라집니다.
