## 6.3.2 인덱스 파티션

### 핵심 개념

테이블 파티션과 인덱스 파티션은 **구분해야** 합니다. 인덱스 파티션은 테이블 파티션과 맞물려 다양한 구성이 존재합니다. 두 축으로 분류합니다.

```text
테이블: 비파티션 테이블 / 파티션 테이블
인덱스
├─ 파티션 인덱스
│   ├─ 로컬(Local)    : 테이블 파티션과 1:1, 오라클 자동 관리
│   └─ 글로벌(Global) : 테이블과 독립적 구성
└─ 비파티션 인덱스    : = 글로벌 비파티션
```

| 테이블 \ 인덱스 | 비파티션 인덱스 | 글로벌 파티션 | 로컬 파티션 |
|------|:---:|:---:|:---:|
| 비파티션 테이블 | O | O | — |
| 파티션 테이블 | O | O | O |

---

### 로컬 파티션 인덱스 (Local Partitioned Index)

테이블 파티션과 **1:1 대응**되도록 오라클이 자동 관리하는 인덱스입니다. 테이블 파티션을 계절별 옷장 서랍에 비유했다면, 로컬 인덱스는 **계절별로 별도 색인을 만드는** 것과 같습니다(그림 6-15).

```sql
-- CREATE INDEX 뒤에 LOCAL
create index 주문_x01 on 주문(주문일자, 주문금액) LOCAL;
create index 주문_x02 on 주문(고객ID, 주문일자)  LOCAL;
```
[그림 6-16] — 각 인덱스 파티션은 테이블 파티션 속성을 **그대로 상속**합니다. 테이블 파티션 키가 주문일자면 인덱스 파티션 키도 주문일자가 됩니다.

**최대 장점은 관리 편의성**입니다.
> "테이블 파티션 구성을 변경(add, drop, exchange 등)하더라도 인덱스를 재생성할 필요가 없다. 변경작업이 순식간에 끝나므로 피크(peak) 시간대만 피하면 서비스를 중단하지 않고도 작업할 수 있다."

---

### 글로벌 파티션 인덱스 (Global Partitioned Index)

파티션을 **테이블과 다르게** 구성한 인덱스입니다. 파티션 수·유형·기준값 정의가 다를 수 있고, 비파티션 테이블에도 만들 수 있습니다.

```sql
-- CREATE INDEX 뒤에 GLOBAL + 독립적 파티션 정의
create index 주문_x03 on 주문(주문금액, 주문일자) GLOBAL
partition by range(주문금액) (
  partition P_01 values less than (100000)      -- 주문금액 < 100000
, partition P_MX values less than (MAXVALUE)     -- 주문금액 >= 100000
);
```
[그림 6-17] — 테이블은 주문일자로, 인덱스는 주문금액으로 파티셔닝 — 서로 다른 키입니다.

**단점은 관리 비용**입니다.
> "글로벌 파티션 인덱스는 테이블 파티션 구성을 변경(drop, exchange, split 등)하는 순간 **Unusable 상태로 바뀌므로 곧바로 인덱스를 재생성**해 줘야 한다. 그동안 해당 테이블을 사용하는 서비스를 중단해야 한다."

⚠️ 주의: DBA가 그림 6-16처럼 테이블과 인덱스를 1:1로 직접 구성해도 그것은 로컬이 **아닙니다.** 오라클이 자동 관리하지 않기 때문입니다 — **모양이 로컬 같아도 글로벌**입니다.

---

### 비파티션 인덱스 (Non-Partitioned Index)

파티셔닝하지 않은 일반 인덱스입니다. 여러 테이블 파티션을 하나의 인덱스가 가리키므로 **글로벌 비파티션 인덱스**라고도 부릅니다(그림 6-18).

```sql
create index 주문_x04 on 주문(고객ID, 배송일자);   -- 일반 CREATE INDEX
```
글로벌 파티션 인덱스와 마찬가지로, 테이블 파티션 구성 변경 시 Unusable이 되어 재생성·서비스 중단이 필요합니다.

---

### Prefixed vs Nonprefixed

또 다른 분류축으로, **인덱스 파티션 키 컬럼이 인덱스 구성 컬럼의 왼쪽 선두에 위치하는가**로 나눕니다.

| 구분 | 의미 |
|------|------|
| **Prefixed** | 인덱스 파티션 키 컬럼이 인덱스 키 컬럼의 **왼쪽 선두**에 위치 |
| **Nonprefixed** | 파티션 키가 선두에 없거나, 인덱스 컬럼에 아예 포함되지 않음 |

로컬/글로벌 × Prefixed/Nonprefixed 조합은 4가지지만, **글로벌은 Prefixed만 지원**합니다.

| | Prefixed | Nonprefixed |
|------|:---:|:---:|
| 로컬 파티션 | ① | ② |
| 글로벌 파티션 | ③ | ④ **(Not Support)** |

따라서 비파티션 인덱스를 포함해 **최종 4가지 유형**으로 정리됩니다.
1. 로컬 Prefixed 파티션 인덱스
2. 로컬 Nonprefixed 파티션 인덱스
3. 글로벌 Prefixed 파티션 인덱스
4. 비파티션 인덱스

```sql
-- 주문 테이블의 4개 인덱스 유형 확인
SQL> select i.index_name, i.partitioned, p.partitioning_type
  2       , p.locality, p.alignment
  3  from user_indexes i, user_part_indexes p
  4  where i.table_name = '주문' and p.index_name(+) = i.index_name;

INDEX_NAME  PAR  PARTITION  LOCALI  ALIGNMENT
----------  ---  ---------  ------  -----------
주문_X01    YES  RANGE      LOCAL   PREFIXED      → 로컬 Prefixed
주문_X02    YES  RANGE      LOCAL   NON_PREFIXED  → 로컬 Nonprefixed
주문_X03    YES  RANGE      GLOBAL  PREFIXED      → 글로벌 Prefixed
주문_X04    NO                                    → 비파티션
```

### ★ 중요한 인덱스 파티션 제약 — Unique 인덱스

> "**Unique 인덱스를 파티셔닝하려면, 파티션 키가 모두 인덱스 구성 컬럼이어야 한다.**"

이 제약의 이유를 두 그림으로 비교합니다.

**✅ 제약 만족 (그림 6-19)** — 주문일자로 파티셔닝, PK = `주문일자 + 주문번호`, 로컬 인덱스
```text
PK 인덱스 키 = 주문일자 + 주문번호 (파티션 키 주문일자가 인덱스 구성 컬럼)
  P2017_Q1   P2017_Q2   P2017_Q3   P2017_Q4 ...
```
2017년 12월 25일 주문번호 123456을 입력하면, 중복 확인을 위해 **P2017_Q4 파티션 인덱스만 탐색**하면 됩니다. 2017년 12월 주문은 P2017_Q4에만 들어갈 수 있기 때문입니다. → 효율적.

**❌ 제약 위반 (그림 6-20)** — PK = `주문번호` 단일 컬럼인데 주문일자로 파티셔닝(가정)
```text
PK 인덱스 키 = 주문번호 (파티션 키 주문일자가 인덱스 구성 컬럼 아님)
  P2017_Q1   P2017_Q2   P2017_Q3   P2017_Q4 ...
```
주문번호 123456을 입력하면, 중복 확인을 위해 **모든 인덱스 파티션을 다 탐색**해야 합니다 — 주문번호 123456은 어떤 파티션에든 들어갈 수 있기 때문입니다. 게다가 입력·커밋 전에 다른 트랜잭션이 같은 주문번호로 다른 파티션에 입력하는 것까지 막으려면 **추가적인 Lock 메커니즘**이 필요합니다. 이런 비효율 때문에 오라클은 이 구성을 **금지**합니다.

핵심 직관: Unique 검증을 **단일 파티션 탐색으로 끝낼 수 있어야** 파티셔닝이 의미가 있습니다. 파티션 키가 인덱스 컬럼에 포함되어야 "이 키는 이 파티션에만 존재할 수 있다"가 보장됩니다.

---

### DML/운영 관점 정리

| 인덱스 종류 | 테이블 파티션 변경 시 | 대량 DML/관리 |
|------|------|------|
| 로컬 파티션 | 인덱스 재생성 **불필요**(자동 관리) | TRUNCATE/DROP PARTITION에 강함 |
| 글로벌 파티션 | **Unusable → 재생성 필요**(서비스 중단) | 관리 부담 큼 |
| 비파티션 | **Unusable → 재생성 필요** | 관리 부담 큼 |

대량 파티션 작업(적재/삭제/교환)이 잦으면 **로컬 인덱스가 운영 안정성에 유리**합니다. 반대로 파티션 키와 무관한 특정 조회가 중요하면 글로벌 인덱스가 답일 수 있습니다.

---

## ✅ 이 절(6.3.2)의 핵심 통찰

1. 인덱스 파티션은 **로컬**(테이블과 1:1, 오라클 자동 관리), **글로벌**(독립 구성), **비파티션** 세 가지로 나뉜다.
2. **로컬은 테이블 파티션 변경에 강하다**(재생성 불필요). 글로벌·비파티션은 Unusable이 되어 재생성·서비스 중단이 필요하다.
3. 모양이 1:1이어도 오라클이 자동 관리하지 않으면 로컬이 아니라 글로벌이다.
4. **Prefixed/Nonprefixed**(파티션 키가 인덱스 선두인가)와 로컬/글로벌을 조합하면 4유형이며, **글로벌 Nonprefixed는 미지원**이다.
5. **Unique 인덱스를 파티셔닝하려면 파티션 키가 모두 인덱스 구성 컬럼이어야 한다** — 그래야 유일성 검증을 단일 파티션 탐색으로 끝낼 수 있다(그림 6-19/6-20).

## 🔗 흐름 속 위치

6.3.1이 테이블 파티션의 범위 축소(Pruning) 효과를 다뤘다면, 6.3.2는 그 효과와 운영 편의를 **인덱스까지 일관되게 유지**하는 방법입니다. 이어지는 6.3.3~6.3.5는 이 파티션·인덱스 구조를 활용한 대량 UPDATE/DELETE/INSERT 튜닝의 실전 기법입니다.

📌 한 줄 요약: 파티션 테이블의 DML·운영 성능은 인덱스를 **로컬로 둘지 글로벌로 둘지**, 그리고 **Unique 인덱스라면 파티션 키를 인덱스에 포함했는지**에 달려 있습니다.
