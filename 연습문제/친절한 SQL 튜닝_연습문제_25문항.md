## 문제

### 1. SQL 최적화 과정에 대한 설명으로 가장 적절한 것은?

① SQL 파싱은 통계정보를 이용해 최저 비용 실행계획을 선택하는 과정이다.  
② SQL 최적화는 옵티마이저가 가능한 실행경로를 비교해 실행계획을 선택하는 과정이다.  
③ SQL 실행계획은 SQL을 실제 실행한 뒤에만 확인할 수 있다.  
④ 구문 오류는 semantic check 단계에서만 발견된다.

### 2. 바인드 변수 사용의 효과로 가장 적절한 것은?

① SQL 텍스트가 매번 달라져 커서 공유 가능성이 낮아진다.  
② 하드 파싱 빈도를 줄여 라이브러리 캐시 재사용성을 높일 수 있다.  
③ 옵티마이저가 항상 동일한 실행계획을 선택하므로 성능 편차가 사라진다.  
④ 테이블 블록을 직접 읽지 않도록 물리 I/O를 완전히 제거한다.

### 3. 논리 I/O와 물리 I/O에 대한 설명으로 옳은 것은?

① 논리 I/O는 디스크에서 읽은 블록 수만 의미한다.  
② 물리 I/O는 버퍼 캐시에서 블록을 찾은 경우에도 항상 발생한다.  
③ 논리 I/O는 버퍼 캐시를 경유해 읽은 블록 접근을 포함한다.  
④ 버퍼 캐시 히트율이 높으면 개별 SQL의 효율성은 항상 좋다고 판단할 수 있다.

### 4. Table Full Scan과 Index Range Scan 비교로 가장 적절한 것은?

① Table Full Scan은 항상 비효율적이므로 피해야 한다.  
② Index Range Scan은 조건에 맞는 데이터가 많을수록 대량 처리에 유리하다.  
③ Table Full Scan은 대량 블록을 순차적으로 읽는 데 상대적으로 유리할 수 있다.  
④ Index Range Scan은 테이블 접근이 전혀 발생하지 않는다.

### 5. Index Range Scan이 정상적으로 시작되기 어려운 조건은?

① 선두 컬럼에 등치 조건이 있다.  
② 인덱스 선두 컬럼이 가공되어 시작 지점을 찾기 어렵다.  
③ 인덱스 구성 컬럼 순서와 조건절 컬럼이 모두 일치한다.  
④ 범위 조건의 시작값과 종료값이 명확하다.

### 6. 다음 SQL에서 인덱스 활용을 저해하는 가장 직접적인 원인은?

```sql
select *
from 주문
where to_char(주문일시, 'YYYYMMDD') = '20260602';
```

① 주문일시 컬럼이 함수로 가공되어 일반적인 인덱스 시작 지점 탐색이 어렵다.  
② 등치 조건이 사용되었기 때문이다.  
③ 문자 리터럴을 사용하면 모든 인덱스가 무효화되기 때문이다.  
④ select-list에 인덱스 컬럼이 없기 때문이다.

### 7. Index Fast Full Scan에 대한 설명으로 옳은 것은?

① 인덱스 리프 블록을 논리적 순서대로 읽으므로 정렬 결과를 보장한다.  
② 인덱스를 테이블처럼 읽어 필요한 컬럼을 처리할 수 있지만 정렬 순서를 보장하지 않는다.  
③ 반드시 단일 블록 I/O만 사용한다.  
④ 유니크 인덱스에서만 사용할 수 있다.

### 8. 테이블 액세스 최소화 관점에서 인덱스 설계 시 고려할 내용으로 가장 적절한 것은?

① 인덱스를 사용하면 테이블 랜덤 액세스 비용은 항상 0이 된다.  
② 인덱스에서 찾은 ROWID로 테이블을 반복 방문하면 비용이 커질 수 있다.  
③ 클러스터링 팩터는 인덱스 스캔 효율과 무관하다.  
④ select-list 컬럼은 인덱스 설계에 영향을 주지 않는다.

### 9. 부분범위 처리의 장점으로 가장 적절한 것은?

① 전체 결과 집합을 모두 만든 뒤 첫 화면을 반환한다.  
② 사용자가 일부 결과만 필요로 할 때 초기 응답 속도를 개선할 수 있다.  
③ 모든 SQL에서 정렬 비용을 제거한다.  
④ 대량 배치 작업에서만 의미가 있다.

### 10. 결합 인덱스 `(고객ID, 주문일시, 상태코드)`가 있을 때 다음 조건의 특징으로 옳은 것은?

```sql
where 고객ID = :cust_id
  and 주문일시 between :from_dt and :to_dt
  and 상태코드 = :status
```

① 고객ID 등치 조건으로 시작 지점을 좁히고 주문일시 범위 조건 이후 상태코드는 필터 성격이 강해질 수 있다.  
② 상태코드가 등치 조건이므로 항상 인덱스 탐색의 첫 번째 기준이 된다.  
③ 주문일시가 범위 조건이면 고객ID 조건도 사용할 수 없다.  
④ 모든 조건이 등치 조건처럼 동일한 탐색 효율을 가진다.

### 11. BETWEEN 조건을 IN-list로 바꾸는 튜닝이 효과적일 수 있는 경우는?

① 범위 내 값이 매우 많고 선두 컬럼 선택도가 낮을 때 무조건 효과적이다.  
② 소수의 이산 값으로 분해하면 후속 컬럼의 탐색 효율을 살릴 수 있을 때 효과적일 수 있다.  
③ IN-list는 항상 Full Scan을 유도하므로 튜닝에 사용할 수 없다.  
④ BETWEEN과 IN-list는 실행계획상 항상 동일하다.

### 12. 인덱스 설계 원칙으로 가장 적절한 것은?

① 선택도가 가장 낮은 컬럼을 무조건 맨 앞에 둔다.  
② 조건절, 조인, 정렬, 그룹핑, 테이블 액세스 감소 효과를 함께 고려한다.  
③ 인덱스 개수는 많을수록 모든 SQL 성능이 좋아진다.  
④ DML 성능은 인덱스 설계와 무관하다.

### 13. NL 조인에 대한 설명으로 옳은 것은?

① 선행 집합의 결과 건수와 후행 테이블 접근 비용에 민감하다.  
② 두 입력 집합을 모두 정렬한 뒤 병합한다.  
③ 항상 대량 데이터 조인에서 해시 조인보다 빠르다.  
④ 후행 테이블의 인덱스 존재 여부와 무관하다.

### 14. Sort Merge Join이 상대적으로 적합할 수 있는 상황은?

① 조인 컬럼에 정렬된 결과가 이미 있거나 정렬 비용을 감수할 만한 경우  
② 한쪽 입력이 매우 작고 해시 테이블 생성 비용이 전혀 없는 경우  
③ 조인 조건이 없는 카테시안 조인만 처리하는 경우  
④ 조인 컬럼의 비교가 불가능한 경우

### 15. Hash Join의 Build Input 선택 기준으로 가장 적절한 것은?

① 항상 SQL에 먼저 적힌 테이블을 Build Input으로 선택한다.  
② 가능한 작은 집합을 Build Input으로 삼아 해시 테이블 생성 부담을 줄이는 것이 일반적이다.  
③ 인덱스가 많은 테이블을 Build Input으로 삼아야 한다.  
④ Build Input과 Probe Input은 성능에 영향을 주지 않는다.

### 16. 조인 방식 선택에 대한 설명으로 옳은 것은?

① OLTP 단건 조회는 무조건 Hash Join을 선택해야 한다.  
② 대량 집합 처리에서는 NL 조인의 반복 랜덤 액세스가 병목이 될 수 있다.  
③ Sort Merge Join은 정렬이 필요하므로 어떤 경우에도 사용하면 안 된다.  
④ 조인 방식은 힌트로만 결정되며 통계정보와 무관하다.

### 17. 서브쿼리 변환에 대한 설명으로 가장 적절한 것은?

① 옵티마이저는 항상 사용자가 작성한 SQL 형태 그대로만 실행한다.  
② 서브쿼리를 조인 형태로 변환하면 동일 결과를 더 효율적으로 만들 수 있는 경우가 있다.  
③ 서브쿼리는 실행계획에서 조인으로 표현될 수 없다.  
④ 쿼리 변환은 결과 집합을 바꾸기 위한 기능이다.

### 18. 소트 작업에 대한 설명으로 옳은 것은?

① 모든 소트는 메모리 내부에서만 끝난다.  
② Sort Area가 부족하면 Temp 영역을 사용하는 외부 소트가 발생할 수 있다.  
③ PGA는 소트와 무관하다.  
④ ORDER BY가 있으면 인덱스 정렬 순서를 절대 활용할 수 없다.

### 19. UNION과 UNION ALL의 차이로 가장 적절한 것은?

① UNION ALL은 중복 제거를 수행하므로 UNION보다 일반적으로 더 많은 소트가 필요하다.  
② UNION은 중복 제거를 위해 정렬 또는 해시 작업이 필요할 수 있다.  
③ 두 연산자는 결과 집합이 항상 같다.  
④ UNION은 두 집합을 단순 연결만 한다.

### 20. Top N 쿼리 튜닝에 대한 설명으로 옳은 것은?

① 필요한 순서와 맞는 인덱스를 활용하면 전체 정렬 없이 빠르게 상위 N건을 얻을 수 있다.  
② Top N 쿼리는 항상 전체 데이터를 정렬해야 한다.  
③ ROWNUM 조건은 인덱스와 함께 사용할 수 없다.  
④ Top N은 OLTP 환경에서 초기 응답 속도와 무관하다.

### 21. Sort Area를 효율적으로 사용하는 SQL 작성법으로 가장 적절한 것은?

① 정렬 전에 불필요한 컬럼과 행을 줄인다.  
② 정렬 대상 컬럼을 select-list에 모두 반복해서 추가한다.  
③ WHERE 조건을 제거해 정렬 입력을 크게 만든다.  
④ 문자열 변환과 포맷팅을 정렬 전에 최대한 많이 수행한다.

### 22. DML 성능에 영향을 주는 요소로 보기 어려운 것은?

① 인덱스 개수와 제약조건  
② Redo와 Undo 발생량  
③ Lock 경합  
④ SQL 문장의 대소문자 사용 여부 자체

### 23. Direct Path I/O에 대한 설명으로 가장 적절한 것은?

① 모든 OLTP 단건 조회에서 기본적으로 사용해야 한다.  
② 버퍼 캐시를 우회해 대량 데이터를 읽거나 적재할 때 유리할 수 있다.  
③ 버퍼 캐시 효율을 높이기 위해 반드시 버퍼 캐시에 모든 블록을 적재한다.  
④ 인덱스를 사용하는 SQL에서는 절대 발생하지 않는다.

### 24. Lock과 동시성 제어에 대한 설명으로 옳은 것은?

① Lock은 데이터 정합성과 동시성을 조율하기 위한 핵심 메커니즘이다.  
② DML Lock은 SQL 튜닝과 무관하므로 고려하지 않아도 된다.  
③ 트랜잭션이 길어져도 경합 가능성은 변하지 않는다.  
④ 모든 Lock은 애플리케이션에서 명시적으로 생성해야 한다.

### 25. 선택도와 카디널리티에 대한 설명으로 옳은 것은?

① 선택도는 전체 로우 중 조건을 만족할 것으로 예상되는 비율이다.  
② 카디널리티는 항상 테이블 전체 로우 수와 같다.  
③ NDV가 커질수록 등치 조건의 단순 선택도는 일반적으로 커진다.  
④ 옵티마이저는 통계정보를 사용하지 않는다.

## 정답 및 풀이

### 1. 정답 ②

SQL 처리 과정은 크게 파싱, 최적화, 실행으로 나눌 수 있다. 최적화 단계에서는 옵티마이저가 여러 실행경로를 평가하고 비용이 낮다고 판단한 실행계획을 선택한다. ①은 최적화 설명이고, ③은 실행 전에도 `EXPLAIN PLAN` 등으로 계획 확인이 가능하므로 틀렸다. ④에서 문법 오류는 syntax check에서 주로 확인한다.

### 2. 정답 ②

바인드 변수는 SQL 텍스트를 안정화해 같은 커서를 재사용할 가능성을 높인다. 이로써 하드 파싱과 라이브러리 캐시 부하를 줄일 수 있다. 다만 바인드 변수 사용이 항상 같은 성능을 보장하는 것은 아니며, 데이터 분포가 치우친 경우 바인드 변수와 실행계획 안정성 이슈도 함께 봐야 한다.

### 3. 정답 ③

논리 I/O는 버퍼 캐시를 통해 블록을 접근한 횟수 관점의 지표다. 물리 I/O는 필요한 블록이 캐시에 없어서 디스크에서 읽을 때 발생한다. 버퍼 캐시 히트율이 높더라도 같은 블록을 비효율적으로 반복 접근하면 SQL 자체는 나쁠 수 있다.

### 4. 정답 ③

Table Full Scan은 많은 블록을 순차적으로 읽는 대량 처리에서 유리할 수 있다. 반대로 Index Range Scan은 소량 데이터를 정확히 찾는 데 강하지만, 많은 ROWID로 테이블을 반복 방문하면 랜덤 액세스 비용이 커질 수 있다. Full Scan 자체를 무조건 악으로 보는 것은 위험하다.

### 5. 정답 ②

인덱스 Range Scan은 시작 지점과 종료 지점을 찾을 수 있어야 효율적이다. 선두 컬럼이 함수나 연산으로 가공되면 일반 B-tree 인덱스의 정렬 구조를 그대로 활용하기 어려워진다. 이 경우 조건을 컬럼 원형 기준으로 바꾸거나 함수 기반 인덱스 등을 검토한다.

### 6. 정답 ①

`to_char(주문일시, ...)`처럼 인덱스 컬럼을 가공하면 인덱스에 저장된 원래 값의 정렬 순서를 이용해 탐색하기 어렵다. 보통은 `주문일시 >= date '2026-06-02' and 주문일시 < date '2026-06-03'`처럼 범위 조건으로 바꾸는 편이 낫다.

### 7. 정답 ②

Index Fast Full Scan은 인덱스 세그먼트를 빠르게 읽어 인덱스에 포함된 컬럼만으로 결과를 만들 수 있을 때 유용하다. 하지만 인덱스 키 순서대로 읽는 방식이 아니므로 정렬 결과를 보장하지 않는다. 정렬 순서 활용은 Index Full Scan이나 Range Scan과 구분해야 한다.

### 8. 정답 ②

인덱스로 조건에 맞는 ROWID를 찾은 뒤 테이블을 반복 방문하면 랜덤 액세스가 발생한다. 결과 건수가 많거나 클러스터링 팩터가 나쁘면 비용이 커진다. 따라서 조건 컬럼뿐 아니라 select-list 컬럼을 인덱스에 포함해 테이블 액세스를 줄일 수 있는지도 검토한다.

### 9. 정답 ②

부분범위 처리는 전체 결과를 끝까지 만들지 않고 필요한 일부 결과를 먼저 반환하는 전략이다. 화면 페이징이나 Top N 조회처럼 초기 응답 속도가 중요한 OLTP 업무에 효과적이다. 전체 결과를 반드시 모두 처리해야 하는 배치성 SQL과는 관점이 다르다.

### 10. 정답 ①

결합 인덱스에서 선두 컬럼 `고객ID`의 등치 조건은 탐색 범위를 크게 좁힌다. 그 다음 `주문일시`가 범위 조건이므로 이후 컬럼인 `상태코드`는 인덱스에서 읽은 범위 안에서 필터 역할이 강해질 수 있다. 범위 조건 뒤 컬럼의 효율이 제한된다는 점이 핵심이다.

### 11. 정답 ②

BETWEEN 범위가 넓어 후속 컬럼을 제대로 활용하지 못할 때, 값의 개수가 적고 명확하다면 IN-list로 분해해 여러 번의 정밀한 탐색을 유도할 수 있다. 다만 값이 많으면 오히려 반복 탐색 비용이 커질 수 있으므로 데이터 분포와 실행계획을 확인해야 한다.

### 12. 정답 ②

인덱스 설계는 단순히 선택도 하나만 보고 결정하지 않는다. 액세스 조건, 필터 조건, 조인 순서, ORDER BY/GROUP BY 생략 가능성, 테이블 랜덤 액세스 감소, DML 부하를 종합적으로 봐야 한다. 인덱스가 많아질수록 조회 일부는 좋아질 수 있지만 DML 비용은 증가한다.

### 13. 정답 ①

NL 조인은 선행 집합의 각 로우마다 후행 테이블을 반복 탐색하는 방식이다. 따라서 선행 집합이 작고 후행 테이블 인덱스 접근이 효율적이면 좋지만, 선행 로우가 많거나 후행 랜덤 액세스가 비싸면 병목이 된다.

### 14. 정답 ①

Sort Merge Join은 양쪽 입력을 조인 키 기준으로 정렬한 뒤 병합한다. 이미 정렬된 경로를 활용할 수 있거나 정렬 비용을 감수할 만한 대량 조인에서 고려할 수 있다. 다만 정렬 부담이 있으므로 항상 좋은 방식은 아니다.

### 15. 정답 ②

Hash Join은 보통 작은 쪽을 Build Input으로 삼아 메모리에 해시 테이블을 만든 뒤 큰 쪽을 Probe한다. Build Input이 과도하게 크면 메모리 부담과 Temp 사용 가능성이 커진다. 테이블이 SQL에 쓰인 순서가 아니라 비용과 통계정보가 중요하다.

### 16. 정답 ②

대량 집합을 NL 조인으로 처리하면 후행 테이블에 대한 반복 랜덤 액세스가 커질 수 있다. 이런 경우 Hash Join이나 Sort Merge Join이 유리할 수 있다. 반대로 단건 조회나 소량 조회에서는 NL 조인이 매우 효율적일 수 있으므로 업무 특성과 데이터량을 같이 봐야 한다.

### 17. 정답 ②

옵티마이저는 사용자가 작성한 SQL을 내부적으로 변환해 더 효율적인 형태의 실행계획을 만들 수 있다. 서브쿼리 Unnesting, 뷰 병합 같은 변환은 결과의 의미를 유지하면서 처리 방식을 바꾸는 것이다. 결과 집합을 바꾸는 기능이 아니다.

### 18. 정답 ②

소트는 PGA 내 Sort Area에서 수행되며, 메모리가 부족하면 Temp 영역을 사용한다. Temp를 쓰는 외부 소트는 비용이 커지므로 정렬 대상 행과 컬럼을 줄이고, 인덱스 순서를 활용할 수 있는지 검토해야 한다.

### 19. 정답 ②

`UNION`은 두 집합을 합친 뒤 중복을 제거해야 하므로 정렬 또는 해시 기반 중복 제거 작업이 필요할 수 있다. `UNION ALL`은 중복 제거 없이 연결하므로 결과가 허용된다면 일반적으로 더 가볍다. 단, 두 연산자의 결과가 같다고 단정하면 안 된다.

### 20. 정답 ①

Top N 조회는 필요한 정렬 순서와 맞는 인덱스를 활용하면 상위 N건만 빠르게 읽고 멈출 수 있다. 전체 데이터를 읽고 정렬한 뒤 N건을 자르는 방식과 성능 차이가 크다. 특히 OLTP 화면 조회에서는 초기 응답 속도 개선에 중요하다.

### 21. 정답 ①

Sort Area를 효율적으로 쓰려면 정렬 입력을 줄여야 한다. 불필요한 행을 WHERE 조건으로 먼저 제거하고, 정렬 전에 필요 없는 컬럼이나 무거운 표현식을 줄이는 것이 좋다. 포맷팅과 문자열 확장은 가능하면 정렬 이후로 미루는 편이 낫다.

### 22. 정답 ④

DML 성능에는 인덱스 유지 비용, 제약조건 검사, Redo/Undo, Lock 경합, 트리거, 서브쿼리 등이 영향을 준다. SQL 문장의 대소문자 자체는 같은 의미의 SQL이라면 본질적인 DML 처리 비용과 직접 관련이 없다.

### 23. 정답 ②

Direct Path I/O는 버퍼 캐시를 우회해 대량 데이터를 처리할 때 효과적일 수 있다. 일반 OLTP 단건 조회처럼 캐시 재사용성이 큰 업무에는 버퍼 캐시 경유가 유리한 경우가 많다. 대량 적재나 병렬 처리, Full Scan 중심 작업에서 주로 검토한다.

### 24. 정답 ①

Lock은 여러 트랜잭션이 동시에 데이터를 다룰 때 정합성을 지키는 장치다. 트랜잭션이 오래 유지되거나 동일 자원을 많이 갱신하면 경합 가능성이 커진다. SQL 튜닝은 I/O뿐 아니라 Lock 대기와 트랜잭션 범위까지 함께 봐야 한다.

### 25. 정답 ①

선택도는 조건을 만족할 것으로 예상되는 비율이고, 카디널리티는 예상 로우 수다. 단순 등치 조건에서 선택도는 대략 `1 / NDV`로 추정할 수 있으므로 NDV가 커질수록 선택도는 작아진다. 옵티마이저는 통계정보를 바탕으로 비용과 카디널리티를 추정한다.
