## 문제 1. SQL 처리 과정과 실행계획 해석

다음 SQL과 실행계획을 보고 판단한 내용으로 가장 적절한 것은?

```sql
select /* SQL-01 */
       e.empno, e.ename, d.dname
from   emp e, dept d
where  e.deptno = d.deptno
and    d.loc = :loc;
```

```text
---------------------------------------------------------------------------------
| Id | Operation                    | Name         | Starts | E-Rows | A-Rows |
---------------------------------------------------------------------------------
|  0 | SELECT STATEMENT             |              |      1 |        |     12 |
|  1 |  NESTED LOOPS                |              |      1 |     10 |     12 |
|  2 |   TABLE ACCESS BY INDEX ROWID| DEPT         |      1 |      1 |      2 |
|  3 |    INDEX RANGE SCAN          | DEPT_LOC_IDX |      1 |      1 |      2 |
|  4 |   TABLE ACCESS BY INDEX ROWID| EMP          |      2 |     10 |     12 |
|  5 |    INDEX RANGE SCAN          | EMP_DEPT_IDX |      2 |      5 |     12 |
---------------------------------------------------------------------------------
```

① `EMP`가 먼저 읽히고 `DEPT`가 후행 테이블로 반복 탐색된다.  
② `DEPT_LOC_IDX`로 `loc` 조건을 만족하는 부서를 찾고, 그 결과 건수만큼 `EMP`를 반복 탐색한다.  
③ `NESTED LOOPS`가 있으므로 두 테이블 모두 Full Scan으로 읽힌다.  
④ `Starts`가 모두 1이 아니므로 이 실행계획은 실제 실행 전에는 생성될 수 없다.

## 문제 4. 조인 방식 선택

다음은 일 배치에서 전일 주문 300만 건과 주문상세 1,200만 건을 조인해 집계하는 SQL이다. `주문상세.주문번호` 인덱스는 존재하지만, 결과 대상 주문 건수가 매우 많다.

```sql
select o.주문일자, sum(d.수량 * d.단가) as 매출금액
from   주문 o
       join 주문상세 d
       on d.주문번호 = o.주문번호
where  o.주문일자 = :target_date
group by o.주문일자;
```

다음 판단 중 가장 적절한 것은?

① `주문상세.주문번호` 인덱스가 있으므로 NL 조인이 항상 최선이다.  
② 대상 주문 건수가 많다면 NL 조인의 반복 랜덤 액세스가 커질 수 있으므로 Hash Join을 검토할 수 있다.  
③ Hash Join은 인덱스를 전혀 사용할 수 없으므로 대량 처리에 부적합하다.  
④ Sort Merge Join은 정렬을 수행하므로 실행계획에 절대 나타나면 안 된다.

## 문제 5. 소트 생략과 Top N 실행계획

다음 인덱스가 존재한다.

```sql
create index 거래_X1 on 거래(종목코드, 거래일시 desc);
```

아래 SQL의 성능 특성으로 가장 적절한 것은?

```sql
select *
from (
    select 거래번호, 종목코드, 거래일시, 체결수량, 체결가격
    from   거래
    where  종목코드 = :code
    order by 거래일시 desc
)
where rownum <= 10;
```

① `ORDER BY`가 있으므로 반드시 전체 거래 데이터를 정렬한 뒤 10건을 반환한다.  
② 인덱스 선두 컬럼 등치 조건과 후속 컬럼 정렬 순서가 맞아 전체 소트 없이 상위 10건을 빠르게 얻을 수 있다.  
③ `rownum` 조건이 있으면 인덱스 Range Scan을 사용할 수 없다.  
④ 인덱스가 `(종목코드, 거래일시 desc)` 순서이면 `종목코드` 조건은 필터로만 처리된다.

# 정답 및 해설

## 문제 1 해설

> 정답: **②**

### 핵심 개념

이 문항은 `실행계획의 들여쓰기`, `NL 조인의 반복 구조`, `Starts`의 의미를 함께 묻습니다.  
NL 조인은 보통 선행 집합에서 얻은 로우마다 후행 집합을 반복 탐색합니다.

### 정답 도출 원리

실행계획을 보면 `DEPT_LOC_IDX`가 먼저 사용됩니다.

```text
3 INDEX RANGE SCAN DEPT_LOC_IDX
2 TABLE ACCESS BY INDEX ROWID DEPT
```

즉 `d.loc = :loc` 조건으로 `DEPT`를 먼저 찾습니다. 실제 로우 수 `A-Rows`는 2건입니다.  
그 다음 `EMP_DEPT_IDX`의 `Starts`가 2입니다.

```text
5 INDEX RANGE SCAN EMP_DEPT_IDX Starts = 2
```

이는 선행 `DEPT` 결과 2건 각각에 대해 `EMP` 쪽 인덱스 탐색이 반복되었음을 의미합니다.

### 오답 분석

| 선택지 | 판단 | 이유 |
|---|---|---|
| ① | 오답 | 실행계획상 `DEPT`가 먼저 읽히고 `EMP`가 후행 반복 탐색됩니다. |
| ③ | 오답 | `TABLE ACCESS BY INDEX ROWID`, `INDEX RANGE SCAN`이 명시되어 있어 Full Scan이 아닙니다. |
| ④ | 오답 | `Starts`, `A-Rows`는 실제 실행 통계가 포함된 계획에서 확인되는 값입니다. 실행계획 생성 자체가 불가능하다는 의미는 아닙니다. |

### 학습 포인트

- NL 조인은 **선행 로우 수 × 후행 접근 비용**이 핵심입니다.
- `Starts`가 후행 인덱스 반복 탐색 횟수를 드러낼 수 있습니다.
- 실행계획은 위에서 아래로만 읽는 것이 아니라, 트리 구조와 반복 관계를 함께 봐야 합니다.

## 문제 2 해설

> 정답: **②**

### 핵심 개념

B-tree 인덱스는 키 값이 정렬된 구조입니다.  
따라서 인덱스를 효율적으로 Range Scan하려면 인덱스 컬럼의 원형 값 기준으로 **시작 지점**과 **종료 지점**을 찾을 수 있어야 합니다.

### 정답 도출 원리

인덱스는 `(주문일시, 고객ID)` 순서입니다.  
②는 선두 컬럼 `주문일시`를 가공하지 않고 날짜 범위로 표현합니다.

```sql
where 주문일시 >= date '2026-06-02'
and   주문일시 <  date '2026-06-03'
```

이 조건은 인덱스에서 `2026-06-02 00:00:00` 이상, `2026-06-03 00:00:00` 미만 범위를 바로 찾을 수 있게 합니다.

### 오답 분석

| 선택지 | 판단 | 이유 |
|---|---|---|
| ① | 오답 | `to_char(주문일시, ...)`로 선두 컬럼을 가공해 일반 인덱스의 정렬 구조를 직접 활용하기 어렵습니다. |
| ③ | 오답 | `고객ID`는 결합 인덱스의 두 번째 컬럼입니다. 선두 컬럼 조건 없이 효율적인 Range Scan이 어렵습니다. |
| ④ | 오답 | 인덱스 선두 컬럼도 아니고, `고객ID`도 함수로 가공했습니다. |

### 학습 포인트

- 날짜 컬럼은 `to_char` 비교보다 **반개방 범위 조건**이 인덱스 활용에 유리합니다.
- 결합 인덱스는 선두 컬럼 조건 유무가 매우 중요합니다.
- 함수 기반 인덱스가 없다면 컬럼 가공은 인덱스 사용성을 크게 떨어뜨릴 수 있습니다.

## 문제 3 해설

> 정답: **②**

### 핵심 개념

이 문항은 `부분범위 처리`, `Top N`, `결합 인덱스 컬럼 순서`, `정렬 생략`을 함께 묻습니다.  
상담 화면처럼 첫 화면 응답이 중요한 SQL은 전체 결과를 모두 만든 뒤 자르는 방식보다, 필요한 순서대로 읽다가 20건에서 멈추는 방식이 유리합니다.

### 정답 도출 원리

조회 조건의 핵심은 다음 세 가지입니다.

```sql
where 고객ID = :cust_id
and   주문상태 in (...)
order by 주문일시 desc
where rownum <= 20
```

가장 중요한 시작 조건은 `고객ID = :cust_id`입니다. 특정 고객의 주문만 좁힌 뒤 `주문일시 desc` 순서로 읽으면, 별도 정렬 없이 최근 주문부터 가져올 수 있습니다.

```text
주문_X2(고객ID, 주문일시 desc, 주문상태)
```

이 인덱스는 `고객ID`로 범위를 좁히고, 그 범위 안에서 이미 `주문일시 desc` 순서로 정렬되어 있습니다.  
따라서 `주문상태`는 일부 필터가 될 수 있지만, 최근 20건을 빠르게 찾는 목적에는 가장 잘 맞습니다.

### 오답 분석

| 선택지 | 판단 | 이유 |
|---|---|---|
| ① | 오답 | `주문상태`가 선두이면 상태별 넓은 범위를 먼저 읽을 수 있습니다. 특정 고객의 최신 20건이라는 화면 목적과 덜 맞습니다. |
| ③ | 오답 | `주문일시`가 선두이면 전체 주문의 시간 순서를 먼저 타게 되어 특정 고객 조건이 뒤로 밀립니다. |
| ④ | 오답 | `결제금액`은 조건, 정렬, 조인 핵심 컬럼이 아니므로 이 SQL의 액세스 경로로 부적합합니다. |

### 학습 포인트

- Top N 화면 SQL에서는 **정렬 순서와 인덱스 순서의 일치**가 매우 중요합니다.
- `IN` 조건 컬럼을 무조건 앞에 두는 것보다, 업무의 반환 방식과 멈출 수 있는 지점을 고려해야 합니다.
- 부분범위 처리는 “전체를 빨리 처리”가 아니라 “필요한 만큼 먼저 반환”하는 전략입니다.

## 문제 4 해설

> 정답: **②**

### 핵심 개념

조인 방식은 인덱스 존재만으로 결정하지 않습니다.  
NL 조인은 후행 테이블을 반복 탐색하므로 선행 집합이 커지면 랜덤 액세스 비용이 폭증할 수 있습니다.

### 정답 도출 원리

이 SQL은 전일 주문 300만 건과 주문상세 1,200만 건을 조인합니다.  
`주문상세.주문번호` 인덱스가 있더라도 주문 300만 건 각각에 대해 주문상세를 반복 탐색하면 다음 형태가 됩니다.

```text
주문 300만 건 × 주문상세 인덱스 탐색 + 테이블 액세스
```

이런 대량 집합 처리에서는 Hash Join으로 한쪽 집합을 해시 테이블로 만들고 다른 쪽을 스캔하며 조인하는 방식이 더 유리할 수 있습니다.

### 오답 분석

| 선택지 | 판단 | 이유 |
|---|---|---|
| ① | 오답 | 인덱스가 있어도 반복 횟수가 많으면 NL 조인은 비효율적일 수 있습니다. |
| ③ | 오답 | Hash Join은 대량 조인에서 자주 유리합니다. 인덱스를 전혀 못 쓴다는 표현도 부정확합니다. |
| ④ | 오답 | Sort Merge Join은 정렬 비용이 있지만, 상황에 따라 선택될 수 있는 정상적인 조인 방식입니다. |

### 학습 포인트

- NL 조인의 핵심 위험은 **반복 랜덤 액세스**입니다.
- 대량 집합 조인은 Hash Join 또는 Sort Merge Join을 함께 검토해야 합니다.
- “인덱스가 있으니 NL”은 SQL 튜닝에서 자주 나오는 위험한 단순화입니다.

## 문제 5 해설

> 정답: **②**

### 핵심 개념

인덱스는 키 순서로 정렬된 구조입니다.  
조건과 정렬 방향이 인덱스 구성과 맞으면 `ORDER BY`를 위한 별도 Sort 작업을 생략할 수 있습니다.

### 정답 도출 원리

인덱스는 다음 순서입니다.

```text
거래_X1(종목코드, 거래일시 desc)
```

SQL은 `종목코드 = :code`로 선두 컬럼을 등치 검색하고, 그 범위 안에서 `거래일시 desc` 순서를 요구합니다.

```sql
where 종목코드 = :code
order by 거래일시 desc
```

따라서 인덱스를 타면 이미 원하는 순서로 읽을 수 있습니다. 여기에 `rownum <= 10`이 있으므로 상위 10건을 읽고 멈추는 부분범위 처리가 가능합니다.

### 오답 분석

| 선택지 | 판단 | 이유 |
|---|---|---|
| ① | 오답 | 인덱스 정렬 순서를 활용하면 전체 정렬이 필요 없을 수 있습니다. |
| ③ | 오답 | `rownum`은 Top N 처리에서 인덱스 Range Scan과 함께 매우 유용하게 쓰일 수 있습니다. |
| ④ | 오답 | `종목코드`는 인덱스 선두 컬럼이며 등치 조건이므로 필터가 아니라 액세스 범위를 결정하는 핵심 조건입니다. |

### 학습 포인트

- `ORDER BY`가 있다고 항상 Sort가 발생하는 것은 아닙니다.
- Top N 최적화는 **인덱스 순서 + 멈춤 조건**이 결합될 때 강력합니다.
- 실행계획에서는 `SORT ORDER BY STOPKEY` 또는 인덱스 Range Scan 기반 Stopkey 처리 여부를 확인합니다.
